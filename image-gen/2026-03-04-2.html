From Topic to TikTok in One Script

We're building a pipeline that takes a topic — like "why octopuses are aliens" — and spits out a fully narrated, AI-illustrated short video. No editing software. No stock footage. Just prompts, APIs, and a bit of glue.
40 min video-generation automation TTS content-pipeline ffmpeg

What's happening

You've probably scrolled past those AI-generated "fact" videos on YouTube Shorts or TikTok — a robotic voice narrating over a slideshow of AI images. Love them or hate them, they're everywhere, and projects like MoneyPrinterV2 have turned the whole pipeline into an open-source recipe.

What's actually interesting isn't the "make money online" angle — it's the architecture. These pipelines chain together 4–5 different AI services into a single automated flow: script writing → voice synthesis → image generation → video assembly. That's a real production pipeline, and understanding how the pieces connect is genuinely useful whether you're building educational content, product demos, or just want to understand how modern AI automation works.

Today we're going to reverse-engineer that pipeline and build a minimal version from scratch — well, from prompts. Your AI agent does the heavy lifting. You do the thinking.

1

The Script: Turning a Topic into Timed Scenes

Every video starts with a script, but for an automated pipeline, we need more than just text. We need structured output — the script broken into scenes, each with narration text AND an image description.

Think of it like a storyboard, except instead of drawings, each panel has a prompt that'll get fed to an image generator later. This is the master document that drives everything downstream.

The mental model here is a recipe card: the narration is the instructions you read aloud, and the image prompt is the photo on the card. They work together but serve different purposes.

Ask your agent
Get your agent to build a script generator that takes a topic and returns structured scenes with narration + image prompts
Think about it
  • What should each 'scene' contain? Think about what the TTS engine needs vs. what the image generator needs.
  • How long is a YouTube Short? How does that constrain the number of scenes and the length of each narration chunk?
  • What output format would make it easiest for the NEXT step in the pipeline to consume this data?
  • Should the image prompts describe exactly what's in the narration, or something more visually interesting?
What the agent gives back

The agent should give you a Python function that calls an LLM (OpenAI or Anthropic) with a system prompt that enforces structured output. You feed in a topic like "why honey never spoils" and get back a list of 5–6 scene objects, each with a narration field (what gets spoken aloud, ~10–15 seconds worth) and an image_prompt field (a detailed visual description for the image generator). The output comes back as JSON so downstream steps can parse it programmatically.

The key design decision: the system prompt should tell the LLM to write image prompts that are cinematic and specific — not just "honey in a jar" but "close-up golden honey dripping from a wooden dipper, warm backlight, macro photography style." Vague prompts make ugly images.

Tip
Tell your agent to make the system prompt enforce a consistent visual style across all scenes — like "documentary photography" or "watercolor illustration." Otherwise each image will look like it's from a different universe and the video feels janky.
API Key Note
You'll need an API key for an LLM service (OpenAI, Anthropic, etc.). If you're just learning, OpenAI's GPT-4o-mini is cheap and solid for script generation.
Why structured output instead of just parsing text?

You could have the LLM write a script and then try to split it up with regex or another LLM call. But that's fragile — it's like giving someone driving directions as a paragraph instead of a numbered list. Structured output (JSON with a defined schema) means every downstream step knows exactly where to find what it needs. MoneyPrinterV2 does something similar: each stage of the pipeline expects a specific data format from the previous stage. This is a core pattern in any automation pipeline — contracts between stages.

2

The Voice: Text-to-Speech That Doesn't Sound Like a Robot

Now we need to turn those narration strings into actual audio files. TTS has gotten shockingly good in the last couple years — we're way past the GPS-lady-from-2008 era.

The basic idea: you send text to a TTS API, it sends back an audio file. Do that for each scene, and you've got your voiceover track. Simple in concept, but there are a few things to think about.

Honestly, the hardest part here isn't the code — it's picking a voice that doesn't make people hit the skip button.

Ask your agent
Get your agent to build a TTS module that takes your scene list and generates an MP3 file for each scene's narration
Think about it
  • Each scene needs its own audio file — why not just one big file for the whole script? (Think about what happens at the video assembly stage.)
  • What TTS API should you use? Think about cost, quality, and whether you need the audio duration back.
  • How will you match each audio file to its scene? Naming convention matters here.
  • What happens if a scene's narration is too long and would make that segment drag? Should you validate?
What the agent gives back

The agent should give you a function that loops through your scenes, sends each narration string to a TTS API (like OpenAI's audio/speech endpoint or ElevenLabs), and saves the resulting audio as numbered files — scene_001.mp3, scene_002.mp3, etc. It should also return the duration of each audio clip, because you'll need that to know how long each image should be displayed.

The key piece is getting audio duration — either from the API response or by reading the file with a library like mutagen or pydub. Without durations, you can't sync images to audio later.

Tip
OpenAI's TTS is ~$0.015 per 1K characters and sounds great. ElevenLabs has more voice variety but costs more. For prototyping, OpenAI's 'nova' or 'shimmer' voices work well for narration.
Warning
Watch out for TTS rate limits. If you fire off 6 API calls simultaneously for 6 scenes, some services will throttle you. Ask your agent to add a small delay between calls or use sequential processing.
How MoneyPrinterV2 handles TTS

MoneyPrinterV2 uses a library called KittenTTS and also integrates with various free and paid TTS services. The clever bit is that it's modular — you can swap TTS providers without changing the rest of the pipeline. That's a pattern worth stealing: each stage of your pipeline should talk to the others through a simple interface ("give me text, I'll give you audio + duration") so you can upgrade individual pieces later without rewriting everything.

✓
At this point, you should have two working pieces: a script generator that outputs structured scenes (narration + image prompts), and a TTS module that converts narration into audio files with known durations. Feed in a test topic and make sure you get back a folder of MP3s and a list of scenes with durations. If you're getting errors, the most common culprits are API key issues or the LLM returning malformed JSON — ask your agent to add error handling for both.
3

The Visuals: Generating Scene Images with AI

Here's where it gets fun. Each scene has an image prompt — now we feed those prompts to an image generation API and get back the visuals for our video.

Think of this stage like hiring an illustrator who works instantly but needs very specific instructions. The quality of your image prompts from Step 1 directly determines whether your video looks professional or looks like an AI fever dream.

One thing that tripped me up at first: image generation APIs return different sizes and aspect ratios. For vertical short-form video (YouTube Shorts, TikTok, Reels), you need 9:16 portrait images. Forget to specify this and you'll get square images that look terrible in a vertical video.

Ask your agent
Get your agent to build an image generation module that creates a portrait-orientation image for each scene
Think about it
  • What aspect ratio do you need for vertical video? How do you specify this in an image API?
  • Should you enhance the image prompts before sending them, or use them as-is from the script generator?
  • How should you handle failures? Image generation can be flaky — what's your retry strategy?
  • Think about visual consistency: if Scene 1 shows a cartoon bee and Scene 3 shows a photorealistic bee, that's jarring. How do you prevent this?
What the agent gives back

The agent should give you a function that takes the scene list, sends each image prompt to a generation API (DALL-E 3, Stability AI, or Replicate's SDXL), and saves the results as numbered PNG files matching the audio files — scene_001.png, scene_002.png, etc. For DALL-E 3, the key is specifying size='1024x1792' for portrait orientation. For Stability AI or SDXL, you'd set height/width parameters accordingly.

The agent should also prepend a style prefix to every prompt — something like "Digital illustration in documentary style, highly detailed:" — so all images share a cohesive visual language. This is the secret to videos that don't look like a random collage.

API Key Note
Image generation is the most expensive step. DALL-E 3 at 1024x1792 is ~$0.080 per image. A 6-scene video costs about $0.48 in images alone. Stability AI via their API or Replicate can be cheaper. For testing, consider generating smaller images first.
Tip
A pro move: ask your agent to add a 'Ken Burns effect' flag. Instead of static images, you'll later tell ffmpeg to slowly zoom/pan across each image. This makes the video feel way more dynamic with zero extra generation cost.
The prompt engineering trick that makes this work

The real unlock is that your Step 1 script generator is actually doing two levels of prompt engineering — you're prompting an LLM to write prompts for an image generator. This is a prompt chain, and the quality compounds at each step. If your system prompt for the script generator says "write image prompts that are cinematic, with specific lighting and camera angle descriptions," the image generator will produce dramatically better results than if you just said "describe what the viewer should see." It's like the difference between telling a photographer "take a picture of a sunset" vs. "golden hour, low angle, silhouette of a lone tree against orange-pink gradient sky, shot on 35mm."

This is cheap (no extra API calls) and effective. It's the same technique professional prompt engineers use for consistent character design.

The Assembly: Stitching It All Together with ffmpeg

This is where the magic happens. We have audio files, we have images, and we know how long each scene should last. Now we need to combine them into an actual video file.

ffmpeg is the Swiss Army knife of video processing. It's a command-line tool that can do basically anything with audio and video — and crucially, it's free and runs everywhere. The catch? Its command-line syntax looks like someone encrypted a treasure map. This is where your AI agent really earns its keep.

The assembly logic is conceptually simple: for each scene, display the image for the duration of the audio, play the audio underneath, then move to the next scene. Concatenate all scenes. Export as MP4. Done.

But the ffmpeg command to do this is… not simple. Let the agent handle it.

Ask your agent
Get your agent to build the video assembly step — combining scene images and audio into a final MP4 video using ffmpeg
Think about it
  • Each image needs to be displayed for exactly the duration of its matching audio. How do you tell ffmpeg how long to show a static image?
  • Should you assemble each scene into its own video clip first, then concatenate? Or do it all in one ffmpeg command?
  • What video resolution and framerate make sense for a YouTube Short?
  • Bonus: how would you add a subtle zoom (Ken Burns effect) to keep static images from feeling boring?
What the agent gives back

The agent should give you a function that does the assembly in two phases. Phase 1: For each scene, create a short video clip by combining the image with its audio — ffmpeg takes the PNG, loops it for the exact duration of the MP3, and outputs a clip. Phase 2: Concatenate all the clips into one final MP4.

The agent will likely use Python's subprocess module to call ffmpeg. The key ffmpeg flags you'll see: -loop 1 (repeat the image), -t <duration> (how long), -c:v libx264 (H.264 video codec), -pix_fmt yuv420p (compatibility), and -shortest (stop when the shorter input ends). For the concat step, it'll create a text file listing all clips and use ffmpeg's concat demuxer.

For the Ken Burns effect, the agent might add a -vf "zoompan=z='min(zoom+0.001,1.5)':d=1" filter — this slowly zooms into each image over its duration, adding just enough motion to keep viewers engaged.

Warning
ffmpeg must be installed on your system. On Mac: brew install ffmpeg. On Ubuntu: apt install ffmpeg. On Windows: download from ffmpeg.org. If you skip this, everything else works but the final assembly step will fail.
Tip
Ask your agent to add a 0.5-second crossfade between scenes. It's one extra ffmpeg filter but makes the video feel 10x more polished than hard cuts.
Why two-phase assembly instead of one giant ffmpeg command?

You can do the entire assembly in a single ffmpeg command with complex filter chains. But here's the thing — if scene 4 out of 6 has a corrupt image or a weird audio format, a single-command approach fails entirely and gives you a cryptic error. The two-phase approach (make individual clips, then concatenate) lets you:

  1. Debug per-scene — if clip 4 fails, you know exactly which scene has the problem
  2. Cache intermediate results — if you re-run with a different scene 3, you don't re-encode scenes 1, 2, 4, 5, 6
  3. Preview individual scenes — you can watch scene clips before committing to the full assembly

This is a general engineering principle: break complex operations into debuggable stages. MoneyPrinterV2 does the same thing — each processing step produces intermediate files.

5

The Orchestrator: One Function to Rule Them All

We've got four modules: script generation, TTS, image generation, and video assembly. Now we need a pipeline orchestrator — a single entry point that chains everything together.

This is actually the simplest step code-wise, but the most important one architecturally. The orchestrator is where you handle the flow: run steps in order, pass data between them, handle errors, and maybe even add a progress indicator so you know what's happening.

Think of it like a factory assembly line supervisor. Each station (module) does its job; the supervisor makes sure parts move from one station to the next and stops the line if something breaks.

Ask your agent
Get your agent to build a main pipeline function that chains all four steps together and can be run with a single command
Think about it
  • What's the input? Just a topic string? Or should it also accept options like voice choice, visual style, and video length?
  • Where should intermediate files live? Think about cleanup and debugging.
  • What happens if image generation fails for one scene? Should the whole pipeline stop, or skip that scene?
  • How would you make this runnable from the command line? Think about argparse or a simple CLI.
What the agent gives back

The agent should give you a clean orchestrator function — something like generate_short(topic, style, voice) — that creates a temp directory, runs each stage in sequence, passes the outputs forward, and returns the path to the final MP4. It should include basic logging (printing which stage is running) and error handling (catching failures at each stage with a clear message).

The agent should also wire it up as a CLI: python pipeline.py "why honey never spoils" --style documentary --voice nova and the whole thing runs end-to-end. The output is a single MP4 file ready to upload.

Bonus: the agent might add timing — printing how long each stage took — which is super useful for identifying bottlenecks (spoiler: image generation is usually the slowest).

Tip
Ask your agent to create an output folder structure like output/<topic>/<timestamp>/ with the final video AND all intermediate files (script JSON, audio files, images). This makes it easy to debug issues and reuse assets.
How to add batch production and scheduling

Once your pipeline works for one video, scaling to batch production is straightforward. You just need a list of topics and a loop. The interesting part is scheduling — MoneyPrinterV2 uses CRON jobs to automatically generate and upload videos on a schedule.

The pattern is:

  1. Keep a topics list (a text file, a spreadsheet, whatever)
  2. A scheduler (cron, or even a simple Python loop with time.sleep()) picks the next topic at a set interval
  3. The pipeline runs, produces a video, saves it to an output folder
  4. Optionally, an upload step pushes it to YouTube/TikTok via their APIs

This is exactly how content farms work — and it's also how legitimate creators batch-produce educational content. The technology is neutral; the ethics depend on what you're making and whether you're transparent about AI involvement.

✓
You should now have a complete end-to-end pipeline. Run it with a test topic and check: (1) Does the script have well-structured scenes with both narration and image prompts? (2) Do the audio files sound natural and match the narration? (3) Are the images portrait-oriented and visually consistent? (4) Does the final MP4 play correctly with synced audio and visuals? If any step is off, isolate which module is the problem — that's why we built it in stages.

Subtitles are basically mandatory for short-form video in 2026. Most people scroll with sound off, so if your video doesn't have captions, 80% of viewers skip it. The challenge is syncing the text to the audio timing — each word or phrase needs to appear at the right moment.
Think about it
  • You already have the narration text and the audio duration for each scene. What additional timing information would you need for word-level subtitles?
  • ffmpeg has text overlay filters (`drawtext`). What parameters would you need — font, size, position, timing?
  • Should captions show one word at a time (karaoke-style), one phrase at a time, or the whole sentence? Think about readability on a phone screen.
  • Where should captions be positioned on a 9:16 vertical video? (Hint: not at the very bottom — that's where platform UI elements live.)

One way you could prompt it 
Add a subtitle system to my video pipeline. For each scene, I want the narration text displayed as a caption overlay on the video. Use ffmpeg's drawtext filter. Position the text in the lower-third area but not at the very bottom (about 70% down the screen). Use a bold sans-serif font, white text with a dark semi-transparent background box for readability. Break long narration into chunks of 5-7 words each, evenly distributed across the scene's duration. The text should be large enough to read on a phone — at least 48px for 1080x1920 resolution. Add this as a new function in the pipeline that runs after the basic video assembly, taking the assembled clips and adding subtitle overlays before final concatenation.

Recap

We just built a fully automated video pipeline — from a bare topic string to a finished, narrated, illustrated MP4. Let's trace the data flow one more time:

Topic → LLM → Structured Script → TTS API → Audio Files → Image API → Scene Images → ffmpeg → Final Video

Each arrow is a clean handoff between modules. Each module has a simple contract: here's what I need, here's what I produce. That modularity is what makes the whole thing work — and what makes it upgradeable. Want better voices? Swap the TTS module. Want photorealistic images instead of illustrations? Change the image model. Want to add background music? Add a new module between assembly steps.

This is the same architecture that powers MoneyPrinterV2 and countless content automation tools. Now you understand not just that they work, but how — and you can build your own version tuned to exactly what you need.

  • AI video pipelines are just chains of API calls with structured data flowing between them — script generation, TTS, image gen, and ffmpeg assembly. No single step is complicated; the power is in the orchestration.
  • Structured output from your script generator is the foundation everything else builds on. Get the scene format right (narration + image prompt + metadata) and every downstream step becomes straightforward.
  • Visual consistency across scenes comes from prompt engineering, not from the image model itself. A style prefix on every image prompt is cheap and dramatically improves the final product.
  • ffmpeg is the unsung hero of automated video. It's ugly to write but incredibly powerful. This is a perfect use case for AI agents — let them write the ffmpeg commands while you focus on the creative decisions.
  • Building in stages with intermediate files isn't just good engineering practice — it's essential for debugging and iteration. If scene 4 looks wrong, you can re-generate just that image without re-running the whole pipeline.

Where to go next

  • Add background music: mix in a looping ambient track at low volume underneath the narration using ffmpeg's amix filter
  • Build an upload module: use the YouTube Data API or TikTok's developer API to automatically publish finished videos
  • Experiment with video-native AI: try Runway Gen-3 or Pika to generate short video clips instead of static images for each scene — same pipeline structure, just swap the image module for a video module
  • Add a feedback loop: use an LLM to 'review' the generated script before sending it to TTS, catching awkward phrasing or factual issues
