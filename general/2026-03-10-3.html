Train Your Own GPT-2 ChatBot for Under $50

Dissecting Karpathy's nanochat — a single hackable codebase covering the entire LLM pipeline, from raw tokens to a ChatGPT-style web UI
40 min LLM-training nanochat pretraining finetuning karpathy GPT-2

What's happening

In 2019, OpenAI spent approximately $43,000 to train GPT-2. Seven years of compounding advances — in hardware, numerical precision, data curation, optimizer design, and training recipes — have collapsed that cost by nearly three orders of magnitude. Andrej Karpathy's nanochat repository, which has rapidly ascended GitHub's trending charts, demonstrates this dramatically: it trains a model matching GPT-2's capability on the DCLM CORE benchmark in under two hours on a single 8×H100 node, for roughly $48 at on-demand rates and closer to $15 on spot instances.

What makes nanochat distinctive is not merely cost reduction but pedagogical design. The codebase is deliberately minimal and hackable, covering every major stage of the LLM lifecycle — tokenization, pretraining, supervised finetuning, evaluation, inference, and a chat web UI — in a single repository with a single complexity dial: the --depth parameter. Set --depth 26, and every other hyperparameter (model width, number of attention heads, learning rate schedule, weight decay, training horizon) is auto-calculated for compute-optimal training.

This session dissects the architecture and pipeline, equipping you to understand, reproduce, and customize each stage using an AI agent as your primary engineering tool.

1

Understanding the Architecture: The --depth Dial and Compute-Optimal Scaling

The central design principle of nanochat is that a single integer — the number of transformer layers — should determine the entire model configuration. This is a deliberate departure from the typical LLM setup, where practitioners must independently tune dozens of interrelated hyperparameters: embedding dimension, number of attention heads, learning rate, warmup steps, batch size, weight decay, and training duration.

nanochat's approach rests on scaling laws — empirical relationships between model size, dataset size, and compute budget that predict optimal allocation. Given a target depth (number of layers), the codebase derives the remaining parameters using known scaling relationships. For example, transformer width typically scales as a fixed multiple of depth, and learning rate decreases as model size increases. The --depth 26 configuration corresponds roughly to GPT-2's 1.6-billion-parameter scale.

Tip
The DCLM CORE benchmark (Data-Centric Language Model) provides a standardized evaluation score. nanochat's leaderboard defines "GPT-2 grade" as matching or exceeding the original GPT-2 checkpoint's CORE score of 0.2565. This is the objective metric that determines whether a training run has succeeded.
Why auto-configuration from a single parameter matters

Manual hyperparameter tuning is one of the most expensive and error-prone parts of LLM training. Each failed experiment on an 8×H100 node wastes hours and hundreds of dollars. By encoding scaling-law relationships into the codebase, nanochat eliminates an entire class of human error. The --depth parameter functions as a complexity dial: increasing it produces a larger, more capable model with proportionally more training, while decreasing it produces a cheaper, faster experiment. This design enables rapid iteration — you can run a --depth 6 experiment in minutes to validate a hypothesis before committing to a full --depth 26 run.

What are scaling laws, concretely?

Scaling laws, most notably formalized by Kaplan et al. (2020) and refined by Hoffmann et al. (2022, the 'Chinchilla' paper), are empirical power-law relationships between three quantities: model parameter count (N), training dataset size (D), and total compute budget (C). The key finding is that for a given compute budget, there exists an optimal ratio of N to D — training a model that is too large on too little data wastes compute, and vice versa. nanochat encodes these relationships so that given a depth (which determines N), it automatically selects a training horizon (D) that is compute-optimal. This is why the single --depth dial works: it is not merely setting the layer count but implicitly selecting a point on the scaling curve.

Dissecting the Speedrun Pipeline: From Raw Data to Trained Model

The file runs/speedrun.sh is nanochat's reference implementation — a single shell script that executes the entire pipeline end-to-end. Understanding this script is equivalent to understanding the full training process. It proceeds through discrete stages: environment setup, data acquisition and tokenization, pretraining with distributed data parallelism across 8 GPUs, supervised finetuning, evaluation against DCLM CORE, and finally serving the chat UI.

Several technical choices in the pipeline deserve attention. FP8 training uses 8-bit floating-point arithmetic (available on H100 GPUs via the FP8 Transformer Engine) to nearly double throughput compared to BF16 with negligible quality loss. Distributed training across 8 GPUs uses PyTorch's native parallelism. The dataset has evolved through the leaderboard's history — the current best result uses NVIDIA's ClimbMix corpus, which replaced earlier datasets for better downstream performance.

Tip

Each change targeted a different bottleneck — precision, scale, throughput, data quality — and each delivered measurable wall-clock improvement.
What is FP8, and why does it matter here?

FP8 (8-bit floating point) is a numerical format introduced with NVIDIA's Hopper architecture (H100 GPUs). It represents numbers using only 8 bits instead of the 16 bits used by BF16 or the 32 bits used by FP32. The H100's Transformer Engine can perform matrix multiplications in FP8, effectively doubling the arithmetic throughput compared to BF16. The challenge is that 8 bits provide much less dynamic range and precision, which can cause training instability. The Transformer Engine addresses this by dynamically scaling tensors and selectively using higher precision for sensitive operations (like gradient accumulation). In practice, FP8 pretraining produces models of comparable quality to BF16 for transformer architectures at this scale, while cutting training time nearly in half — which is why it appears as a key optimization in the nanochat leaderboard.

Why did switching to ClimbMix improve results?

The training dataset matters as much as the model architecture. NVIDIA's ClimbMix is a curated blend of high-quality web text, code, and other sources specifically assembled for efficient language model training. Compared to earlier datasets, ClimbMix provides better data quality per token — meaning the model extracts more useful knowledge from each training example. This is a practical illustration of the 'data-centric AI' principle: improving data quality is often more efficient than increasing data quantity or model size. The leaderboard entry that switched to ClimbMix cut training time from 2.76 hours to 2.02 hours while maintaining benchmark performance.

Provisioning Infrastructure and Launching the Pipeline

Running nanochat requires an 8×H100 GPU node — a machine with eight NVIDIA H100 GPUs connected via NVLink, typically available from cloud GPU providers such as Lambda, CoreWeave, RunPod, or major cloud platforms. At current market rates, these nodes cost approximately $24 per hour on-demand, or as low as $6–8 per hour on spot/preemptible instances.

The provisioning decision involves a trade-off between cost and reliability. Spot instances offer dramatic savings (potentially bringing the total GPT-2 training cost to ~$15) but can be interrupted at any time. Since nanochat's pretraining run takes approximately two hours and does not currently include automatic checkpointing for spot recovery, an interruption would require restarting from scratch. On-demand instances guarantee availability but cost three to four times more.

Once the node is provisioned, the actual setup is remarkably simple — nanochat uses uv (a fast Python package manager) to manage its virtual environment, and the entire pipeline is a single shell script invocation.

If you are experimenting for the first time, consider running at --depth 6 first (minutes, negligible cost) to validate your setup.
Tip
The uv package manager (from Astral, the creators of ruff) is significantly faster than pip — it resolves and installs Python dependencies in seconds rather than minutes. nanochat's use of uv is both a practical choice (faster setup) and a pedagogical signal: modern Python tooling has improved dramatically.
Spot instance strategy for longer runs

If you intend to experiment with depths beyond 26 (larger models requiring longer training), spot instance interruption becomes a more serious risk. A robust strategy involves: (1) modifying the training script to checkpoint every N steps (nanochat saves checkpoints; the question is frequency), (2) using a provider with persistent storage that survives instance termination, and (3) adding resume-from-checkpoint logic to the launch script. This is a natural extension project and a good exercise in using an agent to modify infrastructure scripts.

From Base Model to Chatbot: Supervised Finetuning and the Chat UI

Pretraining produces a base model — a powerful text completion engine that has absorbed statistical patterns from billions of tokens of web text. However, a base model is not a chatbot. If you prompt it with a question, it is equally likely to continue the text as if it were a Wikipedia article, a Reddit thread, or a code comment. It has no notion of a conversation, no concept of helpfulness, and no instinct to answer questions directly.

Supervised finetuning (SFT) bridges this gap. The model is trained on a curated dataset of conversations in the format the chat UI expects — alternating user and assistant turns, with the model learning to produce helpful, coherent responses. This is a relatively short training phase (minutes rather than hours) because the base model already "knows" the language; SFT merely teaches it a new behavioral pattern.

The final stage is nanochat's chat web UI, launched via python -m scripts.chat_web. This provides a ChatGPT-style browser interface for interacting with the model you just trained — a tangible, interactive artifact that makes the entire pipeline concrete.

What the agent gives back

The agent should illustrate the distinction clearly. A base model given 'What is the capital of France?' might continue with '\n\nWhat is the capital of Germany?\n\nWhat is the capital of...' (treating it as a quiz document), or 'The answer depends on...' (mimicking a web article), or simply '... Paris is a city known for...' (continuing as prose). It has no commitment to answering the question. A finetuned chat model responds directly: 'The capital of France is Paris.' The agent should explain that SFT works because the base model already possesses the knowledge (it has seen 'Paris is the capital of France' thousands of times in pretraining data); finetuning merely teaches the format — that when a user asks a question, the model should provide a direct answer. The agent should also note that nanochat performs SFT only, omitting RLHF/DPO alignment stages, which limits the model's ability to refuse harmful requests or follow complex instructions, but keeps the pipeline simple.

Warning
A GPT-2-scale model finetuned only with SFT — without RLHF or other alignment techniques — has no safety guardrails. It will attempt to answer any question without refusal. This is acceptable for educational experimentation but means the model should not be deployed publicly or used in production.
The chat template: how conversations are formatted for training

Chat models require a structured format that delineates user and assistant turns. Common formats include ChatML (<|im_start|>user\n...) and Llama-style templates. The tokenizer must understand these special tokens so that the model learns to generate responses only in the assistant role, never fabricating user turns. During SFT, the loss function is typically applied only to assistant tokens — the model learns to predict what the assistant should say given the conversation history, but is not penalized for 'predicting' the user's messages (which are provided as context). This asymmetric loss is a subtle but critical implementation detail.

Why the chat UI matters pedagogically

The web UI (scripts.chat_web) is not merely a convenience feature. It closes the feedback loop between training decisions and observable behavior. When you modify the training data, adjust hyperparameters, or change the finetuning approach, the chat UI provides immediate, qualitative feedback — you can ask your model questions, probe its knowledge boundaries, and observe its failure modes. This rapid iteration cycle is one of nanochat's most valuable design choices, making the abstract (loss curves, benchmark scores) tangible and explorable.

✓
You should now understand the complete lifecycle: pretraining on web-scale data produces a base model that is a powerful but undirected text completer; SFT on conversation data transforms it into a chatbot; and the web UI provides an interactive interface. The key insight is that most of the 'knowledge' lives in pretraining — finetuning teaches format and behavior, not facts.

Customizing the Pipeline: Your Own Data, Your Own Model

The true value of nanochat is not in reproducing GPT-2 — it is in providing a hackable platform for experimentation. The most impactful customization is changing the training data. The leaderboard itself demonstrates this: switching from the original dataset to NVIDIA ClimbMix reduced training time by 27% while maintaining benchmark performance. Data quality is the highest-leverage variable in LLM training.

Other productive customization vectors include: modifying the SFT dataset to specialize the chatbot for a particular domain (e.g., medical Q&A, code generation, customer support), adjusting the model architecture within the constraints of the --depth framework, and extending the evaluation suite beyond DCLM CORE to include task-specific benchmarks.

An AI agent is particularly well-suited to this kind of work — the modifications involve understanding the data format, writing preprocessing scripts, and adjusting configuration parameters, all of which are well within an agent's capabilities.

Ask your agent
Ask your agent to outline a plan for adapting nanochat to train a domain-specific chatbot — for instance, a coding assistant — including what data you would need, how to format it, and which pipeline stages require modification.
Think about it
  • What format does nanochat's SFT stage expect its training data to be in? How would you convert existing datasets (e.g., from Hugging Face) into that format?
  • Should you modify the pretraining data, the finetuning data, or both? What are the trade-offs of each approach?
  • If you want a coding assistant specifically, what public datasets contain high-quality code instruction-following examples?
  • How would you evaluate a domain-specific model — is DCLM CORE still the right benchmark?
What the agent gives back

The agent should propose a structured plan: (1) Keep pretraining on the general corpus (ClimbMix) to maintain broad language understanding — modifying pretraining data is high-effort, high-risk, and rarely necessary for domain specialization. (2) Replace or augment the SFT dataset with domain-specific conversation data. For a coding assistant, suitable sources include datasets like Code Alpaca, OSS-Instruct, or filtered subsets of ShareGPT containing code discussions. (3) Convert the data into nanochat's expected conversation format — typically a JSON or JSONL structure with alternating user/assistant turns. (4) Replace DCLM CORE evaluation with domain-relevant benchmarks — for code, this might be HumanEval or MBPP pass rates. (5) Run a low-depth experiment (e.g., --depth 6) to validate the pipeline with the new data before committing to a full training run. The agent should emphasize that SFT data quality matters far more than quantity — a few thousand high-quality examples often outperform tens of thousands of noisy ones.

Tip
When adapting the SFT dataset, resist the temptation to maximize dataset size. Research consistently shows that a small dataset of high-quality, diverse examples produces better chat behavior than a large dataset of low-quality or repetitive examples. A useful heuristic: if you would not want your chatbot to imitate a particular conversation, do not include it in the training data.
The economics of experimentation at different depths

nanochat's depth dial enables a rational experimentation budget. At --depth 6, experiments complete in minutes at negligible cost — ideal for validating data pipelines, debugging format issues, and testing code changes. At --depth 12, training takes perhaps 15–20 minutes and produces a model with basic but observable language ability — useful for quick qualitative checks. At --depth 26, you are committing approximately $48 and two hours for a GPT-2-grade result. A disciplined workflow is: iterate rapidly at depth 6, validate promising approaches at depth 12, and reserve depth 26 runs for configurations you have high confidence in. This is analogous to the common software engineering practice of running unit tests locally before triggering full CI pipelines.

Recap

This session examined Karpathy's nanochat from architecture to deployment. We traced the six-stage pipeline — tokenization, pretraining, supervised finetuning, evaluation, inference, and chat UI — and understood how the --depth parameter encodes scaling laws to auto-configure the entire model from a single integer. We explored the economics (GPT-2 capability for $48, down from $43,000 in 2019), the key optimizations (FP8 precision, ClimbMix dataset, 1M-token batch sizes), and the critical distinction between a pretrained base model and a finetuned chatbot. Finally, we considered how to customize the pipeline for domain-specific applications and how to monitor expensive training runs effectively.

The deeper lesson is structural: nanochat demonstrates that the LLM pipeline is no longer a research-lab endeavor requiring teams of specialists. With modern hardware, well-encoded scaling laws, and a hackable codebase, a single practitioner can train, evaluate, and deploy a capable language model in an afternoon. The agent-assisted workflow we practiced — defining objectives, formulating precise prompts, and evaluating outputs against known benchmarks — is precisely the methodology that makes this tractable.

  • The --depth parameter is a compute-optimal complexity dial: it sets the transformer layer count, and all other hyperparameters (width, heads, learning rate, training horizon) are derived automatically from scaling laws.
  • Pretraining teaches knowledge; supervised finetuning teaches behavior. A base model has no concept of helpfulness — SFT on conversation data transforms it into a chatbot. Data quality at both stages matters more than data quantity.
  • FP8 training on H100 GPUs nearly doubles throughput compared to BF16 with negligible quality loss, and dataset choice (e.g., ClimbMix) can be as impactful as architectural changes — the leaderboard progression from 3.04 hours to 1.80 hours was driven by data, precision, and throughput optimizations, not model size changes.
  • The rational experimentation workflow uses the depth dial as a cost control: validate at --depth 6 (minutes, pennies), check at --depth 12 (tens of minutes, dollars), commit at --depth 26 (hours, tens of dollars).
  • DCLM CORE provides an objective, reproducible benchmark for comparing training runs. Without a fixed evaluation methodology, it is impossible to distinguish genuine improvements from noise.

Where to go next

  • Provision an 8×H100 node and run the speedrun at --depth 6 to validate the full pipeline end-to-end — this takes minutes and costs almost nothing, and gives you direct experience with every stage.
  • Study the nanochat leaderboard contributions to understand the optimization landscape: each entry represents a specific technical insight (FP8, batch size, dataset) that you can apply to your own experiments.
  • Explore domain-specific finetuning by curating an SFT dataset in your area of interest and running a customized pipeline — this is where nanochat transitions from a learning exercise to a practical tool.
