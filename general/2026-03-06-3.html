Build and Ship Your First Claude Code Plugin

From scaffold to marketplace: create a custom plugin with slash commands, agent definitions, and MCP server integration — then package it for the official Anthropic plugin directory.
35 min claude-code plugins mcp developer-tools hands-on

What's happening

Anthropic has launched an official, curated plugin directory for Claude Code — a structured marketplace where developers can publish and discover extensions that add slash commands, agent configurations, skill definitions, and MCP server integrations directly into the Claude Code environment.

This matters for two reasons. First, it establishes a standard plugin architecture: a well-defined directory structure (plugin.json, commands/, agents/, skills/, .mcp.json) that any developer can follow. Second, it creates a distribution channel: users can install plugins with a single command (/plugin install {name}@claude-plugin-directory) or browse them through the /plugin > Discover interface.

The plugin system builds on the Model Context Protocol (MCP) — the open standard for connecting AI models to external tools and data sources. A Claude Code plugin can bundle an MCP server alongside its commands, giving users seamless access to new capabilities without manual configuration.

In this session, we will scaffold a complete plugin from scratch, wire up a custom slash command, configure an MCP server, and package everything to the specification required for directory submission. The entire process will be driven through agent interactions — we will define objectives, prompt the agent, and evaluate its output at each stage.

1

Understand the Plugin Architecture

Before we build anything, we need a clear mental model of what a Claude Code plugin actually is.

A plugin is a directory that conforms to a specific structure. At minimum, it contains a .claude-plugin/plugin.json file — this is the plugin's manifest, declaring its name, version, description, and the capabilities it provides. Beyond this required manifest, a plugin may optionally include:

  • commands/ — Slash commands that users invoke directly (e.g., /my-command). Each command is defined in a Markdown file that specifies its behavior.
  • agents/ — Agent definitions that configure specialized Claude behaviors for particular tasks.
  • skills/ — Skill definitions that enable pattern-matched automatic behaviors.
  • .mcp.json — An MCP server configuration that exposes external tools and data sources to Claude.

Think of a plugin as a portable configuration bundle. It packages everything a user needs — tools, commands, agent personas, and skill triggers — into a single installable unit. The plugin system handles discovery, installation, and activation.

We will begin by asking our agent to generate the foundational scaffold.

Ask your agent
Have the agent generate the complete directory structure and plugin.json manifest for a new Claude Code plugin.
Think about it
  • What metadata does a plugin manifest need to declare? Consider name, version, description, author, and capabilities.
  • Which optional directories should be included in the scaffold if we plan to add both a slash command and an MCP server?
  • How should the plugin be named — what conventions does a directory listing impose on naming?
What the agent gives back

The agent should produce a directory tree showing the standard plugin layout — a root folder (e.g., my-first-plugin/) containing .claude-plugin/plugin.json, .mcp.json, commands/, agents/, skills/, and README.md. The plugin.json should be a small JSON object with fields for name, version, description, author, and a capabilities array listing commands and mcp. Expect roughly 8–12 lines of JSON for the manifest.

Tip
The .claude-plugin/ directory (note the leading dot) is what signals to Claude Code that this folder is a plugin. Without it, the plugin system will not recognize the directory.
What is the relationship between plugins and MCP?

MCP (Model Context Protocol) is the underlying protocol that allows Claude to interact with external tools and data sources. A plugin's .mcp.json file is simply an MCP server configuration — the same format used elsewhere in the Claude Code ecosystem. The plugin system's contribution is packaging and distribution: it bundles an MCP configuration alongside commands and agents so that users can install everything in one step, rather than manually editing their MCP settings.

2

Create a Custom Slash Command

Slash commands are the most visible feature a plugin provides. When a user types / in Claude Code, installed plugin commands appear alongside built-in ones. Each command is defined as a Markdown file in the commands/ directory.

A command file has a specific structure: a YAML front-matter block specifying the command's name, description, and any arguments it accepts, followed by a Markdown body that serves as the system prompt for that command's execution. When a user invokes the command, Claude reads this file and executes according to its instructions.

This is a powerful pattern. The command definition is essentially a structured prompt — you are authoring instructions that Claude will follow when the command is triggered. The quality of the command depends entirely on the clarity and precision of these instructions.

For this workshop, we will create a command called /review-deps that analyzes a project's dependency files and reports potential issues — outdated packages, security concerns, or unnecessary dependencies.

Ask your agent
Have the agent create a slash command definition file for a `/review-deps` command that analyzes project dependencies.
Think about it
  • What front-matter fields does a command file need? Think about how Claude Code identifies and presents the command to users.
  • The Markdown body is the prompt Claude follows when the command runs. What instructions would make a thorough dependency review? Consider scope, output format, and what 'issues' means in this context.
  • Should the command accept arguments — for instance, a path to a specific dependency file? How would that be declared?
What the agent gives back

The agent should produce a Markdown file (e.g., commands/review-deps.md) with YAML front-matter declaring the command name, a one-line description, and an optional argument for the file path. The body should contain clear instructions directing Claude to locate dependency files (package.json, requirements.txt, Cargo.toml, etc.), analyze them for outdated versions, known vulnerabilities, and unused dependencies, and produce a structured summary. The entire file should be concise — roughly 15–20 lines of Markdown.

Warning
Command names must be unique within a user's installed plugin set. Choose descriptive, namespaced names (e.g., review-deps rather than review) to avoid collisions with other plugins.
Why Markdown and not code?

Slash commands are declarative, not imperative. You do not write a program that executes when the command is invoked. Instead, you write a prompt — a set of instructions in natural language — that Claude follows within its existing capabilities. This is a deliberate design choice: it means anyone who can write a clear prompt can author a command, without needing programming expertise. The Markdown format provides structure (via front-matter) and expressiveness (via the body) in a single, human-readable file.

✓
At this point, you should have a plugin scaffold with a valid `plugin.json` manifest and a `commands/review-deps.md` file defining your first slash command. The directory structure should match the standard layout: `.claude-plugin/`, `commands/`, and the supporting directories.
3

Configure an MCP Server for the Plugin

An MCP server extends Claude's capabilities by providing access to external tools and data sources. In the context of a plugin, the .mcp.json file declares which MCP servers should be activated when the plugin is installed.

The .mcp.json format specifies server entries, each with a transport type (typically stdio for local servers or sse for remote ones), a command to start the server, and any arguments or environment variables it requires.

For our plugin, we will configure an MCP server that provides a check-npm-package tool — a tool that can query the npm registry for package metadata, enabling our /review-deps command to look up the latest versions of JavaScript dependencies. This demonstrates how MCP servers and slash commands can work together: the command provides the user-facing interface, while the MCP server provides the underlying capability.

Note that we are not implementing the MCP server itself in this session — we are configuring an existing one. The plugin's job is to declare the server; the server's implementation is a separate concern.

Ask your agent
Have the agent generate an `.mcp.json` configuration file that declares an MCP server for querying the npm registry.
Think about it
  • What fields does an MCP server configuration require? Consider the transport mechanism, the startup command, and any environment variables.
  • If the MCP server is a Node.js script bundled with the plugin, how should the command path be specified — relative or absolute?
  • What happens when a user installs a plugin with an `.mcp.json`? When does the server start, and who manages its lifecycle?
What the agent gives back

The agent should produce a .mcp.json file containing a mcpServers object with one entry. The entry should specify stdio transport, a command like node with arguments pointing to a server script (e.g., ./mcp-server/index.js), and any necessary environment configuration. The file should be compact — roughly 8–10 lines of JSON. The agent should also explain that this configuration is read by Claude Code at plugin activation time, and the server process is managed automatically.

API Key Note
If your MCP server requires API keys or secrets (e.g., for authenticated registry access), these should be referenced as environment variables in the .mcp.json — never hardcoded. Users will be prompted to provide values during plugin installation.
stdio vs. SSE transport — when to use which

The stdio transport runs the MCP server as a local subprocess, communicating via standard input/output. This is appropriate for servers bundled with a plugin — they run on the user's machine and have access to the local filesystem. The sse (Server-Sent Events) transport connects to a remote HTTP server, which is appropriate for hosted services. For plugins distributed through the directory, stdio is the common choice because it keeps everything self-contained and avoids dependency on external infrastructure.

Quick Check

You want your plugin's MCP server to access a user's GitHub token for querying private repository data. Where should this token be specified?
✗ Not quite. Hardcoding secrets in configuration files is a security violation. The `.mcp.json` file is distributed with the plugin and would expose the token to anyone who installs or inspects it.
✓ Correct! Environment variable references (e.g., `${GITHUB_TOKEN}`) allow the plugin to declare its requirements without embedding secrets. The user provides the value in their own environment, keeping credentials out of distributed files.
✗ Not quite. The `plugin.json` manifest is for metadata — name, version, description, and capabilities. It has no mechanism for secret management. Secrets belong in the runtime environment, not the manifest.
4

Package and Install the Plugin Locally

With our scaffold, command, and MCP configuration in place, we can now install the plugin locally to verify that it works before considering directory submission.

Local installation serves as both a functional test and a preview of the user experience. Claude Code's plugin system can install from a local directory path, which means we can iterate rapidly — modify a command file, reinstall, and test immediately.

The installation process validates the plugin structure: it checks for the required .claude-plugin/plugin.json, verifies the manifest fields, and registers any declared commands, agents, skills, and MCP servers. If anything is malformed, the installation will fail with a diagnostic message.

Once installed, the plugin's commands should appear in Claude Code's slash command menu, and its MCP server should be available for tool calls.

Ask your agent
Have the agent verify the plugin structure is complete and correct, then provide the exact installation command for local testing.
Think about it
  • What validation checks should be performed before installation? Think about required files, JSON syntax, and field completeness.
  • What is the command syntax for installing a plugin from a local directory path versus from the marketplace?
  • After installation, how would you verify that the command is registered and the MCP server is active?
What the agent gives back

The agent should first review the directory structure and validate that plugin.json is well-formed, command files have proper front-matter, and .mcp.json is valid JSON. It should then provide the local installation command — something like /plugin install ./my-first-plugin — and explain how to verify the installation by checking the slash command list and testing the /review-deps command on a sample project. The agent should also note common installation errors (missing manifest, invalid JSON, conflicting command names) and their remedies.

Tip
During development, you can modify plugin files in place and reinstall without uninstalling first. Claude Code will update the registration with the latest version of all files.
What does the submission process for the official directory look like?

To publish a plugin to the official Anthropic plugin directory, you submit it through the plugin directory submission form referenced in the repository. External plugins land in the /external_plugins directory and must meet Anthropic's quality and security standards. The review process evaluates the plugin's structure, documentation (a thorough README.md is essential), security posture (no hardcoded secrets, no excessive permissions), and overall utility. Approved plugins become discoverable via /plugin > Discover and installable via /plugin install {name}@claude-plugin-directory.

✓
You should now have a fully structured plugin installed locally in Claude Code, with a working `/review-deps` slash command and an MCP server configuration. The plugin is ready for testing and, after validation, submission to the official directory.

Your Turn

Create a second slash command for your plugin — `/audit-env` — that scans a project for environment variable usage, identifies any referenced variables that lack default values, and flags potential security risks such as secrets committed to version control.
Environment variable management is a common source of deployment failures and security incidents. A command that systematically audits `.env` files, code references to `process.env` or `os.environ`, and `.gitignore` patterns would be immediately useful to any developer. This exercise tests your ability to formulate a precise, scoped prompt that produces a well-structured command definition.
Think about it
  • What scope should the command cover? Should it scan all files, or focus on specific patterns and file types?
  • How should the command's output be structured — a flat list, a categorized report, or severity-ranked findings?
  • What constitutes a 'security risk' in the context of environment variables? How precisely can you define this for the agent?
  • Should the command accept arguments, such as a target directory or a strictness level?
See a sample prompt
One way you could prompt it 
Create a Claude Code slash command file at `commands/audit-env.md`. The command should be named `audit-env` with the description 'Scan project for environment variable issues and security risks.' It should accept an optional `--strict` flag. The command body should instruct Claude to: (1) find all `.env`, `.env.*` files and check if they are in `.gitignore`, (2) scan source files for references to environment variables (`process.env`, `os.environ`, `System.getenv`, etc.), (3) identify variables referenced in code but not defined in any `.env` file, (4) flag any `.env` files containing values that appear to be real secrets (API keys, tokens, passwords) rather than placeholders, and (5) produce a structured report with three sections: Missing Variables, Security Risks, and Recommendations. In strict mode, also flag variables without default/fallback values in code.
5

Prepare for Directory Submission

A plugin that works locally is only half the story. Publishing to the official directory requires meeting Anthropic's quality and documentation standards. The two most common reasons for submission rejection are insufficient documentation and poor security practices.

The README.md is the plugin's public face. It must clearly explain what the plugin does, what prerequisites it requires, how to configure it, and what each command or capability provides. Think of it as the documentation a competent developer would need to evaluate and adopt the plugin without any other context.

Security review focuses on several areas: the plugin should not include hardcoded credentials, it should not request unnecessary permissions, its MCP server should not perform actions beyond its stated scope, and all external dependencies should be pinned to specific versions.

Finally, the plugin must be self-contained and reproducible — another developer should be able to clone the repository, install the plugin, and have it work without undocumented manual steps.

Ask your agent
Have the agent generate a comprehensive README.md and perform a pre-submission audit of the plugin, identifying any issues that would prevent directory approval.
Think about it
  • What sections should a plugin README include? Think about what a reviewer and a potential user each need to know.
  • What security checks should be performed before submission? Consider secrets, permissions, and dependency management.
  • How should prerequisites and configuration steps be documented — especially for the MCP server's environment variable requirements?
What the agent gives back

The agent should produce a README.md with sections for Overview, Installation, Commands (documenting each slash command with examples), Configuration (explaining any required environment variables), Prerequisites, and License. It should also produce an audit checklist confirming: no hardcoded secrets, .mcp.json uses environment variable references, command names are properly namespaced, plugin.json has all required fields, and dependencies (if any) are version-pinned. The agent should flag any issues it finds and suggest remediations.

Warning
The submission form is for the official curated directory. Anthropic reviews all submissions for quality and security. Ensure your plugin is thoroughly tested locally before submitting — the review process evaluates real functionality, not just structural compliance.
Internal vs. external plugins — what's the difference?

The directory distinguishes between internal plugins (in /plugins) and external plugins (in /external_plugins). Internal plugins are developed and maintained by Anthropic team members. External plugins come from third-party partners and the community. Both undergo review, but external plugins go through the submission form and must meet published quality and security standards. The installation mechanism is identical for both — users install from the directory using the same command regardless of origin.

Recap

In this session, we constructed a complete Claude Code plugin from the ground up — from understanding the architecture through scaffolding, command creation, MCP configuration, local testing, and submission preparation.

The central insight is that a Claude Code plugin is fundamentally a structured collection of prompts and configurations, not a traditional software package. Slash commands are prompt files. Agent definitions are behavioral specifications. Skill definitions are pattern-matched triggers. The MCP configuration is a server declaration. The plugin system's contribution is packaging these elements into a discoverable, installable, and updatable unit.

This architecture means that the skill of building good plugins is largely the skill of writing precise, well-scoped instructions — exactly the competency this workshop develops. Every interaction with the agent required us to think carefully about scope, structure, and specificity before prompting.

  • A Claude Code plugin is a directory conforming to a standard structure: .claude-plugin/plugin.json (required), plus optional commands/, agents/, skills/, and .mcp.json.
  • Slash commands are Markdown files with YAML front-matter — they are prompts, not programs. The quality of a command depends on the clarity of its instructions.
  • MCP server configurations in plugins use environment variable references for secrets, never hardcoded values. The plugin system manages server lifecycle automatically.
  • Local installation (/plugin install ./path) provides rapid iteration during development. The official directory requires a submission review covering documentation, security, and functionality.
  • Effective agent prompting for plugin creation requires specifying scope, output format, naming conventions, and error-handling expectations — generic prompts yield generic results.

Where to go next

  • Add agent and skill definitions to your plugin — explore the agents/ and skills/ directories to create specialized behaviors that activate automatically.
  • Build a more sophisticated MCP server that exposes multiple tools, and configure it with the stdio transport for local development.
  • Study existing plugins in the official directory (both /plugins and /external_plugins) to understand patterns and conventions used by approved submissions.
  • Submit your plugin to the Anthropic plugin directory via the submission form and iterate based on review feedback.
