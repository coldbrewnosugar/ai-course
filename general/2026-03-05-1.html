OpenAI Goes Open — And It's Actually Good

gpt-oss gives you full chain-of-thought reasoning, native tool use, and it runs on hardware you can actually afford. Let's set it up.
40 min model-release open-weights local-inference reasoning fine-tuning

What's happening

Okay, so here's the headline: OpenAI just released two open-weight models under Apache 2.0. Yes, that OpenAI. The one that's been synonymous with "closed-source" for the past three years.

The models are called gpt-oss-120b and gpt-oss-20b, and they're genuinely impressive:

  • gpt-oss-120b has 117B total parameters but only 5.1B active at any time (thanks to Mixture-of-Experts architecture). It fits on a single H100.
  • gpt-oss-20b has 21B parameters with 3.6B active — and runs in 16GB of VRAM. That's a consumer GPU.
  • Both come with full chain-of-thought reasoning, configurable reasoning effort (low/medium/high), and native function calling.
  • They use a new thing called the Harmony response format, which is essential — the models literally won't work right without it.

The real kicker? They were post-trained with MXFP4 quantization baked in, meaning the quantization isn't an afterthought — all the benchmarks were run with it. You're getting the real deal, not a degraded version.

Today we're going to spin up gpt-oss-20b locally, understand what makes it tick, test its agentic capabilities, and peek at fine-tuning. Let's go.

1

Understand the Architecture Before You Install Anything

Before we touch any code, let's build a mental model of what gpt-oss actually is. This matters because it affects every decision you'll make — from which GPU you need to how you format your prompts.

Mixture-of-Experts (MoE) — the restaurant analogy:

Imagine a restaurant with 20 specialized chefs (math chef, writing chef, coding chef, etc.), but for any given dish, only 2-3 chefs actually work on it. That's MoE. gpt-oss-120b has 117 billion parameters total, but only routes through ~5 billion for each token. You get big-model intelligence at small-model speed.

MXFP4 quantization — built in, not bolted on:

Most quantized models feel like photocopies of photocopies. gpt-oss was post-trained with MXFP4 quantization, meaning the model learned to work well at lower precision. Think of it like a musician who practiced on a slightly out-of-tune piano — they've adapted to it, so the performance still sounds great.

The Harmony response format — this is the big one:

Harmony is a structured output format that gpt-oss was trained on exclusively. It handles things like separating the model's internal reasoning (chain-of-thought) from its final answer, and it structures tool calls in a specific way. If you don't use Harmony, the model will produce garbage. It's not optional — it's like trying to speak French to someone who only understands Mandarin.

What exactly is MXFP4 and why should I care?

MXFP4 stands for Microscaling Floating Point 4-bit. Normal FP16 uses 16 bits per parameter. MXFP4 uses just 4 bits but with a clever twist: groups of values share a scaling factor, which preserves more information than naive 4-bit quantization. The 'microscaling' part means each small block of weights gets its own scale, reducing the error that accumulates with cruder approaches. The practical result: gpt-oss-120b uses about 60GB of memory instead of 230GB+. That's the difference between 'you need a cluster' and 'you need one GPU.'

Spin Up gpt-oss-20b with vLLM

vLLM is the recommended way to serve gpt-oss. It gives you an OpenAI-compatible API server, which means any code or tool that talks to the OpenAI API can talk to your local model with zero changes. You just point it at localhost instead of api.openai.com.

The gpt-oss team released a special build of vLLM (0.10.1+gptoss) that has all the necessary kernels for MXFP4 and the Harmony format baked in. This isn't the standard vLLM pip package — you need to install from their custom index.

Honestly, this part confused me at first too — there are three different pip index URLs involved. But it's just one install command, and then one serve command. Let's get our agent to walk us through it.

Tip

Why vLLM instead of just using Transformers directly?

You can use HuggingFace Transformers directly — the repo has examples for that. But vLLM gives you three big advantages: (1) PagedAttention manages GPU memory like virtual memory manages RAM, so you can serve more concurrent requests, (2) continuous batching means new requests don't have to wait for the current batch to finish, and (3) the OpenAI-compatible API means you can swap between your local model and OpenAI's hosted version with one line change. For a workshop, Transformers is fine. For anything production-adjacent, vLLM is the move.

✓
At this point, you should have vLLM running with gpt-oss-20b, serving on localhost:8000. You can verify it's working by hitting the `/v1/models` endpoint — you should see `openai/gpt-oss-20b` listed. If you're getting CUDA errors, double-check your GPU has at least 16GB of VRAM and that you installed the right PyTorch nightly.

Talk to Your Local Model — With Harmony and Reasoning Effort

Now for the fun part — actually talking to the model. But there are two things that make gpt-oss different from just chatting with Llama or Mistral:

1. The Harmony format is mandatory.

When you use the vLLM OpenAI-compatible server, Harmony is handled for you automatically. But if you ever use the model directly (through Transformers, for example), you need the openai-harmony package to format your messages correctly. Think of Harmony as the envelope your letter has to go in — the model was trained to read messages in this specific packaging.

2. Reasoning effort is configurable.

You can set reasoning to low, medium, or high. Low is fast but shallow — good for simple lookups or classification. High gives you deep chain-of-thought but takes longer. It's like asking someone 'quick, what's 2+2?' versus 'walk me through this proof step by step.'

The coolest part? You get the full chain-of-thought in the response. Unlike OpenAI's hosted models where reasoning is hidden, here you can see exactly how the model thought through a problem. This is gold for debugging and understanding model behavior.

Tip
Try the same question at low and high reasoning effort and compare the responses. For a math problem, low might give you a quick answer while high shows you every step of the derivation. The latency difference is real — plan accordingly for your use case.
What's actually in the Harmony format?

Harmony structures the conversation into distinct sections: system instructions, user messages, assistant reasoning (the chain-of-thought), and assistant responses. It also has specific formatting for tool calls and tool results. The key innovation is that it cleanly separates 'thinking' from 'answering' — the model generates its reasoning in a dedicated block, then produces the user-facing response in another. This is why you get full chain-of-thought access. Without Harmony, the model tries to generate everything in one stream and gets confused about what's internal reasoning versus what to show the user.

Test Agentic Capabilities — Function Calling That Actually Works

Here's where gpt-oss gets really interesting. These models have native function calling — they were trained to understand tool definitions, decide when to call tools, and format the calls correctly. This isn't a hack or a prompt engineering trick; it's baked into the model's training.

What does 'native' mean in practice? You define your tools (functions with names, descriptions, and parameter schemas), include them in the request, and the model will:

  1. Decide if any tool is relevant to the user's question
  2. Generate a properly formatted tool call with the right arguments
  3. Wait for the result, then incorporate it into its reasoning

This is the foundation of agentic AI — a model that can take actions, not just generate text. And now you can run it entirely on your own hardware, with no API calls leaving your network.

The vLLM server handles the tool calling protocol just like OpenAI's API does, so if you've ever used function calling with GPT-4, the interface is identical.

What the agent gives back

The agent should give you a small example with a weather tool definition and show the back-and-forth flow. The key insight: the model never actually runs the tools — it just decides what to call and with what arguments. You're the one executing the functions.

Tip
The model can call multiple tools in a single response and handle parallel function calls. Try giving it a question that requires two lookups — like 'What's the weather in Tokyo and convert it to Fahrenheit?' — and watch it generate both calls at once.
How does function calling actually work under the hood?

When you include tool definitions in your request, the Harmony format injects them into the system prompt in a structured way the model was trained to recognize. The model then generates special tokens that signal 'I want to call a tool' instead of producing regular text. The vLLM server intercepts these tokens and packages them as a tool_calls object in the API response. It's essentially the model having a vocabulary that includes 'call this function with these args' alongside regular words. The training process taught it when to use this vocabulary versus just answering directly.

Fine-Tune gpt-oss with Unsloth — The 70% Less VRAM Trick

So you've got gpt-oss running locally and it's pretty good out of the box. But what if you want to make it yours — trained on your data, optimized for your specific use case?

Unsloth already supports gpt-oss fine-tuning, and the numbers are wild: 2x faster training with 70% less VRAM compared to standard approaches. They even have GRPO (Group Relative Policy Optimization) support for reinforcement learning, which is how you'd train your own reasoning behaviors.

The secret sauce is LoRA (Low-Rank Adaptation). Instead of updating all 21 billion parameters, LoRA freezes the base model and trains small adapter matrices — typically adding less than 1% new parameters. Unsloth then optimizes the heck out of this process with custom Triton kernels.

Think of it like this: the base model is a skilled chef who knows how to cook everything. Fine-tuning with LoRA is like giving them a small recipe card for your specific dish — they still use all their general cooking knowledge, but now they have your precise instructions on top.

The practical impact? You can fine-tune gpt-oss-20b on a single consumer GPU with ~6GB of VRAM. That's a $200 graphics card territory.

Tip
Unsloth provides free Colab notebooks specifically for gpt-oss. If you don't have a local GPU, start there — you can fine-tune the 20b model on Colab's free T4 GPU.
SFT vs GRPO — a quick comparison

SFT (Supervised Fine-Tuning) is like teaching by example: you show the model input-output pairs and it learns to mimic them. Great for style adaptation, domain knowledge, and specific formatting. GRPO (Group Relative Policy Optimization) is like teaching through feedback: the model generates multiple responses, you score them, and it learns to produce higher-scored outputs. GRPO is how you'd train better reasoning — you can reward the model for getting math problems right, writing better code, or following complex instructions more carefully. Unsloth's GRPO support for gpt-oss uses 80% less VRAM than standard implementations, making it feasible on consumer hardware.

Recap

Let's step back and appreciate what just happened. OpenAI — the company that literally started the 'closed-source AI' era — dropped two genuinely competitive open-weight models with Apache 2.0 licensing. And they're not toy models. They have configurable reasoning, full chain-of-thought, native tool use, and they're small enough to run on hardware real people actually own.

We went from understanding the architecture (MoE + MXFP4 = big brain, small footprint) to serving it locally with vLLM, to making actual API calls with reasoning effort control, to testing agentic function calling, to understanding how to fine-tune it on our own data with Unsloth.

The bigger picture here: the gap between cloud AI and local AI just got significantly smaller. You can now run a reasoning model with tool use on a single consumer GPU, fine-tune it for your domain, and keep all your data on your machine. That's a fundamentally different proposition than what we had six months ago.

  • gpt-oss uses MoE + MXFP4 to deliver big-model reasoning at small-model cost — the 20b runs in 16GB VRAM with only 3.6B parameters active per token
  • The Harmony response format is NON-NEGOTIABLE — always use it, whether through vLLM's automatic handling or the openai-harmony package. Without it, the model produces garbage.
  • Configurable reasoning effort (low/medium/high) lets you trade latency for depth — match the effort to your task complexity
  • Native function calling means you can build real agentic applications that run entirely on your own hardware — no API keys, no data leaving your network
  • Unsloth makes fine-tuning accessible on consumer GPUs with LoRA — use SFT for knowledge and style, GRPO for teaching better reasoning
