Hands-On with GPT-5.4: Exploring 1M-Token Context, Tool Search, and Computer Use via the API

Build an agent that leverages GPT-5.4's new computer-use and million-token context capabilities end-to-end
35 min model-release gpt-5.4 openai computer-use long-context tool-use api

What's happening

On March 5, 2026, OpenAI released GPT-5.4 — their latest frontier model and a substantial architectural upgrade over the GPT-5 series. Three capabilities distinguish this release from its predecessors and from competing models:

  1. A 1-million-token context window. Previous GPT models topped out at 128K tokens. GPT-5.4 extends this by roughly an order of magnitude, enabling ingestion of entire codebases, lengthy legal corpora, or multi-document research collections in a single prompt.
  1. Native computer-use capability. The model can observe screenshots, reason about graphical interfaces, and emit structured actions (click, type, scroll) to operate desktop and web applications autonomously — without external orchestration frameworks.
  1. Built-in tool search. Rather than requiring developers to pre-register every tool in the system prompt, GPT-5.4 can dynamically discover and invoke tools from a searchable registry, reducing prompt overhead and enabling more flexible agentic architectures.

Taken together, these features represent a meaningful shift in what a single API call can accomplish. This session examines each capability through structured experimentation, with the objective of building a lightweight autonomous agent that combines all three.

1

Establishing API Access and Verifying Model Availability

Before exercising any of GPT-5.4's new capabilities, we must establish a working connection to the OpenAI API and confirm that the model identifier resolves correctly. This step is deliberately simple — its purpose is to eliminate environmental issues before they compound with more complex tasks.

The model identifier for GPT-5.4 follows OpenAI's standard naming convention: gpt-5.4. It is accessible through the Chat Completions endpoint, which remains the primary interface for conversational and agentic workloads. The Responses API also supports the model, and we will use it in later steps when exercising tool-use features.

A key architectural detail: GPT-5.4 supports the same message format as its predecessors (system, user, assistant roles), but the max_tokens parameter now accepts values up to 1,000,000. The default remains conservative — typically 4,096 — so explicit configuration is required when working with extended context.

  • How should the API key be supplied — hardcoded, or through an environment variable? What are the security implications?
  • What is the simplest possible prompt you could send to verify the model is operational?
  • How will you confirm from the response object that `gpt-5.4` specifically handled the request?
What the agent gives back

The agent should produce a short Python script — no more than ten lines — that instantiates the OpenAI client, sends a trivial chat completion request specifying model="gpt-5.4", and prints both the response text and the model field from the response object. The key verification is that response.model returns a string containing gpt-5.4, confirming the correct model served the request.

```python

from openai import OpenAI

client = OpenAI() # reads OPENAI_API_KEY from environment

response = client.chat.completions.create(

model="gpt-5.4",

messages=[{"role": "user", "content": "Say hello."}]

)

print(response.choices[0].message.content)

print(f"Served by: {response.model}")

```

API Key Note
GPT-5.4 requires an active OpenAI API key with access to the model. As of launch, it is available on all paid API tiers. Set your key via export OPENAI_API_KEY=sk-... before running any scripts.
Tip
The response.model field may return a versioned identifier such as gpt-5.4-2026-03-05 rather than the bare string gpt-5.4. This is expected behavior — OpenAI appends snapshot dates to model identifiers.
Why verify the model identifier explicitly?

OpenAI's API silently falls back to available models if a requested identifier is invalid or the account lacks access. Without checking response.model, you could unknowingly run an entire session against GPT-4o or another fallback. This is a common source of confusing benchmark results and unexpected behavior differences. Always verify.

2

Exercising Computer Use: Screen Observation and Action Emission

GPT-5.4's computer-use capability allows the model to observe the state of a graphical interface — via screenshots — and respond with structured actions that an execution environment can carry out. The model does not control the computer directly; rather, it participates in an observe-reason-act loop where a host process captures screenshots, sends them to the model, receives action instructions, executes those actions, and captures the next screenshot.

This architecture separates perception and reasoning (the model's responsibility) from execution (the host's responsibility). The distinction matters: the model never has raw access to the operating system. It sees pixels and emits JSON-structured actions such as click(x, y), type(text), scroll(direction), and key(combination). The host process interprets and executes these.

OpenAI exposes computer use through the Responses API rather than the Chat Completions endpoint. The request includes a computer_use tool definition specifying the screen dimensions, and the model returns computer_call output items containing the actions to perform.

Conceptually, this is analogous to pair programming with a colleague who can see your screen but cannot touch your keyboard — they describe exactly what to click and type, and you carry out the instructions.

  • How would you structure the loop: capture → send → parse action → execute → repeat? What terminates the loop?
What the agent gives back

The script captures a screenshot (or loads one from file for testing), base64-encodes it, sends it to the Responses API with a computer_use_preview tool definition specifying screen width and height, and parses the returned computer_call action. The key structural elements are: (1) the tool definition with type: "computer_use_preview", display_width, and display_height fields; (2) the image sent as a base64-encoded content block; and (3) parsing the response for type: "computer_call" items that contain an action object with fields like type: "click", x, y or type: "type", text. The agent should note that actually executing the actions requires a system-level automation library such as pyautogui.

Warning
Computer use is a powerful capability that executes actions on real systems. When experimenting, use a sandboxed environment — a virtual machine or Docker container with a virtual display — to prevent unintended modifications to your primary system.
Tip
For testing without a live desktop, capture a static screenshot of any application and use it as input. The model will still produce valid actions relative to the UI elements it observes in the image.
How does computer use differ from function calling?

Function calling provides the model with a structured interface: defined parameters, typed inputs, and predictable outputs. Computer use is fundamentally different — the model operates on unstructured visual input (pixels) and must infer the state of the interface, identify interactive elements, and reason about spatial coordinates. This makes it more flexible (it can operate any GUI application without a pre-built API) but also less reliable (pixel-level reasoning introduces ambiguity). In practice, function calling is preferable when a structured interface exists; computer use fills the gap for applications that expose only a graphical interface.

Built-in Tool Search: Dynamic Tool Discovery at Runtime

Traditional tool-use architectures require all available tools to be declared in the system prompt or tool registry before the conversation begins. This creates a tension: registering many tools inflates prompt size and can degrade model performance, but registering too few limits the agent's capability.

GPT-5.4 introduces built-in tool search as a resolution to this tension. Instead of enumerating every tool upfront, developers register tools in a searchable catalog. The model, when it determines that a task requires a tool it does not currently have access to, can query this catalog and retrieve relevant tool definitions on demand.

The mechanism operates as follows: the developer includes a tool_search tool in the request's tool list. When the model invokes tool_search, it provides a natural-language query describing the capability it needs. The host application searches the catalog, returns matching tool definitions, and the model can then invoke those tools in subsequent turns. This is analogous to a craftsperson who, rather than carrying every tool at all times, knows how to locate the right tool from a well-organized workshop when the task demands it.

This pattern is particularly valuable for agents that operate across diverse domains — a single agent might need file-system tools, database connectors, web scrapers, and communication APIs, but rarely all at once.

  • How should the search function rank results — exact match, semantic similarity, or keyword overlap?
  • What happens when tool search returns multiple candidates? How does the model select the right one?
  • How do you prevent the model from searching for tools endlessly instead of completing the task?
What the agent gives back

The agent should describe a tool registry architecture with three components: (1) a catalog — a list of tool definitions, each containing a name, description, parameter schema, and tags for searchability; (2) a search function that accepts a natural-language query and returns the top-N matching tools, using either keyword matching or embedding-based semantic search; and (3) integration with the Responses API where tool_search is defined as an available tool, and when invoked, the host intercepts the call, runs the search, and injects the matched tool definitions into the conversation. The agent should emphasize that matched tools are added to the available tools for subsequent turns, not executed immediately.

Tip
Start with a simple keyword-based search over tool names and descriptions. Semantic search using embeddings is more powerful but adds latency and complexity. Optimize only when the simpler approach proves insufficient.
How large can a tool catalog be before search becomes necessary?

Empirical testing across the GPT-4 and GPT-5 series suggests that model performance with tool use begins to degrade noticeably beyond 15-20 simultaneously available tools. The model struggles to consistently select the correct tool when presented with many similar options, and the token cost of including all definitions grows linearly. Tool search becomes valuable when the total catalog exceeds this threshold, even if only a handful of tools are needed for any given task. It is a form of retrieval-augmented tool use — the same principle as RAG, applied to capabilities rather than knowledge.

4

Stress-Testing the Million-Token Context Window

A 1-million-token context window fundamentally changes what can be accomplished in a single API call. To provide a sense of scale: one million tokens corresponds to approximately 750,000 words, or roughly 10-15 full-length novels, or the entire codebase of a medium-sized software project.

However, context window size and context window utilization are distinct concerns. Three practical considerations govern effective use of extended context:

1. Attention distribution. Research on long-context models consistently demonstrates a "lost in the middle" phenomenon — models attend most strongly to information near the beginning and end of the context, with reduced recall for material in the middle. GPT-5.4 improves on this, but the pattern has not been entirely eliminated.

2. Cost and latency. Token pricing applies to the entire context, including input. A 1M-token prompt at GPT-5.4's input pricing represents a non-trivial cost per request. Latency also scales with context length — time-to-first-token increases as the model processes more input.

3. Retrieval vs. ingestion. For many use cases, the question is not "can the model hold all this context?" but "should it?" Retrieval-augmented generation (RAG) — where relevant passages are extracted and injected into a shorter prompt — often yields comparable accuracy at lower cost. The million-token window is most valuable when the task genuinely requires holistic understanding of the entire corpus: cross-document reasoning, comprehensive summarization, or global pattern detection.

  • How would you structure the system prompt to guide the model's attention across a very long context? Consider explicit instructions about where to look.

What the agent gives back

The agent should describe a pipeline that: (1) reads multiple documents from a directory, (2) concatenates them with clear delimiters (e.g., XML-style tags like <document id="1" title="...">) to preserve boundaries, (3) prepends a system prompt instructing the model to treat the collection as a unified corpus and reference documents by their identifiers, and (4) poses an analytical question that requires synthesizing information from multiple documents. The agent should also suggest a verification strategy: embed a unique "needle" phrase in a document in the middle of the context and ask the model to locate it, confirming that the full context is being processed. The key API parameter is max_tokens set high enough for a substantive response, and the agent should note that the total input token count can be checked via response.usage.prompt_tokens.

Warning
A 1M-token input at GPT-5.4's pricing represents significant cost per request. During development, test with a subset of your corpus (50K-100K tokens) and only scale to full context when the pipeline is validated.
Tip
Use tiktoken with the encoding for GPT-5.4 to count tokens before sending a request. This prevents unexpected truncation or API errors from exceeding the context limit.
When should you use RAG instead of the full context window?

The million-token context window does not obsolete retrieval-augmented generation. RAG remains preferable when: (1) the total corpus exceeds 1M tokens, (2) the query targets specific factual information rather than requiring global synthesis, (3) cost sensitivity is high and the task can be served with a smaller context, or (4) low latency is critical and the overhead of processing a large context is unacceptable. The full context window excels at tasks requiring holistic reasoning — comparing themes across an entire book, identifying contradictions across a legal filing, or understanding the full dependency graph of a codebase. Think of RAG as a targeted search and the full context as reading the entire document shelf.

Assembling the Agent: Combining Computer Use, Tool Search, and Extended Context

The individual capabilities we have examined — computer use, tool search, and extended context — are each useful in isolation. Their real power emerges when combined in a single agentic workflow.

Consider a realistic scenario: an analyst needs to review a large collection of research papers, extract key findings, and enter a summary into a web-based project management tool that has no API. This task requires: (1) ingesting the papers into the context window for cross-document analysis, (2) dynamically discovering the right summarization and formatting tools via tool search, and (3) using computer use to navigate the web application and enter the results.

The architectural pattern for this combined agent is an orchestration loop with three phases per iteration:

  • Perceive: Gather input — load documents, capture screenshots, receive user instructions.
  • Reason: Send all available context to GPT-5.4, including tool definitions and any computer-use observations. The model decides what to do next.
  • Act: Execute the model's instructions — run a tool, perform a computer action, or return a result to the user.

The loop continues until the model signals completion or a maximum iteration count is reached. The critical design decision is the state management between iterations: what context carries forward, what gets summarized, and what gets discarded.

Ask your agent

Think about it
  • How should the agent's state be represented between loop iterations? What information must persist, and what can be discarded?
  • When the model returns both a tool call and a computer action in the same response, what is the correct execution order?
  • How do you prevent the context from growing unboundedly across iterations, even with a 1M-token window?
  • What termination conditions should the loop enforce — both success conditions and safety limits?
What the agent gives back

The agent should describe a main loop function that maintains a conversation history and iterates through perceive-reason-act cycles. Key design elements include: (1) a message history that carries forward across iterations, with a summarization strategy to compress older messages when the context approaches capacity; (2) a response parser that handles three output types — text responses, tool calls (including tool search), and computer-use actions — and dispatches each to the appropriate handler; (3) a tool executor that intercepts tool search calls to augment the available tools dynamically; (4) a computer-use executor that captures screenshots after each action for the next observation cycle; and (5) termination conditions including a maximum iteration count, explicit completion signals from the model, and an error threshold. The agent should emphasize that the loop is model-driven — GPT-5.4 decides what to do at each step, and the orchestrator simply executes.

Tip
Implement a sliding-window summarization strategy for the conversation history: after every N iterations, ask the model to summarize the oldest messages into a concise state description, then replace those messages with the summary. This prevents context exhaustion in long-running tasks.
Warning
An agent with computer use and tool access can take consequential actions — modifying files, submitting forms, sending messages. Always implement a human-in-the-loop confirmation step for irreversible actions during development.
How does this architecture compare to existing agent frameworks?

The perceive-reason-act loop is a well-established pattern in agent design, shared by frameworks such as LangChain's AgentExecutor, AutoGPT, and Anthropic's computer-use reference implementation. The distinguishing factor with GPT-5.4 is that all three capability types — tool use, computer use, and extended context — are handled natively within a single model call, rather than requiring separate model invocations or specialized sub-agents. This reduces architectural complexity and latency, though it concentrates more responsibility (and potential failure modes) in a single component. The tradeoff is simplicity versus resilience: a multi-model architecture can route different capabilities to specialized models, while a single-model approach is easier to build and debug but has a single point of failure.

Recap

In this session, we examined three capabilities that define GPT-5.4 as a meaningful advancement for agentic applications.

We began by establishing API access and verifying model availability — a simple but essential foundation. We then explored computer use, understanding the observe-reason-act loop that allows GPT-5.4 to interact with graphical interfaces through structured action emission. We examined tool search as a solution to the tool registration scalability problem, enabling dynamic capability discovery at runtime. We stress-tested the 1-million-token context window, analyzing its practical limits and the strategies required to use it effectively — including attention distribution patterns, cost considerations, and the continued relevance of RAG. Finally, we assembled these capabilities into a cohesive orchestration loop that represents the core architectural pattern for GPT-5.4-powered agents.

The central insight is that these capabilities are individually useful but combinatorially powerful. An agent that can see a screen, discover tools on demand, and reason over vast context simultaneously occupies a qualitatively different capability tier than one limited to text-in, text-out interactions.

  • GPT-5.4's computer use operates through an observe-reason-act loop where the model sees screenshots and emits structured actions — it never directly controls the system. Use it only when no structured API alternative exists.
  • Tool search is retrieval-augmented generation applied to capabilities rather than knowledge. It resolves the tension between tool availability and prompt efficiency by enabling dynamic discovery from a searchable catalog.
  • A million-token context window enables holistic cross-document reasoning, but it does not replace retrieval-augmented generation. Cost, latency, and attention distribution patterns mean that RAG remains optimal for targeted queries over very large corpora.
  • The perceive-reason-act orchestration loop is the fundamental architectural pattern for combining these capabilities. State management — particularly context summarization across iterations — is the key design challenge.
  • Always implement safety boundaries for autonomous agents: sandboxed execution environments for computer use, maximum iteration limits for loops, and human-in-the-loop confirmation for irreversible actions.
