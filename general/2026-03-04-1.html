RAG Pipelines, But Make Them Lego Bricks

UltraRAG 3.0 turns every RAG component into an independent MCP server you wire together with YAML — no orchestration spaghetti required
40 min RAG MCP low-code retrieval agents YAML

What's happening

Here's what's been happening: UltraRAG 3.0 dropped in late January from a collaboration between Tsinghua (THUNLP), Northeastern University (NEUIR), OpenBMB, and AI9stars. The big idea? Instead of building RAG pipelines as monolithic chains where your retriever, reranker, and generator are all tangled together in one codebase, each component runs as its own MCP Server — a standalone service with a clean interface.

Why should you care? Because the Model Context Protocol (MCP) is quickly becoming the USB-C of AI tooling. It's a standard way for AI models to talk to external tools and data sources. UltraRAG takes that concept and applies it to RAG specifically: your retriever is one MCP server, your reranker is another, your generator is a third. You connect them with a YAML config file that supports loops, conditionals, and branches.

Meanwhile, AgentScope has added native MCP support, and ReMe offers a memory layer with hybrid retrieval (vector + BM25) that pairs nicely with this whole ecosystem. The trend is clear: RAG is moving from "write a big script" to "compose small services."

Today we'll build a document Q&A pipeline using this architecture — and you'll see why a few dozen lines of YAML can replace hundreds of lines of orchestration code.

1

Get the Mental Model: MCP as RAG Architecture

Before we touch any tools, let's make sure we're on the same page about what's actually happening here.

Think of a traditional RAG pipeline like a restaurant where one person takes your order, cooks the food, and serves it. It works, but if you want to swap out the chef or add a dessert course, you're rewriting the whole operation.

UltraRAG's MCP approach is more like a food truck court. Each truck (MCP server) does one thing well — one retrieves documents, one reranks them, one generates answers. They don't know about each other's internals. A coordinator (the MCP client) takes your order and routes it to the right trucks in the right sequence.

The magic ingredient is that this routing logic lives in a YAML file, not buried in Python code. Want to add a reranking step? Add three lines of YAML. Want to loop back and retrieve more documents if the first answer isn't confident enough? That's a loop construct in YAML.

Here are the key pieces:

  • MCP Servers: Atomic services — each one registers "tools" (functions) that the orchestrator can call
  • MCP Client: The orchestrator that reads your YAML and calls the right tools in the right order
  • YAML Workflow: Your pipeline definition — sequential steps, conditional branches, loops
  • Pipeline Builder UI: A visual canvas that syncs bidirectionally with the YAML (edit either one, the other updates)

What exactly is MCP and why does it matter for RAG?

The Model Context Protocol is an open standard (originally from Anthropic) that defines how AI models interact with external tools and data sources. Think of it like HTTP for AI tools — a shared language so any model can talk to any tool.

For RAG, this is a game-changer. Traditionally, if you built a retriever in LangChain and wanted to swap it into a LlamaIndex pipeline, you'd rewrite the integration. With MCP, your retriever just exposes a standard interface: "give me a query, I'll give you documents." Any MCP-compatible orchestrator can use it.

UltraRAG is the first framework to go all-in on this: every component IS an MCP server. The framework itself is basically a smart MCP client that knows how to read YAML and call these servers in the right order.

2

Define Your First Pipeline in YAML

Alright, this is where it gets fun. In UltraRAG, a pipeline is just a YAML file that says: "First do this, then do that, and if this condition is true, do this other thing."

The simplest RAG pipeline has three steps:

  1. Retrieve — find relevant documents for the user's query
  2. Rerank — rescore and filter those documents for relevance
  3. Generate — feed the top documents + query to an LLM for an answer

In traditional code, this is 100+ lines of Python with error handling, prompt templates, and glue code. In UltraRAG, it's a YAML workflow where each step calls a tool on an MCP server.

The YAML supports three control structures you'll use constantly:

  • Sequential: Steps run one after another (the default)
  • Conditional branches: "If the confidence score is below 0.7, retrieve more documents"
  • Loops: "Keep refining the answer until quality checks pass"

Honestly, when I first saw this I thought "there's no way YAML is expressive enough for real pipelines." But the trick is that the complex logic lives inside the MCP servers — the YAML just wires them together. It's like how a circuit board doesn't need to understand transistor physics; it just needs to connect the right components.

How does YAML handle loops and branches?

UltraRAG extends standard YAML with control flow keywords. A conditional branch looks something like a step with a condition field — if the condition evaluates to true, execute one set of steps; otherwise, execute another. A loop wraps a set of steps with a loop construct that includes a termination condition.

For example, an iterative retrieval strategy might say: "Retrieve documents → Generate answer → Check confidence → If confidence < threshold, reformulate query and loop back to retrieval." In traditional code, this is a messy while loop with state management. In UltraRAG YAML, it's a loop block with a break_condition.

The visual Pipeline Builder makes this even clearer — you literally drag and drop nodes and draw connections, and the YAML updates in real time (and vice versa).

Build a Knowledge Base and Wire Up Hybrid Retrieval

A RAG pipeline without documents is like a library with no books. Let's fix that.

UltraRAG includes Knowledge Base Management built right into the UI — you upload documents (PDFs, markdown, text files), and it handles chunking, embedding, and indexing. But what's interesting is how it retrieves.

Most tutorials show you pure vector search: embed the query, find the nearest document embeddings, done. That works okay, but it has a blind spot. Vector search is great at semantic similarity ("car" matches "automobile") but terrible at lexical matching (if someone asks for "error code E-4012" and that exact string is in your docs, vector search might miss it).

Hybrid retrieval combines two approaches:

  • Vector search (dense retrieval): Good at meaning, bad at exact matches
  • BM25 (sparse retrieval): Good at exact keyword matching, bad at synonyms

Think of it like having two friends help you search — one who understands what you mean and one who's great at finding the exact words you said. Together, they cover each other's weaknesses.

ReMe's memory system uses this same hybrid pattern, which is why the tools pair well together. The retriever MCP server in UltraRAG can be configured to use either or both strategies.

Warning
Chunk size matters more than you'd think. Too small and you lose context. Too large and you dilute the relevant signal with noise. 512 tokens with some overlap is a solid starting point, but ask your agent about how to tune this for your specific documents.
Deep dive: How Reciprocal Rank Fusion works

When you have two ranked lists of documents (one from vector search, one from BM25), you need a strategy to combine them. Reciprocal Rank Fusion (RRF) is elegant in its simplicity:

For each document, its RRF score = sum of 1 / (k + rank) across all lists where it appears. The constant k (usually 60) prevents top-ranked documents from having disproportionate scores.

So if a document is ranked #1 in vector search and #5 in BM25, its score is 1/61 + 1/65 = 0.0318. A document ranked #2 in both gets 1/62 + 1/62 = 0.0323 — slightly higher because consistent ranking is rewarded.

The beauty is that RRF doesn't care about the actual scores from each system (which might be on totally different scales). It only cares about rank positions. This makes it robust and easy to implement.

How does ReMe's hybrid retrieval compare?

ReMe (from the AgentScope ecosystem) uses a similar hybrid approach for its vector-based memory system. It supports three memory types — personal memory, task/procedural memory, and tool memory — all searchable via a memory_search method that combines vector and BM25.

The interesting connection: you could use ReMe as the memory layer alongside UltraRAG's document retrieval layer. ReMe handles what the agent has learned from past conversations; UltraRAG handles querying your knowledge base. Together, they give an agent both long-term memory and access to your documents — which is honestly where this whole space is heading.

Add Iterative Retrieval with YAML Loops

Here's where UltraRAG's YAML orchestration really flexes. In a basic pipeline, you retrieve once and generate once. But what if the first retrieval doesn't find great documents? Or the generated answer contradicts itself?

Iterative retrieval is the strategy where your pipeline can loop back: generate an answer, evaluate it, and if it's not good enough, reformulate the query and try again. It's like when you Google something, don't find what you need, rephrase your search, and try again — except automated.

Traditionally, implementing this requires writing a state machine in Python: track attempt counts, manage query reformulation, handle termination conditions, store intermediate results. It's easily 200+ lines of orchestration code.

In UltraRAG, it's a loop construct in your YAML. You define:

  • The steps that repeat (retrieve → evaluate)
  • A break condition (confidence > 0.8 OR attempts > 3)
  • How the query gets reformulated between iterations

This is the kind of pattern that separates "I built a RAG demo" from "I built a RAG system that actually works." And it's also where the MCP architecture pays dividends — because each iteration calls the same MCP servers with different inputs, and those servers don't need to know they're being called in a loop.

What the agent gives back

The agent should modify your YAML to wrap the retrieve-rerank-generate steps in a loop construct. After generation, a new 'evaluate' step checks the answer quality (using the LLM itself as a judge — asking 'does this answer address the query using the provided documents?'). If the score is below a threshold, a 'reformulate' step asks the LLM to rephrase the query based on what was missing. The loop breaks when quality is sufficient or after 3 iterations. The agent should explain each new section and warn you about the cost implications — each loop iteration means more LLM calls.

Tip
LLM-as-judge (using the model to evaluate its own outputs) is a pragmatic pattern, not a perfect one. It's good enough for most cases but can be blind to its own mistakes. For production, consider adding rule-based checks alongside LLM evaluation.
Warning
Each loop iteration costs money (LLM API calls). Set a max iteration limit and consider adding a cost tracking step. Ask your agent about adding logging to track how many iterations each query actually needs — you'll be surprised how often one pass is enough.
When is iterative retrieval actually worth it?

Iterative retrieval shines for complex, multi-faceted questions where a single query can't capture everything. For example: "Compare the pricing models of AWS Lambda and Google Cloud Functions for a workload that processes 10,000 images per day."

A single retrieval might find documents about Lambda pricing OR Cloud Functions pricing, but rarely both in the right context. An iterative approach can:

  1. First pass: Retrieve Lambda pricing docs
  2. Reformulate: "Now find Google Cloud Functions pricing for image processing"
  3. Second pass: Retrieve Cloud Functions docs
  4. Generate: Compare using both sets of documents

For simple factual questions ("What's the default timeout for Lambda?"), iteration is wasteful. The art is knowing when to enable it — and that's often a conditional branch in your YAML: if the query looks complex (multi-entity, comparative), use the iterative path; otherwise, use the simple one.

✓
You now have a complete iterative RAG pipeline defined in YAML — retriever, reranker, generator, evaluator, and query reformulator, all wired together with loop logic. This is a production-grade retrieval pattern described in a config file. Take a moment to appreciate that this would have been 300+ lines of Python orchestration code without the MCP/YAML approach.

Use the Pipeline Builder UI and Test Your Pipeline

UltraRAG's Pipeline Builder is where the visual and code worlds merge. It's a canvas-based editor — think flowchart tool — where you can:

  • Drag and drop MCP server nodes onto a canvas
  • Draw connections between them to define data flow
  • Edit parameters inline (click a node, change top_k, swap the model)
  • See the YAML update in real time as you rearrange the visual layout
  • Edit the YAML directly and watch the visual canvas rearrange itself

This bidirectional sync is honestly the killer feature for learning. You can build visually to understand the flow, then switch to YAML to understand the config syntax, and they always stay in sync.

The UI also has a built-in AI Assistant that can help you design pipelines, tune parameters, and even generate prompts. Once your pipeline is ready, you can convert it into an interactive chat interface with one click — meaning you can go from YAML config to "chat with your documents" demo in minutes.

Connecting UltraRAG to AgentScope and ReMe

Since AgentScope has native MCP support, you can use UltraRAG's MCP servers as tools within an AgentScope agent. The agent gets ReAct-style reasoning (from AgentScope), document retrieval (from UltraRAG), and persistent memory (from ReMe).

The setup: point AgentScope's MCP client at UltraRAG's server addresses. Now your AgentScope agent can call retrieve, rerank, and generate as tools, and AgentScope's built-in planning decides when to use them. Add ReMe for memory, and your agent remembers past conversations while also querying your document knowledge base.

This composability is the whole point of MCP-based architecture. Each piece is independent, reusable, and swappable.

Recap

Let's step back and look at what we actually built today.

We started with a mental model: RAG components as independent MCP servers, wired together with YAML instead of Python glue code. We built a pipeline that retrieves documents using hybrid search (vector + BM25), reranks them for precision, generates answers, and — the cool part — loops back to try again if the answer isn't good enough.

The bigger insight isn't about UltraRAG specifically. It's about composable AI architectures. The MCP protocol means your retriever doesn't care if it's being called by UltraRAG, AgentScope, or something you build yourself. Your reranker doesn't know if it's part of a three-step pipeline or a ten-step one. Everything is a service with a standard interface.

This is how the best software gets built — small, focused pieces that compose cleanly. And now it's how RAG pipelines get built too.

  • MCP servers turn RAG components into reusable, swappable building blocks — the retriever, reranker, and generator don't know about each other and don't need to
  • YAML orchestration with loops and conditional branches replaces hundreds of lines of state management code, and the visual Pipeline Builder makes it explorable
  • Hybrid retrieval (vector + BM25 with Reciprocal Rank Fusion) covers both semantic and keyword matching — always use both unless you have a specific reason not to
  • Iterative retrieval with LLM-as-judge evaluation is the pattern that separates demo-quality RAG from production-quality RAG
  • Composability across tools (UltraRAG for retrieval, AgentScope for agent reasoning, ReMe for memory) is the real payoff of the MCP architecture
