Turn Your AI Agent into a Lab Partner

170+ open-source scientific skills just dropped. Let's install them and run a real research workflow — no PhD required.
35 min agent-skills scientific-computing open-source claude-code

What's happening

So here's the deal: a company called K-Dense just open-sourced a collection of 170+ scientific agent skills that plug into Claude Code, Cursor, and Codex via something called the Agent Skills standard. Think of it like an app store for your AI agent's brain — except instead of apps, you're installing expertise.

Why does this matter? Because LLMs are already decent at science-y tasks, but they're inconsistent. Ask Claude to do a cheminformatics workflow and sometimes it nails the RDKit calls, sometimes it hallucinates a function that doesn't exist. These skill files act like curated cheat sheets — they tell your agent exactly which libraries to use, what the correct function signatures are, and how to chain steps together.

The skills cover everything from single-cell RNA-seq to drug discovery to astronomical data analysis. We're going to install the whole pack, peek under the hood at how a skill is structured, and then actually run a multi-step scientific workflow to see the difference it makes.

Honestly, this is one of those things that sounds incremental on paper but feels magical in practice. Let's dig in.

1

What Even Is an Agent Skill?

Before we install anything, let's get a mental model for what's happening here.

Think of your AI agent like a really smart generalist doctor. They know about a lot of things, but if you hand them a mass spectrometer and say "analyze this proteomics data," they might fumble. An agent skill is like handing that doctor a detailed protocol sheet — it doesn't make them smarter, it makes them more reliable at a specific task.

Technically, a skill is just a markdown file that lives in your project's .claude/skills/ directory (or ~/.claude/skills/ for global access). When your agent starts working, it reads these files as extra context. Each skill contains:

  • What the skill does — a clear description of the workflow
  • Which libraries and tools to use — no guessing, no hallucinating APIs
  • Step-by-step patterns — the correct order of operations with real function calls
  • Common pitfalls — things the agent would normally get wrong

The key insight: you're not writing code. You're giving your agent a better map of the territory.

Tip
The Agent Skills standard is open and tool-agnostic. These same skill files work in Claude Code, Cursor, Codex, and any agent that reads from .claude/skills/. You're not locked into one ecosystem.
How is this different from a system prompt?

Great question. A system prompt is one big blob of instructions that applies to everything. Skills are modular and contextual — your agent can have hundreds of them but only the relevant ones get activated for a given task. It's the difference between memorizing an entire textbook vs. having a well-organized reference shelf. The agent reads the skill that matches what you're asking it to do, not all 170+ at once.

Install the Scientific Skill Pack

Alright, let's get these skills onto your machine. The K-Dense repo is a regular Git repository — you clone it and copy the skill files where your agent can find them.

Here's where you start working with your agent instead of doing everything manually. Let's ask it to handle the installation for us.

Ask your agent
Get your agent to clone the K-Dense scientific skills repo and install the skills into your global Claude skills directory.
Think about it
  • Where do global agent skills live on your filesystem? (Hint: think about the `.claude/` directory in your home folder)
  • What's the Git command to clone a repo? Does your agent need the full URL?
  • After cloning, you need to copy the skill files to the right location — what directory structure does the Agent Skills standard expect?
What the agent gives back

Your agent should clone the repo from https://github.com/K-Dense-AI/claude-scientific-skills, then copy the skill markdown files into ~/.claude/skills/. It might create that directory if it doesn't exist. After running, you should be able to list the skills directory and see 170+ .md files organized by category — things like cheminformatics.md, single-cell-rnaseq.md, proteomics.md, etc.

Tip
Installing to ~/.claude/skills/ makes skills available globally across all your projects. If you only want them in one project, use .claude/skills/ inside that project directory instead.
Warning
These skills tell your agent how to use scientific libraries, but they don't install the libraries themselves. When you actually run a workflow, your agent will need to pip install the relevant packages (like RDKit, scanpy, biopython, etc.) — it'll handle that as part of the workflow.
What does a skill file actually look like inside?

Each skill is a markdown file with structured sections. Imagine something like:

Title & description — what this skill enables (e.g., "Molecular property prediction using RDKit")

Prerequisites — which Python packages are needed

Workflow steps — numbered instructions with actual function calls and correct parameters

Example patterns — short code snippets showing the right way to call specific APIs

Gotchas — common mistakes and how to avoid them

The beauty is you never need to read these files yourself. They're written for the agent, not for you. But peeking inside helps you understand why your agent suddenly got so much better at a specific task.

✓
At this point you should have the K-Dense scientific skills installed in your `~/.claude/skills/` directory. Run `ls ~/.claude/skills/ | head -20` to confirm you see skill files. If you see markdown files with scientific-sounding names, you're golden.

Instead of manually grepping through files, let's ask our agent to do the inventory for us.

Ask your agent
Ask your agent to survey the installed skills and give you a categorized summary — how many skills per domain, and what kinds of workflows each category supports.
Think about it
  • What would you ask a librarian if you walked into a new library? You'd want the map, not every book.
  • Think about what grouping would be most useful to you — by scientific domain? By complexity? By the libraries they use?
  • Can you get the agent to highlight which skills chain together for multi-step workflows?
What the agent gives back

The agent should read through the skill files and give you a nicely organized breakdown — something like: Bioinformatics (25 skills): single-cell RNA-seq, gene regulatory networks, variant annotation... Cheminformatics (20 skills): molecular property prediction, ADMET analysis, virtual screening... Clinical Research (15 skills): pharmacogenomics, clinical trials, drug safety... and so on. The best responses will also note which skills work well together — like how the cheminformatics skills can feed into the drug discovery pipeline skills.

Tip
Try asking your agent: "Which of these skills would be useful for someone studying [your field]?" Personalizing the exploration makes it way more actionable.
The 'multi-step workflow' superpower

Individual skills are useful, but the real magic is chaining them. For example, a drug discovery workflow might use:

  1. Cheminformatics skill — to parse SMILES strings and compute molecular descriptors
  2. ADMET prediction skill — to predict absorption, distribution, metabolism, excretion, and toxicity
  3. Molecular docking skill — to simulate how a molecule binds to a target protein
  4. Data visualization skill — to create publication-quality figures of the results

Without skills, the agent might get each step roughly right but mess up the handoffs between them. With skills, the agent knows exactly what format each step expects and produces. It's like the difference between handing someone a recipe vs. handing them four separate recipes that were designed to work together.

Run a Real Scientific Workflow: Drug Property Prediction

Here's where it gets fun. We're going to ask our agent to run a cheminformatics workflow — specifically, taking a molecule (represented as a SMILES string — that's the text format chemists use to describe molecular structures) and predicting its drug-like properties.

If you've never done cheminformatics before, don't worry. That's literally the point — the skill handles the domain expertise so you can focus on the scientific question.

A SMILES string is just a text encoding of a molecule. For example, CC(=O)OC1=CC=CC=C1C(=O)O is aspirin. Think of it like GPS coordinates for a chemical structure — compact, precise, and machine-readable.

Let's ask our agent to take aspirin's SMILES string and run a full drug-likeness analysis.

Ask your agent
Get your agent to take aspirin's SMILES string and predict its drug-like properties — molecular weight, LogP (lipophilicity), hydrogen bond donors/acceptors, and whether it passes Lipinski's Rule of Five.
Think about it
  • What information does the agent need to get started? (The SMILES string and what properties you want)
  • Should you tell the agent which library to use, or let the skill guide it?
  • What format would make the results easiest to interpret — a table? A summary with pass/fail?
  • Lipinski's Rule of Five is a quick filter for whether a molecule might work as an oral drug. Can you ask the agent to explain the results, not just compute them?
What the agent gives back

The agent should use RDKit (guided by the cheminformatics skill) to compute molecular descriptors for aspirin. You'll get back something like:

  • Molecular Weight: ~180.16 (passes — under 500)
  • LogP: ~1.2 (passes — under 5)
  • H-Bond Donors: 1 (passes — under 5)
  • H-Bond Acceptors: 4 (passes — under 10)
  • Verdict: Passes all Lipinski criteria ✓

A good agent response will also explain what these numbers mean — that LogP measures how well the molecule dissolves in fats vs. water (important for absorption), and that Lipinski's rules are a rough filter, not a guarantee. The agent might also generate a 2D structure image of the molecule.

API Key Note
No API keys needed for this workflow! RDKit is a free, open-source cheminformatics library. Your agent will pip install it as part of the workflow. It's a hefty install (~100MB) but totally free.
Tip
Try swapping in different molecules after aspirin works. Caffeine is CN1C=NC2=C1C(=O)N(C(=O)N2C)C. Ibuprofen is CC(C)CC1=CC=C(C=C1)C(C)C(=O)O. See how the properties differ!
Why does having a skill make this so much better?

Without the cheminformatics skill, here's what typically goes wrong when you ask an agent to do this:

  1. It might use the wrong RDKit functions (there are dozens of descriptor calculators)
  2. It might compute LogP using a deprecated method
  3. It might forget to sanitize the molecule first (a critical RDKit step that catches invalid structures)
  4. It might apply Lipinski's thresholds incorrectly

The skill file contains the exact correct workflow: parse SMILES → sanitize → compute descriptors using the right functions → apply the right thresholds. The agent follows the protocol instead of improvising. In scientific work, that reliability difference is everything.

5

Chain It: From Single Molecule to Mini Screening Pipeline

One molecule is cool. But real drug discovery involves screening many molecules. Let's level up — ask the agent to build a mini virtual screening pipeline that takes a list of molecules and ranks them by drug-likeness.

This is where multi-step workflows shine. You're asking the agent to:

  1. Parse multiple SMILES strings
  2. Compute properties for each one
  3. Filter by Lipinski's rules
  4. Rank the survivors
  5. Visualize the results

Five steps, each with its own potential for errors. But with the skill guiding the agent, it chains them together correctly.

Ask your agent
Ask your agent to take a list of 5-10 molecules (by SMILES string), compute drug-likeness properties for each, filter out any that fail Lipinski's rules, and create a ranked table plus a visualization comparing the passing molecules.
Think about it
  • Where should the list of molecules come from? You could ask the agent to use well-known drugs, or provide your own list.
  • What should the ranking criterion be? Drug-likeness isn't one number — how should the agent prioritize?
  • What kind of visualization would be most informative? A scatter plot of LogP vs. molecular weight? A radar chart?
  • Should the agent save results to a file, or just display them inline?
What the agent gives back

The agent should produce a complete mini-pipeline that processes multiple molecules. You'll get:

  1. A table showing each molecule's name, SMILES, molecular weight, LogP, H-bond donors/acceptors, and pass/fail status
  2. A filtered list of molecules that pass all Lipinski criteria
  3. A visualization — likely a scatter plot with molecular weight on one axis and LogP on the other, with each molecule labeled and color-coded by pass/fail
  4. A brief analysis explaining which molecules look most promising and why

The key thing to notice: the agent chains all five steps together seamlessly because the skill defines how data flows between them. No weird format mismatches, no dropped molecules, no plotting errors.

Tip
If you want to go further, try asking the agent to add ADMET predictions (absorption, distribution, metabolism, excretion, toxicity) to the pipeline. The ADMET skill should chain naturally with the cheminformatics one.
What if I want to do biology instead of chemistry?

Swap the domain entirely! The same pattern works for bioinformatics workflows. For example:

  • Single-cell RNA-seq: Give the agent a gene expression matrix and ask it to cluster cells, identify marker genes, and create a UMAP visualization using scanpy
  • Genomics: Provide a VCF file and ask for variant annotation and pathogenicity prediction
  • Proteomics: Supply mass spec data and ask for peptide identification

Each of these has a dedicated skill that guides the agent through the correct library calls and data formats. The prompting pattern is identical — describe your scientific question, reference the relevant skill, and let the agent handle the implementation.

✓
You should now have a working drug-likeness screening pipeline that processes multiple molecules, filters by Lipinski's rules, and produces a visualization. If your agent generated a scatter plot and a ranked table, you're in great shape. If something broke, try explicitly telling the agent to 'use the cheminformatics skill' in your prompt.

Recap

Let's step back and appreciate what just happened. We took a vanilla AI agent, installed 170+ scientific skill files (literally just markdown documents), and turned it into something that can reliably execute multi-step research workflows across chemistry, biology, medicine, and more.

We didn't write any scientific code ourselves. We didn't need to know RDKit's API or scanpy's preprocessing pipeline. We learned to describe what we want scientifically and let curated skill definitions guide the agent through the correct implementation.

The bigger picture: this is what the Agent Skills standard enables. Skills are modular, shareable, and composable. Someone with deep domain expertise writes the skill once, and then everyone's agent gets better at that task. It's like open-source, but for agent competence rather than code.

  • Agent skills are curated context files (markdown) that make AI agents dramatically more reliable at specific tasks — they're protocol sheets, not code libraries
  • The Agent Skills standard is open and portable — same skill files work across Claude Code, Cursor, Codex, and any compatible agent
  • Multi-step scientific workflows benefit the most from skills because they eliminate the error accumulation that happens when agents improvise each step
  • You don't need domain expertise to run sophisticated analyses — you need to know how to describe your scientific question and point the agent to the right skill
  • Skills are composable: chaining cheminformatics → ADMET → visualization skills creates pipelines that would take days to code manually
