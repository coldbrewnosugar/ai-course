Your LLM Is Doing the Same Math Over and Over Again

LMCache fixes that — caching KV states across requests so your GPU stops wasting cycles on repeated context
40 min inference-optimization vLLM KV-cache open-source serving

What's happening

Here's a pattern that's quietly eating GPU budgets everywhere: you've got a RAG pipeline, or a chatbot with a big system prompt, or multi-turn conversations — and every single request makes your model re-process the same long context from scratch. That's like re-reading the entire Harry Potter series every time someone asks you a question about Hogwarts.

LMCache just hit trending on GitHub, and for good reason. It's an open-source extension that plugs into vLLM or SGLang and caches the internal KV states (the model's 'working memory' of text it's already processed) across requests and even across different serving instances. The result? 3-10x reduction in time-to-first-token for long-context workloads.

It's already been adopted by GMI Cloud, Google Cloud, CoreWeave, and integrated with Redis, NVIDIA Dynamo, and KServe. This isn't a toy project — it's production infrastructure that's solving a real cost problem.

Today we're going to understand why this matters, set it up with vLLM, and benchmark the difference ourselves.

1

Understand the Problem: Why KV Cache Reuse Is a Big Deal

Before we touch any tools, let's build a mental model for what's happening inside your LLM when it processes a request.

When a transformer model reads your prompt, it builds up something called KV (Key-Value) cache — think of it as the model's scratch notes. For each token in your input, the model computes attention keys and values that capture how that token relates to everything else. These scratch notes are expensive to create — they require full forward passes through the model.

Here's the kicker: if you send the same system prompt with every request (which almost everyone does), the model rebuilds those exact same scratch notes every single time. It's like a student re-deriving calculus from first principles before answering each homework question.

LMCache says: "Hey, we already computed those notes. Let me just hand them back to you." It stores KV caches in a multi-tier system — GPU memory (fastest), CPU RAM, disk, or even S3 (cheapest) — and serves them back when matching text appears in a new request.

The really clever part? It works for any reused text, not just prefixes. And it works across different serving instances in your cluster. That's wild.

Tip
TTFT = Time To First Token — the delay before the model starts streaming its response. For interactive applications, this is the metric users feel most. Nobody minds a slow stream, but a 2-second blank stare kills the UX.
Why not just use vLLM's built-in prefix caching?

vLLM does have automatic prefix caching (APC), and it works well — for a single instance. But APC only helps when the same instance sees repeated prefixes, and it doesn't persist across restarts. LMCache extends this idea to work across multiple instances, survive restarts, and handle non-prefix reuse (like shared context that appears in the middle of a prompt). Think of APC as a single-machine L1 cache, and LMCache as the distributed L2/L3 hierarchy.

```yaml

local_cpu:

enabled: true

max_size: 30 # GB of CPU RAM for cache

chunk_size: 256 # tokens per cache chunk

```

Then you launch vLLM with --kv-cache-config lmcache_config.yaml and you're rolling.

Warning
GPU requirement: You need an NVIDIA GPU with CUDA support. LMCache works on Linux only. If you're on a Mac or don't have a GPU, you can still follow along conceptually — the config and benchmarking patterns are what matter most here.
Tip
Chunk size matters. LMCache breaks KV caches into chunks (default 256 tokens). Smaller chunks = more flexible reuse but more overhead. Bigger chunks = less overhead but you might miss partial matches. 256 is a solid default for most workloads.
What about multi-tier storage?

LMCache supports a storage hierarchy: GPU → CPU → Disk → S3. In production, you'd typically use CPU as the hot tier and disk or S3 for persistence. The tiering is automatic — hot entries stay in CPU RAM, cold ones spill to disk. For our workshop we'll stick with CPU to keep things simple, but adding disk storage is just a few more YAML lines. The real power move is using Redis or a shared storage backend so all your serving instances share the same cache pool.

Our benchmark scenario:

Without LMCache, every request pays the full prefill cost. With LMCache, the first request pays full price, but requests 2-10 get the system prompt's KV cache for free. We should see TTFT drop dramatically after the first request.

Use streaming mode for TTFT measurement. When you enable stream=True on the API call, you can timestamp when the first chunk arrives — that's your TTFT. Without streaming, you only get total latency, which includes generation time too.
What kind of speedups should I actually expect?

LMCache claims 3-10x TTFT reduction, and that tracks with the math. The speedup depends on:

  • What fraction of your prompt is reusable? If 80% is shared context, you skip 80% of prefill.
  • How long is the shared context? Longer shared prefixes = bigger absolute savings. A 500-token system prompt saves less than a 5,000-token one.
  • Your GPU speed. Slower GPUs show a bigger relative improvement because prefill takes longer in the first place.
  • Cache tier. GPU memory gives near-zero overhead. CPU adds some transfer time. Disk/S3 adds more, but it's still way faster than recomputing.

For a typical RAG workload with a 2-4K shared prefix, expect 3-5x TTFT reduction on the cached portion.

4

Scale It Up: Multi-Instance Cache Sharing

Here's where LMCache gets really interesting — and where it pulls ahead of simpler prefix caching solutions.

In production, you don't run one vLLM instance. You run a fleet behind a load balancer. Without shared caching, each instance maintains its own KV cache, and a request that hits Instance B gets zero benefit from the cache that Instance A already built. It's like every cashier at a grocery store having to memorize the price list independently.

LMCache solves this with a shared storage backend. All instances read and write to the same cache pool — typically backed by Redis or a shared filesystem. When Instance A processes a new system prompt, it writes the KV cache to the shared store. When Instance B gets a request with the same prompt, it pulls the cached KV states instead of recomputing them.

This is especially powerful for:

  • Scaling events — new instances come up with a warm cache immediately
  • Multi-tenant platforms — popular system prompts get cached once, benefit everyone
  • Disaggregated prefill — you can have dedicated prefill workers that just generate KV caches for other instances to consume

For cutting-edge setups, LMCache supports NIXL (NVIDIA's inter-node transfer library) for near-zero-copy GPU-to-GPU transfers.

Tip
Disaggregated prefill is the power-user pattern here. You dedicate some instances purely to computing KV caches (prefill workers) and others purely to generating tokens (decode workers). LMCache is the glue that transfers KV states between them. This lets you independently scale prefill and decode capacity.
How does NIXL fit into this?

NIXL (NVIDIA Inter-node eXchange Library) is NVIDIA's low-level library for high-speed data transfer between GPUs across machines. LMCache integrates with NIXL to transfer KV caches directly from GPU memory on one machine to GPU memory on another — skipping the CPU entirely. This matters at scale because KV caches for large models are huge (tens of GB). Going GPU→network→GPU instead of GPU→CPU→network→CPU→GPU cuts transfer time significantly. It's like having a dedicated express lane on the highway.

5

Tune for Production: Cache Eviction, Monitoring, and Gotchas

Alright, we've got LMCache running and benchmarked. Before you take this to production, let's talk about the things that bite people.

Cache sizing: Your cache has finite space. LMCache uses LRU (Least Recently Used) eviction by default — old, unused entries get kicked out to make room. But if your cache is too small for your workload's diversity, you'll get poor hit rates and wasted memory. The rule of thumb: monitor your cache hit rate and size up until it stabilizes above 80%.

Model updates: When you update or fine-tune your model, the old KV caches are invalid — same text produces different internal states. LMCache handles this through cache keys that include model identity, so a model swap automatically invalidates the old cache. But be aware: your cache goes cold after every model update.

Token boundaries matter: LMCache works at the token level, and different tokenizers produce different tokens for the same text. If you're sharing caches across model variants, make sure they use the same tokenizer.

Memory pressure: KV caches are big. A single 8K-context request on a 70B model can generate 5-10 GB of KV data. If you're caching thousands of unique contexts on CPU, you'll need serious RAM. Plan your hardware accordingly.

What about cost savings in practice?

The cost math is straightforward: if LMCache reduces your prefill compute by 70% (because 70% of your average prompt is cached), you need ~30% of the GPU capacity for prefill that you'd otherwise need. For a fleet running 100 A100s on a RAG workload, that could mean dropping to 50-60 GPUs for the same throughput. At $2-3/hour per A100, that's $80-120/hour saved. The cache infrastructure (extra CPU RAM, maybe a Redis cluster) costs a fraction of that. The ROI is real, which is why cloud providers are adopting this.

Recap

We just went from "what even is a KV cache?" to understanding a production-grade caching system that major cloud providers are deploying right now. Let's step back and look at the big picture.

The core insight: LLM serving is full of redundant computation. Every time your model processes the same system prompt, the same RAG context, or the same conversation history, it's burning GPU cycles to produce identical internal states. LMCache intercepts that waste by storing and reusing KV caches across requests and across your entire serving fleet.

What we built: A vLLM deployment with LMCache, a benchmark showing 3-10x TTFT improvement on cached content, and an understanding of multi-instance cache sharing and production tuning.

The bigger trend: This is part of a broader shift in LLM infrastructure from "throw more GPUs at it" to "be smarter about what you compute." KV cache reuse, speculative decoding, prompt caching at the API level (like Anthropic's and OpenAI's built-in prompt caching) — they're all attacks on the same problem. Understanding KV cache reuse gives you a mental model for all of them.

  • KV caches are the model's expensive scratch notes — reusing them for repeated context skips the most expensive part of inference (prefill)
  • LMCache adds a multi-tier (GPU/CPU/Disk/S3) cache layer to vLLM or SGLang with minimal config — it's not a fork, it's a plugin
  • The real power is cross-instance sharing: all your serving instances can share a cache pool via Redis or shared storage, so one instance's work benefits the whole fleet
  • Production tuning is about cache sizing and hit rate — monitor these metrics and you'll know exactly when LMCache is earning its keep
  • This same KV-cache-reuse concept underlies prompt caching features in commercial APIs (Anthropic, OpenAI) — understanding the mechanism helps you use those features better too

Where to go next

  • Try LMCache with SGLang if that's your serving engine of choice — it supports KV cache offloading there too
  • Explore disaggregated prefill: separate your fleet into prefill workers and decode workers connected by LMCache
  • Read the LMCache technical report for the CacheGen compression algorithm — it compresses KV caches for cheaper storage and faster transfer
  • Benchmark on your actual workload — the speedup varies dramatically based on how much context you reuse, so real numbers from your traffic matter more than synthetic benchmarks
