Build a Super Agent with DeerFlow 2.0

ByteDance's open-source framework just hit #1 on GitHub Trending. Let's spin it up and see what all the fuss is about.
40 min multi-agent orchestration sandboxes open-source

What's happening

What just happened?

On February 28th, 2026, ByteDance dropped DeerFlow 2.0 — a ground-up rewrite of their open-source agent framework — and it immediately rocketed to #1 on GitHub Trending.

Here's why that matters: most AI agent frameworks give you one agent that tries to do everything. DeerFlow takes a fundamentally different approach. It's a super agent harness — think of it like a project manager that delegates work to specialized sub-agents, each with their own skills, their own sandbox to run code in, and access to shared long-term memory.

The name stands for Deep Exploration and Efficient Research Flow, but version 2.0 has evolved well beyond research. It can research, code, create content, run analysis — basically anything you can break into sub-tasks.

The key ingredients:

  • Sub-agent orchestration — multiple agents working together, not one agent doing everything
  • Extensible skills — plug in new capabilities without touching the core
  • Sandboxed code execution — agents can write and run code safely (Docker or local)
  • Long-term memory — agents remember context across sessions
  • LangChain model integration — swap in GPT-4, Claude, Gemini, or open-source models

And it's fully open-source. Let's get it running.

1

Clone and configure DeerFlow

First things first — we need to get DeerFlow on our machine and tell it which AI model to talk to.

DeerFlow uses a config.yaml file as its brain configuration. This is where you define which models are available, how they behave, and what API keys to use. Think of it like setting up a new phone — before it does anything useful, it needs to know your accounts.

The setup is refreshingly simple: clone, run a config generator, edit one file. The make config command creates local config files from templates so you don't accidentally commit your API keys to GitHub (we've all been there).

What the agent gives back

Your agent should walk you through the three commands (git clone, cd deer-flow, make config) and then help you fill in config.yaml. The key part is the model definition — it needs a LangChain class path, a model identifier, and an API key reference. Here's what one model entry looks like:

```yaml

models:

  • name: gpt-4

display_name: GPT-4

use: langchain_openai:ChatOpenAI

model: gpt-4

api_key: $OPENAI_API_KEY

max_tokens: 4096

```

The use field is the magic — it's a LangChain class path, meaning you can swap in any LangChain-compatible model provider.

API Key Note
You'll need at least one model provider API key (OpenAI, Anthropic, etc.) and a Tavily API key for the search/research capabilities. Tavily has a free tier that's plenty for experimenting. Set these in a .env file in the project root — never hardcode keys in config.yaml for anything beyond local tinkering.
Tip
The use: langchain_openai:ChatOpenAI syntax is a LangChain class path. This means DeerFlow supports any model that LangChain supports — including local models via Ollama. Ask your agent how to configure a local model if you want to go that route.
What's the deal with LangChain class paths?

The use field in config.yaml follows the pattern module_path:ClassName. So langchain_openai:ChatOpenAI means "from the langchain_openai package, use the ChatOpenAI class." This is DeerFlow's way of being model-agnostic without writing adapter code for every provider.

For Anthropic models, you'd use langchain_anthropic:ChatAnthropic. For Google, langchain_google_genai:ChatGoogleGenerativeAI. If a LangChain integration exists for a model, DeerFlow can use it.

This is a genuinely clever design decision — instead of building their own model abstraction layer, they piggyback on LangChain's massive ecosystem of integrations.

2

Launch DeerFlow with Docker

Now let's actually run this thing. DeerFlow gives you two options: Docker (recommended) or local development. We're going with Docker because it handles all the dependencies, sets up the sandbox environment, and gets you a consistent setup regardless of what's on your machine.

Here's a mental model for what happens when you start DeerFlow: imagine spinning up a small office. The main app is the receptionist — it takes your requests through a web UI at localhost:2026. Behind the scenes, there's a provisioner service that manages sandbox containers (think: disposable workspaces where agents can safely run code). The Docker setup wires all of this together.

The interesting bit: make docker-start is smart enough to read your config.yaml and only spin up the provisioner if you've configured DeerFlow to use container-based sandboxing. No wasted resources.

Tip
Prefer running locally without Docker? Run make check first to verify you have Node.js 22+, pnpm, uv, and nginx installed. Then make dev starts everything. But honestly, Docker saves you from dependency headaches — especially with the sandbox setup.
Warning
If make docker-start hangs or errors, check that Docker Desktop is running and has enough allocated memory (at least 4GB recommended). Also verify your .env file exists and has valid API keys — DeerFlow won't start properly without at least one configured model.
What's this 'provisioner' thing?

The provisioner is DeerFlow's container manager for sandboxed code execution. When a sub-agent needs to run code (say, a Python script to analyze some data), the provisioner spins up a fresh Docker container, runs the code inside it, captures the output, and tears the container down.

This is crucial for safety — you don't want an AI agent running arbitrary code directly on your machine. The sandbox is like giving the agent a disposable laptop: it can install packages, write files, run scripts, but when it's done, everything gets thrown away. Your host machine stays clean.

The provisioner only starts if your config.yaml specifies src.community.aio_sandbox:AioSandboxProvider with a provisioner_url. Otherwise, DeerFlow uses local execution mode, which runs code directly on your host (faster but less isolated).

✓
At this point, you should have DeerFlow running and accessible at `http://localhost:2026`. Open it in your browser — you should see a clean chat interface. If you see a connection error, double-check that your Docker containers are running (`docker ps`) and that your API keys are set correctly.
3

Understand the sub-agent architecture

Okay, this is where DeerFlow gets genuinely interesting. Most agent frameworks work like a solo contractor — one agent, one brain, trying to juggle everything. DeerFlow works like a general contractor with specialized subcontractors.

Here's the mental model: when you give DeerFlow a complex task (say, "research the latest trends in solar energy and create a report with charts"), the orchestrator breaks it into sub-tasks and delegates each one to the right sub-agent. One agent handles web research. Another writes the report. Another generates charts by writing and executing Python code in a sandbox.

Each sub-agent has its own skills — think of these as the agent's toolbox. A research agent has search tools. A coding agent has sandbox access. A writing agent has formatting tools. The orchestrator knows which agent has which skills and routes work accordingly.

The clever part? These sub-agents can talk to each other. The research agent's findings get passed to the writing agent. The writing agent can ask the coding agent to generate a visualization. It's collaborative, not just parallel.

Honestly, this part confused me at first too. The key insight is that DeerFlow isn't just running multiple agents independently — it's orchestrating them. The orchestrator maintains a shared context so sub-agents aren't working in the dark.

What the agent gives back

Your agent should explain that DeerFlow uses a planning-then-execution pattern. The orchestrator first creates a task plan (breaking the request into sub-tasks), then assigns each sub-task to the appropriate sub-agent based on required skills. Sub-agents communicate through a shared context/memory layer — not direct messages. The orchestrator manages dependencies ("don't start the chart generation until the data analysis is done"). This is fundamentally different from a simple chain because the orchestrator can adapt the plan mid-execution if a sub-agent discovers something unexpected.

Skills vs. Tools — what's the difference?

In DeerFlow's terminology, tools are atomic capabilities — a single function like "search the web" or "read a file." Skills are higher-level packages of behavior that might use multiple tools.

Think of it like cooking: a tool is a knife. A skill is "prep vegetables" — it involves using the knife, a cutting board, knowing how to julienne, etc.

DeerFlow's skills are extensible — you can write your own and plug them in. This is one of the main reasons the framework is getting attention. Instead of forking the whole project to add a capability, you just define a new skill.

How does this compare to CrewAI or AutoGen?

If you've played with other multi-agent frameworks, here's the key differentiator: DeerFlow bundles sandboxed execution and persistent memory directly into the orchestration layer. CrewAI focuses on role-based agent collaboration but doesn't include built-in sandboxing. AutoGen excels at conversational agent patterns but takes a different approach to task decomposition.

DeerFlow's sandbox integration means agents can safely write and run code as a first-class operation, not an afterthought. And the long-term memory means agents get smarter across sessions — they remember what worked before. That combo is what makes it a "super agent harness" rather than just another multi-agent framework.

4

Explore memory and context engineering

Here's something that separates DeerFlow from a lot of agent frameworks: long-term memory.

Most AI agents have the memory of a goldfish. Every conversation starts fresh. DeerFlow's agents can remember things across sessions — what you asked for before, what strategies worked, what your preferences are.

Think of it like the difference between calling a random customer service rep every time versus having a dedicated account manager who knows your history. The first interaction might be similar, but by the fifth time, the agent with memory is dramatically more useful.

DeerFlow calls this broader concept context engineering — it's not just about stuffing things into the prompt window. It's about strategically deciding what context each sub-agent needs, when to pull from long-term memory, and how to keep the shared context clean as a complex task unfolds.

This is one of those features that doesn't sound flashy until you experience it. The first time DeerFlow references something you told it three sessions ago without you having to repeat it — that's when it clicks.

Tip
DeerFlow also supports MCP (Model Context Protocol) servers — this means you can plug in external tools and data sources that sub-agents can access. If you've used MCP with Claude Desktop, the same concept applies here but across multiple collaborating agents.
What is 'context engineering' really?

Context engineering is the practice of carefully managing what information goes into an AI's prompt window. It's become a hot term in 2026 because as agent systems get more complex, naive approaches ("just dump everything into the context") break down fast.

Imagine you're managing a team of five people working on a project. You wouldn't CC everyone on every email — the designer doesn't need the database migration details, and the backend dev doesn't need the color palette discussion. You'd curate who gets what information.

That's context engineering for agents. DeerFlow's orchestrator decides: what does this specific sub-agent need to know to complete its sub-task? What from long-term memory is relevant? What from the current conversation matters? It assembles a focused context window for each agent, keeping them efficient and on-task rather than drowning in irrelevant information.

Give DeerFlow a real multi-step task

Time to see the whole system work together. Let's give DeerFlow a task that actually requires multiple sub-agents collaborating.

A good test is something that involves research + analysis + output creation — that forces the orchestrator to use at least three different sub-agents and coordinate their work.

Go to localhost:2026 and try giving it a complex request. Watch how the UI shows you the task being decomposed, sub-agents being assigned, and results flowing between them. This is the payoff for all the setup.

The key thing to observe: DeerFlow doesn't just run things in parallel for speed. It runs things in the right order — respecting dependencies. The analysis agent waits for the research agent. The report writer waits for the analysis. But independent sub-tasks DO run in parallel when possible.

What models work best with DeerFlow?

DeerFlow's docs recommend using capable models for the orchestrator (it's doing the hardest job — planning and coordination) and potentially lighter models for simpler sub-agents. GPT-4, Claude 3.5 Sonnet/Opus, and Gemini Pro are all solid choices for the orchestrator.

An interesting optimization: you can configure different models for different roles in config.yaml. Use a heavy model for orchestration and a faster, cheaper model for straightforward sub-tasks like formatting or simple searches. This keeps costs down without sacrificing quality where it matters.

The LangChain integration means you can even mix providers — Claude for orchestration, GPT-4 for coding tasks, a local Ollama model for simple classification. DeerFlow doesn't care, as long as LangChain can talk to it.

Recap

What we built

We went from zero to a running DeerFlow 2.0 instance — ByteDance's open-source super agent harness that just hit #1 on GitHub Trending.

More importantly, we understood why it's architecturally interesting:

The orchestrator pattern — instead of one agent struggling with complex tasks, DeerFlow breaks work into sub-tasks and routes each one to a specialized sub-agent. It's the difference between one overwhelmed generalist and a well-coordinated team of specialists.

Sandboxed execution — agents that can safely write AND run code, not just generate it. Docker containers give agents disposable workspaces where they can install packages, run scripts, and produce real output without touching your machine.

Long-term memory — agents that learn and remember across sessions. This is the feature that turns DeerFlow from a cool demo into something genuinely useful for daily work.

Extensible skills — add new capabilities without forking the project. DeerFlow's plug-in architecture means it can grow with your needs.

The real takeaway isn't about DeerFlow specifically — it's about a shift in how agent systems are being built. We're moving from single-agent-does-everything to orchestrated teams of specialized agents. DeerFlow 2.0 is one of the best open-source implementations of this pattern available today.

  • Multi-agent orchestration beats single-agent complexity — break hard tasks into sub-tasks handled by specialized agents
  • Sandboxed code execution is essential for agent safety — always prefer Docker isolation when agents run code autonomously
  • Long-term memory transforms agents from stateless tools into persistent collaborators that improve over time
  • LangChain class paths make DeerFlow model-agnostic — you can mix and match providers and even use local models
  • Context engineering (curating what each agent sees) is as important as the model you choose

Where to go next

  • Write a custom skill for your specific use case and register it with the orchestrator
  • Experiment with different model configurations — try a heavy model for orchestration and lighter models for sub-agents to optimize cost vs. quality
  • Set up MCP servers to give DeerFlow access to your own tools and data sources
  • Explore the 1.x branch if you're specifically interested in DeerFlow's Deep Research roots
  • Join the DeerFlow community on GitHub — the project is actively developed and contributions are welcome
