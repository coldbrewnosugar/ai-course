Build a Super Agent with DeerFlow 2.0

Clone, configure, and run ByteDance's trending multi-agent framework — then understand how it orchestrates sub-agents, sandboxes, and memory under the hood
40 min multi-agent orchestration sandboxes open-source

What's happening

What happened: On February 28th, 2026, ByteDance dropped DeerFlow 2.0 — a complete ground-up rewrite of their open-source agent framework — and it immediately claimed the #1 spot on GitHub Trending.

Why it matters: Most "agent frameworks" give you one agent with a bunch of tools bolted on. DeerFlow is different. It's a super agent harness — think of it like a conductor leading an orchestra. The conductor (DeerFlow) doesn't play every instrument. Instead, it orchestrates sub-agents that each have their own specialties: one researches, one codes, one creates content. They share long-term memory, run code in sandboxes so nothing blows up your machine, and can learn new tricks through an extensible skills system.

The 2.0 rewrite shares zero code with v1 (which was more of a "deep research" tool). This version is about doing almost anything autonomously — research, write code, generate reports, create artifacts — by breaking big tasks into smaller ones and handing them to the right sub-agent.

What we're doing today: We'll get DeerFlow running locally, understand how its architecture connects the pieces, and learn to think about multi-agent systems by prompting our AI agent to help us every step of the way.

1

Get the Lay of the Land — Clone and Configure

Before we spin anything up, let's understand what we're working with. DeerFlow uses a pretty standard setup flow: clone the repo, run a config generator, then fill in your API keys and model preferences.

The key file is config.yaml — this is where you tell DeerFlow which LLM to use. The cool part? It uses LangChain class paths to specify models, which means you can plug in basically any model provider (OpenAI, Anthropic, local models, whatever) as long as there's a LangChain integration for it.

Think of config.yaml as the brain transplant menu. You're choosing which brain your super agent gets.

What's a LangChain class path?

When you see use: langchain_openai:ChatOpenAI in the config, that's a LangChain class path. It's like an address that says "go to the langchain_openai package, find the ChatOpenAI class, and use that as the model interface."

This is clever because DeerFlow doesn't need to know about every LLM provider. It just says "give me a LangChain-compatible chat model" and the class path tells it which one. Want to use Anthropic? Change it to langchain_anthropic:ChatAnthropic. Local model via Ollama? langchain_ollama:ChatOllama. The framework doesn't care — it just needs something that speaks the LangChain chat interface.

It's like how a USB port doesn't care if you plug in a keyboard or a mouse — it just needs something that speaks USB.

Configure Your Model — The Brain Transplant

Here's where it gets fun. The config.yaml file is DeerFlow's master control panel. You define one or more models, and DeerFlow can assign different models to different sub-agents (imagine giving your researcher a fast cheap model and your coder a powerful expensive one).

Each model entry needs a few things: a name, which LangChain class to use, the actual model identifier, your API key, and some generation parameters like temperature and max tokens.

Let's get your agent to help you write a solid config.

Ask your agent
Ask your agent to generate a `config.yaml` model entry for your preferred LLM provider
Think about it
  • Which model provider do you actually have an API key for?
  • What parameters matter for an agent that needs to reason carefully vs. one that needs to be creative?
  • Why would you use an environment variable reference like `$OPENAI_API_KEY` instead of pasting the key directly?
What the agent gives back

The agent should give you a YAML snippet — around 8 lines — defining a model with name, display_name, use (the LangChain class path), model, api_key (referencing an env var), max_tokens, and temperature. It should explain that $OPENAI_API_KEY syntax means DeerFlow reads from your environment, keeping secrets out of the config file. If you ask for Anthropic, the use field changes to langchain_anthropic:ChatAnthropic.

Tip
Temperature tip: for agent tasks, stick with 0.3–0.7. Too low and the agent becomes rigid. Too high and it starts hallucinating tool calls. The default 0.7 is a solid starting point.
Warning
Never paste raw API keys into config.yaml if you might push this to git. Use the .env file or environment variables — DeerFlow's $VAR_NAME syntax makes this painless.
Can I use multiple models at once?

Yes! This is one of DeerFlow's superpowers. You can define multiple models in your config.yaml and assign them to different roles. The models field is an array, so you can list as many as you want.

Why would you? Cost optimization. Your research sub-agent might just need to summarize web pages — a smaller, cheaper model handles that fine. But your coding sub-agent needs to write correct, complex code — you want your best model there.

This is like hiring a team where you don't pay senior engineer rates for someone doing data entry. Right tool for the right job.

Fire It Up — Docker vs. Local Dev

DeerFlow gives you two ways to run: Docker (recommended) or local development. Honestly, Docker is the way to go here — not just because it's easier, but because DeerFlow's sandbox feature uses Docker containers to isolate code execution. So you kinda need Docker anyway.

Here's the mental model: when DeerFlow's coding sub-agent writes and runs code, you probably don't want that code running directly on your machine. What if the agent decides to rm -rf something? The sandbox runs that code inside a Docker container — a little sealed-off mini-computer that can't touch your real files. It's like giving a toddler finger paint but putting newspaper on the floor first.

The Docker setup has two commands: one to pull the sandbox image (one-time thing), and one to start everything.

Deep dive: How sandbox modes work

DeerFlow supports multiple sandbox execution modes, and this is worth understanding because it's core to how the agent safely runs code:

Local Execution — Code runs directly on your host machine. Fast, but dangerous. Only use this for trusted tasks or development.

Docker Execution — Code runs in isolated Docker containers. Each execution gets a fresh container, so nothing persists or leaks. This is the default and recommended mode.

Provisioner Mode — A more advanced setup where a separate provisioner service manages container lifecycle. This is for production-scale deployments where you need better resource management.

The sandbox isn't just about safety — it also gives the agent a file system to work with. The agent can write files, run scripts, install packages — all inside the container. When it's done, results get extracted back out. It's like giving the agent its own temporary laptop to work on.

Understand the Architecture — Sub-Agents, Skills, and Memory

Okay, DeerFlow is running. Let's slow down and understand what's actually happening under the hood, because this is the really interesting part.

DeerFlow's architecture has five core pillars:

1. Sub-Agents — Instead of one mega-agent trying to do everything, DeerFlow breaks tasks into pieces and hands them to specialized sub-agents. One researches, one codes, one writes. The main orchestrator decides who gets what.

2. Skills & Tools — Skills are like plugins the agents can use. A skill might be "search the web," "run Python code," or "generate an image." The extensible skills system means you can add new capabilities without rewriting agent logic.

3. Sandbox & File System — When an agent needs to run code, it gets a sandboxed environment. This is huge for safety and reproducibility.

4. Context Engineering — This is DeerFlow's fancy term for managing what information each sub-agent sees. Not every agent needs every piece of context. Good context engineering = efficient token usage = cheaper and faster.

5. Long-Term Memory — Agents can remember things across sessions. Ask it to research something today, and it can recall that research tomorrow.

The orchestrator tracks subtask status and chains outputs — the researcher's findings become the coder's input. Think of it like a project manager breaking down a sprint into tickets and assigning them to the right developer.

Tip
DeerFlow also supports MCP (Model Context Protocol) servers — this means you can extend it with any MCP-compatible tool. If you've been building MCP tools for Claude Desktop or other clients, they plug right in.
What is 'context engineering' and why should I care?

Context engineering is arguably the most underrated concept in agent design. Here's the problem: LLMs have context windows (the amount of text they can process at once). If you dump everything into every agent's context, you waste tokens, slow things down, and actually hurt performance — the model gets confused by irrelevant information.

DeerFlow's context engineering means each sub-agent only sees what it needs. The researcher sees the original query and search results. The coder sees the researcher's summary and a task description, but not the raw search results. The writer sees the coder's output and the original brief.

It's like a relay race — each runner only carries the baton, not the entire team's equipment. This makes each agent faster, cheaper, and more focused.

This matters to you because as these systems get more complex, naive 'throw everything in the prompt' approaches will hit walls. Context engineering is how you scale.

How does long-term memory work?

Long-term memory in DeerFlow means agents can persist information across sessions. Without it, every conversation starts from zero — like talking to someone with amnesia.

The memory system typically works by:

  1. Extracting key facts and decisions from a session
  2. Storing them in a structured way (usually a vector database or similar)
  3. Retrieving relevant memories when a new session starts or a related topic comes up

So if you ask DeerFlow to research competitor pricing on Monday, then on Wednesday ask it to 'update the pricing comparison,' it can recall the Monday research instead of starting over. This is especially powerful for ongoing projects where context builds over time.

Think of it like a team wiki that the agents maintain and consult automatically.

Give It a Real Task — Watch the Orchestra Play

Now for the fun part. Let's actually use DeerFlow and watch how it decomposes a task and coordinates sub-agents.

A good test task is something that requires multiple skills: research, reasoning, and producing an artifact. Something like "Research the top 5 open-source agent frameworks released in 2026, compare their architectures, and generate a summary table."

This forces DeerFlow to:

  • Use its research sub-agent to search the web
  • Use its analysis capabilities to compare findings
  • Use its sandbox to potentially run code for generating the table
  • Use context engineering to pass the right info between steps

Watch the UI at localhost:2026 — you'll see the task decomposition in real time.

  • What task would test the handoff between a research sub-agent and a coding sub-agent?
  • Think about tasks that would be genuinely hard for a single agent but natural for a team...
What the agent gives back

Your agent should suggest diverse tasks like: (1) a research + visualization task (e.g., 'research Y Combinator's latest batch and create a chart of company categories'), (2) a pure coding task that requires iteration (e.g., 'build a Python script that scrapes and summarizes the top HN posts'), and (3) a multi-step creative task (e.g., 'research how RAG works, then write a beginner-friendly tutorial with code examples'). The key insight is that good test tasks cross skill boundaries — they're not something a single tool call can handle.

Tip
Start with a simpler task first to make sure everything is wired up correctly. "Summarize the top 3 stories on Hacker News right now" is a good smoke test — it exercises web search and summarization without needing the sandbox.
What to watch for in the DeerFlow UI

When you give DeerFlow a task, watch the UI carefully. You should see:

  1. Task decomposition — The orchestrator breaks your request into subtasks. This is the "planning" phase.
  2. Sub-agent assignment — Each subtask gets routed to a specialist. You'll see which agent is handling what.
  3. Tool/skill usage — Watch for web searches, code execution, file creation. Each one is a skill being invoked.
  4. Handoffs — The output of one sub-agent becomes the input for another. This is the context engineering in action.
  5. Final synthesis — The orchestrator collects all sub-agent outputs and assembles the final response.

If something goes wrong, it's usually at the handoff point — a sub-agent produced output that the next agent couldn't use well. That's the hardest part of multi-agent orchestration to get right.

Recap

We went from zero to running DeerFlow 2.0 — ByteDance's trending super agent framework. But more importantly, we built a mental model for how modern multi-agent systems work.

The key insight isn't the specific tool — it's the pattern: instead of building one massive agent that tries to do everything, you build an orchestrator that routes tasks to specialized sub-agents, each with their own skills and sandboxed execution environment. Memory ties sessions together. Context engineering keeps each agent focused and efficient.

This is where agent development is heading. Not bigger models, but smarter orchestration.

  • DeerFlow 2.0 is a super agent harness — an orchestrator that coordinates specialized sub-agents, not a single monolithic agent
  • The five pillars are sub-agents, skills/tools, sandboxed execution, context engineering, and long-term memory — and they work together as a system
  • Sandbox execution isn't optional for agents that write code — it's a safety requirement. Docker containers give agents a disposable workspace
  • LangChain class paths in config.yaml mean you can swap LLM providers without touching agent code — configure, don't hardcode
  • Context engineering (what each agent sees) is as important as the model choice — focused agents outperform overloaded ones
