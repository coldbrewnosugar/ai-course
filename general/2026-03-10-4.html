Build Multi-Tool AI Agents with the Strands Agents SDK

Go from zero to a working agentic RAG pipeline in Python using Strands' model-driven framework
35 min agents sdk rag python tools

What's happening

AWS open-sourced Strands Agents in mid-2025, and the project has since climbed steadily through GitHub's trending charts. The SDK represents a deliberate philosophical departure from the graph-and-chain paradigm popularised by LangChain and similar frameworks. Where those tools ask the developer to define explicit control flow — retrieval node, then summarisation node, then output node — Strands delegates orchestration entirely to the language model. The developer supplies tools and a system prompt; the model decides which tools to invoke, in what sequence, and when to stop.

This architectural choice has significant implications. It reduces framework-level boilerplate, makes agents easier to prototype, and — critically — allows the same agent definition to exhibit different behaviours depending on the query. A single agent equipped with retrieval, web search, and calculator tools might use all three for one question and only one for the next. The trade-off is that debugging becomes less deterministic: you must reason about model behaviour rather than tracing a fixed graph.

In this session, we will use an AI coding agent to construct a series of increasingly capable Strands agents, culminating in an agentic RAG pipeline. Each step focuses on the prompting decisions required to guide an AI agent toward correct, well-structured output — not on writing code by hand.

1

Understand Model-Driven Architecture and Build a Minimal Agent

Before constructing anything, we must establish a clear mental model of how Strands agents operate.

In a graph-driven framework, the developer is an architect drawing a blueprint: each room (node) has a defined purpose, and hallways (edges) connect them in a fixed layout. The model fills each room with content but never rearranges the floor plan.

In Strands' model-driven approach, the developer is more like a hiring manager. You define a role (system prompt), provide a toolkit (available tools), and assign a task (user message). The agent — powered by the underlying language model — decides how to accomplish the task. It may use every tool available, or none at all. The SDK's event loop simply executes whatever the model requests and feeds results back, continuing until the model signals completion.

The core abstraction is the Agent class, which accepts three primary inputs: a model provider (Bedrock, OpenAI, Anthropic, Ollama, or others), a system prompt, and a list of tools. The agent is callable — you invoke it with a string, and it returns the model's final response after any tool-use cycles have resolved.

Our first task is to have our AI coding agent generate a minimal Strands agent that demonstrates this loop.

Ask your agent
Get your AI agent to produce a minimal Strands agent that uses a single custom tool, so you can observe the model-driven tool-use loop in action.
Think about it
  • What are the three components every Strands agent requires — and how should each be configured for a simple demonstration?
  • Strands uses a `@tool` decorator that introspects function signatures and docstrings. What does the decorator need from your function to generate a correct tool schema?
  • How will you know the tool was actually invoked rather than the model answering from its own knowledge? Consider choosing a tool that returns information the model could not fabricate.
  • What model provider should you specify, and what happens if you omit it entirely?
What the agent gives back

The agent should produce a short Python script — no more than 15–20 lines total — that installs the SDK, defines a single custom tool using the @tool decorator (for example, a function returning the current system time or a random number), and creates an Agent instance that uses it. The key structural insight to verify: the tool function must have a Google-style docstring with parameter descriptions, since the decorator parses this to build the JSON schema the model receives. If no model provider is specified, Strands defaults to Amazon Bedrock with Claude.

```python

from strands import Agent, tool

@tool

def current_time() -> str:

"""Return the current system time."""

from datetime import datetime

return datetime.now().isoformat()

agent = Agent(tools=[current_time])

print(agent("What time is it right now?"))

```

API Key Note
Strands defaults to Amazon Bedrock, which requires AWS credentials configured via aws configure or environment variables. To use OpenAI or Anthropic directly, install the corresponding extra (pip install strands-agents[openai]) and pass an explicit model provider to the Agent constructor.
Tip
The @tool decorator is not merely cosmetic. It inspects your function's type hints, argument names, and docstring to auto-generate the JSON schema that the language model uses to decide when and how to call your tool. Omitting the docstring or type hints produces a degraded schema that the model may misinterpret.
How does the event loop actually work?

When you call agent("What time is it?"), the SDK sends the conversation history (including the system prompt and your message) plus all tool schemas to the model provider. The model returns a response that may contain tool_use blocks — structured JSON objects naming a tool and providing arguments. The SDK intercepts these blocks, executes the corresponding Python functions, wraps the return values in tool_result messages, and appends them to the conversation. It then sends the updated conversation back to the model. This cycle repeats until the model returns a response with no tool_use blocks — a stop signal — at which point the SDK returns the final text to the caller. The entire orchestration logic lives in approximately 50 lines of framework code; the intelligence is in the model.

Equip the Agent with Built-in Tools from Strands Tools

A custom @tool function is useful when you need domain-specific logic, but Strands ships a companion package — strands-agents-tools — containing a curated library of pre-built tools for common capabilities: file I/O, shell command execution, HTTP requests, a Python REPL, a calculator, and notably a retrieve tool that connects to Amazon Bedrock Knowledge Bases.

These tools follow the same interface as custom tools and can be mixed freely with them. Importing a tool from strands_tools gives you a callable that is already decorated and schema-ready.

The architectural significance is composability. Because the model — not the developer — decides which tools to invoke, adding a new tool to the list immediately expands the agent's capabilities without requiring any changes to control flow logic. The model simply sees an additional entry in its tool catalogue and can choose to use it when appropriate.

In this step, we will have our AI agent construct a multi-tool agent that combines several built-in tools to act as a general-purpose assistant.

What the agent gives back

The agent should produce a Python script that imports four or five built-in tools from strands_tools, constructs an Agent with a system prompt that instructs the model to prefer tool-verified answers over parametric guesses, and demonstrates the agent handling a query that requires multiple tools. The system prompt is the most important piece — it should explicitly instruct the agent to use the calculator for arithmetic, http_request for live data, and to cite which tools produced which parts of its answer. The code itself remains under 15 lines; the intellectual work is in the prompt design.

```python

from strands import Agent

from strands_tools import calculator, http_request, file_read, python_repl

agent = Agent(

tools=[calculator, http_request, file_read, python_repl],

system_prompt="You are a research assistant. Always use tools to verify facts. Use calculator for math, http_request for live data. State which tool produced each finding.",

)

```

Warning
The shell and python_repl tools execute arbitrary code on the host machine. In any deployment beyond local experimentation, these tools require sandboxing — containerised execution, restrictive file system permissions, or network isolation. The Strands samples repository explicitly states that its examples are not production-ready.
Why does the system prompt matter so much in a model-driven architecture?

In a graph-driven framework, control flow is encoded in the graph itself: the developer decides when retrieval happens, when summarisation happens, and when output is produced. In Strands, the system prompt is the only mechanism through which the developer influences the model's orchestration decisions. A vague prompt like 'You are helpful' gives the model no guidance on when to use tools versus relying on parametric knowledge. A well-crafted prompt — specifying tool preferences, verification requirements, and output format — effectively becomes the control flow specification. This is why prompt engineering is not a supplementary skill in the model-driven paradigm; it is the primary form of programming.

What tools are available in strands-agents-tools?

The package includes: calculator (arithmetic), file_read / file_write (filesystem access), shell (command execution), http_request (HTTP client), python_repl (code execution), retrieve (Amazon Bedrock Knowledge Base retrieval), current_time (timestamps), memory / load_memory (persistent key-value storage across sessions), editor (code editing), journal (structured note-taking), swarm (multi-agent swarm coordination), and agent_graph (graph-based multi-agent orchestration). Each is a drop-in tool that requires no configuration beyond import — though some (like retrieve) expect environment variables or AWS credentials.

Connect External Capabilities via MCP Servers

The Model Context Protocol (MCP) is an open standard for connecting language models to external tool servers. Strands implements MCP through its MCPClient class, which supports two transport mechanisms: stdio (launching a subprocess that communicates via standard input/output) and SSE (connecting to a remote HTTP server via Server-Sent Events).

The integration pattern is straightforward: you create an MCPClient, call list_tools() to discover what the server offers, and pass those tools to the Agent alongside any local tools. From the model's perspective, MCP tools are indistinguishable from local @tool functions — they appear as entries in the same tool catalogue with the same JSON schema format.

This design has a powerful implication: you can extend an agent's capabilities without writing any new tool code. If someone publishes an MCP server for a database, an API, or a documentation corpus, you simply point your agent at it and the model gains access to those capabilities immediately.

The MCPClient is used as a context manager (with block) to ensure proper lifecycle management of the subprocess or HTTP connection.

Ask your agent

Think about it
  • The `MCPClient` requires either a `command` and `args` (for stdio transport) or a `url` (for SSE transport). Which transport would you use for a locally installed MCP server package versus a remote service?
  • MCP tools must be discovered at runtime via `list_tools()`. How does this affect the structure of your code — specifically, why is a context manager necessary?
  • When combining MCP tools with local tools, you pass both to the same `tools` list. How would you instruct the model (via the system prompt) to prefer one source over another?
  • What happens if the MCP server is unavailable at runtime? Consider how your prompt should address error handling.
What the agent gives back

The agent should produce a script that creates an MCPClient using stdio transport (launching an MCP server as a subprocess), discovers its tools, and passes them alongside local built-in tools to a Strands Agent. The structural point to verify: the Agent construction and invocation must occur inside the with mcp_client: block, because MCP tools are only available while the client connection is active. The system prompt should reference both tool categories and provide guidance on when to prefer each.

```python

from strands import Agent

from strands.tools.mcp import MCPClient

from strands_tools import calculator

mcp = MCPClient(command="uvx", args=["some-mcp-server@latest"])

with mcp:

agent = Agent(

tools=[calculator, *mcp.list_tools()],

system_prompt="Use MCP tools for domain queries; use calculator for arithmetic.",

)

print(agent("Your query here"))

```

Tip
The spread operator pattern *mcp.list_tools() unpacks MCP tools into the same list as local tools. This is idiomatic Strands — the Agent treats all tools uniformly regardless of origin. You can combine tools from multiple MCP servers and multiple local definitions in a single agent.
What MCP servers are available?

The MCP ecosystem is growing rapidly. AWS publishes several MCP servers for their documentation and services (e.g., awslabs.aws-documentation-mcp-server). Community-maintained servers exist for databases (PostgreSQL, SQLite), file systems, web search APIs, and various SaaS platforms. The Strands samples repository demonstrates integration with several of these. Because MCP is an open protocol, any server conforming to the specification works with Strands' MCPClient without modification.

Build an Agentic RAG Pipeline

Retrieval-Augmented Generation (RAG) in its basic form follows a fixed two-step pattern: retrieve relevant documents, then generate an answer conditioned on those documents. Agentic RAG elevates this by letting the model control the retrieval process — deciding when to retrieve, what queries to use, whether to retrieve again with a refined query, and when it has gathered sufficient context to produce a final answer.

Strands is particularly well-suited to agentic RAG because the model-driven event loop naturally supports iterative, multi-step retrieval without requiring the developer to encode retrieval logic in a graph. The model simply calls the retrieval tool as many times as it deems necessary.

The retrieve tool from strands-agents-tools connects to Amazon Bedrock Knowledge Bases, but the same pattern applies with any retrieval tool — a custom @tool function that queries Elasticsearch, Pinecone, pgvector, or any other vector store. The model does not know or care about the underlying retrieval engine; it sees a tool that accepts a query string and returns relevant text passages.

The system prompt is critical in agentic RAG. It must instruct the model to: (1) reformulate user questions into effective retrieval queries, (2) evaluate whether retrieved context is sufficient, (3) perform follow-up retrievals with refined queries when initial results are inadequate, and (4) cite sources in its final answer.

  • The system prompt must encode the retrieval strategy. What specific instructions distinguish agentic RAG from naive single-shot retrieval? Think about query reformulation, sufficiency evaluation, and iterative refinement.
  • Should the agent have access to tools beyond retrieval? Consider what complementary tools (web search, calculator, code execution) might improve answer quality for different query types.
  • How would you instruct the model to handle conflicting information from multiple retrieval passes? The system prompt needs explicit guidance here.
  • What does 'citing sources' mean in the context of tool-returned content? How should the model attribute information to specific retrieval results?
What the agent gives back

The agent should produce a Strands agent definition centered on a carefully crafted system prompt that encodes the full agentic RAG strategy. The prompt should instruct the model to: decompose complex questions into sub-queries, perform initial retrieval, assess whether results adequately address the question, issue refined follow-up queries if gaps remain, and synthesise a final answer with source attribution. The code itself is minimal — perhaps 10 lines — because the intelligence lives entirely in the system prompt and tool selection.

The system prompt is the primary deliverable. It should contain explicit instructions like: 'When you receive a question, first formulate 2-3 retrieval queries that capture different aspects of the question. After each retrieval, assess whether the returned passages contain sufficient information. If not, reformulate your query using more specific terms or alternative phrasings and retrieve again. Synthesise your final answer from all retrieved passages and cite which retrieval produced each claim.'

Optionally, the agent may include complementary tools (calculator for quantitative questions, http_request for supplementary web data) to handle queries that require more than document retrieval alone.

Tip
The quality of agentic RAG depends almost entirely on the system prompt, not the retrieval engine. A mediocre vector store with a well-prompted iterative retrieval agent frequently outperforms a superior vector store with naive single-shot RAG. The model's ability to reformulate queries and evaluate retrieval quality is the differentiating factor.
Warning
Iterative retrieval increases both latency and token consumption. Each retrieval cycle requires a round trip to the model (to evaluate results and decide next steps) plus a round trip to the retrieval backend. Set expectations accordingly — an agentic RAG response may take 10–30 seconds versus 2–5 seconds for single-shot RAG.
How does agentic RAG compare to other advanced RAG patterns?

Agentic RAG is one of several techniques that improve upon naive RAG. Query expansion (generating multiple search queries from a single question) can be implemented within the system prompt by instructing the model to formulate diverse queries before its first retrieval call. Re-ranking (scoring retrieved passages for relevance before generation) can be approximated by asking the model to evaluate passage relevance in a separate tool call. Hypothetical Document Embeddings (HyDE) — where the model generates an ideal answer first and uses it as a retrieval query — is another strategy easily encoded in the system prompt. The model-driven architecture makes all of these patterns accessible through prompt engineering alone, without requiring framework-level changes.

What if I'm not using Amazon Bedrock Knowledge Bases?

The retrieve tool from strands-agents-tools is Bedrock-specific, but the agentic RAG pattern works with any retrieval backend. Define a custom @tool function that accepts a query string, calls your vector store (Pinecone, Weaviate, pgvector, Elasticsearch, ChromaDB, or any other), and returns the top-k passages as a formatted string. The model interacts with it identically — it has no awareness of the underlying implementation. You could also expose a retrieval endpoint via an MCP server and connect it through MCPClient.

✓
You now have an agentic RAG pipeline — a Strands agent that iteratively retrieves, evaluates, and synthesises information guided by a carefully designed system prompt. The retrieval strategy lives in the prompt, not in framework code. This is the central design principle of model-driven agents.

Coordinate Multiple Agents for Complex Workflows

Some tasks exceed the scope of a single agent — not because one agent lacks tool access, but because distinct phases of a workflow benefit from different system prompts, tool sets, or even model providers. Strands supports several multi-agent coordination patterns.

The most straightforward is the agent-as-a-tool pattern: wrap one Agent inside a @tool function and pass that tool to another Agent. The outer (coordinator) agent invokes the inner (specialist) agent as if it were any other tool. This is hierarchical delegation — the coordinator decides when to call the specialist, and the specialist operates independently within its own system prompt and tool context.

Strands also provides built-in tools for more dynamic coordination: the swarm tool enables an orchestrator agent to spawn and manage worker agents at runtime, and agent_graph supports graph-based multi-agent topologies where agents are nodes connected by defined communication edges.

The key architectural question in multi-agent design is when to split. A single well-prompted agent with many tools is simpler to build, debug, and maintain. Multi-agent patterns introduce coordination overhead and potential for miscommunication. The split is justified when specialist agents need fundamentally different instructions (a code reviewer should not share a system prompt with a code writer) or when different phases require different model providers (a fast, cheap model for classification feeding into a capable model for generation).

Ask your agent
Get your AI agent to produce a multi-agent system using the agent-as-a-tool pattern — a coordinator agent that delegates research and analysis tasks to specialist sub-agents.
Think about it
  • Each specialist agent needs its own system prompt optimised for its role. How should a 'researcher' prompt differ from an 'analyst' prompt in terms of tool preferences and output format?
  • When wrapping an agent as a `@tool`, the tool function must extract the text content from the agent's response object. What does the response structure look like, and how do you access the final text?
  • The coordinator agent needs a system prompt that understands what its sub-agent tools do. How should you describe the specialist capabilities so the coordinator delegates effectively?
  • What are the failure modes? Consider what happens if a specialist agent produces an error or irrelevant output — how should the coordinator's prompt handle this?
What the agent gives back

The agent should produce a multi-agent architecture with two or three specialist agents (e.g., a researcher with web/retrieval tools and an analyst with calculator/Python REPL tools), each wrapped in a @tool function, and a coordinator agent whose tool list consists of these wrapped specialists. The coordinator's system prompt should describe each specialist's capabilities and specify when to delegate versus answer directly. The response extraction pattern — accessing result.message['content'][0]['text'] from the inner agent — should be clearly present in each wrapper function.

The conceptual output is a three-tier system: user → coordinator agent → specialist agents → tools. The coordinator never directly accesses low-level tools; it operates entirely through delegation.

Tip
Each agent in a multi-agent system maintains its own conversation history. The specialist agents do not share context with each other or with the coordinator beyond what is passed through tool arguments and return values. Design your tool wrapper functions to pass sufficient context and return structured, complete responses.
When should you use multi-agent vs. single-agent?

Prefer a single agent when: (1) all tasks share the same domain context and instructions, (2) the tool set is manageable (under 10–15 tools), and (3) the workflow does not require fundamentally different model behaviours at different stages. Split into multiple agents when: (1) specialists need contradictory instructions (a devil's advocate agent should not share a system prompt with an advocate agent), (2) you want to use different model providers for cost or capability optimisation, (3) the total tool set exceeds what the model can effectively select from (tool confusion increases with catalogue size), or (4) you need isolation — a code execution agent should not have access to your production database tools.

Recap

This session traced a path from a single-tool agent to a multi-agent agentic RAG pipeline, all built through the Strands Agents SDK's model-driven architecture. The recurring theme: in a model-driven framework, the system prompt is the program. Where graph-based frameworks encode orchestration logic in developer-defined control flow, Strands places that responsibility entirely with the language model, guided by the prompt and constrained by the available tools.

We built each component by prompting an AI coding agent — formulating clear objectives, specifying architectural constraints, and evaluating the output against our understanding of Strands' abstractions. This workflow — goal definition, prompt formulation, output validation, iterative refinement — is the transferable skill underlying every step.

  • Model-driven agents delegate orchestration to the language model: the developer supplies tools and a system prompt, not a control flow graph. This reduces boilerplate but shifts debugging from graph tracing to prompt analysis.
  • The system prompt is the primary programming interface in Strands. A well-crafted prompt encoding retrieval strategy, tool preferences, and output format is more impactful than any framework-level configuration.
  • Agentic RAG — where the model controls iterative retrieval, query reformulation, and sufficiency evaluation — is naturally expressed in Strands because the event loop already supports multi-step tool invocation without developer-defined sequencing.
  • MCP integration allows capability extension without new code: point the agent at an MCP server and the model gains access to its tools immediately, alongside any local tools.
  • Multi-agent coordination (via agent-as-a-tool or the built-in swarm/graph tools) is warranted when specialists need different prompts, tools, or model providers — not merely because a task has multiple steps.
