Build a Local Voice Pipeline with MLX-Audio on Apple Silicon

Ship a TTS + STT app that runs entirely on your Mac — no API keys needed
35 min tts stt apple-silicon mlx local-inference voice-cloning

What's happening

The economics of cloud-based speech APIs are straightforward: you pay per character synthesized and per second transcribed, your audio traverses the public internet, and your application breaks whenever the provider's endpoint is unavailable. For prototyping, experimentation, and privacy-sensitive workloads, this trade-off is increasingly difficult to justify — particularly when the hardware sitting on your desk is capable of running these models locally.

MLX-Audio is a recently trending open-source library that brings text-to-speech (TTS), speech-to-text (STT), and speech-to-speech (STS) capabilities to Apple Silicon through Apple's MLX framework. It supports multiple model architectures — Kokoro for fast multilingual synthesis, Dia for dialogue, Chatterbox for expressive speech, Whisper for transcription — and exposes an OpenAI-compatible REST API, meaning existing applications that call the OpenAI speech endpoints can be redirected to your local machine with a single URL change.

The library also supports model quantization (3-bit through 8-bit), voice cloning from short audio samples, and a Gradio-based web interface. In this session, we will install the library, generate speech across multiple models, transcribe audio with Whisper, and stand up a local API server — constructing a complete voice-in/voice-out pipeline with zero cloud dependencies.

1

Set Up the MLX-Audio Environment

Before generating any audio, we need a clean Python environment with MLX-Audio installed and verified. This step is foundational: MLX-Audio depends on Apple's MLX framework, which requires macOS on Apple Silicon (M1 or later). The library is pip-installable, but there are subtle choices around installation method — pip for project integration, uv tool install for CLI-only usage, or a development clone for the web interface.

The distinction matters. A uv tool install gives you the command-line tools in an isolated environment without polluting your project's dependencies. A pip install into a virtual environment is appropriate when you intend to use the Python API programmatically. The development install adds the Gradio web UI and additional tooling.

Why MLX instead of PyTorch or ONNX?

Apple's MLX framework is designed specifically for the unified memory architecture of Apple Silicon. Unlike PyTorch (which uses MPS as a backend translation layer) or ONNX Runtime (which treats the GPU as a separate device), MLX treats CPU and GPU memory as a single shared pool. This eliminates the copy overhead that plagues other frameworks on Apple hardware. For audio workloads — where models are moderate in size but inference is latency-sensitive — this architectural advantage translates directly into lower time-to-first-audio. MLX also supports lazy evaluation and composable function transformations, making it well-suited for the kind of streaming generation that speech synthesis requires.

2

Generate Speech with Kokoro and Explore Voice Options

With the environment established, we now move to the core capability: text-to-speech synthesis. MLX-Audio supports multiple TTS model architectures, each with different trade-offs. We will begin with Kokoro (mlx-community/Kokoro-82M-bf16), an 82-million-parameter model that is fast, lightweight, and supports eight languages.

Kokoro is a good starting point for three reasons. First, at 82M parameters in bf16, it downloads quickly and runs with minimal memory overhead. Second, it supports multiple voices (such as af_heart, af_bella, am_michael) and adjustable speech speed, giving us immediate control surfaces to experiment with. Third, it accepts a lang_code parameter that selects language-specific phoneme handling — a for American English, b for British English, j for Japanese, and so on.

The mental model here is important: TTS models do not simply read text aloud. They convert text to an intermediate representation (phonemes or tokens), predict acoustic features (mel spectrograms or audio codecs), and then synthesize a waveform. The voice parameter selects a speaker embedding that conditions this entire pipeline, which is why the same text with different voices produces not just a different pitch, but different rhythm, emphasis, and prosody.

Understanding voice codes and language codes

Kokoro's voice identifiers follow a convention: the first letter indicates gender (a for female, b for male in some conventions), and the remainder is a descriptive name. For example, af_heart is a female voice with a warm, expressive quality, while am_michael is a male voice. The lang_code parameter is separate from the voice — it controls phoneme conversion rules. Setting lang_code='a' uses American English phonemization regardless of the voice. This separation means you can, in principle, use an English voice with Japanese phonemization, though results will vary. The full list of available voices is model-specific and can be inspected in the model's Hugging Face repository.

How does Kokoro compare to cloud TTS?

Kokoro at 82M parameters is significantly smaller than cloud TTS systems (which typically range from 500M to several billion parameters). The quality gap is noticeable in long-form narration: Kokoro may exhibit less natural prosody over paragraphs and occasionally mispronounce uncommon words. However, for short-form generation — UI feedback, notifications, brief narration — the quality is entirely adequate, and the latency advantage of local inference (no network round-trip) can make the application feel more responsive than a cloud alternative.

Transcribe Audio with Whisper on MLX

A voice pipeline requires both directions: speech out (TTS) and speech in (STT). MLX-Audio includes speech-to-text support through Whisper, OpenAI's robust transcription model, optimized to run on Apple Silicon via MLX.

The Whisper model available through MLX-Audio is mlx-community/whisper-large-v3-turbo-asr-fp16 — a tuned variant of Whisper Large V3 that balances accuracy and speed. Whisper supports over 99 languages and can perform both transcription (speech in language X → text in language X) and translation (speech in any language → text in English).

This step closes the loop: we will transcribe the audio file we generated in Step 2, verifying that the TTS output is intelligible to a STT model. This round-trip test is a practical validation technique — if the STT model accurately transcribes the TTS output, we have confidence that both halves of our pipeline are functioning correctly.

MLX-Audio also supports Qwen3-ASR (mlx-community/Qwen3-ASR-1.7B-8bit) as an alternative STT backend, which uses an 8-bit quantized model for lower memory usage, and Parakeet from NVIDIA for high-accuracy English transcription.

Why round-trip testing matters

Round-trip testing — generating speech from text and then transcribing it back — is a surprisingly effective integration test for voice pipelines. It validates that: (1) the TTS model produces phonetically intelligible audio, (2) the audio encoding and file format are correct, (3) the STT model can handle the synthetic voice without degradation. In production pipelines, monitoring the round-trip word error rate over time can surface model drift or configuration regressions before they affect users. This technique is borrowed from signal processing, where loopback tests are standard practice for verifying communication channels.

Stand Up the Local OpenAI-Compatible API Server

The practical value of a local voice pipeline increases dramatically when it is accessible as a network service rather than a standalone script. MLX-Audio ships with an OpenAI-compatible REST API server that implements the same endpoint contracts as OpenAI's speech and transcription APIs. This means any application, library, or tool that speaks the OpenAI protocol can be redirected to your local machine with a single configuration change.

This is architecturally significant. The OpenAI speech API has become a de facto standard interface for TTS and STT — dozens of libraries and frameworks integrate with it. By implementing this interface locally, MLX-Audio transforms your Mac into a drop-in replacement for the cloud service, with the benefits of zero latency, no usage costs, and complete data privacy.

The server supports model selection, voice configuration, speed adjustment, and streaming responses. It can serve multiple concurrent requests and automatically manages model loading and caching.

Tip

API Key Note
No API key is required for the local server. The OpenAI SDK mandates an api_key parameter, but the local server ignores it. You can pass any non-empty string.
Streaming vs. non-streaming responses

The OpenAI TTS API supports streaming responses, where audio bytes are sent as they are generated rather than waiting for the full synthesis to complete. This is critical for real-time applications — a voice assistant, for example, should begin speaking before the entire response is synthesized. MLX-Audio's server supports this streaming mode. When using the OpenAI SDK, set response_format='pcm' for raw streaming audio or response_format='mp3' for compressed output. The trade-off is that streaming PCM has lower latency but higher bandwidth, while MP3 adds encoding latency but reduces data size. For local use (where bandwidth is not a constraint), PCM streaming typically provides the best user experience.

If this produces a playable WAV file, your pipeline is fully operational.

Optimize with Quantization and Explore Larger Models

The models we have used so far — Kokoro at 82M parameters, Whisper Large V3 Turbo — are moderate in size. But MLX-Audio also supports substantially larger models: Chatterbox (1.6B parameters, 16 languages, expressive prosody), Dia (1.6B parameters, dialogue-focused with speaker turn handling), and Qwen3-TTS (1.7B parameters, multilingual with voice design capabilities). These models produce higher-quality speech but require more memory.

This is where quantization becomes essential. Quantization reduces the precision of model weights from 16-bit floating point to lower bit widths (8-bit, 6-bit, 4-bit, or even 3-bit), reducing memory footprint proportionally. A 1.6B parameter model in bf16 requires approximately 3.2 GB of memory; in 4-bit quantization, this drops to roughly 800 MB — comfortably fitting on an 8 GB machine alongside the operating system.

MLX-Audio supports quantized model variants published by the mlx-community on Hugging Face. You select a quantized model simply by using the appropriate model identifier — no additional configuration is needed. The trade-off is straightforward: lower bit width means smaller memory footprint and faster inference, at the cost of some audio quality degradation. In practice, 8-bit and 6-bit quantization are perceptually transparent for most use cases; 4-bit introduces subtle artifacts; 3-bit is suitable only for experimentation.

Warning

How quantization works in MLX

MLX implements quantization through weight-only compression: the model weights are stored and loaded at the reduced precision, but computation is performed by dequantizing to a higher precision on the fly. This approach preserves the mathematical structure of the forward pass while reducing memory bandwidth requirements — which is often the true bottleneck on Apple Silicon, where the Neural Engine and GPU share memory bandwidth with the CPU. The mlx-community organization on Hugging Face publishes pre-quantized checkpoints using MLX's quantization tools, so users download already-compressed weights rather than performing quantization themselves. This is important because quantization is a lossy process that benefits from calibration data — the pre-quantized models have been calibrated by the community for optimal quality at each bit width.

Model selection guide

Kokoro (82M): Best for low-latency, low-memory scenarios. Eight languages. Suitable for notifications, short responses, and resource-constrained devices.

Chatterbox (1.6B): Best for expressive, natural-sounding speech across 16 languages. Supports voice cloning. Higher quality but significantly more memory and compute.

Dia (1.6B): Optimized for dialogue — handles speaker turns and conversational rhythm. Best for chatbot and assistant applications where the speech needs to sound like one side of a conversation.

Qwen3-TTS (1.7B): Alibaba's multilingual model with voice design capabilities — you can describe the voice characteristics you want in natural language. Unique among the supported models for this feature.

Spark (0.5B): Mid-range option supporting English and Chinese. Good balance of quality and resource usage.

CSM (1B): Conversational Speech Model with voice cloning from short audio samples. English only.

Recap

In this session, we constructed a complete local voice pipeline on Apple Silicon using MLX-Audio — from environment setup through TTS generation, STT transcription, API server deployment, and model optimization via quantization.

The central insight is architectural: by running speech models locally through an OpenAI-compatible API, we achieve a setup that is simultaneously private (no audio leaves the device), free (no per-character or per-second billing), and interoperable (any OpenAI-compatible client works without modification). The mlx-community ecosystem on Hugging Face provides a growing catalog of pre-optimized and pre-quantized models, making it straightforward to trade quality against resource usage for any given deployment target.

This pipeline is not merely a development convenience — it represents a meaningful shift in how voice-enabled applications can be prototyped and, for many use cases, deployed. The combination of Apple Silicon's unified memory architecture and MLX's framework-level optimizations makes local inference practical in a way that was not feasible even twelve months ago.

  • MLX-Audio provides TTS, STT, and STS through a unified library with multiple model architectures, each optimized for different use cases — Kokoro for speed, Chatterbox for expressiveness, Dia for dialogue, Whisper for transcription.
  • The OpenAI-compatible API server is the highest-leverage feature for integration: it transforms any application that uses the OpenAI speech endpoints into a locally-served application with a single URL change.
  • Quantization (8-bit through 3-bit) enables larger, higher-quality models to run on memory-constrained hardware, with pre-quantized checkpoints available from the mlx-community on Hugging Face.
  • Round-trip testing (TTS → STT → compare) is a practical validation technique for voice pipelines that catches integration issues across the full stack.
  • Voice cloning from short audio samples is supported by several models (CSM, Chatterbox, Spark) and opens applications in accessibility and personalization, with important ethical considerations.
