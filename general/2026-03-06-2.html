Train Your Own Reasoning Agent with AReaL's Async RL System

Go from zero to a custom RL-trained agent by swapping a single base_url
40 min reinforcement-learning agentic-ai open-source reasoning LLM-training

What's happening

Why This Matters Now

On March 2, 2026, the AReaL team—a collaboration between Tsinghua IIIS and Ant Group—released a complete recipe for training agentic models using reinforcement learning with a remarkably simple interface: swap a base_url and an api_key, and your existing agent runtime becomes an RL training environment.

This is significant for three reasons:

  1. RL for reasoning is no longer exotic infrastructure. AReaL-lite reduces the codebase by 80% while retaining 90% of the full system's performance, making RL-based agent training accessible to individual researchers and small teams.
  1. Asynchronous RL delivers real speedups. The boba² release demonstrated a 2.77× training speedup over synchronous approaches—a critical efficiency gain when training runs consume thousands of GPU-hours.
  1. The base_url pattern changes who can do this. By exposing the RL training loop as an OpenAI-compatible API endpoint, AReaL decouples the training system from the agent runtime. Any framework that can call an LLM via HTTP can now serve as an RL environment—no framework lock-in, no custom integration code.

The project has already produced state-of-the-art results: their AReaL-SEA model (235B MoE) surpasses GPT-5 and matches Gemini 3.0 Pro on the τ²-bench benchmark. In this session, we will understand the architecture, set up an AReaL-lite training loop, and replicate the OpenClaw agent recipe.

1

Understand the Architecture: Synchronous vs. Asynchronous RL

Before touching any configuration, we need a clear mental model of what AReaL does differently.

The Synchronous RL Bottleneck

In conventional RL training for LLMs, the pipeline operates in lockstep: generate a batch of rollouts (the model produces outputs for a set of prompts), compute rewards, update the model weights, then repeat. Each phase must complete before the next begins. GPUs sit idle during reward computation; reward servers sit idle during generation.

AReaL's Asynchronous Approach

AReaL decouples these phases entirely. Generation, reward computation, and weight updates run concurrently on separate worker pools. As soon as a rollout completes, it enters a replay buffer; the trainer consumes from this buffer continuously. The key insight is that slightly stale rollouts (generated by a model a few gradient steps behind the current one) still provide useful training signal—and the throughput gain far outweighs the staleness cost.

Think of it as the difference between a restaurant where every table orders, eats, and pays before the next table is seated (synchronous) versus a normal restaurant where the kitchen, servers, and cashier all operate in parallel (asynchronous).

The base_url Abstraction

For agentic RL specifically, AReaL introduces an elegant abstraction. The agent runtime—whatever framework runs your agent's tool-use loop—simply calls an LLM endpoint. AReaL replaces that endpoint with its own RL-managed model server. The agent runtime becomes the environment; each episode of tool use becomes a rollout. Rewards flow back through the same API boundary. No modifications to the agent code are required.

What the agent gives back

The agent should produce a concise technical comparison covering three points: (1) synchronous RL serializes generation → reward → update, causing each component to idle while the others work; (2) asynchronous RL pipelines these stages, allowing generation workers to produce new rollouts while the trainer processes previous ones; (3) staleness—the gap between the model version that generated a rollout and the model version that trains on it—introduces a small bias but is empirically manageable, and the throughput gain (AReaL reports 2.77×) more than compensates. The agent may reference importance sampling or off-policy corrections as mitigation strategies.

Tip
The 2.77× speedup figure comes from AReaL's boba² release (v0.3, June 2025). This was measured against synchronous baselines on equivalent hardware. The speedup scales further with larger model sizes, where the ratio of generation time to training time increases.
How does AReaL handle staleness technically?

AReaL uses a bounded staleness model. Each rollout is tagged with the model version (parameter checkpoint) that generated it. The trainer can be configured to reject rollouts older than a threshold number of gradient steps. In practice, the team found that allowing 2–4 steps of staleness has negligible impact on final model quality while maintaining full pipeline throughput. For agentic RL, where each rollout involves multi-turn tool interactions and is therefore slower, the asynchronous design is especially beneficial—the trainer never waits for a long-running agent episode to complete.

Set Up AReaL-lite for Rapid Prototyping

AReaL-lite is the lightweight entry point to the system, designed for researchers and rapid prototyping. It retains the core asynchronous training loop in 80% fewer lines of code than the full AReaL system.

What AReaL-lite Provides

  • An algorithm-first API that exposes RL primitives (rollout collection, reward computation, policy update) as composable Python objects.
  • Native support for fully asynchronous agentic RL without requiring distributed systems expertise.
  • Compatibility with standard model formats (HuggingFace Transformers) and standard serving frameworks (SGLang, vLLM).

What You Need

The minimum requirements are a machine with at least one GPU (the system scales to multi-node, but a single-GPU setup suffices for AReaL-lite), Python 3.10+, and a base model in HuggingFace format. For this workshop, we will use a 7B parameter model—small enough to train on a single node but large enough to exhibit meaningful reasoning behavior.

The agent should provide a short setup sequence: clone the repository, install dependencies (the project uses a standard pip install -e . pattern with optional extras for lite mode), and run a minimal smoke test. The key detail is that AReaL-lite lives within the same repository but is invoked through a separate entry point. The agent should mention checking GPU availability via a brief Python import check and confirm that the areal package is importable. Expect 4–6 shell commands total, not a lengthy script.

Warning
AReaL requires PyTorch with CUDA support. CPU-only installations will install correctly but cannot run training. Verify your CUDA setup before proceeding: python -c "import torch; print(torch.cuda.is_available())".
Tip
If you lack local GPU access, the conceptual understanding from this session transfers directly. The configuration files and prompting patterns we discuss are hardware-independent—you can prepare everything locally and execute on a cloud GPU instance later.
AReaL-lite vs. full AReaL: when to use which

AReaL-lite is appropriate when you are developing a new reward function, experimenting with a novel agent environment, or training models up to approximately 14B parameters on a single node. The full AReaL system adds multi-node orchestration, advanced fault tolerance, and optimized communication patterns necessary for models at the 32B+ scale or for production training runs. The algorithm-level behavior is identical—AReaL-lite is not a simplified algorithm, it is a simplified deployment and API surface.

3

Configure and Launch a Reasoning RL Training Loop

With AReaL-lite installed, we now configure a training loop for a reasoning task. The core configuration specifies four things:

  1. The base model — a HuggingFace model identifier or local path.
  2. The dataset — a collection of prompts with verifiable answers (math problems, coding challenges, or structured QA).
  3. The reward function — how to score the model's outputs. For reasoning tasks, this is typically outcome-based: did the model arrive at the correct answer?
  4. The training hyperparameters — learning rate, batch size, number of rollouts per update, and the staleness bound.

Why Outcome-Based Rewards Work for Reasoning

A common question is why we can use simple outcome verification (correct/incorrect) rather than dense, step-by-step reward shaping. The answer lies in the chain-of-thought structure: the model generates an extended reasoning trace before producing a final answer. RL with outcome-based rewards incentivizes the model to develop whatever internal reasoning strategies maximize final-answer accuracy. The model discovers useful intermediate steps on its own—this is precisely the phenomenon that makes RL-trained reasoners interesting.

The Configuration File

AReaL-lite uses a declarative configuration file (YAML or Python dataclass) that specifies the components above. The key design choice is that the reward function is a Python callable—you can implement any scoring logic you need.

What the agent gives back

The agent should produce a configuration object (either YAML or a Python dataclass instantiation) with the following elements: a model reference (e.g., a 7B model from HuggingFace), a dataset reference pointing to a math reasoning dataset, a reward function that extracts the final numerical answer from the model's output and compares it to the ground truth (returning 1.0 for correct, 0.0 for incorrect), and training hyperparameters including a conservative learning rate (around 1e-6), a rollout batch size, generation temperature (typically 0.7–1.0 for exploration), and max generation length. The configuration should be concise—roughly 10–15 lines at most.

Tip
Temperature during rollout generation is a critical hyperparameter. Too low (< 0.5) and the model explores too little—it reinforces its existing behavior. Too high (> 1.2) and the outputs become incoherent, yielding sparse reward signal. The range 0.7–1.0 is a well-established starting point for reasoning tasks.
Why not use process reward models (PRMs)?

Process reward models score individual reasoning steps rather than just the final answer. They can accelerate training by providing denser signal, but they introduce a significant dependency: you need a reliable PRM, which itself requires training data of annotated reasoning steps. For initial experiments, outcome-based rewards are simpler and sufficient. AReaL supports both approaches—you can upgrade to a PRM-based reward function later by changing the reward callable in the configuration without modifying any other component.

4

Replicate the OpenClaw Agent Recipe via base_url Swap

This is the centerpiece of the session: using AReaL to train an agentic model by pointing an existing agent runtime at AReaL's model server.

The OpenClaw Pattern

OpenClaw is a lobster-themed agent example (hence the 🦞) that demonstrates AReaL's agentic RL capability. The pattern is straightforward:

  1. You have an agent runtime—any system where an LLM interacts with tools over multiple turns. This could be a LangChain pipeline, an OpenAI function-calling loop, a custom orchestrator, or any framework that calls an LLM via an HTTP API.
  1. AReaL starts a model server that exposes an OpenAI-compatible /v1/chat/completions endpoint. Instead of serving a static model, this endpoint serves the currently training model—the weights update continuously as RL progresses.
  1. You replace the base_url and api_key in your agent runtime's LLM client to point at AReaL's server. The agent runtime now generates rollouts using the training model.
  1. A reward function evaluates completed episodes. The reward, the prompt, and the model's outputs flow back to AReaL's trainer.

The elegance of this design is that your agent code does not change at all. The LLM client still makes the same API calls. AReaL interposes itself as the model provider and collects the training data it needs from the interaction.

What Changes in Configuration

Compared to the reasoning task in Step 3, the agentic configuration adds two elements: (1) the agent runtime command—how to start your agent environment, and (2) a multi-turn rollout specification, since agentic tasks involve multiple LLM calls per episode rather than a single generation.

Second, the agent runtime's LLM client is reconfigured to point at this endpoint—typically by changing base_url to the AReaL server's address and api_key to the AReaL-issued key. The agent should clarify that no agent code changes are needed beyond these two values. It should explain that episode boundaries and rewards are communicated through AReaL's reward callback mechanism, and that the asynchronous design naturally handles variable-length episodes since rollouts enter the training buffer as they complete.

API Key Note
The api_key used with AReaL's model server is not a cloud API key—it is a locally generated token used for authentication between the agent runtime and the AReaL server running on the same machine or cluster. No external API charges are incurred; inference runs on your own GPUs.
Tip
The base_url swap pattern means you can develop and test your agent with a commercial API (OpenAI, Anthropic) for rapid iteration, then switch to AReaL for RL training. Your agent code remains identical across both phases.
What makes this different from fine-tuning on traces?

A common alternative is to collect successful agent traces and fine-tune a model on them (behavioral cloning). This approach is simpler but fundamentally limited: the model can only learn to imitate the traces you collected. RL, by contrast, allows the model to explore novel strategies and discover behaviors that no existing trace demonstrates. In the AReaL paper, RL-trained agents consistently outperform models fine-tuned on the same tasks' demonstration data, particularly on out-of-distribution scenarios where the agent must generalize beyond observed patterns.

Evaluate Training Progress and Know When to Stop

RL training for LLMs does not converge in the same predictable way as supervised learning. The reward curve is noisier, and there are failure modes specific to RL that require monitoring.

Key Metrics to Track

  • Mean episode reward — The primary signal. For outcome-based rewards, this is the fraction of problems the model solves correctly. Expect a slow initial climb followed by faster improvement as the model discovers effective reasoning strategies.
  • Reward variance — High variance indicates the model is inconsistent. Some variance is expected (and even desirable for exploration), but if variance increases while mean reward plateaus, the model may be oscillating rather than improving.
  • KL divergence from the base model — RL training can cause the model to drift far from its pre-trained distribution, degrading general capabilities. AReaL includes a KL penalty term to control this. If KL divergence grows too rapidly, increase the KL penalty coefficient.
  • Response length — A known failure mode in RL for reasoning is reward hacking through verbosity: the model learns to produce extremely long outputs that happen to contain the correct answer somewhere. Monitor mean response length and consider a length penalty if it grows unboundedly.

When to Stop

Unlike supervised learning, there is no clean loss convergence. Common stopping criteria include: mean reward has plateaued for N updates, KL divergence has reached a predetermined budget, or a held-out evaluation set shows no improvement across successive checkpoints.

What the agent gives back

Key items include: mean reward (should increase monotonically on a smoothed curve), reward variance (should decrease as training stabilizes), KL divergence (should grow slowly and remain bounded), mean response length (should be stable or grow modestly, not explode), and rollout throughput (should remain stable—a drop indicates infrastructure issues). The agent should specifically flag verbosity hacking and reward gaming as failure modes to watch for.

Warning
Do not evaluate solely on the training reward. Always maintain a held-out evaluation set that the reward function never sees during training. RL models can overfit to the reward function's specific patterns without developing generalizable reasoning.
The KL divergence tradeoff in depth

KL divergence from the base model acts as a regularizer: it prevents the RL-trained model from collapsing into a narrow behavioral mode that maximizes reward at the expense of general language capability. However, setting the KL penalty too high constrains the model from learning new behaviors at all. The AReaL team's recommended approach is to start with a moderate KL coefficient and adjust based on the reward-KL Pareto frontier: plot mean reward against KL divergence across checkpoints, and select the checkpoint that achieves the best reward for an acceptable level of divergence. This is analogous to the bias-variance tradeoff in supervised learning, where the KL penalty controls the 'bias' toward the original model's behavior.

Recap

What We Covered

This session traced a path from the conceptual foundations of asynchronous RL for LLMs through to the practical setup of an RL training loop using AReaL-lite.

We began with the architectural insight that motivates AReaL: by decoupling rollout generation, reward computation, and model updates into concurrent processes, asynchronous RL achieves a 2.77× throughput improvement over synchronous approaches with negligible impact on training quality.

We then examined AReaL-lite as a prototyping entry point, configured a reasoning RL training loop with outcome-based rewards, and—critically—replicated the OpenClaw agent pattern, where any agent runtime becomes an RL environment through a simple base_url swap. This pattern is the key practical takeaway: it means RL-based agent training is no longer a monolithic, framework-specific endeavor.

Finally, we addressed the evaluation challenge: monitoring reward curves, KL divergence, and response characteristics to detect failure modes and determine when training has yielded a useful model.

The Broader Significance

AReaL represents a maturation point for RL-based LLM training. The combination of open-source infrastructure, a simple integration pattern, and demonstrated state-of-the-art results lowers the barrier from 'requires a dedicated ML infrastructure team' to 'requires a GPU and a clear training objective.' The question is no longer whether RL can improve reasoning agents, but which tasks justify the training investment over prompt engineering alone.

  • Asynchronous RL decouples generation, reward, and training into concurrent pipelines, delivering substantial speedups (2.77×) with bounded staleness as the only tradeoff.
  • The base_url swap pattern is AReaL's key ergonomic innovation: any agent that calls an OpenAI-compatible API can become an RL training environment without code changes.
  • Outcome-based rewards (did the agent reach the correct final answer?) are sufficient to train strong reasoning and agentic behaviors—dense step-level rewards are an optimization, not a prerequisite.
  • RL training requires active monitoring of reward, KL divergence, and response characteristics. Unlike supervised learning, there is no single loss curve that tells you when to stop.
  • AReaL-lite provides 90% of the full system's capability with 80% less code, making it suitable for individual researchers and algorithm development.
