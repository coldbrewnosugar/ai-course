Give Your AI Agents a Safe Place to Run Code

Set up Alibaba's OpenSandbox to spin up isolated containers where agents can execute code without blowing up your machine
35 min sandboxing agents code-execution docker kubernetes

What's happening

Here's the problem every agent builder eventually hits: your AI can write code, but where does it run it?

Letting an LLM execute arbitrary code on your actual machine is... let's just say it's a speedrun to a very bad day. You need a sandbox — an isolated throwaway environment where code can run, fail, even do weird stuff, and your host stays perfectly safe.

Alibaba just open-sourced OpenSandbox, a platform designed exactly for this. It's not the first sandbox tool out there (you might've seen E2B or Code Interpreter API), but it brings some interesting things to the table:

  • Multi-language SDKs — Python, TypeScript, Java, C#, Go (coming soon). Not just a Python-only story.
  • Docker AND Kubernetes runtimes — run it locally on your laptop, or scale to hundreds of concurrent sandboxes on a cluster.
  • A unified "Sandbox Protocol" — lifecycle management, file operations, command execution, and a full Code Interpreter, all through one consistent API.
  • Built-in environments for coding agents, browser automation (Playwright, Chrome), and even desktop environments with VNC.

Today we're going to get it running locally with Docker, spin up a Code Interpreter sandbox, execute some Python code through the SDK, and understand the mental model well enough that you could wire this into any agent system you're building. Let's get into it.

1

Get the Sandbox Server Running Locally

Before we can run any code in a sandbox, we need the server — the thing that manages sandbox lifecycles, talks to Docker, and exposes the API that our SDK calls will hit.

Think of it like this: OpenSandbox has a client-server architecture. The server is the bouncer at the club — it creates containers, watches them, tears them down when they're done. The SDK is you outside the club, asking the bouncer to let your code in.

The server runs locally and talks to your Docker daemon. So step one: make sure Docker is running on your machine. (If you're on Mac or Windows, Docker Desktop needs to be open. On Linux, the Docker service should be active.)

Now, instead of memorizing install commands, let's have our agent walk us through it.

The agent should walk you through three commands: installing the server package with uv pip install opensandbox-server, generating a config file with opensandbox-server init-config ~/.sandbox.toml --example docker, and then starting the server with opensandbox-server. It should mention that Docker must be running first, and that the config file at ~/.sandbox.toml tells the server to use Docker as its runtime. Warning
Docker must be running before you start the sandbox server. On Mac, make sure Docker Desktop is open (check the whale icon in your menu bar). The server talks directly to the Docker daemon — no daemon, no sandboxes.
Tip
The uv tool is a fast Python package manager from Astral (the Ruff folks). If you don't have it, pip install uv or curl -LsSf https://astral.sh/uv/install.sh | sh. It's dramatically faster than plain pip.
What's actually in that config file?

The ~/.sandbox.toml config tells the server which container runtime to use. The --example docker flag generates a config pre-filled for local Docker. If you were deploying to production, you'd swap this for a Kubernetes config instead — same server, different runtime backend.

This is a nice design choice: your SDK code doesn't change at all between local dev (Docker) and production (Kubernetes). The server handles that abstraction for you.

Why not just use Docker directly?

You could just docker run containers yourself and exec into them. But OpenSandbox gives you: automatic cleanup (sandboxes have timeouts and get killed), a file API (read/write files without docker cp), structured output from code execution (not just raw stdout), and the Code Interpreter layer which handles things like preserving state between code executions. It's the difference between raw sockets and an HTTP library — same underlying tech, much nicer interface.

2

Understand the Sandbox Lifecycle

Before we write any SDK code, let's build a mental model of how sandboxes work. This is the part that honestly confused me at first, because there are a few layers.

Imagine a sandbox as a rented apartment:

  1. Create — You sign the lease. A fresh Docker container spins up from an image. It's empty, isolated, yours.
  2. Use — You move in. You can run shell commands, read/write files, execute code. The apartment is fully furnished (the container image has Python, system tools, etc.).
  3. Kill — Lease is up. The container is destroyed. Everything inside it vanishes. No cleanup needed.

The key insight: sandboxes are ephemeral by design. They're meant to be created, used for a task, and thrown away. Your agent might spin up dozens during a single session — one per code execution attempt, or one per user request. They're cheap to create (a few seconds) and completely isolated from each other.

Now there's a layer on top of the raw sandbox: the Code Interpreter. Think of it as a tenant who already lives in the apartment. The sandbox gives you a container with shell access. The Code Interpreter gives you a running Python/JS/etc. process inside that container, with a REPL-like interface where you can send code and get structured results back (stdout, stderr, return values, even plots).

Tip

Why async? Can't I just use regular Python?

OpenSandbox's Python SDK is async-first because sandbox operations involve network calls to the server and waiting for containers to start. In a real agent scenario, you might be managing multiple sandboxes concurrently (say, one per user), and async lets you do that efficiently without threading headaches. If you're not comfortable with async Python, the good news is the pattern is always the same: async def main(), await the SDK calls, asyncio.run(main()) at the bottom.

What's that entrypoint parameter doing?

The entrypoint=['/opt/opensandbox/code-interpreter.sh'] tells Docker what process to start inside the container. For the code interpreter image, this launches the interpreter service that listens for code execution requests. Without it, the container would start but wouldn't know how to handle Code Interpreter SDK calls. Think of it as the difference between starting a computer (sandbox) and launching an application on it (code interpreter).

✓
At this point you should have: (1) the OpenSandbox server running locally with Docker, and (2) a mental model of the sandbox lifecycle — create, use, kill. The server is your control plane, and sandboxes are ephemeral containers you spin up and throw away. If the server starts without errors and you can see it listening, you're good to go.
3

Wire Up the Code Interpreter

Now we get to the fun part. The raw sandbox gives you shell commands and file operations — useful, but kind of like having a terminal. The Code Interpreter layer gives you something much richer: a persistent Python (or JavaScript, or other language) session where you can send code, get structured results, and even handle things like plots and data frames.

This is exactly what tools like ChatGPT's Code Interpreter or Claude's analysis tool are doing under the hood — running your code in an isolated environment and streaming back the results.

Here's the mental model: the Code Interpreter is a REPL session living inside your sandbox. It remembers variables between executions. If you run x = 42 and then later run print(x), it prints 42. State persists within a session, but vanishes when the sandbox dies.

The SDK for this is a separate package: opensandbox-code-interpreter. It wraps the sandbox and gives you a clean interpreter.codes.run() method.

What the agent gives back

The agent should give you a script that first creates a sandbox (same as before), then creates a Code Interpreter with CodeInterpreter.create(sandbox), then runs Python code using interpreter.codes.run('your code here', language=SupportedLanguage.PYTHON). The result comes back as a structured object — result.result[0].text gives you the return value of the last expression, and result.logs.stdout[0].text gives you anything that was printed. The key insight: the interpreter separates return values from printed output, which is way cleaner than just capturing stdout like a shell command would.

Tip
The Code Interpreter's result vs logs distinction is huge for agents. An agent can check the return value programmatically (was it an error? a number? a dataframe summary?) without having to parse messy stdout strings. This is what makes it better than just running python -c 'your code' in a shell.
Warning
Don't forget to install both packages: opensandbox (for sandbox management) AND opensandbox-code-interpreter (for the interpreter layer). They're separate because you might want sandboxes without the interpreter — for example, if you're doing browser automation or running shell scripts.
How does state persistence work across runs?

When you call interpreter.codes.run() multiple times, each execution happens in the same Python process. Variables, imports, and function definitions from earlier runs are still available. This is crucial for agent workflows where the AI might say "let me try a different approach" — it can build on what it already set up.

But here's the gotcha: if the sandbox dies (timeout, crash, explicit kill), everything is gone. State persists within a sandbox's lifetime, not across sandboxes. If you need durable state, write it to a file and read it back in a new sandbox.

What languages does the Code Interpreter support?

The SupportedLanguage enum includes Python and JavaScript at minimum. The image opensandbox/code-interpreter:v1.0.1 comes pre-configured with Python (you can even set the version via the PYTHON_VERSION environment variable). Language support depends on the container image — the interpreter itself is language-agnostic, but the runtime needs the right toolchain installed. You could build a custom image with R, Julia, or anything else.

4

File Operations — Reading, Writing, and Moving Data

Real agent workflows don't just run code in a vacuum. They need to get data into the sandbox, process it, and get results out. Maybe your agent needs to analyze a CSV, modify a config file, or generate a report.

OpenSandbox gives you a file API that works like a simple cloud storage interface:

  • Write files into the sandbox (upload data for processing)
  • Read files from the sandbox (download results)
  • Think of it as FTP for containers, but way simpler

This is the glue between your agent's world (where it has user data, API responses, etc.) and the sandbox's world (where code actually runs). Without this, the sandbox would be isolated in a bad way — safe, but useless.

The pattern is always: write input files → run code that processes them → read output files.

Think about it

  • If you write a CSV into the sandbox, how would the Code Interpreter access it?
  • After the code generates output, how do you get that output file back to your host?
  • Think about file permissions — what mode should sandbox files have?
What the agent gives back

The agent should show you two key operations: writing files with sandbox.files.write_files() which takes a list of WriteEntry objects (each with a path, data, and mode like 644), and reading with sandbox.files.read_file('/path/to/file'). A practical example would be writing a CSV into the sandbox, running Python code in the interpreter that reads and processes it (using pandas or even just csv module), writing results to a new file, then reading that file back. The key insight is that the sandbox filesystem is completely separate from your host — files only get in or out through the SDK's file API.

Tip
For larger files, think about whether you actually need to transfer the file out, or if you can just have the Code Interpreter compute a summary/result and return it directly via stdout. Transferring a 100MB dataset out of the sandbox is slower than having the interpreter compute the answer inside.
How does this compare to Docker volumes?

You might be thinking "why not just mount a Docker volume?" You could, but the file API is better for agent scenarios because:

  1. Security — Mounted volumes give the sandbox access to your host filesystem. The file API keeps the boundary strict — you explicitly choose what goes in and comes out.
  2. Multi-tenant — If you're running sandboxes for multiple users, volumes are a shared-state nightmare. The API keeps each sandbox's files isolated.
  3. Auditing — Every file transfer goes through the API, so you can log exactly what data entered and left the sandbox.

For local dev it doesn't matter much, but in production this distinction is critical.

5

Connecting Sandboxes to Your Agent Loop

Alright, let's zoom out. We've been looking at OpenSandbox in isolation, but the whole point is to wire this into an agent system. Let's talk about how that actually works.

The typical agent-with-code-execution loop looks like this:

  1. User asks the agent to do something that requires code (analyze data, build a chart, debug a script)
  2. LLM decides it needs to execute code and generates a code string
  3. Your system takes that code string and runs it in a sandbox
  4. Results (stdout, errors, return values) go back to the LLM
  5. LLM interprets the results, either responding to the user or trying again

OpenSandbox slots into step 3. It's the execution layer. The LLM never talks to the sandbox directly — your orchestration code does.

If you're using a framework like LangChain, CrewAI, or building a custom agent, you'd register the sandbox executor as a tool — something the LLM can choose to invoke. The tool description might say: "Execute Python code in an isolated sandbox. Input: code string. Output: stdout, stderr, and return value."

The response sent back to the LLM should include stdout, the return value, and any errors — enough for the LLM to understand what happened and iterate.

Tip
When sending sandbox results back to the LLM, truncate long outputs! If the code prints 10,000 lines, sending all of that back to the model wastes tokens and confuses it. Send the first ~50 lines of stdout and the last ~20, with a note about truncation.
Warning
OpenSandbox has egress controls (per-sandbox network policies) and an ingress gateway. In production, ALWAYS restrict network access for sandboxes running untrusted code. A malicious code snippet could exfiltrate data, mine crypto, or attack other services on your network.
What about other sandbox types beyond Code Interpreter?

OpenSandbox isn't just for running Python. The platform supports:

  • Browser environments — Sandboxes with Chrome and Playwright pre-installed, perfect for GUI agents that need to navigate websites
  • Desktop environments — Full desktop with VNC access, for agents that need to interact with GUI applications
  • Custom environments — Build your own Docker image with whatever tools you need

The Code Interpreter is the most common use case, but if you're building something like a web scraping agent or a QA testing agent, those other environments are gold. Same SDK, same lifecycle, different container image.

Docker vs Kubernetes — when do you need K8s?

Docker is perfect for local development and small-scale deployment. But if you need to run many sandboxes concurrently (like serving multiple users), Docker on a single machine hits limits fast.

OpenSandbox's Kubernetes runtime lets you schedule sandboxes across a cluster, with proper resource limits, auto-scaling, and isolation. The documentation mentions a 'high-performance Kubernetes runtime' — this is for scenarios like RL training where you might spin up hundreds of sandboxes simultaneously.

The beautiful part: your SDK code is identical. You just change the server's config file from Docker to Kubernetes. The SDK doesn't know or care which runtime is underneath.

Recap

Let's step back and look at what we covered today.

We started with a real problem: AI agents can write code, but they need somewhere safe to run it. OpenSandbox gives us that — a server that manages isolated Docker containers, an SDK that makes working with them feel almost trivial, and a Code Interpreter layer that turns those containers into full REPL environments.

The architecture is clean: server (manages containers) → sandbox (isolated environment) → Code Interpreter (structured code execution). Files go in and out through an explicit API. Sandboxes are ephemeral — create them, use them, throw them away.

The real power isn't any single feature — it's that this gives you a standard, reusable execution layer for any agent system. Whether you're building a coding assistant, a data analysis agent, or an evaluation harness, the pattern is the same: create sandbox, run code, get results, iterate.

And because it runs on Docker locally and Kubernetes in production (with the same SDK), you can prototype on your laptop and deploy without rewriting anything. That's a genuinely nice developer experience.

  • Sandboxes are ephemeral containers — create, use, destroy. They're cheap and isolated, perfect for untrusted code execution.
  • The Code Interpreter layer gives you structured results (return values + stdout + stderr), not just raw terminal output. This is what makes it agent-friendly.
  • The SDK is async-first and uses context managers for lifecycle management — async with sandbox: ensures cleanup even if things crash.
  • File operations are explicit (write in, read out) — the sandbox filesystem is completely isolated from your host, which is a security feature, not a limitation.
  • Same SDK code works on Docker (local dev) and Kubernetes (production). The runtime is a server config, not a code change.
