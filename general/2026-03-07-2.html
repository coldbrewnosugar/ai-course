Build Interactive Replays of AI Agent Sessions with claude-replay

Every Claude Code session produces a rich transcript — prompts, tool calls, thinking blocks, timestamps. claude-replay converts those logs into self-contained, interactive HTML replays you can share, embed, and learn from.
30 min developer-tools claude-code session-replay documentation cli

What's happening

The Problem with Sharing Agent Work

AI-assisted development sessions are inherently temporal. A developer prompts an agent, the agent reasons through a problem, invokes tools, produces code, encounters errors, and iterates — sometimes across dozens of exchanges spanning an hour or more. This process is where the real value lies: not just in the final artifact, but in the reasoning trajectory that produced it.

Yet sharing this trajectory has been remarkably difficult. The prevailing methods — terminal screenshots, screen recordings, or raw log dumps — each lose something essential. Screenshots strip away sequence and context. Screen recordings are bulky, unsearchable, and unnavigable. Raw JSONL logs are machine-readable but impenetrable to human readers.

claude-replay, released on March 6, 2026, addresses this gap directly. It is a CLI tool that reads the JSONL session transcripts that Claude Code already stores locally (in ~/.claude/projects/) and produces a single, self-contained HTML file — an interactive replay with playback controls, collapsible tool calls, expandable thinking blocks, and chapter bookmarks. No external dependencies. No hosting infrastructure required. Email the file, drop it into a blog post, or embed it via iframe.

The tool is community-built and open-source. In this session, we will install it, understand the JSONL log format it consumes, generate replays with various customization options, and explore practical embedding strategies for documentation and technical writing.

1

Understand the JSONL Session Log Format

Before generating replays, we need to understand what we are working with. Claude Code stores a complete transcript of every session as a JSONL file — one JSON object per line, each representing a single conversational event.

These files reside in ~/.claude/projects/, organized by project path. Each JSONL file corresponds to one session and contains a sequence of records that collectively describe the entire agent interaction: user prompts, assistant responses, tool invocations (with both the call and the result), and — when available — the agent's internal thinking blocks.

Each record carries a timestamp, a role identifier (user, assistant, or tool), and a content payload. Tool call records are particularly rich: they include the tool name, input parameters, and the result returned. This is the raw material that claude-replay transforms into a navigable visual timeline.

The key insight is that you already have this data. Every Claude Code session you have ever run has produced these logs. The question is simply how to surface them in a useful form.

Warning
Session logs may contain sensitive information — API keys, environment variables, file contents, and internal reasoning. The claude-replay tool includes automatic secret redaction by default, but you should always review replays before sharing them publicly.
What makes JSONL particularly well-suited for session logs?

JSONL (JSON Lines) stores one complete JSON object per line, which makes it ideal for append-only logging. Each conversational turn can be written to disk immediately without needing to maintain a valid enclosing structure (as standard JSON would require with its array brackets). This means logs survive crashes gracefully — you never lose previously written records. It also enables efficient streaming reads: tools like claude-replay can process the file line by line without loading the entire transcript into memory, which matters for long sessions that may contain hundreds of exchanges.

Install claude-replay and Generate Your First Replay

With an understanding of the source data, we can now install the tool and produce our first replay. claude-replay is distributed as an npm package and can be used either as a global installation or invoked directly via npx for zero-install usage.

The fundamental operation is straightforward: point the tool at a JSONL session file, specify an output path, and receive a self-contained HTML file. This file embeds the entire session data (compressed by default), the player interface, all styling, and all JavaScript — no CDN links, no external fonts, no network requests at runtime. The result is a single artifact that functions identically whether opened from a local filesystem or served from a web server.

API Key Note
claude-replay requires Node.js (version 18 or later). If you do not have Node.js installed, your agent can guide you through installation via your platform's package manager (Homebrew on macOS, apt on Ubuntu, etc.).
Why does self-contained output matter so much?

A single-file HTML replay with no external dependencies is a remarkably portable artifact. It can be attached to a GitHub issue, sent via email, embedded in a static site, or archived indefinitely without worrying about CDN availability, version drift, or broken links. This is the same philosophy behind tools like single-file HTML exports from Jupyter notebooks: the goal is to produce an artifact whose fidelity is guaranteed by its self-containment. For claude-replay specifically, the session data is embedded as a compressed base64 blob, which the player decompresses at load time. This keeps file sizes reasonable even for long sessions.

3

Customize Replays with Filters, Bookmarks, and Themes

A raw replay of a full session can be long and unfocused. Real sessions include false starts, tangential explorations, and repetitive debugging cycles. For a replay intended for an audience — a blog post, a demo, a teaching resource — you need editorial control.

claude-replay provides three primary mechanisms for shaping your output:

Turn and time filters let you extract a contiguous slice of the session. The --turns flag accepts a range (e.g., 5-15), while --from and --to accept ISO 8601 timestamps. These are essential for isolating the interesting portion of a longer session.

Bookmarks add labeled chapter markers at specific turns, giving viewers a table of contents they can click to jump directly to key moments. You can specify them inline with --mark "N:Label" or provide a JSON file.

Themes control the visual presentation. The default tokyo-night theme suits dark backgrounds, but several alternatives are available (dracula, among others), and you can supply a custom theme JSON file for full control over colors and typography.

What the agent gives back

The agent should construct a command that combines multiple flags, such as: claude-replay session.jsonl --turns 12-28 --mark "12:Problem Setup" --mark "18:Root Cause Found" --mark "25:Fix Verified" --theme dracula -o polished-replay.html. It should explain that --list-themes displays the available built-in themes. It should also note that bookmarks appear as clickable chapter markers in the player's timeline, enabling viewers to jump directly to labeled moments rather than scrubbing through the entire replay.

Tip
For teaching or documentation replays, consider using --no-thinking to hide the agent's internal reasoning blocks by default. Viewers can still expand them on demand, but the initial view stays focused on the visible conversation flow. Conversely, for sessions where the reasoning is the point, leave thinking blocks visible.
Bookmarks as a JSON file — when and why?

For replays with many bookmarks, inline --mark flags become unwieldy. The --bookmarks flag accepts a JSON file containing an array of objects, each with a turn (integer) and label (string) field. This is particularly useful when you are scripting replay generation — for instance, if you are producing a series of annotated replays from multiple sessions for a documentation site. The bookmark file can be version-controlled alongside your documentation source, and generated programmatically if you build tooling that identifies interesting turns automatically (e.g., turns where a test suite transitions from failing to passing).

Embed Replays in Documentation, Blogs, and Bug Reports

Generating a replay file is only half the value. The other half lies in putting that replay where people will encounter it in context — inside a blog post, a documentation page, a GitHub issue, or a pull request description.

Because the output is a single HTML file, embedding strategies are straightforward:

Direct hosting: Place the HTML file alongside your other static assets. Link to it directly, or open it in a new tab.

Iframe embedding: Use an <iframe> tag to embed the replay inline within a larger page. This is the approach best suited for blog posts and documentation, where the replay should appear as one component among prose, diagrams, and code samples.

GitHub and issue trackers: Attach the HTML file to a GitHub issue or pull request. Reviewers can download and open it locally. For bug reports, a replay that shows the exact sequence of agent actions leading to a failure is far more informative than a textual description.

The key consideration for iframe embedding is sizing. The replay player is responsive, but you should set a reasonable height (600–800px is typical) and allow full width.

It should note that the sandbox attribute should allow scripts (allow-scripts) since the player requires JavaScript, but can restrict other capabilities. The agent may also suggest wrapping the iframe in a responsive container div with max-width and margin: auto for centered presentation.

Tip
For blog posts, consider generating a focused replay of just the most illustrative 5–10 turns rather than the full session. Readers are more likely to engage with a concise, well-bookmarked replay than a 50-turn marathon. Use --turns and --mark to curate the experience.
Using replays for code review and bug reports

One of the most practical applications of claude-replay is in code review workflows. When a developer uses an AI agent to implement a feature, the replay provides reviewers with the full context of how the code was produced — what the developer asked for, what the agent tried, where it made mistakes, and how it corrected them. This context is invaluable for evaluating whether the resulting code reflects genuine understanding or a lucky guess. Similarly, for bug reports, a replay can show the exact sequence of tool invocations that led to an unexpected state, eliminating the ambiguity inherent in natural-language descriptions of 'what happened.'

Automate Replay Generation for Teams and CI Pipelines

For individual use, running claude-replay manually after a notable session is sufficient. But in team environments — where multiple developers use AI agents daily and documentation or knowledge sharing is a priority — manual generation becomes a bottleneck.

The tool's CLI interface is designed for scriptability. Every option is available as a flag, output can be piped to stdout, and the bookmarks file format enables programmatic annotation. This makes claude-replay suitable for integration into automated workflows.

Consider several scenarios:

Post-session hooks: A shell alias or wrapper script that automatically generates a replay after each Claude Code session, storing it alongside the project's documentation.

CI integration: A pipeline step that generates replays from sessions used during development and attaches them to pull requests as artifacts, giving reviewers access to the full reasoning context.

Batch generation: A script that iterates over all session logs in a project directory, generates replays with standardized bookmarks and themes, and publishes them to an internal documentation site.

The common thread is that the CLI's composability — flags for filtering, annotation, theming, and output control — maps naturally onto the needs of automation.

What the agent gives back

The agent should outline a short shell script (described in prose, not as a full listing) that identifies the most recently modified JSONL file in the project's session directory, generates a replay with team-standard defaults (a chosen theme, a descriptive title derived from the project and date, thinking blocks hidden by default), and saves the output to a designated directory such as docs/replays/ or a shared drive. The agent should note that the script could be invoked manually, aliased to a short command, or triggered by a file-watch mechanism. It might also suggest naming the output file with a timestamp for easy chronological browsing.

Warning
Automated replay generation should always use the default secret redaction (do not pass --no-redact). In team environments, session logs from different developers may contain credentials, internal URLs, or other sensitive material that the developer did not consciously expose. The redaction default exists precisely for this scenario.
Replays as organizational memory

Beyond immediate sharing, replays serve as a form of organizational memory. When a complex debugging session resolves a subtle issue, the replay preserves not just the fix but the entire diagnostic reasoning process. Six months later, when a similar issue arises, a teammate can watch the original replay and understand the thought process — something that a commit message or Jira ticket rarely captures. This is particularly valuable for onboarding: new team members can study replays of significant sessions to understand both the codebase and the team's problem-solving patterns. In this sense, claude-replay transforms ephemeral agent sessions into durable knowledge artifacts.

Recap

In this session, we explored the full lifecycle of working with claude-replay — from understanding the JSONL session logs that Claude Code produces, through generating and customizing interactive replays, to embedding them in documentation and automating their production.

The core insight is that AI agent sessions produce rich, structured data as a byproduct of normal use. That data — the sequence of prompts, reasoning traces, tool invocations, and results — represents genuine intellectual work that is typically discarded or buried in inaccessible log files. claude-replay surfaces this data in a form that is navigable, shareable, and archivable.

This matters for several reasons. It makes AI-assisted work legible to others: reviewers, teammates, students, and future selves. It shifts the documentation of agent interactions from lossy formats (screenshots, recordings) to a faithful, interactive representation. And it creates the possibility of treating agent sessions as first-class artifacts — not just means to an end, but records worth preserving and studying.

  • Claude Code stores complete session transcripts as JSONL files in ~/.claude/projects/ — every prompt, response, tool call, and thinking block, timestamped and structured. This data is already available; claude-replay makes it accessible.
  • A single self-contained HTML file with no external dependencies is a remarkably portable and durable artifact. It can be emailed, embedded, hosted, or archived without dependency management or infrastructure concerns.
  • Editorial control — turn filtering, bookmarks, theme selection, thinking block visibility — transforms a raw session log into a purposeful narrative. The difference between a useful replay and a useless one is curation, not technology.
  • Automatic secret redaction is a critical default for any tool that surfaces agent session data. Always review replays before sharing, especially in team environments where sessions may contain credentials or internal information.

Where to go next

  • Build a custom theme JSON file that matches your team's or blog's visual identity, and establish it as a standard for all generated replays.
  • Investigate integrating replay generation into your pull request workflow — attaching replays as CI artifacts so reviewers can see how AI-assisted code was produced.
  • Experiment with the --from and --to time-range filters to produce replays of specific time windows, useful for correlating agent sessions with deployment incidents or production events.
