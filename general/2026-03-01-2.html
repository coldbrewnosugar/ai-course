Your Own AI That Never Forgets

Nous Research's Hermes Agent is a persistent, self-hosted agent that learns your workflow, connects to your messaging apps, and gets smarter the longer it runs. Let's set it up.
40 min autonomous-agents open-source self-hosting tool-calling persistent-memory

What's happening

Here's the situation: most AI tools you've used are amnesiac. You open ChatGPT, have a brilliant conversation about your project architecture, close the tab, and next time it has no idea who you are. Every session starts from zero.

Nous Research — the folks behind some of the best open-source fine-tuned models — just dropped Hermes Agent, and it flips that model completely. This is an agent that:

  • Lives on your server — it's a process that keeps running, not a browser tab
  • Remembers everything — your preferences, your projects, your environment, across sessions
  • Builds skills — when it solves a hard problem, it writes itself a reusable playbook for next time
  • Reaches you anywhere — Telegram, Discord, Slack, WhatsApp, CLI — all from one gateway
  • Runs tasks on a schedule — "every morning at 8am, check my repos for dependency updates" in plain English
  • Uses any model — OpenRouter, your own VLLM server, or Nous Portal. Switch with a single command.

Think of it less like a chatbot and more like hiring a junior dev who lives in your terminal, never sleeps, and has a perfect notebook of everything you've ever discussed. That's the vibe.

It's fully open-source, and the same architecture powers Nous's internal training pipelines for their next-gen tool-calling models. So this isn't a weekend project — it's production infrastructure they're sharing with the world.

1

Get Hermes Running on Your Machine

First things first — let's get the agent installed and talking to a model. The installer is a one-liner that handles Python, dependencies, and symlinks. But before we run it blindly, let's understand what we're setting up.

Hermes Agent has a clean separation between three things:

  1. The agent runtime — the brain that reasons, plans, and uses tools
  2. The model backend — where the actual LLM lives (cloud API, local server, etc.)
  3. The gateway — how messages get in and out (CLI, Telegram, Discord, etc.)

Think of it like a car: the runtime is the driver, the model is the engine, and the gateway is the steering wheel. You can swap any of them independently.

For this workshop, we'll start with the CLI gateway (simplest) and an OpenRouter backend (most flexible). You can wire up Telegram later — it's the same process with different credentials.

Ask your agent
Get your agent to walk you through the Hermes Agent installation and initial model configuration, customized for your setup.
Think about it
  • What OS are you on? The install script differs between Linux/macOS and Windows.
  • Which model backend makes sense for you — do you have an OpenRouter key, a Nous Portal subscription, or a local VLLM instance?
  • What should happen after install — how do you verify it's actually working?
What the agent gives back

The agent should give you the one-line install command for your OS — curl -fsSL ... | bash for Linux/macOS or the PowerShell equivalent for Windows. It should explain that the installer handles uv (a fast Python package manager), Python 3.11, cloning to ~/.hermes/hermes-agent, and symlinking the hermes command globally. Then it should walk you through connecting a model backend — the simplest being hermes model to switch between providers. For OpenRouter, you'd set your API key as an environment variable. The verification step is just running hermes in your terminal and seeing the TUI come up.

API Key Note
You'll need at least one model backend. OpenRouter is the easiest start — sign up at openrouter.ai and grab an API key. It gives you access to 200+ models including Claude, GPT-4, Llama, and Nous's own Hermes models. If you want zero-config, a Nous Portal subscription handles everything.
Tip
The installer uses uv instead of pip — it's a Rust-based Python package manager that's roughly 10-100x faster. If you've ever waited five minutes for pip to resolve dependencies, you'll appreciate this choice.
What's actually in ~/.hermes/?

The install creates ~/.hermes/hermes-agent with the main codebase, plus two submodules: mini-swe-agent (a lightweight software engineering agent) and tinker-atropos (their RL training environment). It also creates a virtual environment with Python 3.11 managed by uv. The hermes command in your PATH is just a symlink into ~/.local/bin, so you don't need to activate any virtualenv manually. Your personal data — memory, skills, conversation history — lives separately from the code, which means updates don't wipe your agent's brain.

Why uv instead of pip?

Written in Rust by the folks at Astral (who also made ruff, the fast Python linter), uv resolves dependencies deterministically and installs packages in parallel. For a project like Hermes Agent that has a lot of dependencies (LLM client libraries, messaging SDKs, terminal UI frameworks), this cuts install time from minutes to seconds. It also manages Python versions directly, so you don't need pyenv or conda. If you're not already using uv for your own projects, this is a good excuse to start.

2

Understand the Memory and Skill System

Here's where Hermes Agent stops being "just another chatbot wrapper" and starts getting genuinely interesting.

Most agent frameworks give you a conversation history and call it memory. Hermes does something fundamentally different: it has persistent memory across sessions and a skill accumulation system.

Let me explain with an analogy. Imagine you hire a new assistant. On day one, you explain your entire project setup, your coding conventions, which repos matter, and how you like your commit messages. A normal chatbot forgets all of this when you close the window. Hermes writes it down in a notebook — and that notebook persists forever.

But the skill system is the really clever part. When the agent solves a complex problem — say, figuring out how to parse a gnarly API response or set up a tricky Docker networking config — it doesn't just solve it and move on. It writes a skill document: a structured, searchable playbook that says "here's the problem, here's how I solved it, here's when to use this approach." Next time a similar problem comes up, the agent searches its skill library before reasoning from scratch.

These skills follow the agentskills.io open standard, which means they're portable and shareable. You can publish your agent's best skills to a Skills Hub, or install community skills that other people's agents have learned.

Ask your agent
Ask your agent to explain how you'd interact with Hermes Agent's memory and skill system — what commands exist, how skills get created, and how to manage what the agent remembers.
Think about it
  • How would you tell the agent to remember something specific about your workflow?
  • What happens when the agent creates a skill — where does it go, and how is it structured?
  • Can you browse or edit the agent's memory? What if it remembered something wrong?
What the agent gives back

The agent should explain that Hermes stores memory as persistent context that carries across sessions — you can naturally tell it things like "remember that I prefer TypeScript over JavaScript" and it'll retain that. Skills are structured documents the agent writes after solving complex problems, stored locally and searchable via semantic matching. You can browse skills, edit or delete them, and share them via the Skills Hub. The key mental model: memory is about you (preferences, context), skills are about how to do things (reusable solutions).

Tip
The skill system is essentially self-improving documentation. Every hard problem the agent solves becomes a searchable how-to guide for future problems. Over weeks and months, your agent builds up a personalized knowledge base that no generic LLM has.
How is this different from RAG?

RAG (Retrieval-Augmented Generation) is about searching your documents to answer questions. Hermes skills are about searching the agent's own experience to solve problems. The difference is authorship — RAG retrieves information that humans wrote, while skills retrieve solutions that the agent itself discovered and validated. It's closer to how a developer builds up muscle memory over years: you don't re-derive how to set up a reverse proxy every time, you remember the pattern from last time. The agent does the same thing, but with structured documentation instead of fuzzy human memory.

What's the agentskills.io standard?

It's an open format for describing agent capabilities as structured documents. Each skill has metadata (name, description, trigger conditions), a procedure (step-by-step instructions), and examples. By using a standard format, skills become portable between different agent systems — in theory, a skill learned by Hermes Agent could be used by any agent that supports the standard. The Skills Hub is a community repository where people publish and discover skills, kind of like npm but for agent capabilities. It's still early days, but the idea of agents sharing learned behaviors is genuinely exciting.

You've had a conversation with it and understand that it remembers things across sessions and builds reusable skills over time. If the TUI isn't launching, double-check that `~/.local/bin` is in your PATH.

Wire Up a Messaging Gateway

Running Hermes in the terminal is great for focused work, but the real magic happens when it meets you where you already are — your phone.

The gateway system is beautifully simple in concept: one agent process, multiple messaging frontends. You can start a conversation on Telegram from your phone, continue it on Discord from your desktop, and check results in the CLI — all the same conversation thread, same memory, same context.

Here's the mental model: think of Hermes Agent as a person with multiple phone numbers. You can call any number and reach the same person. They remember what you said on the other line.

For this step, we'll wire up Telegram because it's the most common choice and the setup is straightforward. The process is similar for Discord and Slack — you create a bot/app on the platform side, grab credentials, and feed them to Hermes.

The gateway runs as a single background process that bridges all your connected platforms. It handles message format conversion (Telegram's markdown is different from Discord's), media attachments (voice memos get transcribed automatically), and rate limiting.

Ask your agent
Get your agent to walk you through setting up a Telegram bot and connecting it to Hermes Agent as a messaging gateway.
Think about it
  • Where do you create a Telegram bot — what's the process on Telegram's side?
  • What credentials does Hermes need, and where do you configure them?
  • How do you verify the gateway is working — what should happen when you message the bot?
What the agent gives back

The agent should walk you through: (1) messaging @BotFather on Telegram to create a new bot and get a bot token, (2) configuring Hermes with that token — typically via a config file or environment variable, (3) starting the gateway process, and (4) sending a test message to your bot on Telegram to verify it responds through Hermes. It should mention that the gateway handles voice memos (auto-transcription), images, and documents — not just text. The key thing: once the gateway is running, your Telegram bot IS your Hermes Agent.

Warning
The gateway process needs to stay running for your bot to respond. If you're on a server, you'll want to run it as a systemd service or in a tmux/screen session. If you're just testing locally, keep the terminal open.
Tip
Voice memos are a killer feature here. You can literally talk to your AI agent while walking the dog — send a voice note on Telegram, Hermes transcribes it, reasons about it, and sends back a text response. It's the closest thing to having an AI assistant in your pocket without a dedicated app.
How does cross-platform conversation mirroring work?

When you start a conversation on Telegram and then switch to Discord, Hermes maintains a single conversation thread internally. It tracks which platform each message came from but treats them as one continuous dialogue. The agent's persistent memory means it doesn't matter where the message originated — context is preserved. This is different from running separate bot instances on each platform, which would create isolated conversations with no shared state. The gateway architecture is essentially a message bus: all platforms feed into the same agent loop, and responses route back to whichever platform the most recent message came from.

What about WhatsApp and Slack?

Slack uses a Slack App with OAuth tokens — you create an app in Slack's developer portal, install it to your workspace, and configure the bot token in Hermes. WhatsApp requires the WhatsApp Business API, which is more involved (you need a Meta Business account and a verified phone number). Discord uses a bot token similar to Telegram. Each platform has its own quirks — Slack has threads, Discord has channels, Telegram has groups — and the gateway handles the mapping between these concepts and Hermes's internal conversation model.

Set Up a Scheduled Automation

This is my favorite part, honestly. Scheduled automations turn Hermes from a tool you use into an assistant that works for you while you sleep.

The idea is dead simple: you tell the agent, in plain English, what to do and when. "Every morning at 8am, check my GitHub repos for new issues and send me a summary on Telegram." "Every Friday at 5pm, review this week's commits and write a changelog." "Every night at midnight, run the test suite and alert me on Discord if anything fails."

Under the hood, this maps to a cron scheduler built into the gateway. But you never write cron syntax — you describe the schedule in natural language and the agent figures out the timing.

What makes this powerful isn't just the scheduling — it's the combination with persistent memory and skills. Your 8am GitHub summary gets better over time because the agent remembers which repos you care about, which issue labels matter, and how verbose you like your summaries. It's not a static script; it's a learning automation.

Let's set one up.

Ask your agent
Get your agent to help you design and configure a useful scheduled automation in Hermes — something that runs daily and delivers results to your messaging platform.
Think about it
  • What repetitive task do you actually do every day that an AI could handle? Think about information gathering, monitoring, or summarization.
  • How should the agent deliver results — which platform, how detailed, what format?
  • What happens if the automation fails — should the agent retry, alert you, or skip?
What the agent gives back

The agent should help you pick a practical automation — common examples include daily repo summaries, dependency vulnerability checks, news/RSS digests, or server health reports. It should explain that you configure these by telling Hermes something like "schedule: every weekday at 9am, check my GitHub notifications and send me a summary on Telegram." The agent should note that scheduled tasks run in the gateway process, so it needs to be running continuously. The agent handles natural language time parsing — "every weekday at 9am", "nightly", "every Monday" all work. Results get delivered to whichever platform you specify.

Tip
Start with one simple automation and iterate. A daily summary of your GitHub notifications is a great first choice — low stakes, immediately useful, and you'll quickly see how the agent's memory makes it smarter over time (it learns which notifications you actually care about).
How do scheduled automations use the skill system?

This is where the architecture gets elegant. When a scheduled automation runs, it has full access to the agent's skill library. So if your daily GitHub check encounters a new type of issue (say, a security advisory format it hasn't seen before), it can check skills for relevant parsing logic. If it figures out how to handle it, it writes a new skill. Next time, it already knows what to do. Over weeks, your automations get more capable without you touching anything. It's like a cron job that debugs and improves itself.

What about complex multi-step automations?

Hermes supports subagent delegation for complex tasks. Your scheduled automation can spawn isolated subagents for parallel workstreams — for example, a weekly report automation might spawn one subagent to check GitHub, another to scan dependency vulnerabilities, and a third to review server logs, then synthesize the results. Each subagent gets its own conversation and terminal, so they can run in parallel without stepping on each other. The parent automation collects results and composes the final delivery.

✓
You should now have Hermes Agent running with a messaging gateway (Telegram, Discord, or at minimum the CLI), and at least one scheduled automation configured. Try messaging your bot from your phone and verifying it responds. If the gateway isn't responding, check that the process is running and your bot token is correct.

Explore Sandboxed Terminals and Subagent Delegation

Let's talk about the safety and power architecture, because this is where Hermes Agent's design philosophy really shows.

When you give an AI agent access to a terminal, you're handing it the keys to your machine. Hermes addresses this with five terminal backends: local, Docker, SSH, Singularity, and Modal. Each offers a different tradeoff between convenience and isolation.

  • Local: no isolation, full access to your machine. Fine for testing, scary for production.
  • Docker: runs in a container with a persistent workspace. Can't touch your host system. This is the sweet spot for most people.
  • SSH: runs on a remote machine. Great for giving the agent access to a server without exposing your laptop.
  • Modal: ephemeral cloud containers that spin up on demand. Pay per second, zero maintenance.

The key insight: you can run the agent against a remote machine so it can't modify its own code or read private API keys. That's real sandboxing, not just "please don't do bad things" in the system prompt.

Subagent delegation is the other superpower here. The main agent can spawn isolated subagents for parallel work — each gets its own conversation and terminal. Think of it like a manager who delegates tasks to specialists: "you go research this API, you go write the tests, you go check the deployment config." Results come back to the parent agent, which synthesizes them into a single response.

Ask your agent
Ask your agent to explain the tradeoffs between Hermes's terminal backends and recommend a setup for your use case.
Think about it
  • What's your threat model — are you worried about the agent accidentally deleting files, or is this more about structured isolation?
  • Do you have Docker installed? A remote server you could SSH into?
  • How do you want to handle persistent state — should the agent's workspace survive restarts?
What the agent gives back

The agent should help you think through which backend fits your needs. For most developers experimenting locally, Docker with a persistent volume is the recommendation — it gives you real isolation without the overhead of a remote server. The agent should explain that switching backends is a configuration change, not a code change. It should also explain subagent delegation: how the main agent can spin up parallel workstreams, each in its own sandboxed terminal, and collect results. The mental model is a team of agents working in separate rooms, reporting back to a coordinator.

Warning
If you're running in local mode, the agent has full access to your filesystem and can execute any command. This is fine for testing but not recommended for production or scheduled automations that run unattended. Use Docker or SSH for anything you're not actively supervising.
How does subagent RPC work?

Here's a neat trick: the agent can write Python scripts that call its own tools via RPC (Remote Procedure Call). This means instead of a multi-turn conversation where the agent calls a tool, reads the result, calls another tool, reads that result, and so on — it can write a single script that does all of that in one shot. This collapses multi-step pipelines into a single turn with zero intermediate context cost. For complex tasks, this can dramatically reduce both latency and token usage. It's like the difference between telling someone each step versus handing them a recipe and saying 'go.'

What's the research/training use case?

Hermes Agent includes a batch runner for generating thousands of tool-calling trajectories in parallel — this is how Nous Research trains their next-gen models. The Atropos RL environment lets you set up reinforcement learning tasks where the agent learns from its own agentic behavior. There's even trajectory compression for fitting training data into token budgets. If you're doing ML research on tool-calling models, this is essentially the same infrastructure Nous uses internally. For most users this isn't relevant day-to-day, but it's a signal that this project is backed by serious research infrastructure, not just weekend hacking.

Recap

We just set up something that didn't really exist in this form a year ago: a fully open-source, self-hosted AI agent that persists across sessions, learns from its own experience, connects to your messaging apps, runs scheduled automations, and sandboxes its own execution.

Let's be real about what this is and isn't. Hermes Agent isn't going to replace a human engineer — it's going to handle the repetitive, context-heavy tasks that eat your mornings. The daily standup prep. The dependency audit. The issue triage. The monitoring checks. All the stuff that's important but not interesting.

What makes this different from a cron job or a GitHub Action is the learning loop. Your Monday morning issue triage gets better over time because the agent remembers which issues you actually cared about, which ones you closed without reading, and what your real priority signals are. The skill system means hard-won solutions don't get lost — they become reusable knowledge.

And because it's model-agnostic, you're not locked into any vendor. When a better model comes out next month, you run hermes model and switch. Same memory, same skills, same automations — just a smarter engine underneath.

  • Persistent agents that accumulate memory and skills across sessions are fundamentally different from stateless chatbots — they get more useful the longer they run, like a colleague who learns your workflow.
  • The gateway architecture (one agent, many messaging platforms) means your AI assistant lives where you already communicate — no new app to install, no context switching.
  • Sandboxed terminal backends (especially Docker) let you give an agent real system access without real system risk — essential for unattended automations.
  • Model-agnostic design protects you from vendor lock-in: swap between OpenRouter, VLLM, Nous Portal, or any OpenAI-compatible endpoint with a single command.
  • The skill system (following the agentskills.io open standard) turns solved problems into shareable, searchable playbooks — your agent's experience becomes a portable knowledge base.

Where to go next

  • Set up a second messaging gateway (Discord or Slack) and test cross-platform conversation mirroring — start a thread on Telegram, continue it on Discord.
  • Explore the Skills Hub at agentskills.io and install a community skill to see how external knowledge integrates with your agent's existing capabilities.
  • Try the Docker terminal backend for your scheduled automations — configure a persistent volume so the agent's workspace survives container restarts.
  • Experiment with subagent delegation: give Hermes a complex task (like auditing a codebase for security issues) and watch how it spawns parallel workstreams.
  • If you're into ML research, explore the batch runner and Atropos RL environments — this is the same infrastructure Nous uses to train their next-gen tool-calling models.
