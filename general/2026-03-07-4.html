Build a Multi-Provider AI Gateway with LiteLLM and A2A Agent Routing

Stand up a unified proxy that routes requests across LLMs and agents using a single API
35 min litellm ai-gateway a2a-protocol multi-provider proxy

What's happening

The Gateway Problem

Most organizations working with large language models now consume multiple providers simultaneously — OpenAI for certain tasks, Anthropic for others, open-source models via Bedrock or Vertex AI for cost-sensitive workloads. Each provider exposes a different API surface, different authentication schemes, and different pricing structures. The operational burden of managing these integrations grows linearly with each new provider.

LiteLLM has matured from a lightweight Python SDK into a full-featured AI Gateway — a proxy server that sits between your applications and every LLM provider, presenting a single OpenAI-compatible interface regardless of the downstream model. Recent additions include first-class support for the A2A (Agent-to-Agent) protocol, which allows autonomous agents to be registered behind the gateway and invoked through the same unified endpoint, and MCP tool bridging, which exposes MCP tool servers to any model routed through the proxy.

The implications are significant: a team can issue virtual API keys with per-key spend limits, route requests across providers with automatic fallback, track costs in a single dashboard, and now orchestrate agent-to-agent communication — all without modifying application code. This session walks through standing up that infrastructure.

1

Generate the LiteLLM Proxy Configuration

The LiteLLM proxy server is configured through a YAML file that declares which models are available, how they map to upstream providers, and what policies govern their use. This configuration file is the single source of truth for the entire gateway.

The core abstraction is the model list — an array of model definitions, each specifying a model_name (the alias your applications use) and litellm_params (the actual provider, model identifier, and API key reference). When a client sends a request to model: "claude", the proxy resolves that alias to the corresponding upstream provider and forwards the request in the correct format.

This indirection is powerful: you can swap providers, add fallback models, or introduce load balancing without any client-side changes. The client always speaks OpenAI-format to a single endpoint.

What the agent gives back

The agent should produce a YAML configuration file (typically named litellm_config.yaml) containing a model_list with at least three entries — for example, an OpenAI GPT-4o model aliased as gpt-4o, an Anthropic Claude model aliased as claude, and a cheaper model aliased as gpt-4o-mini for cost-sensitive routing. Each entry should reference API keys through os.environ/ syntax rather than hardcoding secrets. The file should also include a general_settings block specifying the master key and completion model. The structure follows this pattern:

```yaml

model_list:

  • model_name: claude

litellm_params:

model: anthropic/claude-sonnet-4-20250514

api_key: os.environ/ANTHROPIC_API_KEY

```

The critical detail is that model_name is the client-facing alias, while litellm_params.model uses the provider/model format that LiteLLM requires for routing.

API Key Note
You will need valid API keys for each provider you configure. Set them as environment variables (e.g., export OPENAI_API_KEY=sk-...) before starting the proxy. The os.environ/ prefix in the YAML tells LiteLLM to read the value from the environment at startup.
Tip
The model_name alias is what clients specify in their requests. You can map multiple upstream models to the same alias — LiteLLM will load-balance across them automatically. This is how you implement redundancy without client changes.
How does the provider/model format work?

LiteLLM uses the provider/model-id convention to determine which API translation layer to activate. When you write anthropic/claude-sonnet-4-20250514, LiteLLM knows to format the request using Anthropic's Messages API, handle the authentication header correctly, and parse the response back into OpenAI-compatible format. Over 100 providers are supported, each with its own translation module. The full list is maintained in LiteLLM's provider documentation. If you omit the provider prefix, LiteLLM assumes OpenAI format.

2

Launch the Gateway and Verify Cross-Provider Routing

With the configuration file in place, the next step is to start the proxy server and confirm that requests are correctly routed to different providers through a single endpoint.

The LiteLLM proxy runs as a local HTTP server — by default on port 4000 — and exposes the standard OpenAI-compatible endpoints: /chat/completions, /embeddings, /responses, and others. Any HTTP client, OpenAI SDK, or tool that speaks the OpenAI format can connect to it without modification. This is the fundamental value proposition: one endpoint, many providers.

The proxy also exposes a /model/info endpoint that returns metadata about all configured models, including their providers, supported parameters, and current status. This is useful for programmatic discovery.

Warning
The api_key parameter in the OpenAI SDK client must match the master_key defined in your LiteLLM configuration's general_settings. This is not your OpenAI API key — it is the gateway's own authentication token. Confusing these two is a common source of 401 errors.
What does the proxy actually do to each request?

When a request arrives at the proxy, LiteLLM performs several transformations:

  1. Authentication — validates the provided key against virtual keys or the master key.
  2. Model resolution — maps the client-facing alias to an upstream provider and model ID.
  3. Format translation — converts the OpenAI-format request into the provider's native format (e.g., Anthropic's Messages API uses system as a top-level parameter, not a message role).
  4. Request forwarding — sends the transformed request to the upstream provider.
  5. Response normalization — converts the provider's response back into OpenAI format.
  6. Logging and metering — records token usage, latency, and cost for the virtual key that made the request.

This pipeline is transparent to the client. From its perspective, it is talking to an OpenAI-compatible server.

3

Configure Virtual Keys and Spend Controls

A gateway without access control is just a transparent proxy. LiteLLM's virtual key system introduces a management layer that mirrors how organizations actually operate: different teams, different budgets, different access levels.

A virtual key is a proxy-level API key that LiteLLM issues and manages independently of any upstream provider key. When a request arrives bearing a virtual key, the proxy checks whether that key is authorized to use the requested model, whether its spend limit has been reached, and which upstream provider key to use on its behalf. This decouples consumer identity from provider credentials entirely.

The practical consequence is substantial: you can give a frontend team a key limited to gpt-4o-mini with a $50/month budget, give the research team access to claude with a $500 budget, and revoke or rotate any key without touching the underlying provider accounts. Cost tracking is per-key, per-model, and per-team — all through the gateway's admin API.

Tip
Virtual keys require a database backend for persistence. For local development, LiteLLM uses SQLite by default — no additional setup required. For production, PostgreSQL is recommended. Add database_url to your configuration's general_settings to specify the connection string.
How does cost tracking work across different providers?

LiteLLM maintains an internal pricing table that maps each model to its per-token input and output costs. After each request completes, the proxy calculates the cost based on the actual token usage reported by the upstream provider and debits it against the virtual key's budget. This pricing data is updated regularly but can also be overridden in the configuration for custom or self-hosted models. The /spend/logs endpoint provides detailed breakdowns by key, model, and time period — effectively a cost dashboard accessible via API. For models with non-standard pricing (e.g., per-image or per-character), LiteLLM maps to the closest token-equivalent cost.

4

Register an A2A Agent Behind the Gateway

The Agent-to-Agent (A2A) protocol is an open specification — originally proposed by Google — that defines how autonomous agents discover each other's capabilities and exchange messages. An A2A-compliant agent publishes an Agent Card (a JSON manifest describing its name, capabilities, and endpoint) and exposes a message-passing interface for receiving tasks.

LiteLLM's gateway now supports A2A natively: you can register external A2A agents in your proxy configuration, and they become accessible through the gateway at /a2a/{agent-name}. This means the same virtual key system, spend tracking, and access controls that govern LLM requests also govern agent invocations. The agent itself can be built with any framework — LangGraph, Pydantic AI, a custom implementation — as long as it speaks A2A.

The architecture is composable: an agent behind the gateway can itself call back into the gateway to access LLMs, creating a loop where the proxy serves as both the agent's interface to the outside world and its access point for language model capabilities.

```yaml

```

The agent should clarify that the registered agent must be running independently and serving the A2A protocol — LiteLLM does not host the agent, it merely routes to it.

Tip
The A2A protocol distinguishes between synchronous and streaming responses. An agent can return a complete response immediately or stream partial updates via Server-Sent Events (SSE). LiteLLM supports both modes and passes the streaming preference through from the client.
Warning
A2A is still a relatively young protocol. While LiteLLM supports it as a first-class routing target, the ecosystem of production-ready A2A agents is still developing. Expect the configuration syntax and protocol details to evolve.
How does A2A differ from MCP?

A2A and MCP (Model Context Protocol) address different layers of the agent architecture:

  • MCP defines how an LLM accesses external tools and data sources. It is a protocol between a model (or its host application) and a tool server. The model decides when to invoke a tool based on the conversation context.
  • A2A defines how agents communicate with each other. It is a protocol between two autonomous agents, each of which may internally use LLMs, tools, or other agents. The key difference is that an A2A agent has its own agency — it interprets the message, decides on a plan, and may perform multiple internal steps before responding.

Think of MCP as giving an agent hands (access to tools), and A2A as giving agents the ability to delegate to and collaborate with other agents. LiteLLM's gateway can broker both: MCP tools are bridged to any model via the proxy, and A2A agents are registered as routable endpoints.

5

Bridge MCP Tools Through the Gateway

The final piece of the gateway architecture is MCP tool bridging — the ability to connect Model Context Protocol tool servers to any LLM routed through the proxy, regardless of whether that LLM natively supports tool use.

MCP defines a standard interface for tools: each tool has a name, a description, and an input schema. LiteLLM's experimental_mcp_client can connect to any MCP server, load its available tools, and translate them into the OpenAI function-calling format. Once translated, these tools can be passed to any model in the gateway's model list. The proxy handles the round-trip: it sends the tool definitions with the completion request, receives the model's tool-call response, executes the tool via MCP, and feeds the result back into the conversation.

This bridging is significant because it decouples tool availability from model choice. A tool server built for Claude can be used with GPT-4o, Gemini, or any other model that supports function calling — the gateway handles the format translation.

Tip

What is the security model for MCP tools behind the gateway?

MCP tool execution introduces a significant trust boundary. When a tool is bridged through the gateway, the LLM can invoke it based on user input — which means a user can indirectly trigger actions on the MCP server (file reads, API calls, database queries) by crafting their prompt appropriately.

LiteLLM's virtual key system provides the first layer of defense: you can restrict which models (and therefore which tool configurations) a given key can access. However, fine-grained per-tool authorization is not yet part of the proxy layer — if a key has access to a model with MCP tools, it can invoke any of those tools.

For production deployments, the recommendation is to run MCP servers with minimal permissions, use separate model configurations for tool-augmented and tool-free access, and audit tool invocation logs through the gateway's spending and logging endpoints.

Recap

This session constructed a complete AI gateway using LiteLLM's proxy server — a single infrastructure component that abstracts the complexity of multi-provider LLM access, agent orchestration, and tool bridging.

We began with the core abstraction: a YAML configuration that maps client-facing model aliases to upstream providers, allowing any OpenAI-compatible client to access any supported LLM without modification. We then layered on operational controls — virtual API keys with model restrictions and spend limits — that mirror real organizational access patterns. The A2A protocol integration demonstrated how autonomous agents can be registered behind the same gateway, inheriting the same access controls and cost tracking that govern LLM requests. Finally, MCP tool bridging showed how external tools can be made available to any model through the proxy, decoupling tool availability from provider choice.

The resulting architecture is a single pane of glass for all AI interactions: one endpoint, one authentication system, one cost ledger, one place to add providers, agents, or tools.

  • An AI gateway decouples application code from provider specifics — clients speak one format, the proxy handles translation, authentication, and routing for 100+ providers.
  • Virtual keys create an organizational layer above provider credentials, enabling per-team budgets, model restrictions, and key rotation without touching upstream accounts.
  • The A2A protocol and MCP serve complementary roles: A2A enables agent-to-agent delegation, while MCP bridges tools to models. Both can be routed through the same gateway.
  • Load balancing and failover are configured at the alias level — multiple upstream models sharing one alias gives you resilience with zero client-side changes.
  • Cost tracking is automatic and per-key, using LiteLLM's internal pricing tables to meter spend across all providers in a single ledger.
