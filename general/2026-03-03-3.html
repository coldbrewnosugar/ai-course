Teaching an AI to Bet Smarter Than You

We're going to build an autonomous agent that reads the news, evaluates prediction markets, and decides whether the crowd is wrong — all without you lifting a finger.
40 min agents RAG prediction-markets autonomous-trading LLM-tools

What's happening

Why prediction markets? Why now?

Prediction markets like Polymarket let people bet real money on future events — "Will the Fed cut rates in June?" or "Will SpaceX launch Starship by April?" The prices on these markets basically represent the crowd's probability estimate. If a market trades at $0.72, the crowd thinks there's a 72% chance it happens.

Here's where it gets interesting: crowds are often wrong. Not wildly wrong, but 5-10% wrong — and that's where money is made.

The Polymarket Agents project (open source, MIT licensed) gives us a framework for building AI agents that do three things:

  1. Research — pull live data from news APIs, betting services, and web search
  2. Reason — use an LLM to evaluate whether a market is mispriced
  3. Trade — actually place orders on Polymarket's decentralized exchange

What we're really building today is an autonomous decision-making loop: gather information → form an opinion → act on it → repeat. That pattern shows up everywhere — from trading bots to customer service agents to scientific research assistants. Prediction markets are just a particularly fun (and measurable) playground for it.

Let's dig in.

1

Understanding the Architecture — What Are We Actually Building?

Before we ask our agent to write a single line of code, let's get a mental model of what's going on.

Think of our prediction market agent like a sports analyst with a research assistant. The research assistant (RAG) runs around collecting articles, statistics, and expert opinions. The analyst (the LLM) reads everything the assistant brought back and forms an opinion. Then a broker (the trading module) actually places the bet.

The Polymarket Agents architecture has three layers:

  • Data connectors — These pull information from Polymarket's API (what markets exist, current prices), news sources, and web search. Think of these as the agent's "senses."
  • RAG pipeline — This takes all that raw data, chunks it up, embeds it into vectors, and stores it in ChromaDB so the LLM can search through it semantically. It's like giving your agent a searchable notebook of everything it's read.
  • LLM reasoning layer — This is where the magic happens. The LLM gets a market question, retrieves relevant context from the RAG store, and outputs a probability estimate plus a confidence score.

Let's start by getting our agent to build the data layer — the part that actually talks to Polymarket.

Ask your agent
Get your AI agent to build a Polymarket data client that can fetch active markets and their current prices
Think about it
  • What API does Polymarket expose, and what kind of data comes back? (Think: market question, current price, volume, end date)
  • How should we filter markets — do we want every market ever, or just ones that are actively trading?
  • What's a clean way to represent a market in our code? What fields matter for a trading agent?
  • Should we sort or rank the results somehow so we see the most interesting markets first?
What the agent gives back

The agent should give you a Python class (something like PolymarketClient) that wraps the Gamma API. It'll have methods to fetch active markets, filter by trading volume, and return clean data objects with fields like question, outcome_prices, volume, and end_date. The key insight is that it uses Polymarket's Gamma API (their public market data API) — you don't need authentication just to read market data. The agent will probably also include a simple Pydantic model to represent a market, which keeps the data tidy as it flows through the rest of the system.

Tip
You don't need a Polymarket account or API key just to read market data. The Gamma API is public. You only need keys when you want to actually place trades.
API Key Note
For the full agent (with trading), you'll eventually need: an OpenAI API key for the LLM, a Polygon wallet private key for placing trades, and USDC loaded in that wallet. For this workshop, we're focusing on the research and reasoning parts — no real money needed.
What's the Gamma API?

Polymarket is built on the Polygon blockchain, but you don't interact with the blockchain directly for most things. The Gamma API is Polymarket's REST API that gives you clean JSON data about markets, events, and prices. Think of it as the friendly front door — behind it, there's a whole decentralized exchange running on smart contracts, but you don't need to worry about that unless you're placing trades. The Gamma API endpoint is https://gamma-api.polymarket.com and it returns standard JSON with fields like question, outcomes, outcomePrices, volume, and endDate.

2

Building the Brain — RAG for Real-Time Research

Okay, so now our agent can see what markets exist and what prices the crowd has set. But a trading agent that only looks at prices is like a poker player who only looks at their own cards. We need context.

This is where RAG comes in, and honestly, this is the part that confused me at first too. So let's break it down with an analogy.

Imagine you're trying to answer: "Will the EU pass its AI regulation bill by July?" You could guess, or you could:

  1. Gather a bunch of recent news articles about EU AI regulation
  2. Highlight the most relevant passages (not entire articles — just the useful bits)
  3. Reason about what those passages tell you about the timeline

That's literally what RAG does:

  1. Retrieve — search your knowledge base for relevant documents
  2. Augment — stuff those documents into the LLM's prompt as context
  3. Generate — let the LLM reason with that fresh information

The "retrieval" part is the clever bit. Instead of keyword search (which would miss "Brussels lawmakers debate artificial intelligence rules"), we use vector embeddings — think of them as GPS coordinates for meaning. "EU AI regulation" and "Brussels lawmakers debate artificial intelligence rules" end up at nearly the same coordinates, even though they share almost no words.

The Polymarket Agents framework uses ChromaDB as its vector store. It's lightweight, runs locally, and is perfect for this kind of thing.

Ask your agent
Get your agent to build a RAG pipeline that collects news articles and makes them searchable by meaning
Think about it
  • What are our data sources? (Think: news APIs, web search, maybe even tweets or blog posts)
  • How do we break articles into useful chunks? A whole article is too big for context — what's the right size?
  • What embedding model should we use, and what does 'embedding' actually mean here?
  • When a market question comes in, how do we decide which chunks are relevant?
What the agent gives back

The agent should give you two main pieces: a data ingestion module that fetches articles from news APIs (like NewsAPI or a web search wrapper), splits them into chunks of ~500 tokens, generates embeddings using OpenAI's embedding model, and stores them in ChromaDB. And a retrieval function that takes a market question, embeds it, searches ChromaDB for the top-k most similar chunks, and returns them as context strings. The key design decision is the chunk size — too small and you lose context, too big and you waste the LLM's context window. The framework defaults to about 500 tokens per chunk with some overlap between chunks so you don't accidentally split a key sentence in half.

Tip
ChromaDB stores vectors locally in a SQLite-backed file. No cloud database needed. This is great for development but means your knowledge base lives on your machine. The Polymarket framework also supports remote RAG if you want to scale up later.
Warning
Embeddings cost money (a tiny amount per API call to OpenAI), and they're only as good as your data sources. Garbage in, garbage out — if you feed it tabloid clickbait, your agent will reason based on tabloid clickbait.
How do vector embeddings actually work?

Here's the intuition: an embedding model (like OpenAI's text-embedding-3-small) reads a piece of text and outputs a list of ~1,536 numbers. Those numbers represent the meaning of the text in a high-dimensional space. Texts with similar meanings end up close together in that space. When you search ChromaDB with a query like "EU AI regulation timeline," it embeds your query into the same space and finds the stored chunks that are closest (usually measured by cosine similarity). It's like if every document had GPS coordinates based on what it's about, and searching meant finding the nearest neighbors to your question's coordinates. The beautiful thing is it works across languages, paraphrases, and synonyms — because meaning is more stable than specific word choices.

Why not just dump everything into the LLM's context window?

Great question! Modern LLMs have huge context windows (100k+ tokens), so why bother with RAG at all? Three reasons: Cost — you pay per token, and stuffing 50 articles into every prompt gets expensive fast. Noise — more context isn't always better. Irrelevant articles can actually confuse the model and lead to worse reasoning. Studies have shown LLMs perform worse when relevant info is buried in a sea of irrelevant context (the 'lost in the middle' problem). Freshness — RAG lets you continuously add new articles without retraining or restructuring anything. Your vector store is a living, breathing knowledge base.

✓
Nice work. At this point you should have two pieces: a **Polymarket data client** that fetches live markets and prices, and a **RAG pipeline** that collects and searches news articles by meaning. These are your agent's eyes and memory. Next, we'll give it a brain.
3

The Reasoning Engine — Getting an LLM to Think Like a Trader

Here's where the art of prompt engineering really matters. We need to take a market question, relevant context from our RAG pipeline, and the current market price, then get an LLM to tell us: "I think the true probability is X%, and here's my confidence level."

This is harder than it sounds. LLMs are notoriously bad at raw probability estimation. Ask GPT "what's the probability it rains tomorrow in Seattle in November?" and you'll get a wishy-washy non-answer.

The Polymarket framework solves this with a clever prompt structure that essentially forces the LLM to think like an analyst:

  1. State the question clearly — no ambiguity
  2. Present the evidence — the RAG-retrieved context
  3. Show the current market price — this anchors the analysis
  4. Ask for structured output — a probability estimate AND a confidence score AND reasoning

The confidence score is crucial. It's the difference between "I have no idea but I'll guess 60%" and "Based on three corroborating sources, I'm fairly confident it's 60%." Our agent should only trade when confidence is high AND the market disagrees with its estimate.

Ask your agent
Get your agent to build the LLM reasoning module — the part that evaluates a market and outputs a probability estimate with confidence
Think about it
  • What goes into the prompt? Think about all the context the LLM needs to make a good judgment.
  • How do we prevent the LLM from just parroting the market price back at us? (That would make us a very expensive index fund.)
  • What format should the output be in? We need to parse it programmatically, not just read it.
  • How do we define 'confidence'? What makes a high-confidence vs low-confidence prediction?
What the agent gives back

The agent should give you a function (something like evaluate_market) that takes a market question, the current market price, and a list of context chunks from RAG. It constructs a detailed system prompt that tells the LLM to act as a quantitative analyst, then feeds everything in. The key design choice is the output format — it should return structured JSON with fields like estimated_probability, confidence (low/medium/high), and reasoning. The prompt should explicitly tell the LLM to challenge the market price, not defer to it. Something like: "The market currently prices this at X%. Your job is to determine whether this is correct, too high, or too low based on the evidence." The agent will probably use OpenAI's function calling or JSON mode to ensure the output is parseable.

Tip
The Polymarket Agents framework includes pre-built prompt templates in its LLM utilities. But the real edge comes from customizing them — if you know a lot about, say, tech regulation, you can add domain-specific reasoning instructions to the prompt.
Why structured output matters here

If the LLM returns a free-text response like "I think this is likely to happen, maybe around 65-70%," our trading module has no idea what to do with that. We need clean, parseable output: {"estimated_probability": 0.67, "confidence": "high", "reasoning": "Three major outlets report..."}. This is where OpenAI's JSON mode or function calling comes in handy — you can define the exact schema you want and the model is constrained to output valid JSON matching that schema. Without this, you'd spend half your time writing regex to parse fuzzy LLM responses, and you'd still get errors.

The anchoring problem

If you show the LLM the current market price (say, 72%), there's a real risk it just says "yeah, 72% sounds about right" — that's called anchoring bias, and LLMs are susceptible to it just like humans. The Polymarket framework handles this a few ways: the prompt explicitly instructs the model to form its OWN estimate before considering the market price, and in some configurations it actually asks the model twice — once without the price, once with — and compares the answers. If they're very different, that's a signal the model is anchoring rather than reasoning independently.

✗ Not quite. That's what the *market* thinks, but our agent disagrees. The whole point of the agent is to find mispricings — situations where our research-backed estimate differs from the crowd. If we just follow the market price, we don't have an edge.

4

Wiring It Together — The Autonomous Trading Loop

We've got our three pieces: data fetching, RAG research, and LLM reasoning. Now we need to wire them into an autonomous loop — an agent that continuously:

  1. Scans for interesting markets
  2. Researches each one
  3. Forms an opinion
  4. Decides whether to trade
  5. Goes back to step 1

This is the leap from "tool" to "agent." A tool does one thing when you ask it. An agent has a loop and makes its own decisions about what to do next.

The Polymarket framework's trade.py script is basically this loop. It filters markets by volume (no point trading on dead markets), runs each through the reasoning pipeline, and only places trades when the estimated probability differs from the market price by more than a threshold AND confidence is high.

The crucial design decision here is risk management. An agent with no guardrails will happily bet your entire balance on a single prediction. We need to build in limits: maximum bet size, maximum total exposure, and a kill switch.

Ask your agent
Get your agent to build the main trading loop with risk management guardrails
Think about it
  • What's the loop structure? Think about how often it should run and what triggers a trade.
  • What guardrails prevent the agent from going rogue? (Max bet size, total exposure limits, etc.)
  • How do we decide the *size* of a bet? Should we bet the same amount every time, or scale by confidence?
  • What should we log? If something goes wrong at 3am, how do we debug it?
What the agent gives back

The agent should give you a main loop function that: fetches top markets by volume, filters for markets ending soon enough to be interesting, runs each through the RAG + reasoning pipeline, and places a trade when the edge (difference between our estimate and market price) exceeds a threshold like 10%. The risk management piece should include a MAX_BET_SIZE constant, a check against total portfolio exposure, and Kelly Criterion-inspired position sizing (bigger edge = bigger bet, but never more than the max). The agent should also include comprehensive logging — every decision should be recorded with its reasoning so you can review what the agent did and why. Don't forget a DRY_RUN mode flag so you can test the full loop without risking real money.

Warning
NEVER run a trading agent with real money until you've tested it extensively in dry-run mode. Even then, start with very small amounts. Autonomous trading agents can lose money fast if the reasoning pipeline has a subtle bug — like consistently misinterpreting a news source, or anchoring to market prices despite the prompt telling it not to.
Tip
The Kelly Criterion is a formula from information theory that tells you the optimal bet size given your edge and the odds. In simplified form: bet_fraction = edge / odds. If your edge is 10% and the odds are even money, bet 10% of your bankroll. It's mathematically optimal but aggressive — most traders use 'half Kelly' to reduce variance.
What makes this an 'agent' and not just a script?

The line between a script and an agent is blurry, but the key difference is autonomous decision-making. A script follows a predetermined path — if X then Y. An agent uses an LLM to make judgment calls that couldn't be hardcoded. "Is this news article relevant to this market?" "Does this evidence suggest the probability should be higher or lower?" "Am I confident enough to act?" These are decisions that require reasoning, not just branching logic. The LLM is the engine that turns data into decisions. That said, the guardrails should absolutely be hardcoded — max bet sizes, exposure limits, and kill switches should never depend on the LLM's judgment.

The Polymarket trade execution flow

Under the hood, placing a trade on Polymarket involves interacting with a decentralized exchange on the Polygon blockchain. The framework's Polymarket.py module handles all of this: building the order (market, side, amount, price), signing it with your wallet's private key, and submitting it to Polymarket's order book. You're essentially signing a smart contract transaction. The framework abstracts away all the blockchain complexity — you just call something like client.place_order(market_id, 'buy', 'no', amount=10, price=0.40) and it handles the rest. But remember: once a trade is on the blockchain, it's irreversible. There's no 'undo' button.

✓
You now have the full picture: a **data layer** that talks to Polymarket, a **RAG pipeline** that researches markets using news sources, an **LLM reasoning engine** that forms probability estimates, and an **autonomous trading loop** that ties it all together with risk management. The architecture is: Fetch Markets → Research via RAG → Reason with LLM → Trade if confident → Repeat.
5

Making It Yours — Adding Custom Data Sources

The default Polymarket Agents setup uses generic news APIs and web search. That's fine, but here's the thing — everyone else using this framework has access to the same generic data sources. Your edge comes from feeding the agent better information.

Think about what data sources might give you an informational advantage for specific market categories:

  • Crypto markets → on-chain data, whale wallet movements, exchange inflow/outflow
  • Politics → polling aggregators, campaign finance data, local news from key districts
  • Tech → SEC filings, patent databases, GitHub commit activity
  • Sports → injury reports, weather data, historical matchup statistics

The Polymarket framework makes this extensible by design. The ChromaDB vector store doesn't care where the data comes from — it just needs text chunks and metadata. Adding a new data source means writing a fetcher that pulls data and feeds it into the same ingestion pipeline.

This is also where domain expertise becomes your superpower. The LLM is great at reasoning, but you know which data sources actually matter for the markets you care about.

Ask your agent
Get your agent to add a custom data source connector — pick a domain you're interested in and wire it into the RAG pipeline
Think about it
  • What domain are you most knowledgeable about? Where could you spot data sources that others might miss?
  • What API or data feed would give useful signal for that domain?
  • How often does this data need to be refreshed? Some data is daily, some is real-time.
  • Should the data be treated differently in the RAG pipeline? (Maybe financial data needs different chunking than news articles)
What the agent gives back

The agent should give you a new data connector class (following the same pattern as the existing ones) that fetches data from your chosen API, processes it into text chunks with metadata (source, timestamp, category), and feeds it into ChromaDB. For example, if you chose politics, it might be a connector for FiveThirtyEight polling data that pulls the latest polls, formats each as a text chunk like "ABC/WaPo poll (March 1): Candidate A leads Candidate B 52-47 among likely voters in Michigan," and tags it with metadata. The agent should also update the main ingestion pipeline to call this new connector alongside the existing ones.

Tip
Don't overlook metadata. When you store chunks in ChromaDB, tag them with source, date, and category. This lets you do filtered searches later — like 'find me relevant chunks, but only from the last 48 hours.' Stale data is worse than no data in prediction markets.
The 'information diet' of your agent matters more than the model

Here's a hot take: the difference between a mediocre trading agent and a good one is rarely the LLM. GPT-4 and Claude are both excellent reasoners. The difference is the information diet — what data goes into the RAG pipeline. A GPT-3.5 agent with access to obscure but high-signal data sources (like local regulatory filings for a zoning question) will outperform a GPT-4 agent reading the same CNN headlines as everyone else. This is true of human analysts too — the best fund managers aren't necessarily smarter, they just have better information sources. Your job as the agent builder is to be a curator of high-quality, timely, relevant data. The LLM handles the reasoning; you handle the inputs.

Recap

What We Built (Conceptually)

We walked through the full architecture of an autonomous prediction market agent:

  1. Data Layer — A client that talks to Polymarket's Gamma API to fetch live markets and prices
  2. RAG Pipeline — A system that collects news articles, chunks them, embeds them into ChromaDB, and retrieves relevant context for any market question
  3. Reasoning Engine — An LLM module with carefully engineered prompts that takes a market question + context and outputs a structured probability estimate with confidence
  4. Trading Loop — An autonomous cycle that scans markets, researches them, reasons about them, and trades when it spots an edge — all with risk management guardrails
  5. Custom Data Sources — The extensibility layer that lets you feed specialized, high-signal data into the pipeline

The bigger picture here isn't really about prediction markets — it's about the pattern of autonomous AI agents. Gather data → retrieve relevant context → reason with an LLM → take action → loop. This same architecture powers customer support agents, research assistants, code review bots, and more. Prediction markets are just a uniquely satisfying application because you get a clear, measurable signal of whether your agent is actually right.

  • RAG is the bridge between an LLM's frozen training data and the real-time world. Without it, your agent is trading on yesterday's news (literally).
  • The quality of your agent's decisions depends more on its 'information diet' (data sources) than on which LLM you use. Curating good inputs is your highest-leverage activity.
  • Autonomous agents need hardcoded guardrails (max bet sizes, kill switches) even though the decision-making is soft (LLM reasoning). Never let the LLM control the safety limits.
  • Prompt engineering for probability estimation requires fighting anchoring bias — explicitly telling the LLM to form its own view before considering the market's view.
  • The two-pass screening pattern (cheap model for broad scan, expensive pipeline for promising candidates) is a general optimization you'll see in many agent architectures.

Where to go next

  • Clone the Polymarket Agents repo and run the CLI in dry-run mode to see the agent reason about live markets
  • Experiment with different prompt templates for the reasoning engine — try adding chain-of-thought instructions or asking for bull/bear arguments before a final estimate
  • Build a backtesting module: run your agent's reasoning on historical markets where you know the outcome, and see how well it would have done
  • Explore the multi-agent pattern: one agent specializes in politics, another in crypto, another in sports — each with domain-specific data sources and prompts
