Build a Real-Time Voice AI Agent with TEN Framework

You're going to talk to something you built yourself — in your browser, with under 500ms latency. Let's wire up speech-to-text, an LLM brain, and text-to-speech into a voice assistant that actually feels conversational.
40 min voice-ai real-time multimodal open-source conversational-ai

What's happening

What's going on: TEN (Transformative Extensions Network) is an open-source framework that's been climbing GitHub's trending charts for real-time multimodal AI. Think of it as the plumbing that connects your microphone → a speech recognizer → an LLM → a voice synthesizer → your speakers, all in real time.

Why does this matter? Because the gap between chatting with an AI and talking to an AI is enormous. Text chat is forgiving — you can take 2 seconds to respond and nobody notices. Voice is brutal. Anything over ~500ms of silence feels broken. TEN handles all the gnarly real-time orchestration (WebRTC streaming, voice activity detection, turn-taking) so you can focus on making your agent actually smart.

The ecosystem includes the core framework, pre-built agent examples, a VAD (Voice Activity Detection) module, turn detection, and a browser-based portal. Today we're spinning up the voice assistant example and then making it our own.

1

Understand the Pipeline Before You Build It

Before we touch Docker, let's get the mental model right. A real-time voice agent is basically a relay race with three runners:

  1. STT (Speech-to-Text) — Deepgram listens to your microphone stream and converts audio chunks to text, in real time. Not "record and transcribe" — more like live captions.
  2. LLM (Language Model) — OpenAI gets the transcribed text and streams back a response, token by token. Streaming is key — we don't wait for the full response.
  3. TTS (Text-to-Speech) — ElevenLabs takes each chunk of LLM output and converts it to audio, which gets streamed back to your browser.

The magic of TEN is that it wires these three together with proper buffering, interruption handling, and turn detection. Imagine a relay race where the baton gets passed before the previous runner finishes — that's the streaming overlap that makes it feel fast.

WebRTC vs WebSocket: TEN supports both. WebRTC is the same tech behind Zoom and Google Meet — it's optimized for low-latency audio/video. WebSocket is simpler but adds a bit more latency. The default examples use Agora's RTC infrastructure for the real-time audio transport.

Why do we need Agora AND Deepgram AND OpenAI AND ElevenLabs?

Each piece does one thing well:

  • Agora = the transport layer (carries audio between browser and server, handles network jitter)
  • Deepgram = ears (converts audio stream to text with very low latency)
  • OpenAI = brain (generates the response)
  • ElevenLabs = mouth (converts text back to natural-sounding speech)

TEN is the orchestrator that wires them all together. You could swap any of these out — use Azure STT instead of Deepgram, or use Anthropic instead of OpenAI. TEN's extension system is designed for exactly that kind of mix-and-match.

Get Your API Keys and Spin Up Docker

Alright, let's get the boring-but-necessary stuff out of the way. TEN runs inside Docker, which is great because it means you don't have to install a bunch of dependencies on your machine. Think of Docker as a mini-computer inside your computer that comes pre-configured.

You'll need four sets of API keys — one for each service in the pipeline. Honestly, this is the most tedious part of the whole workshop. The actual building is the fun part.

What's the deal with Agora?

Agora is a real-time communication platform — think of it as the infrastructure that Clubhouse was built on. In TEN, Agora handles the WebRTC transport: getting audio from your browser's microphone to the server and back with minimal latency. You need both an App ID (identifies your project) and an App Certificate (for token-based authentication). The free tier is generous — 10,000 minutes per month, which is way more than you'll use in development.

Launch the Voice Assistant

Here's where it gets real. We're going to clone the repo, drop in our .env file, and spin up Docker containers. TEN's Docker setup gives you a development environment with everything pre-installed — the framework, the build tools, and a web portal to interact with your agent.

The process is: clone → configure → docker compose up → build inside container → run. About 10 minutes of waiting for Docker to do its thing, then you're talking to your agent.

What the agent gives back

The agent should give you a clean sequence: clone the repo, cd into ai_agents, copy .env.example to .env (and fill in your keys), run docker compose up -d, then docker exec -it ten_agent_dev bash to get inside the container. From there, cd into agents/examples/voice-assistant, run task install then task run. The agent should tell you to open localhost:3000 for the web portal and that the agent backend runs on port 49483. The whole Docker build takes about 5-8 minutes the first time.

Tip
The first docker compose up downloads a lot of stuff (~2-4 GB). Grab coffee. Subsequent runs are fast because Docker caches the layers.
Warning
Make sure Docker has at least 4 GB of RAM allocated. On Mac, check Docker Desktop → Settings → Resources. If builds fail with cryptic errors, low memory is usually the culprit.
What's happening inside the Docker container?

The Docker container bundles the TEN framework runtime, a Go/Python/Node build environment, and a web server. When you run task install, it pulls dependencies for the specific agent example you chose. When you run task run, it:

  1. Starts the TEN runtime (the orchestrator that connects all extensions)
  2. Loads the agent graph (which extensions to use and how they're connected)
  3. Starts a web server that serves the portal UI
  4. Listens for WebRTC connections from your browser

The 'graph' is the key concept — it's a JSON config that defines the pipeline. The default voice assistant graph connects: Agora RTC input → Deepgram STT → OpenAI LLM → ElevenLabs TTS → Agora RTC output. Each arrow is a data connection managed by TEN.

voice-assistant vs voice-assistant-realtime — what's the difference?

The voice-assistant example uses the 'chained' approach: STT → LLM → TTS as separate steps. The voice-assistant-realtime example uses OpenAI's real-time voice API, which handles STT + LLM + TTS in a single model. The realtime version has lower latency but less flexibility (you can't swap in different STT/TTS providers). Start with the chained version — it teaches you more about how the pipeline works.

Customize Your Agent's Personality and Behavior

Cool, your agent talks. But right now it's just... generic. It's like having a conversation with a default Siri. Let's make it interesting.

TEN uses a graph-based configuration — think of it as a wiring diagram that says "connect this microphone input to this speech recognizer, connect that to this LLM, and so on." Each node in the graph is an extension, and each extension has properties you can configure.

The system prompt for the LLM, the voice selection for TTS, the language for STT — it's all configurable through properties. And here's the fun part: you don't need to write this config by hand. Ask your agent to help you design the personality and generate the right config changes.

Ask your agent
Ask your agent to help you customize the voice assistant — give it a persona, change the voice, and adjust the system prompt to make it behave like a specific character or specialist.
Think about it
  • What kind of assistant would be fun or useful? A sarcastic cooking assistant? A patient math tutor? A dramatic storyteller?
  • What properties would you need to change to affect the LLM's personality vs. the voice it speaks with?
  • How would you ask about where these configuration files live in the TEN project structure?
What the agent gives back

The agent should explain that the personality lives in the OpenAI extension's system_message property, and the voice lives in the ElevenLabs extension's voice configuration. These are typically in a property.json file inside the agent example directory. The agent might suggest a system prompt like: 'You are a friendly barista named Alex who helps people discover new coffee drinks. Keep responses under 2 sentences — you're in a busy café.' It should also mention that ElevenLabs has dozens of pre-built voices you can select by name or ID.

Tip
For voice agents, shorter responses are better. Add instructions like 'Keep responses under 2-3 sentences' to your system prompt. Nobody wants to listen to a 30-second monologue when they asked a simple question.
How does the graph configuration actually work?

TEN's graph is essentially a directed data flow. Each extension declares what data it can receive (inputs) and what it produces (outputs). The graph config connects outputs to inputs:

  • Agora RTC extension outputs raw audio frames
  • Deepgram STT extension inputs audio frames, outputs text transcripts
  • OpenAI LLM extension inputs text, outputs text (streamed)
  • ElevenLabs TTS extension inputs text, outputs audio frames
  • Agora RTC extension inputs audio frames (to send back to browser)

You can insert new nodes anywhere in this chain. Want to log every conversation? Add a logging extension between STT and LLM. Want to translate? Add a translation extension between LLM and TTS. That's the power of the graph model.

5

Extend Your Agent with Memory and Tools

Right now your agent has the memory of a goldfish. Every time you start a new session, it forgets everything. And it can't do anything — it can only talk. Let's fix both.

TEN has an extension system that lets you plug in new capabilities. Memory extensions give your agent the ability to remember previous conversations. Tool extensions let your agent take actions — check the weather, look up information, or call external APIs.

This is where it gets really powerful. Instead of building these extensions from scratch, we're going to ask our AI agent to help us understand the extension interface and design a custom tool.

Ask your agent
Ask your agent to explain how TEN's extension system works and help you design a simple custom tool extension — something like a weather lookup or a knowledge base query.
Think about it
  • What would you need to know about the extension interface to build a new capability?
  • How does memory work in a stateless pipeline — where does the conversation history get stored and injected?
  • If you wanted your agent to check the weather, what would the data flow look like through the graph?
What the agent gives back

The agent should explain that TEN extensions follow a standard interface — they register what commands they handle and what data they produce. A memory extension typically stores conversation turns in a database (or even a simple file) and injects recent history into the LLM's context before each response. For a custom tool, the agent should describe how OpenAI's function calling works with TEN: the LLM decides it needs to call a tool, TEN routes that to your tool extension, your extension does the work (like calling a weather API), and sends the result back to the LLM to incorporate into its response. The agent should emphasize that the LLM handles the 'when to use this tool' logic — you just need to define what the tool does.

Tip
Start with the memory extension before building custom tools. A voice agent that remembers your name and preferences feels dramatically more human than one that doesn't.
What's the difference between memory and context?

Context is the information the LLM sees right now — the system prompt plus recent conversation turns. It's like short-term memory, limited by the model's context window.

Memory (in the TEN sense) is persistent storage that survives across sessions. A memory extension might store every conversation in a database, then retrieve relevant past conversations and inject them into the context. It's like the difference between remembering what someone said 5 minutes ago (context) vs. remembering that they mentioned they're allergic to nuts last week (memory).

The memory extension acts as a middleware that enriches the LLM's context with relevant historical information before each turn.

Recap

Let's step back and appreciate what we just did. We took four separate AI services — a speech recognizer, a language model, a voice synthesizer, and a real-time transport layer — and wired them into a conversational voice agent that runs in your browser.

The key insight isn't any single technology. It's the pipeline thinking: every real-time voice agent is fundamentally STT → LLM → TTS, and the engineering challenge is making the hand-offs between stages seamless enough that it feels like a conversation instead of a series of API calls.

TEN abstracts away the hardest parts: WebRTC negotiation, audio buffering, voice activity detection, interruption handling. What you're left with is a graph-based system where you plug in the services you want, configure their properties, and focus on making your agent actually useful.

The most powerful thing you built today isn't the agent itself — it's the mental model for how to build one. Next time you need a voice interface for anything, you know the pieces, you know how they connect, and you know how to ask an AI agent to help you wire it all up.

  • Real-time voice agents are a three-stage pipeline (STT → LLM → TTS) where streaming at every stage is critical for low latency
  • TEN Framework handles the hard orchestration — WebRTC, VAD, turn detection, interruptions — so you can focus on the agent's behavior and personality
  • The graph-based extension system means you can swap providers, add memory, or plug in custom tools without rewriting the pipeline
  • Voice agents need shorter responses than text agents — 2-3 sentences max for natural conversation flow
  • Docker-based development means your entire setup is reproducible and shareable — anyone can run your agent with just a .env file

Where to go next

  • Try swapping OpenAI for another LLM provider (like Anthropic's Claude) to compare response quality and latency
  • Explore the voice-assistant-realtime example that uses OpenAI's native real-time voice API for even lower latency
  • Build a custom tool extension that connects to an external API — weather, calendar, or a knowledge base
  • Experiment with speaker diarization to build a multi-user voice agent that knows who's talking
  • Deploy your agent to a cloud server so others can access it — TEN's docs cover deployment on various cloud services
