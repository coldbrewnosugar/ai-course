Build a RAG-Powered Long-Form Writing System That Never Forgets

Architect a context management pipeline that maintains coherence across massive document generation
40 min RAG context-engineering long-form-generation entity-graphs claude-code

What's happening

Large language models are remarkably capable sentence-by-sentence generators, but they possess no durable memory. Ask a model to produce a 50-chapter novel, and by chapter 12 it will have forgotten the protagonist's surname, invented a second love interest, or contradicted a plot point established three chapters earlier. This is not a hypothetical failure mode — it is the central obstacle to any form of long-form sequential generation, whether creative writing, technical documentation, legal drafting, or report synthesis.

The open-source project webnovel-writer (trending on GitHub this week) addresses this problem head-on. Built as a Claude Code plugin, it supports serialized fiction at the 2-million-character scale — roughly the length of the entire Harry Potter series. Its approach is architectural rather than prompt-engineering cosmetic: a full RAG pipeline with embedding and reranking, an entity graph that tracks characters and their relationships, and context snapshots that preserve volume-level coherence.

The underlying techniques — retrieval-augmented generation with hybrid search, entity tracking, and hierarchical context management — are applicable far beyond creative writing. Any task that requires an LLM to produce consistent output across many sequential generation steps faces the same fundamental constraint: the context window is finite, but the document is not. This session examines how webnovel-writer solves that constraint and guides you through building a simplified version of the same architecture.

1

Understanding the Forgetting Problem: Why Context Windows Are Not Enough

Before building anything, we must understand precisely what problem we are solving.

A language model's context window — even a large one — is a fixed-size buffer. When generating chapter 30 of a long document, the model cannot simultaneously hold chapters 1 through 29 in its context. Even if it could, most of the content in those earlier chapters is irrelevant to the sentence currently being generated. The challenge is therefore twofold: selection (which prior content matters right now?) and consistency (does the new content contradict anything established earlier?).

Consider a concrete example. A character named Lin Wei is introduced in chapter 3 as a retired naval officer with a prosthetic left hand. By chapter 25, the system must know — without re-reading all intervening chapters — that Lin Wei exists, what her background is, and what physical attributes have been established. If the generation step for chapter 25 mentions Lin Wei catching a ball with her left hand, something has gone wrong.

The webnovel-writer project decomposes this into three subsystems:

  1. Embedding + Reranking Pipeline — converts prior chapters into vector embeddings, retrieves the most relevant passages for the current generation context, then reranks them for precision.
  2. Entity Graph — a structured knowledge base tracking characters, locations, objects, and their interrelationships, updated after each chapter.
  3. Context Snapshots — volume-level outline summaries that preserve high-level narrative structure without consuming excessive context space.

These three systems work in concert: the entity graph answers "what facts must remain consistent," the embedding pipeline answers "what prior passages are relevant," and the context snapshots answer "where are we in the larger structure."

Ask your agent
Have your agent design the data model for an entity tracking system — the structured representation of entities (characters, locations, objects) and their relationships that persists across generation steps.
Think about it
  • What attributes does each entity need? Consider not just the name, but when and where the entity was introduced, what properties have been established, and which chapters reference it.
  • How should relationships between entities be represented? A character might be related to another character, associated with a location, or linked to an object.
  • What operations will you need to perform on this data structure? Think about lookup by name, retrieval of all entities mentioned in a given chapter, and conflict detection.
  • How should this data be serialized for storage — and how compact does it need to be to fit within a context window when retrieved?
What the agent gives back

The agent should produce a concise data model specification — likely as a set of Python dataclasses or TypeScript interfaces — defining an Entity with fields for name, type (character/location/object), a dictionary of established attributes (e.g., {"occupation": "retired naval officer", "chapter_introduced": 3}), a list of chapter references, and a relationships list linking to other entities with labeled edges (e.g., "sibling_of", "located_in"). The model should also include an EntityGraph class with methods for adding entities, querying by name or chapter, and detecting attribute conflicts. The total specification should be compact — a schema definition, not a full implementation.

Tip
When prompting your agent, specify the purpose of the data model, not just the shape. Saying "design a data model for tracking entities across a 50-chapter document to prevent contradictions" is far more effective than "create a class with name, type, and attributes fields." The agent will make better design decisions when it understands the constraint environment.
Why an entity graph rather than just full-text search?

Full-text search can find prior mentions of "Lin Wei," but it cannot distinguish between the sentence "Lin Wei lost her left hand in the naval accident" (an established fact) and "Lin Wei reached out with both hands" (a narrative action that contradicts the established fact). An entity graph extracts and structures the canonical facts about each entity, making contradiction detection a simple attribute-comparison operation rather than a natural-language inference problem. This is the difference between retrieval (finding relevant text) and knowledge management (maintaining structured, queryable facts).

2

Building the Retrieval Pipeline: Embeddings, BM25, and Reranking

With an entity graph handling structured facts, we now need a retrieval system for unstructured context — the narrative passages, descriptions, dialogue, and exposition that give a document its texture. This is where RAG (Retrieval-Augmented Generation) enters.

The webnovel-writer project employs a hybrid retrieval strategy with three layers:

  1. Semantic search via embeddings — each chapter (or chunk thereof) is converted to a vector embedding using a model such as Qwen3-Embedding-8B. At generation time, a query embedding is produced from the current outline or prompt, and the nearest neighbors in embedding space are retrieved.
  2. Lexical search via BM25 — a traditional keyword-based ranking algorithm that excels at exact-match retrieval. If the current chapter mentions a specific proper noun or technical term, BM25 will find prior occurrences even when the embedding model misses the semantic connection.
  3. Reranking — the combined candidate set from semantic and lexical search is passed through a cross-encoder reranker (e.g., Jina Reranker v3) that scores each candidate against the query with full attention, producing a final ranked list.

This three-layer approach is described in the project as auto/graph_hybrid mode with BM25 fallback. The "auto" designation means the system decides dynamically whether to rely on semantic search alone or invoke the BM25 fallback based on the nature of the query.

The critical insight is that no single retrieval method is sufficient. Embedding search captures thematic relevance ("scenes involving betrayal") but may miss specific names. BM25 captures exact terms but misses paraphrases. The reranker reconciles both, producing a final context window that balances precision and recall.

Ask your agent
Have your agent build a hybrid retrieval module that combines embedding-based semantic search with BM25 keyword search and a reranking step, suitable for retrieving relevant prior chapters during document generation.
Think about it
  • What are the inputs and outputs of this module? The input is a query (the current chapter outline or generation prompt) and a corpus (all prior chapters). The output is a ranked list of the most relevant passages.
  • How should chapters be chunked before embedding? Consider whether whole chapters, paragraphs, or fixed-size windows are most appropriate — and what trade-offs each choice introduces.
  • The module needs to call three external services: an embedding API, a BM25 index, and a reranking API. How should these be abstracted so the module remains testable and the models remain swappable?
  • How should the results from semantic search and BM25 be merged before reranking? Consider score normalization — BM25 scores and cosine similarities are on entirely different scales.
What the agent gives back

The agent should produce a retrieval module with three clearly separated stages. First, a function that takes a query string and returns the top-K semantically similar chunks using an embedding API (with the embedding model and endpoint configurable via environment variables). Second, a BM25 index built from the same corpus that returns top-K lexical matches. Third, a reranking function that takes the union of both candidate sets, calls a reranking API, and returns the final sorted results. The key architectural decision is the interface between stages — each should accept and return a common format (e.g., a list of scored passage objects) so stages can be composed, tested, or replaced independently.

API Key Note
The webnovel-writer project uses ModelScope for embeddings (Qwen3-Embedding-8B) and Jina AI for reranking (jina-reranker-v3). Both offer free tiers suitable for experimentation. You will need API keys for both services, configured via environment variables: EMBED_BASE_URL, EMBED_MODEL, EMBED_API_KEY, RERANK_BASE_URL, RERANK_MODEL, RERANK_API_KEY.
Tip
When prompting your agent, explicitly state that you want the three retrieval stages to be independent and composable. Without this guidance, the agent may produce a monolithic function that intermixes embedding calls, BM25 scoring, and reranking logic — making it difficult to test, debug, or swap components later.
Why BM25 as a fallback rather than a co-equal retrieval path?

In the webnovel-writer architecture, BM25 serves as a fallback rather than a primary retrieval path because semantic search handles the majority of retrieval needs adequately. BM25 becomes essential in specific scenarios: when a rare proper noun appears that the embedding model has not seen during training, when an exact technical term must be matched, or when the embedding service is temporarily unavailable. Treating BM25 as a fallback rather than running both systems for every query reduces latency and API costs during normal operation while preserving robustness for edge cases. The auto mode in webnovel-writer dynamically determines when to activate this fallback based on query characteristics.

What is a cross-encoder reranker and why does it outperform cosine similarity?

An embedding-based retrieval system (a "bi-encoder") independently encodes the query and each document, then compares them via cosine similarity. This is fast but approximate — the query and document never directly attend to each other. A cross-encoder reranker takes a (query, document) pair as a single input and processes both through the same transformer with full cross-attention. This produces dramatically more accurate relevance scores, but at higher computational cost — which is why rerankers are applied only to a small candidate set (typically 20–50 passages) rather than the full corpus. The two-stage pipeline (fast retrieval → precise reranking) combines the efficiency of bi-encoders with the accuracy of cross-encoders.

At this point, you should have two complementary subsystems designed: an entity graph that tracks structured facts about characters, locations, and objects across chapters, and a hybrid retrieval pipeline that finds relevant prior passages using embeddings, BM25, and reranking. These two systems address different aspects of the forgetting problem — the entity graph handles consistency of established facts, while the retrieval pipeline provides relevant narrative context.

Context Snapshots: Maintaining Structural Coherence Across Volumes

The entity graph tracks individual facts. The retrieval pipeline surfaces relevant passages. But neither addresses a third dimension of coherence: structural awareness — knowing where you are in the larger document and what the high-level plan dictates.

Imagine generating chapter 45 of a 60-chapter novel. The entity graph tells you that character X is alive and in city Y. The retrieval pipeline provides the most relevant prior passages. But neither tells you that, according to the volume outline, this chapter is supposed to be the climactic confrontation — not a quiet interlude. Without structural context, the model may generate tonally or narratively inappropriate content that is locally coherent but globally wrong.

The webnovel-writer project addresses this with context snapshots — compressed summaries of the volume-level outline that are injected into the generation context alongside retrieved passages and entity data. These snapshots capture:

  • The volume's narrative arc and current position within it
  • Chapter-level summaries for recently completed chapters
  • Planned plot points for upcoming chapters
  • Tonal and pacing directives

The key design challenge is compression. A volume outline for 30 chapters might be several thousand words. A context snapshot must distill this to a few hundred words — enough to orient the model without consuming precious context space. This is, in effect, a summarization problem: what is the minimum representation of structural context that prevents the model from going off-plan?

Ask your agent
Have your agent design a context snapshot system that compresses volume-level outlines into compact representations suitable for injection into a generation prompt, along with a snapshot manager that updates these summaries as chapters are completed.
Think about it
  • What information from a volume outline is essential for generating the *current* chapter, versus information that is merely archival? Think about which details the model needs right now versus which can be omitted.
  • How should the snapshot change as chapters are completed? A snapshot for chapter 5 of a 30-chapter volume should look different from a snapshot for chapter 28 — the balance between 'what has happened' and 'what is planned' shifts.
  • What is the target size for a snapshot in tokens? Consider that this snapshot will share the context window with entity data, retrieved passages, and the generation prompt itself.
  • Should the snapshot be generated by an LLM (summarizing the full outline) or constructed programmatically (extracting key fields from a structured outline)? What are the trade-offs?
What the agent gives back

The agent should produce a context snapshot system with two components. First, a snapshot schema that includes: the volume title and arc summary (1–2 sentences), a compressed summary of completed chapters (one line each for the most recent 5, omitting earlier ones), the current chapter's outline and objectives, and a brief forecast of the next 2–3 planned chapters. Second, a snapshot manager that, after each chapter is completed, regenerates the snapshot — either by calling an LLM to summarize the updated state, or by programmatically assembling it from structured outline data. The total snapshot should target 300–500 tokens, providing structural orientation without crowding out other context.

Warning
Context window budget management is the single most important architectural decision in a RAG-powered generation system. Every token allocated to the snapshot is a token unavailable for retrieved passages or entity data. When prompting your agent, specify an explicit token budget (e.g., "the snapshot must fit within 400 tokens") — without this constraint, the agent will likely produce a snapshot format that is informative but too large to be practical.
The sliding window pattern for chapter summaries

The webnovel-writer project employs a sliding window approach to chapter summaries within snapshots. Rather than summarizing all completed chapters (which would grow linearly and eventually overflow any budget), the snapshot maintains detailed summaries only for the most recent N chapters (typically 3–5) and progressively compressed summaries for earlier ones. Chapters more than 10 positions behind the current one are represented only by their one-sentence loglines. This mirrors how human authors work — they have vivid recall of what they just wrote, moderate recall of recent chapters, and only thematic recall of much earlier material. The sliding window ensures the snapshot remains within its token budget regardless of how many chapters have been completed.

Assembling the Generation Pipeline: From Retrieval to Prompt Construction

We now have three subsystems — entity graph, retrieval pipeline, and context snapshots. The final architectural challenge is orchestration: how these components are assembled into a single generation prompt for each chapter.

The webnovel-writer project implements this as a multi-agent pipeline within Claude Code. When the user invokes /webnovel-write 25 to generate chapter 25, the system executes the following sequence:

  1. Context Agent — queries the entity graph for all entities relevant to chapter 25 (based on the chapter outline), retrieves the most relevant prior passages via the hybrid retrieval pipeline, and loads the current context snapshot.
  2. Prompt Assembly — constructs a generation prompt that includes, in order: the context snapshot (structural orientation), relevant entity data (factual constraints), retrieved passages (narrative context), and the chapter outline (immediate instructions).
  3. Generation — the assembled prompt is sent to the LLM, which generates the chapter text.
  4. Post-Generation Update — newly introduced entities are added to the entity graph, existing entity attributes are updated, the chapter text is embedded and indexed for future retrieval, and the context snapshot is regenerated.

This four-phase cycle repeats for every chapter. The post-generation update is critical — without it, the system would gradually fall out of sync with its own output, leading to the same consistency failures it was designed to prevent.

The multi-agent architecture (webnovel-writer uses separate Claude Code agents for context retrieval, writing, and review) allows each phase to be independently configured — including which model to use. The context agent might run on a faster, cheaper model since its task is retrieval and assembly, while the writing agent might use a more capable model for generation quality.

Ask your agent
Have your agent design the orchestration layer — the pipeline that, given a chapter number and outline, assembles the full generation context from the entity graph, retrieval pipeline, and context snapshot, constructs the prompt, invokes the LLM, and performs the post-generation update.
Think about it
  • What is the sequence of operations, and which steps depend on the outputs of prior steps? Some operations (e.g., entity lookup and passage retrieval) can run in parallel; others (e.g., post-generation entity extraction) must wait for the generation to complete.
  • How should the prompt template be structured? Consider that the model needs clear section boundaries — it must understand which part of the prompt is structural context, which is factual constraints, and which is the actual writing instruction.
  • What happens when the retrieval pipeline returns no relevant passages — perhaps because this chapter introduces an entirely new setting? The orchestration layer needs a graceful fallback.
  • How should errors in the subsystems (embedding API timeout, entity graph corruption) be handled without aborting the entire generation?
What the agent gives back

The agent should produce an orchestration module — a pipeline function or class that accepts a chapter number and outline as input, then executes a defined sequence: (1) parallel retrieval of entity data and relevant passages, (2) loading the current context snapshot, (3) assembly of these components into a structured prompt with labeled sections, (4) invocation of the generation model, (5) post-generation extraction of new entities and facts, (6) updating the entity graph, embedding index, and context snapshot. The design should specify error handling (e.g., proceed without retrieved passages if the embedding API times out) and should define the prompt template structure — likely using XML-style tags or markdown headers to delineate context sections for the model.

Tip
The prompt template structure is where this system's value is realized. When prompting your agent to design it, emphasize that the template must make the role of each context section explicit to the model. A section labeled ## Established Entity Facts — Do Not Contradict is far more effective than simply pasting entity data into the prompt without framing. The model must understand that certain context is prescriptive (must be obeyed) versus descriptive (for reference).
The review agent and feedback loops

The webnovel-writer project includes a review phase (/webnovel-review) that evaluates generated chapters for consistency, pacing, and adherence to the outline. This is implemented as a separate agent that reads the generated chapter alongside the entity graph and outline, then produces a critique. If issues are found, the chapter can be regenerated with the critique appended to the prompt — creating a feedback loop. This pattern (generate → review → optionally regenerate) adds a quality gate that catches errors the retrieval-and-constraint system missed. It is analogous to the distinction between compile-time checks (the entity graph prevents known contradictions) and runtime testing (the review agent catches emergent problems).

Why multi-agent architecture rather than a single monolithic prompt?

A single prompt that performs retrieval, generation, and self-review would need to be extraordinarily long and would conflate distinct cognitive tasks. The multi-agent approach allows each agent to be specialized: the context agent focuses on retrieval and assembly (a task well-suited to structured reasoning), the writing agent focuses on creative generation (a task requiring fluency and style), and the review agent focuses on analytical evaluation (a task requiring critical reading). Each agent can use a different model, different temperature settings, and different system prompts optimized for its specific role. This decomposition is a direct application of the software engineering principle of separation of concerns.

Operational Considerations: Embedding Lifecycle and Recovery

A production context management system must handle operational realities that architectural diagrams omit. Three merit specific attention.

Embedding drift and reindexing. When you change your embedding model (upgrading from one version to another, or switching providers), all existing vectors become incompatible with new query vectors. The entire corpus must be re-embedded — a potentially expensive and time-consuming operation. The webnovel-writer project manages this implicitly by tying the embedding configuration to environment variables (EMBED_MODEL, EMBED_BASE_URL), but a production system needs an explicit reindexing strategy: versioned embedding stores, migration scripts, and ideally the ability to run old and new embeddings in parallel during transition.

Entity graph corruption and recovery. The entity graph is updated after every generation step. If a post-generation update incorrectly extracts an entity attribute (perhaps the model hallucinates a detail during extraction), that incorrect fact propagates into all subsequent chapters. The webnovel-writer project includes operational recovery procedures, and the /webnovel-review command serves as a partial safeguard. A robust system should version the entity graph, allowing rollback to a known-good state.

Cost management. Every generation step involves multiple API calls: embedding the query, retrieving candidates, calling the reranker, and generating the chapter itself. For a 50-chapter document, this means 200+ API calls at minimum. The auto mode in webnovel-writer's retrieval — which activates BM25 fallback only when needed rather than for every query — is a cost-conscious design decision. When prompting your agent to build these systems, specify cost constraints explicitly; otherwise the agent will optimize for quality alone.

Warning
Entity extraction from generated text is the most fragile link in this architecture. The post-generation step relies on an LLM to identify new entities and attributes in the generated chapter — but LLMs can hallucinate during extraction just as they can during generation. Consider having your agent add a human-in-the-loop confirmation step for entity graph updates, at least for high-stakes attributes (character deaths, relationship changes, key plot points) that would be difficult to correct after propagation.
Tip
The webnovel-writer project includes a visualization dashboard (/webnovel-dashboard) that provides read-only views of the entity graph, chapter status, and outline structure. When building your own system, invest early in observability — the ability to inspect the entity graph and retrieval results is essential for diagnosing coherence failures.
The economics of hybrid retrieval

Consider the cost profile for generating a single chapter: one embedding call for the query vector (~$0.001), one vector similarity search (free if self-hosted, negligible if using a managed service), one BM25 search (free, runs locally), one reranking call with ~30 candidates (~$0.005), and the chapter generation itself (~$0.05–0.50 depending on model and length). The retrieval overhead is roughly 1–10% of the generation cost — a modest premium for a substantial improvement in coherence. The expensive component is not retrieval but re-embedding the entire corpus when the embedding model changes, which is a one-time migration cost rather than a per-chapter cost.

Recap

We have examined the architecture of a system that solves one of the most persistent problems in LLM-powered generation: maintaining coherence across documents that exceed the context window. The webnovel-writer project demonstrates that this is fundamentally an information architecture problem, not a prompt engineering problem.

The solution decomposes into four interdependent components: an entity graph that maintains structured, queryable facts about entities and their relationships; a hybrid retrieval pipeline (embedding + BM25 + reranking) that surfaces relevant prior content; context snapshots that preserve awareness of the document's high-level structure; and an orchestration layer that assembles these into a generation prompt and updates all subsystems after each generation step.

This architecture is domain-agnostic. While webnovel-writer applies it to serialized fiction, the same pattern applies to any sequential generation task where consistency matters: technical documentation, legal document drafting, research paper series, educational curricula, or game narrative design. The core principle is the same in all cases: separate the concerns of factual consistency, contextual relevance, and structural awareness, then compose them at generation time.

  • Context windows are finite, but documents are not — RAG bridges this gap by retrieving relevant prior content rather than attempting to hold everything in memory simultaneously.
  • Hybrid retrieval (semantic + lexical + reranking) outperforms any single retrieval method because different types of relevance (thematic, terminological, structural) require different matching strategies.
  • An entity graph converts the open-ended problem of 'consistency checking' into the tractable problem of 'attribute comparison' — structured facts are easier to enforce than unstructured narrative coherence.
  • Context budget management — deciding how to allocate limited context space among entity data, retrieved passages, and structural snapshots — is the most consequential design decision in any RAG-powered generation system.
  • Post-generation update is essential: the system must learn from its own output, or it will gradually diverge from the document it is producing.
