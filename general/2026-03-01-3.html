Your Agent Needs a Sandbox (Before It Breaks Something)

Alibaba just open-sourced OpenSandbox — a platform that gives AI agents their own isolated playgrounds with shell access, file systems, and code execution. Let's set it up and put it to work.
35 min sandbox agent-infrastructure code-execution docker security

What's happening

Here's the problem nobody talks about enough: when you give an AI coding agent the ability to run code on your machine, you're basically handing a toddler a chainsaw. It might build something beautiful. It also might rm -rf your home directory.

Alibaba just dropped OpenSandbox as an open-source project, and it solves exactly this. Think of it as a hotel room for your AI agent — it gets its own filesystem, its own shell, its own code interpreter, all running inside a Docker container. When it checks out, everything gets cleaned up. No mess left behind.

Why this matters right now:

  • Coding agents are everywhere — Claude Code, Cursor, Copilot Workspace — and they all need safe execution environments
  • Agent evals require isolation — you can't benchmark 50 agents if they can stomp on each other's files
  • Network egress controls are the unsung hero here — you can decide exactly which URLs your sandbox can reach, so your agent can't exfiltrate data or call APIs it shouldn't

OpenSandbox gives you multi-language SDKs (Python, JS, Go, Java, C#), works with both Docker locally and Kubernetes at scale, and has a proper protocol for sandbox lifecycle management. It's the kind of boring-but-essential infrastructure that makes agent development actually safe.

1

Get the Sandbox Server Running Locally

Before we can sandbox anything, we need the server running. OpenSandbox uses Docker under the hood — each sandbox is literally a container. The server manages the lifecycle: creating sandboxes, routing commands into them, and tearing them down.

The mental model here is like a hotel front desk. The server is the concierge — you tell it "I need a room with Python 3.11 and a code interpreter," and it spins up a container matching that description. When you're done, you check out, and the room gets cleaned.

Let's get your agent to help us set this up.

What the agent gives back

The agent should give you a clear 3-step process: (1) make sure Docker is running, (2) pip install opensandbox-server, (3) generate the config with opensandbox-server init-config ~/.sandbox.toml --example docker, and (4) start the server with opensandbox-server. It might also mention Python 3.10+ as a requirement. The config file at ~/.sandbox.toml is where all the runtime settings live — you'll want the docker example template for local development.

Warning
Docker Desktop must be running before you start the server. If you're on Linux, regular Docker Engine works fine. The server talks directly to the Docker daemon to manage containers.
Tip
You can also clone the full repo and run from source if you want to hack on the server itself, but for just using it, the pip package is way simpler.
What's actually in that .sandbox.toml config?

The config file tells the server which runtime to use (Docker or Kubernetes), where to find container images, network policies, resource limits, and timeout defaults. The --example docker flag gives you sensible defaults for local dev — things like a 10-minute sandbox timeout and no egress restrictions. In production, you'd lock this down significantly. Think of it as the hotel's house rules.

2

Spin Up Your First Sandbox and Run Code In It

Now the fun part. We're going to create a sandbox, run a shell command in it, write and read files, and then tear it down. This is the core loop you'll use for everything — whether you're building a coding agent, running evaluations, or just experimenting.

Here's the key concept: a sandbox is not the same as a code interpreter. A sandbox is the isolated container (filesystem + shell). A code interpreter is a service inside the sandbox that understands how to execute Python, JavaScript, etc. and return structured results. You create the sandbox first, then optionally create a code interpreter inside it.

It's like the difference between renting an apartment (sandbox) and setting up a home office inside it (code interpreter). The apartment gives you walls and a door. The office gives you a desk to actually work at.

Ask your agent
Get the agent to write a Python script that creates a sandbox, runs a shell command, writes a file, reads it back, and then cleans up
Think about it
  • The SDK package is called `opensandbox` — what class manages the sandbox lifecycle?
  • Everything is async — how should we structure the script?
  • What container image should we use? (Hint: OpenSandbox has a pre-built code interpreter image)
  • How do we make sure the sandbox gets cleaned up even if something fails?
What the agent gives back

The agent should give you an async Python script using the opensandbox SDK. The key pieces: Sandbox.create() takes an image name like opensandbox/code-interpreter:v1.0.1, environment variables (like Python version), and a timeout. You use async with sandbox: to ensure cleanup. Inside, sandbox.commands.run() executes shell commands, sandbox.files.write_files() writes files, and sandbox.files.read_file() reads them back. The whole thing runs inside asyncio.run(). It's maybe 15-20 lines total but very readable.

Tip
The async with sandbox: pattern is your best friend. It guarantees the container gets killed even if your code throws an exception. Always use it instead of manually calling sandbox.kill().
Why is everything async?

Creating containers, executing commands, reading files — these all involve talking to the Docker daemon over a socket, which means network I/O. Async lets you fire off multiple operations without blocking. This becomes critical when you're managing dozens of sandboxes simultaneously (like during agent evaluations). If this were synchronous, creating 50 sandboxes would take 50x longer because you'd be waiting for each one sequentially.

What images are available out of the box?

OpenSandbox ships with opensandbox/code-interpreter:v1.0.1 which includes Python, Node.js, and basic system tools. But here's the cool part — you can use any Docker image. Need a sandbox with Rust? Point it at a Rust image. Need Chrome for browser automation? There are examples using Playwright in a sandbox. The image is just the starting state of the filesystem.

✓
At this point, you should have the OpenSandbox server running locally and be able to create a sandbox, run commands in it, and manage files. The server is your concierge, the sandbox is the hotel room, and you're about to furnish it with a code interpreter.
3

Add a Code Interpreter — The Agent's Actual Workbench

Shell commands are great, but agents need something more structured. When an agent writes Python code and sends it for execution, it doesn't want to parse raw stdout — it wants structured results, captured output, and even rendered charts.

That's what the Code Interpreter SDK (opensandbox-code-interpreter) gives you. It sits on top of a sandbox and provides a proper interface: send code in, get results back as structured objects with stdout, stderr, return values, and even binary outputs like images.

Think of it this way: the sandbox is a blank room. The code interpreter is a lab bench inside that room — with proper instruments, safety goggles, and a notebook to record results.

Think about it
  • The SDK is `opensandbox-code-interpreter` — what's the main class?
  • How do you specify which language to execute? (Hint: there's a `SupportedLanguage` enum)
  • The result object has `.result` and `.logs` — what's the difference?
  • Can you run multiple code blocks in sequence and have them share state?
What the agent gives back

The agent should show you how to create a CodeInterpreter from an existing sandbox using CodeInterpreter.create(sandbox), then use interpreter.codes.run() to execute Python code. The key insight: .result contains the return value of the last expression (like a REPL), while .logs.stdout contains anything that was print()-ed. The language is specified via SupportedLanguage.PYTHON. The whole addition is maybe 5-6 lines on top of what you already have.

API Key Note
No API keys needed for OpenSandbox itself — it's fully local. But if your sandboxed code calls external APIs (like OpenAI), those keys need to be passed as environment variables when creating the sandbox.
What languages does the code interpreter support?

Out of the box, the code interpreter image supports Python (3.11 by default) and can be configured for other languages. The SupportedLanguage enum tells you what's available. But remember — you can always fall back to sandbox.commands.run() for any language that has a CLI compiler/interpreter. The code interpreter just gives you a nicer structured interface.

How does state work between code runs?

Each interpreter.codes.run() call within the same interpreter shares state — variables defined in one run are available in the next. This is exactly how Jupyter notebooks work, and it's intentional. Agents often need to build up state incrementally: install a package, import it, load data, then analyze it. If each run was isolated, this workflow would break.

4

Lock It Down — Network Policies and Sandbox Security

Here's where OpenSandbox goes from "cool toy" to "production-ready infrastructure." Any sandbox platform can run code in a container. What makes this one interesting is the network policy layer.

OpenSandbox has two networking concepts:

  • Ingress Gateway — how the outside world reaches into sandboxes (multiple routing strategies)
  • Egress Controls — how sandboxes reach the outside world (per-sandbox URL whitelists)

The egress part is the security game-changer. Imagine you're running an AI agent that writes and executes code based on user prompts. That code could try to curl your internal APIs, send data to an attacker's server, or download malware. Egress controls let you say: "This sandbox can reach pypi.org and nothing else."

It's like giving someone a phone that can only call numbers on an approved list. They can use it for legitimate purposes but can't dial out to anyone sketchy.

Ask your agent
Get the agent to explain how to configure egress network policies in OpenSandbox — both at the server config level and per-sandbox
Think about it
  • Where would network policies be configured — the .sandbox.toml file? Per-sandbox creation params? Both?
  • What granularity makes sense — block everything by default and whitelist, or allow everything and blacklist?
  • What domains would a typical coding sandbox need access to? (package registries, maybe an LLM API...)
  • How does this differ between Docker runtime (local dev) and Kubernetes runtime (production)?
What the agent gives back

The agent should explain that OpenSandbox supports both server-level default policies (in .sandbox.toml) and per-sandbox overrides at creation time. The recommended approach is deny-by-default with explicit whitelists — allow pypi.org, npmjs.org, and whatever API endpoints your agent needs, block everything else. On Kubernetes, network policies map to native K8s NetworkPolicy resources, which gives you battle-tested enforcement. On Docker, it uses iptables rules on the container's network namespace. The agent should note that the Ingress Gateway handles routing external traffic into sandboxes, which is useful for browser automation scenarios.

Warning
In local Docker mode, egress controls rely on iptables which requires the server to run with appropriate permissions. In production, always use Kubernetes where NetworkPolicies are enforced by the CNI plugin — much more robust.
Tip
Start with a completely locked-down sandbox (no egress), then add domains as your agent actually needs them. It's much easier to open holes than to close them after something leaks.
Real-world scenario: Agent evaluation harness

Imagine you're running SWE-bench evaluations on 500 coding agents simultaneously. Each agent gets its own sandbox with a git repo cloned inside. You want agents to be able to install pip packages but NOT access the internet otherwise (to prevent cheating by fetching solutions). With OpenSandbox on Kubernetes, you create a network policy allowing pypi.org:443 and nothing else, spin up 500 sandboxes with that policy, and let them rip. Each sandbox is fully isolated — they can't see each other, can't reach the internet, and automatically get cleaned up when the evaluation finishes.

Ingress Gateway deep dive

The Ingress Gateway is the reverse side of the coin — it's about getting traffic into sandboxes. This matters for browser automation: if you're running a headless Chrome inside a sandbox and want to access its DevTools port from outside, the gateway handles the routing. It supports multiple strategies so you can map external URLs to specific sandbox ports. Think of egress as 'what can the sandbox call out to' and ingress as 'what can call into the sandbox.'

You now understand the full OpenSandbox stack: the server manages sandbox lifecycles, the Python SDK creates and controls sandboxes, the Code Interpreter SDK gives agents a structured code execution interface, and network policies keep everything secure. Time to build something real with it.

Beyond Code — Browser Automation and Desktop Sandboxes

Here's where OpenSandbox gets wild. Code execution is table stakes. The examples directory shows sandboxes running headless Chrome with Playwright for browser automation, and even full desktop environments with VNC and VS Code inside containers.

Why? Because the next generation of AI agents aren't just writing code — they're browsing the web, filling out forms, using desktop apps. These "GUI agents" need the same isolation guarantees as coding agents, maybe even more so, since they're interacting with real websites.

The architecture stays the same: create a sandbox with the right image (one that has Chrome or a desktop environment), use the ingress gateway to access the GUI, and let the agent do its thing. The sandbox protocol is generic enough to handle all of this.

This is honestly the part that gets me most excited. We're going from "run this Python snippet safely" to "let the agent use a full computer safely." Same platform, same SDK, same security model.

  • How does the Ingress Gateway help here — what ports need to be exposed?
  • What egress domains should a browser agent be allowed to reach?
What the agent gives back

The agent should explain that you'd use a container image with Chromium and Playwright pre-installed, create the sandbox with that image, and use the ingress gateway to expose the browser's DevTools port (typically 9222) so the agent can control it via the Chrome DevTools Protocol. The agent interacts by taking screenshots and sending click/type commands — or by accessing the DOM directly via DevTools. For egress, you'd whitelist the specific domains the agent needs to browse. The setup code is almost identical to the code interpreter example — just a different image and entrypoint.

Tip
The VNC desktop sandbox is great for debugging — you can actually watch what the agent is doing in real-time through a browser-based VNC client. It's like having a security camera on your AI.
The bigger picture: Sandbox as agent infrastructure

OpenSandbox isn't just about safety — it's about reproducibility and scalability. When every agent interaction happens inside a sandbox, you get: (1) perfect isolation between runs, (2) the ability to snapshot and replay environments, (3) easy horizontal scaling on Kubernetes, and (4) clean resource accounting. This is why platforms like Modal, E2B, and now OpenSandbox exist — they're the cloud runtime layer that agent frameworks need but don't want to build themselves. If you're building anything where AI agents execute actions in the real world, you eventually need something like this.

Recap

We went from "AI agents running code on your bare metal is terrifying" to having a full understanding of how OpenSandbox creates isolated, secure, ephemeral environments for agent execution.

Here's the stack we explored:

  • Server — the orchestrator that manages sandbox lifecycles via Docker or Kubernetes
  • Sandbox SDK — create containers, run commands, manage files
  • Code Interpreter SDK — structured code execution with proper result objects
  • Network Policies — egress controls that prevent data exfiltration and unauthorized access
  • Beyond code — browser automation and desktop environments using the same platform

The key insight isn't about any specific API call — it's that agent infrastructure is becoming its own category. Just like web apps needed load balancers and databases, AI agents need sandboxes and execution environments. OpenSandbox is Alibaba's answer to this, joining E2B, Modal, and others in building the runtime layer for the agentic era.

  • A sandbox is an isolated container (filesystem + shell); a code interpreter is a structured execution service inside it — they're separate layers with separate SDKs
  • Egress network policies are the most important security feature — deny by default, whitelist what you need, and your agent can't phone home to places it shouldn't
  • The sandbox pattern (create → use → destroy) applies to way more than code execution — browser automation, desktop environments, and RL training all use the same lifecycle
  • Everything is async because you'll eventually manage many sandboxes concurrently — evaluations, multi-agent systems, and parallel user sessions all need this
  • OpenSandbox's protocol is generic enough that you can bring your own Docker images — it's not locked to Python or any specific toolchain

Where to go next

  • Try running the browser automation example from the OpenSandbox examples/ directory — it's a great way to see GUI agent sandboxing in action
  • Set up a Kubernetes cluster (even minikube) and test the K8s runtime — network policies are much more robust there than Docker iptables
  • Build a multi-sandbox evaluation harness: spin up 10 sandboxes simultaneously and benchmark how quickly they create and execute code
  • Integrate OpenSandbox into an existing agent framework (LangChain, CrewAI) as a custom tool — give your agents a 'run code safely' capability
