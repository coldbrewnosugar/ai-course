Unified Information Extraction with GLiNER2

One 205M-parameter model. Four extraction tasks. No GPU. No API keys. Just structured data from unstructured text.
35 min information-extraction NER text-classification relation-extraction local-inference

What's happening

Information extraction — the process of converting unstructured text into structured, machine-readable data — has historically required assembling multiple specialized models. A typical pipeline might chain a Named Entity Recognition model, a separate text classifier, and a custom relation extractor, each with its own dependencies, training data format, and inference quirks.

GLiNER2, released by Fastino AI, collapses this complexity into a single 205M-parameter model capable of four tasks: entity extraction, text classification, structured data parsing, and relation extraction. The model operates on CPU without external API dependencies, which makes it immediately relevant for privacy-sensitive applications, edge deployments, and rapid prototyping.

The architectural insight is schema-driven extraction: rather than training separate heads for each task, GLiNER2 accepts a declarative schema — essentially a description of what you want — and produces structured output conforming to that schema. This is conceptually similar to how large language models follow instructions, but implemented in a compact encoder model optimized for extraction rather than generation.

In this session, we will use an AI agent to build a complete extraction pipeline that exercises all four capabilities and combines them into a single-pass multi-task workflow. We will also examine how LoRA adapters enable domain-specific customization without full retraining.

1

Environment Setup and First Entity Extraction

Before we can explore GLiNER2's capabilities, we need a working environment with the library installed and a model loaded into memory. The base model (fastino/gliner2-base-v1) is 205M parameters — small enough that loading and inference are comfortable on any modern laptop CPU.

The fundamental pattern in GLiNER2 is: load the model once, then call task-specific methods with a text and a schema. The schema is the key design element — it tells the model what to look for without requiring any retraining. For entity extraction, the schema is simply a list of entity type labels (e.g., ["person", "organization", "date"]). For more precise extraction, the schema can include natural-language descriptions alongside each label.

This distinction between labels-only and labels-with-descriptions schemas is worth understanding early. Labels alone rely on the model's pretrained understanding of common categories. Descriptions allow you to inject domain-specific semantics — for example, distinguishing "medication" (a drug name) from "supplement" (a non-prescription compound) in medical text, even though both are substances.

Tip
Model loading is the most expensive operation. In production or interactive use, load the model once at startup and reuse the extractor instance across all calls. Each extraction call is fast — typically under 100ms on CPU for moderate-length text.
Warning

How does schema-driven extraction differ from traditional NER?

Traditional NER models are trained on a fixed label set (e.g., the CoNLL-2003 tags: PER, ORG, LOC, MISC). Adding a new entity type requires collecting annotated data and retraining. GLiNER2's schema-driven approach treats entity type labels as inputs to the model rather than outputs of a classification head. The model learns a general mapping between label semantics and text spans, enabling zero-shot generalization to novel entity types. This is analogous to how zero-shot text classifiers use label descriptions as prompts — the model matches text spans against label meanings rather than memorized categories.

2

Text Classification: Single-Label and Multi-Label

GLiNER2's text classification capability applies the same schema-driven approach to document-level labels rather than span-level entities. Given a text and a set of candidate categories, the model assigns one or more labels with confidence scores.

Two modes are available: single-label classification (mutually exclusive categories — e.g., sentiment as positive/negative/neutral) and multi-label classification (non-exclusive categories — e.g., tagging a news article as both "technology" and "business"). The distinction is controlled by a parameter, not by model architecture, which means you decide the classification semantics at inference time.

Confidence scores are particularly important for classification tasks because they enable threshold-based filtering. In a production pipeline, you might accept labels above 0.7 confidence and route ambiguous cases (0.4–0.7) to human review. The model's calibration — how well confidence scores reflect actual accuracy — varies by domain, so empirical threshold tuning is advisable.

Why does one model handle both NER and classification?

Both tasks reduce to the same underlying operation: matching a semantic label against text content. In NER, the match is localized to a specific span; in classification, it covers the entire document. GLiNER2's architecture uses a shared encoder that produces representations for both the input text and the schema labels, then computes compatibility scores. For NER, scores are computed per token span; for classification, scores are computed over the full sequence representation. This weight-sharing is why a single 205M-parameter model can serve both tasks without separate heads or training phases.

Structured Data Extraction and Relation Extraction

Entity extraction and classification produce flat outputs — lists of spans or labels. Many real-world applications require structured output: nested fields, typed attributes, and relationships between entities. GLiNER2 addresses this through two complementary capabilities.

Structured Data Extraction accepts a JSON-like schema definition and parses text into a conforming structure. Think of it as form-filling: you define the fields and their types, and the model populates them from the text. This is useful for extracting structured records from semi-structured documents — invoices, job postings, medical records.

Relation Extraction identifies typed connections between entity pairs. Given entity types and relation types, the model produces triples of the form (entity_1, relation, entity_2). For example, from "Tim Cook is the CEO of Apple," the model would extract (Tim Cook, CEO_of, Apple). This capability is essential for knowledge graph construction and any application requiring an understanding of how entities relate, not merely which entities appear.

These two features round out GLiNER2's four-task repertoire and represent the higher-complexity end of the extraction spectrum.

How does relation extraction scale with entity and relation type counts?

Relation extraction is computationally more expensive than NER or classification because it must evaluate potential relations between all entity pairs. If you define E entity types and R relation types, and the model finds N entity mentions, the search space is approximately N² × R. For short texts with a handful of entities, this is negligible. For long documents with dozens of entities and many relation types, consider chunking the text or narrowing the entity/relation type sets to maintain reasonable latency. The 205M base model handles moderate complexity well on CPU, but extreme cases may benefit from the larger 340M model or the API-hosted 1B model.

Combined Multi-Task Extraction in a Single Pass

Running entity extraction, classification, and relation extraction as separate calls is straightforward but involves repeated encoding of the same text. GLiNER2 supports combined schemas that specify multiple task types in a single call, enabling the model to perform all extractions in one forward pass.

A combined schema is a dictionary that includes keys for each task type — entities, classification labels, structured fields, and relations. The model processes the text once and returns a unified result object containing outputs for all specified tasks.

This is more than a convenience optimization. In production pipelines, single-pass extraction ensures consistency: the entity spans identified for relation extraction are guaranteed to match the entities returned in the NER output, because they were resolved in the same inference context. With separate calls, subtle differences in tokenization or span boundaries can create mismatches that require post-processing to reconcile.

Domain Adaptation with LoRA Adapters

The base GLiNER2 model provides strong zero-shot extraction across general domains, but specialized applications — legal document analysis, biomedical literature mining, financial report parsing — benefit from domain-specific fine-tuning. Full model retraining is expensive and risks catastrophic forgetting of general capabilities.

LoRA (Low-Rank Adaptation) offers an efficient alternative. Rather than updating all 205M parameters, LoRA freezes the base model and trains a small set of rank-decomposed weight matrices (typically 0.5–2% of total parameters) that adapt the model's behavior to a target domain. The resulting adapter is a lightweight file that can be loaded on top of the base model at inference time.

GLiNER2 supports training LoRA adapters from domain-specific annotated data and switching between adapters at runtime. This means a single deployment can serve multiple domains — medical extraction, legal extraction, financial extraction — by swapping adapters without reloading the base model.

The training data format follows a straightforward convention: each example is a text paired with the expected extraction output (entities, labels, or relations) in the same format the model produces at inference time.

Warning
LoRA adapter quality is highly sensitive to annotation consistency. Inconsistent labels in training data — the same span labeled as "drug" in one example and "medication" in another — will degrade adapter performance. Establish a clear annotation guide before collecting training data.
What is Low-Rank Adaptation, conceptually?

A neural network layer performs a matrix multiplication: output = input × W, where W is a large weight matrix. LoRA decomposes the update to W into two small matrices: ΔW = A × B, where A and B have much lower rank than W. During fine-tuning, only A and B are trained while W remains frozen. At inference time, the effective weight is W + A × B. Because A and B are small (e.g., rank 8 means A is d×8 and B is 8×d instead of d×d), the number of trainable parameters drops by orders of magnitude — from millions to thousands. The key insight is that task-specific adaptations often lie in a low-dimensional subspace of the full parameter space, so low-rank updates capture most of the useful signal.

Recap

In this session, we constructed a complete information extraction pipeline using GLiNER2 — a single 205M-parameter model that performs entity extraction, text classification, structured data parsing, and relation extraction on CPU without external dependencies.

We began with the foundational concept of schema-driven extraction: rather than training task-specific models, we define what to extract at inference time through declarative schemas. This paradigm shift — from model-per-task to schema-per-task — dramatically reduces the engineering overhead of information extraction systems.

We progressed through each of GLiNER2's four capabilities in isolation, then combined them into a single-pass multi-task pipeline that ensures cross-task consistency. Finally, we examined LoRA adaptation as a mechanism for domain specialization without sacrificing the base model's generality.

The resulting pipeline is suitable for integration into agentic workflows, data processing jobs, or any application requiring structured data from unstructured text — with the critical advantage of fully local execution.

  • Schema-driven extraction treats entity types, classification labels, and relation types as model inputs rather than fixed outputs, enabling zero-shot generalization to novel domains without retraining.
  • A single 205M-parameter model handling four extraction tasks eliminates the complexity of multi-model pipelines — one installation, one loaded model, one inference call.
  • LoRA adapters provide parameter-efficient domain adaptation (typically under 2% of model parameters), and multiple adapters can be swapped at runtime to serve different domains from a single base model.
  • Combined schemas enable multi-task extraction in a single forward pass, ensuring consistency across entity, classification, and relation outputs while reducing computational cost.
  • CPU-first inference with no external API dependencies makes GLiNER2 immediately deployable in privacy-sensitive, air-gapped, or resource-constrained environments.
