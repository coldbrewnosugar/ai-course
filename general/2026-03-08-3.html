Causal Discovery and Inference with pgmpy

Move beyond correlation: learn causal graph structures from data, then ask interventional and counterfactual questions that standard ML cannot answer.
35 min causal-ai bayesian-networks probabilistic-modeling python

What's happening

Most machine learning models answer the question "What is the probability of Y given that we observed X?" — a purely associational query. But many real-world decisions require a fundamentally different kind of reasoning: "What happens to Y if we intervene and force X to a particular value?" or even "Would Y have been different had X not occurred?"

These are causal questions, and answering them requires a model that encodes not just statistical dependencies but the direction of influence between variables. A Bayesian Network — a directed acyclic graph (DAG) where edges represent direct causal relationships and nodes carry conditional probability distributions — provides exactly this structure.

pgmpy, a mature Python library for probabilistic graphical models, has recently consolidated its causal inference capabilities into a clean, end-to-end pipeline: discover structure from data, validate it, learn parameters, and then compute interventional distributions using Judea Pearl's do-calculus. This session walks through that pipeline, with you directing an AI agent to build each component.

2

Discover causal structure with the PC algorithm

The PC algorithm (named after its inventors, Peter Spirtes and Clark Glymour) is the foundational constraint-based structure learning method. Its logic is elegantly simple:

  1. Start with a fully connected undirected graph — assume every variable could be connected to every other variable.
  2. Test pairs of variables for conditional independence. If X and Y are conditionally independent given some set of other variables Z, remove the edge between them.
  3. Orient the remaining edges using a set of orientation rules. The most important is the v-structure rule: if X — Z — Y with no edge between X and Y, and Z was not in the conditioning set that separated X and Y, then orient as X → Z ← Y.

The result is a completed partially directed acyclic graph (CPDAG) — some edges are directed, others remain ambiguous because the data alone cannot distinguish their orientation. This is a fundamental limitation: observational data can identify the Markov equivalence class (the set of DAGs encoding the same conditional independencies) but often cannot pinpoint a unique DAG without additional assumptions or interventional data.

The choice of conditional independence test matters. For discrete data, the chi-square test is standard. For continuous Gaussian data, partial correlation (Pearson's r) is appropriate.

What is a Markov equivalence class?

Multiple DAGs can encode exactly the same set of conditional independence relationships. These DAGs form a Markov equivalence class, represented compactly by a CPDAG. For example, X → Y and X ← Y are Markov equivalent if there are no other variables — both encode the same (trivial) independence structure. The PC algorithm identifies the equivalence class, not a unique DAG. Edges that are directed in the CPDAG are genuinely identifiable from data; undirected edges represent genuine ambiguity that only domain knowledge or interventional experiments can resolve.

How does the significance level affect structure learning?

The significance level (α) controls the threshold for declaring two variables conditionally independent. A lower α (e.g., 0.01) requires stronger evidence of dependence before retaining an edge, producing a sparser graph with fewer false positives but potentially more false negatives. A higher α (e.g., 0.1) retains more edges, capturing weaker dependencies at the cost of more spurious connections. The default (0.05) is a reasonable starting point, but domain-specific tuning is often necessary.

3

Fit parameters and perform probabilistic inference

A DAG without parameters is a skeleton — it tells us which variables influence which, but not how strongly or in what way. Parameter learning fills in the conditional probability distributions (CPDs) at each node.

For a discrete Bayesian Network, each node's CPD is a conditional probability table (CPT): a table specifying the probability of each state of the node given every combination of its parents' states. With the alarm network's discrete variables, these tables can be estimated directly from data using maximum likelihood estimation (MLE) — simply counting frequencies — or Bayesian estimation, which incorporates a prior (typically a Dirichlet distribution) to handle sparse data more gracefully.

Once parameters are fitted, we can perform probabilistic inference: computing the posterior distribution of any variable given observed evidence on other variables. This is the standard "prediction" task — analogous to what any classifier does, but with the added benefit that a Bayesian Network can predict any variable from any subset of observations, not just a single target from a fixed feature set.

Tip
Probabilistic inference in a Bayesian Network is omnidirectional: you can predict any variable from any subset of others. This contrasts with discriminative models (e.g., logistic regression), which are trained to predict a single target variable. This flexibility is one reason Bayesian Networks are valued in domains like medical diagnosis, where the 'target' variable changes depending on the clinical question.
MLE vs. Bayesian estimation: when does it matter?

Maximum likelihood estimation sets each conditional probability to the observed frequency. This works well with sufficient data but produces zero probabilities for state combinations not observed in the sample — a problem known as data sparsity. Bayesian estimation (with a Dirichlet prior) adds pseudocounts, ensuring no probability is exactly zero. This is analogous to Laplace smoothing in naive Bayes classifiers. With 5,000 samples and 37 variables, some parent-state combinations will be rare, making Bayesian estimation the safer default for production use.

4

Interventional queries with do-calculus

This is where causal inference departs from standard machine learning. The do-operator, introduced by Judea Pearl, formalizes the difference between seeing and doing.

When we observe that X = x (conditioning), we update our beliefs about all variables correlated with X — including X's causes. When we intervene and set X = x (the do-operation), we physically force X to that value, severing it from its natural causes. Graphically, this is represented by graph surgery: deleting all incoming edges to X in the DAG, then computing the resulting probability distribution.

The practical consequence is profound. Consider a medical example: observing that a patient has high ventilator pressure and low oxygen does not mean that reducing ventilator pressure would improve oxygen. The association may be confounded — perhaps both are caused by severe lung disease. The do-calculus answers the interventional question correctly by adjusting for confounding through the graph structure.

pgmpy implements this through its CausalInference class, which takes a fitted Bayesian Network and computes interventional distributions using the adjustment formula derived from the graph.

Warning

This is why structure validation (comparing learned structure against domain knowledge) is not optional; it is a prerequisite for trustworthy causal inference.
The adjustment formula, intuitively

When we cannot run a randomized experiment, the do-calculus tells us how to compute interventional distributions from observational data — if we know the causal graph. The simplest case is the back-door adjustment formula: to estimate P(Y | do(X=x)), identify a set of variables Z that blocks all 'back-door paths' (non-causal paths) between X and Y, then compute the weighted average P(Y | X=x, Z=z) × P(Z=z), summing over all values of Z. This is precisely what pgmpy's CausalInference.query() automates: it identifies the appropriate adjustment set from the graph and applies the formula.

When do-calculus cannot help

The do-calculus is not omnipotent. If a required adjustment set is not available — for example, because a necessary confounder is unmeasured — the interventional distribution is not identifiable from observational data alone. Pearl's completeness results show that the three rules of do-calculus, combined with graph structure, can identify every identifiable causal effect, but some effects are genuinely not identifiable without additional data (e.g., from experiments or instrumental variables).

5

Simulate under interventions and counterfactual scenarios

Beyond computing a single interventional probability, we often want to simulate what an entire dataset would look like under a hypothetical intervention. This is the causal analogue of synthetic data generation: instead of sampling from the observational distribution, we sample from the interventional distribution.

pgmpy supports this through its .simulate() method with a do parameter. When you specify do={'X': value}, the simulator performs graph surgery internally — it fixes X to the specified value, removes its dependence on parents, and then generates samples by ancestral sampling through the modified graph.

This capability has direct practical applications:

  • Policy evaluation: "If we mandate treatment A for all patients, what is the expected distribution of outcomes?"
  • Fairness auditing: "If we intervene to equalize a protected attribute, how do downstream predictions change?"
  • Scenario planning: "If we force a supply chain variable to a stress value, how do downstream metrics respond?"

Counterfactual reasoning goes one step further. Rather than asking about populations, counterfactuals ask about specific individuals: "Given that this particular patient experienced outcome Y, would the outcome have been different had we administered treatment X instead?" This requires a fully specified structural equation model and involves three steps: abduction (infer latent variables for the individual), action (perform the intervention), and prediction (compute the counterfactual outcome).

The three levels of Pearl's causal hierarchy

Judea Pearl organizes causal reasoning into three levels, often called the causal ladder:

  1. Association (seeing): P(Y | X) — standard conditional probability. Answered by any statistical model.
  2. Intervention (doing): P(Y | do(X)) — what happens if we force X. Requires a causal model (DAG + parameters).
  3. Counterfactual (imagining): P(Y_x | X'=x', Y'=y') — what would have happened to a specific unit. Requires a fully specified structural equation model.

Each level subsumes the one below it: a model that can answer counterfactual questions can also answer interventional ones, which can also answer associational ones. The reverse is not true. Most ML models operate at level 1. This workshop has taken you to level 2, with a glimpse of level 3.

Recap

In this session, we moved through the complete pgmpy causal inference pipeline — not by writing code from scratch, but by directing an AI agent with increasingly precise prompts. We began with data loading and domain exploration, progressed through structure learning with the PC algorithm, fitted parameters via maximum likelihood, and then crossed the critical boundary from associational to causal reasoning by computing interventional distributions with do-calculus.

The central insight is that causal inference requires more than data — it requires a model of the data-generating process. A DAG encodes assumptions about which variables cause which, and those assumptions enable questions that no amount of correlational analysis can answer. The do-operator, graph surgery, and the adjustment formula are the formal tools that make this possible.

Equally important: causal discovery algorithms like PC are tools for hypothesis generation, not ground truth. Their output must be validated against domain knowledge, and their assumptions (causal sufficiency, faithfulness) must be critically examined. A learned graph is a starting point for causal reasoning, not an endpoint.

  • The do-operator formalizes the difference between observing and intervening — P(Y|X) and P(Y|do(X)) can differ when confounding is present, and only the latter answers causal questions.
  • The PC algorithm discovers causal structure by systematically testing conditional independence, but it identifies an equivalence class of DAGs, not a unique graph — some edge orientations remain ambiguous from observational data alone.
  • Bayesian Networks enable omnidirectional inference: any variable can be predicted from any subset of observations, and the same model supports both probabilistic conditioning and causal interventional queries.
  • Structure learning is hypothesis generation, not proof. Algorithmic output must be validated against domain knowledge, and the assumptions (causal sufficiency, faithfulness, correct graph) must hold for interventional estimates to be trustworthy.

Where to go next

  • Apply the pipeline to your own dataset: define candidate variables, run structure learning, and compare the learned graph against your domain expertise before trusting interventional queries.
  • Explore the FCI algorithm (pgmpy.estimators.FCI) for settings where latent confounders may be present — it relaxes the causal sufficiency assumption at the cost of less definitive edge orientations.
  • Investigate pgmpy's LinearGaussianBayesianNetwork for continuous data, using Pearson partial correlation as the CI test — the same conceptual pipeline applies, but with different distributional assumptions.
  • Read Judea Pearl's 'The Book of Why' for an accessible introduction to causal reasoning, or 'Causality' (2nd edition) for the full technical treatment of do-calculus and its completeness results.
