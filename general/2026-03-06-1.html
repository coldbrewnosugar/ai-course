Managing AI Agent Configs with Microsoft's Agent Package Manager (APM)

Declare your project's agent skills, prompts, instructions, and tools in a single manifest — and give every developer an identical agent setup from one command.
35 min tooling ai-agents developer-experience package-management

What's happening

The Problem APM Solves

AI coding agents — GitHub Copilot, Claude, Cursor, Codex — are now standard development tools. But each requires its own configuration: instruction files, skill definitions, prompt libraries, MCP server declarations. Today, onboarding a new developer means manually replicating this setup, copying files between repositories, and hoping everyone converges on the same agent behavior.

This is the exact problem that package.json, requirements.txt, and Cargo.toml solved for application dependencies a decade ago. Microsoft's Agent Package Manager (APM), which surfaced on GitHub Trending this week, applies the same paradigm to AI agent configuration.

APM introduces a single manifest file — apm.yml — that declares every agent primitive your project needs: instructions, skills, prompts, agents, hooks, and MCP servers. A new developer clones the repository, runs apm install, and every agent tool is configured identically. Packages can depend on other packages, and APM resolves the full transitive dependency tree automatically.

The key mental model: APM is to your AI agent what npm is to your JavaScript runtime. It does not manage application code. It manages the configuration that makes your AI agents useful in the context of your specific project.

1

Install APM and Understand the Primitive Model

Before writing any manifest, we need APM installed and a clear understanding of what it manages. APM operates on six agent primitives — discrete units of agent configuration that can be declared, versioned, and shared:

| Primitive | Purpose | Example |

|---|---|---|

| Instructions | Coding standards and guardrails | "Use type hints in all Python files" |

| Skills | Reusable AI capabilities or workflows | A form builder, a code reviewer |

| Prompts | Slash commands for common tasks | /security-audit, /design-review |

| Agents | Specialized agent personas | An accessibility auditor agent |

| Hooks | Lifecycle event handlers | Pre-tool validation, post-tool linting |

| MCP Servers | External tool integrations | Database access, API connectors |

These primitives are the vocabulary of APM. Every entry in an apm.yml file resolves to one or more of these types. Understanding this taxonomy is essential before proceeding — it determines how you structure your manifest and how APM organizes the installed output.

Tip

Warning

Why six primitives instead of just 'instructions'?

Early agent configuration tools treated everything as flat instruction text — a single markdown file appended to the agent's context. This is analogous to managing all application dependencies as a single concatenated script. The primitive model introduces separation of concerns: instructions govern behavior, skills define capabilities, prompts provide entry points, agents define personas, hooks handle lifecycle events, and MCP servers provide external tool access. This decomposition enables granular dependency management — you can depend on a single skill from a large package without pulling in its prompts or hooks.

2

Scaffold an apm.yml Manifest

The apm.yml file is the central artifact in any APM-managed project. It serves the same role as package.json in a Node.js project: it declares metadata about your project and lists the agent packages it depends on.

The manifest has a minimal required structure: a name, a version, and a dependencies block. The dependencies block uses the apm key to list packages, each identified by a repository path. These paths follow a consistent format:

  • Full repository: owner/repo — installs all primitives from the package
  • Specific path: owner/repo/path/to/primitive — installs only the targeted primitive
  • Non-GitHub hosts: Full URL like https://gitlab.com/acme/coding-standards.git

This addressing scheme is the key design decision in APM. Rather than a central registry (like npmjs.com), APM treats any Git repository as a potential package source. This eliminates the registry bottleneck and allows teams to share private packages without publishing them.

Tip

How does APM resolve paths within repositories?

When you specify owner/repo/path/to/primitive, APM clones the repository (or fetches the relevant subtree) and looks for recognized primitive files at the given path. A skill is typically a markdown file with a specific front-matter schema. An agent is an .agent.md file. Instructions may be plain markdown or structured YAML. APM identifies the primitive type from file conventions and front-matter metadata, then installs it into the appropriate location in your project's .github/instructions/ directory. This convention-over-configuration approach means package authors need only follow naming conventions — no separate package descriptor is required for simple single-file primitives.

✓
At this point you should have APM installed and a valid `apm.yml` manifest file in your project root. The manifest should declare at least three community packages targeting different primitive types (skills, agents, instructions). You have not yet installed the dependencies — that comes next.
3

Install Dependencies and Inspect the Resolved Tree

Running apm install reads your manifest, resolves all dependencies (including transitive ones), and writes the resulting primitives into your project. This is the core workflow — analogous to npm install reading package.json and populating node_modules.

The resolution process is worth understanding. If package A depends on package B, and your manifest depends on package A, APM will install both A and B. This is transitive dependency resolution — the same mechanism that makes npm, pip, and Cargo practical for large dependency trees. Without it, every consumer would need to manually declare the full closure of all indirect dependencies.

After installation, APM places primitives under .github/instructions/ when editor integration is enabled (e.g., VS Code with GitHub Copilot). This directory is the conventional location where Copilot and other VS Code–integrated agents look for project-specific instructions.

Transitive resolution: a concrete example

Suppose your apm.yml depends on microsoft/apm-sample-package. That sample package itself might declare dependencies on a set of foundational coding standards and a security audit prompt. When you run apm install, APM fetches the sample package, reads its own dependency declarations, and recursively fetches all downstream packages. The result is a flat installation in your .github/instructions/ directory — no nested node_modules-style hierarchy. This flat model avoids the duplication and depth problems that plagued early npm versions. If two packages depend on the same downstream primitive, APM installs it once.

4

Compile Output for Cross-Agent Compatibility

APM's apm compile command is the bridge between the unified primitive store and the divergent formats that different AI agents expect. This is a critical design choice: rather than forcing all agents to read from a single format, APM compiles to each agent's native convention.

The compile step produces two primary outputs:

  • AGENTS.md — consumed by GitHub Copilot, Cursor, and Codex. This file aggregates instructions, skill references, and agent definitions into the markdown format these tools expect.
  • CLAUDE.md — consumed by Claude. Same content, structured according to Claude's instruction conventions.

This compilation model is analogous to how a language compiler targets multiple architectures from a single source. The apm.yml is the source; the agent-specific markdown files are the compiled targets. Any change to the manifest propagates to all targets through a single apm compile invocation.

The recommended workflow is: edit apm.yml → run apm install → run apm compile → commit the results. Some teams add apm install && apm compile to a post-checkout Git hook to keep everything synchronized automatically.

What the agent gives back

The agent should describe apm compile as the step that transforms installed primitives into agent-specific instruction files. It should note that AGENTS.md and CLAUDE.md may differ in structure — for instance, Claude's format may use specific heading conventions or metadata blocks. The agent should recommend committing both files to version control so that agents work immediately upon cloning. For automation, it should suggest a simple Git post-checkout or post-merge hook that runs apm install && apm compile, ensuring that switching branches or pulling changes automatically updates the agent configuration.

Tip
Running apm compile is idempotent — running it multiple times with the same installed primitives produces identical output. This makes it safe to include in CI pipelines or Git hooks without risk of spurious diffs.
Why not a single universal format?

The AI agent ecosystem lacks a universal instruction standard. GitHub Copilot reads from AGENTS.md and .github/instructions/ with specific conventions for file-matching globs and skill references. Claude reads CLAUDE.md with its own structural expectations. Cursor, Codex, and Gemini each have their own conventions. Rather than waiting for standardization — which may never fully converge — APM adopts a pragmatic compile-to-target approach. This insulates package authors from format differences: they write primitives once, and APM handles the format translation. If a new agent emerges with a different convention, only the APM compiler needs updating, not every package in the ecosystem.

5

Create and Share Your Own APM Package

APM's value scales with the ecosystem. Beyond consuming community packages, teams benefit significantly from authoring internal packages that encode their own coding standards, review workflows, and domain-specific prompts.

The apm init command scaffolds a new package. A package is simply a Git repository (or a directory within one) that contains recognized primitive files. There is no separate registry to publish to — any Git-accessible repository can serve as a package source. This means your organization's private coding standards can live in an internal repository and be referenced by any project's apm.yml.

The key architectural insight is that packages compose. A team-level package can depend on organization-level packages, which can depend on community packages. This creates a layered configuration model: community best practices at the base, organization standards in the middle, and team-specific customizations at the top. Each layer is independently versioned and updatable.

What the agent gives back

The agent should outline a directory structure for the package containing three files: an instruction markdown file for Python type hinting conventions, a skill file (likely .md with appropriate front-matter) for the code review workflow, and a prompt file for the /security-audit command. The agent should emphasize that organizing primitives in separate subdirectories (e.g., skills/, prompts/, instructions/) allows consumers to reference specific paths like your-org/your-package/skills/code-review rather than pulling everything.

Tip
Package authoring follows the same conventions as package consumption. If you can read an apm.yml, you can write one. The barrier to creating shareable agent configuration is intentionally low.
Versioning and update strategies

APM currently resolves packages from the default branch of the referenced repository. This means that updates to the source repository are picked up on the next apm install. For teams requiring stability, pinning to a specific Git tag or commit hash in the dependency reference provides deterministic builds. The trade-off is the same as in any dependency manager: latest-branch references give you automatic updates but risk breaking changes, while pinned references give stability but require manual updates. Most teams adopt a hybrid approach — pinning organization-critical packages while tracking latest for community packages that are lower-risk.

Recap

In this session, we established a complete workflow for managing AI agent configuration as a first-class project dependency. We began with APM's primitive model — the six types of agent configuration (instructions, skills, prompts, agents, hooks, MCP servers) that form APM's vocabulary. We then scaffolded an apm.yml manifest referencing community packages, installed those dependencies with transitive resolution, and compiled the output into agent-specific formats (AGENTS.md for Copilot/Cursor/Codex, CLAUDE.md for Claude). Finally, we examined how to author and share custom packages, enabling teams to encode their own standards as reusable, composable dependencies.

The fundamental shift APM represents is from manual, per-developer agent setup to declarative, reproducible agent configuration. This is the same shift that package managers brought to application development — and it addresses the same core problem: ensuring that every contributor to a project operates in an equivalent environment.

  • APM manages six agent primitives — instructions, skills, prompts, agents, hooks, and MCP servers — in a single apm.yml manifest, providing a unified declaration for all agent configuration.
  • Transitive dependency resolution means your manifest only needs to declare direct dependencies; APM automatically fetches and installs the full closure of indirect dependencies.
  • The apm compile step bridges the fragmented agent ecosystem by generating format-specific output (AGENTS.md, CLAUDE.md) from a single source of truth, ensuring cross-agent compatibility without manual duplication.
  • Any Git repository can serve as a package source — no central registry is required — which enables private, organization-internal packages with zero publishing overhead.
  • Agent configuration is now a committable, reviewable, version-controlled project artifact — subject to the same governance as application code.

Where to go next

  • Explore the microsoft/apm-sample-package repository to study a complete, well-structured package with multiple primitive types and declared dependencies.
  • Set up a Git post-checkout hook running apm install && apm compile to automate agent configuration updates when switching branches.
  • Author an organization-level APM package encoding your team's coding standards, then reference it from multiple project repositories to establish a consistent baseline.
  • Investigate APM's behavior with non-GitHub hosts (GitLab, Bitbucket, Azure DevOps) if your organization uses alternative Git platforms.
