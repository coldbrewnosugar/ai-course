Build a Tool-Calling Agent with Qwen-Agent and MCP

From a bare model endpoint to an autonomous, tool-wielding agent — guided entirely through AI-assisted development
40 min agents function-calling MCP Qwen open-source RAG

What's happening

Qwen-Agent, the open-source framework maintained by Alibaba's Qwen team, has matured into one of the most complete agent scaffolds available for open-weight models. With the February 2026 release of Qwen3.5 and the accompanying agent demo, the framework now offers first-class support for Model Context Protocol (MCP) servers, multi-step function calling, code interpretation, and retrieval-augmented generation — all from a single pip install.

This matters for a specific reason: until recently, sophisticated tool-calling agents were tightly coupled to proprietary APIs. Qwen-Agent demonstrates that the same agentic patterns — plan-then-execute loops, parallel tool dispatch, memory-augmented reasoning — can run against a model you host yourself, with no API keys leaving your infrastructure.

In this session, we construct a working tool-calling agent step by step. We will not write code from scratch. Instead, we will formulate precise prompts to an AI coding assistant and evaluate the output at each stage. The objective is twofold: to understand the architecture of a tool-calling agent pipeline, and to develop fluency in directing an AI agent to build one for us.

1

Establish the Model Service and Agent Skeleton

Every Qwen-Agent application begins with two primitives: a model service (an OpenAI-compatible endpoint serving a Qwen model) and an agent class that orchestrates interactions with that service.

Qwen-Agent supports two deployment paths. You can use Alibaba Cloud's DashScope API, which requires only an API key. Alternatively, you can serve an open-weight Qwen model locally via vLLM or Ollama. For this workshop, either path works — the agent code is identical in both cases because Qwen-Agent abstracts the model behind a uniform llm configuration dictionary.

The agent skeleton itself is minimal. Qwen-Agent provides a base class called Assistant that handles the core loop: receive a user message, decide whether to call tools or respond directly, execute any tool calls, and return a final answer. Our first task is to get this skeleton running with a simple greeting exchange — no tools yet, just proof that the model service and the agent framework are connected.

  • Qwen-Agent's `Assistant` class accepts an `llm` configuration — what shape should that configuration take?
  • How should the script invoke the agent and display its response? Consider that Qwen-Agent uses a generator-based response pattern.
  • What installation extras are needed if we plan to add MCP and RAG later?
What the agent gives back

The agent should produce a short Python script — roughly 15 lines — that imports Assistant from qwen_agent.agents, configures it with an llm dictionary specifying model, model_server, and api_key, then calls agent.run() with a simple user message and prints the streamed response. The critical configuration looks like this:

```python

```

The script should confirm that the agent responds coherently, proving the connection is live.

API Key Note

Tip
For local deployment, Ollama is the simplest path: ollama run qwen3.5 starts a compatible server immediately. vLLM offers higher throughput but requires a GPU with sufficient VRAM. For this workshop, model size matters less than having a working endpoint — even a 7B parameter model supports tool calling.
What does `Assistant` actually do under the hood?

The Assistant class implements a ReAct-style loop. On each turn, it sends the conversation history (plus any system prompt and tool descriptions) to the model. If the model's response contains a function call, Assistant executes the corresponding tool, appends the result to the conversation, and loops again. If the response contains no function call, the loop terminates and the final text is yielded to the caller. This loop is the fundamental pattern behind nearly all tool-calling agents, regardless of framework.

2

Define a Custom Tool with Function Calling

A tool, in the context of an LLM agent, is a function that the model can choose to invoke when it determines that its parametric knowledge is insufficient. The model does not execute the function itself — it produces a structured request (a function call) specifying the tool name and arguments, and the agent framework dispatches the actual execution.

Qwen-Agent models tools as subclasses of BaseTool. Each tool declares three things: a name, a description (which the model reads to decide when to use the tool), and a parameters schema (a JSON Schema object defining the inputs). The framework injects these declarations into the system prompt so the model understands what capabilities are available.

The quality of the tool description is decisive. A vague description leads to misuse; an overly narrow one leads to under-use. This is a prompt engineering problem, not a coding problem — and it is where your judgment matters most.

For our first tool, we will build a simple weather lookup. The purpose is not meteorological accuracy but structural clarity: it demonstrates the full lifecycle of a tool call.

  • What parameters does a weather lookup require? What JSON Schema types and constraints are appropriate?
  • How does the `Assistant` class learn about available tools — does it discover them automatically, or must they be explicitly registered?
  • What should the tool's `call()` method return? Consider the format the agent framework expects.
What the agent gives back

The agent should produce a class that extends BaseTool, with name, description, and parameters class attributes. The description should be precise — something like 'Retrieves the current weather for a specified city, returning temperature, conditions, and humidity.' The parameters list defines a required city field of type string. The call() method returns a dictionary with mock data. The tool is then passed to Assistant via the function_list parameter. When the user asks 'What is the weather in Tokyo?', the agent should invoke the tool and incorporate the result into its response.

Tip
Tool descriptions function as prompts to the model. Treat them with the same care you would give a system prompt. Specify what the tool does, what it returns, and — critically — when it should not be used.
Warning
Qwen-Agent parses tool calls from the model output using its own template, not vLLM's built-in parser. When serving Qwen3.5 with vLLM, do not pass --enable-auto-tool-choice or --tool-call-parser hermes — these flags conflict with Qwen-Agent's parsing logic.
How does function calling actually work at the token level?

During fine-tuning, Qwen models are trained to emit special tokens (such as ✿FUNCTION✿ and ✿ARGS✿ in earlier versions, or structured JSON blocks in Qwen3.5) when they determine a tool call is appropriate. The agent framework scans the model output for these patterns, extracts the function name and arguments, executes the corresponding Python method, and injects the return value back into the conversation as an observation. The model then generates a final response that synthesizes the observation with the user's original question. This two-phase generation (call, then summarize) is why tool-calling agents typically require two model forward passes per tool use.

Parallel and multi-step tool calls

Qwen3.5 supports parallel tool calling — the model can emit multiple function calls in a single response, and the agent framework executes them concurrently. It also supports multi-step calling, where the result of one tool informs the next call. The Assistant class handles both patterns transparently: it loops until the model produces a response with no further tool calls. This means a well-described set of tools can enable surprisingly complex workflows without any additional orchestration code.

Connect an MCP Server for External Tool Discovery

The Model Context Protocol (MCP) is an open standard that allows an agent to discover and invoke tools hosted by an external server, rather than defining them in-line within the agent code. Think of MCP as a USB port for AI agents: any MCP-compliant server can plug in, and the agent automatically learns what tools are available, what parameters they accept, and how to call them.

This decoupling is architecturally significant. Without MCP, every tool must be defined as a Python class within your agent codebase. With MCP, tools can be maintained as separate services — potentially written in different languages, operated by different teams, and versioned independently. The agent queries the MCP server at startup, receives a manifest of available tools, and treats them identically to locally defined ones.

Qwen-Agent integrates MCP through its mcpServers configuration key, which mirrors the format used by Claude Desktop and other MCP-aware clients. Each server entry specifies either a command (for stdio-based servers) or a url (for HTTP-based servers).

For this step, we will connect the widely-used filesystem MCP server, which exposes file system operations (read, write, list, search) as tools the agent can call.

Ask your agent
Get your AI agent to modify the existing script so that it connects to an MCP server (the filesystem server) and can use its tools alongside the custom weather tool.
Think about it
  • How does Qwen-Agent's `Assistant` accept MCP server configuration? What key do you add to its constructor?
  • The filesystem MCP server runs as a subprocess via `npx`. What command and arguments does it require, and how do you restrict which directories it can access?
  • Can locally defined tools and MCP-provided tools coexist in the same agent? How does the agent's tool selection logic handle both?
  • What happens at agent startup — does it connect to the MCP server immediately, or lazily on first tool call?
What the agent gives back

The agent should produce an updated Assistant configuration that includes an mcp_servers parameter alongside the existing function_list. The MCP configuration specifies the filesystem server:

```python

mcp_servers = [{

'command': 'npx',

'args': ['-y', '@modelcontextprotocol/server-filesystem', '/tmp/workspace'],

}]

```

When the agent starts, it should connect to the MCP server, discover the available filesystem tools (read_file, write_file, list_directory, etc.), and make them available alongside the custom weather tool. A test query like 'List the files in /tmp/workspace' should trigger the MCP filesystem tool.

Warning
The filesystem MCP server grants the agent real read/write access to the directories you specify. Always scope it to a dedicated workspace directory. Never point it at your home directory or system paths.
Tip
You can connect multiple MCP servers simultaneously. Each one contributes its tools to the agent's available pool. This composability is MCP's primary design advantage — you can assemble an agent's capabilities from a library of independent servers.
stdio vs. HTTP transport for MCP servers

MCP defines two transport modes. The stdio transport launches the server as a child process and communicates via stdin/stdout — this is what the command/args configuration uses. It is simpler to set up and is the default for local development. The HTTP transport connects to a running server via HTTP with Server-Sent Events, suitable for remote or shared servers. Qwen-Agent supports both; the choice depends on your deployment context, not on the agent code itself.

How does MCP tool discovery work?

When the agent initializes, it sends a tools/list request to each configured MCP server. The server responds with a JSON manifest containing each tool's name, description, and input schema — exactly the same information you would define manually in a BaseTool subclass. Qwen-Agent wraps each discovered tool as an internal tool object and adds it to the agent's tool registry. From the model's perspective, there is no difference between a locally defined tool and an MCP-provided one.

4

Add RAG: Give the Agent a Knowledge Base

Retrieval-Augmented Generation (RAG) allows an agent to ground its responses in a specific document corpus rather than relying solely on its training data. In Qwen-Agent, RAG is implemented as a built-in tool — the agent can decide to search a document collection just as it would decide to call any other tool.

The architecture is straightforward. You provide a set of documents (files, URLs, or raw text) to the agent. Qwen-Agent chunks these documents, builds a retrieval index, and exposes a retrieval tool that the model can invoke when it needs information from the corpus. The model formulates a search query, the retrieval tool returns relevant chunks, and the model synthesizes a response from those chunks.

This is conceptually identical to how a human researcher works: faced with a question outside their immediate knowledge, they consult reference materials, extract relevant passages, and compose an answer. The difference is that the model decides autonomously whether retrieval is necessary — it is not forced to retrieve on every query.

For this step, we will give the agent access to a small set of documents and verify that it can answer questions from them.

  • Does the retrieval tool need to be explicitly added to `function_list`, or does specifying documents automatically enable it?
  • What document formats does Qwen-Agent support out of the box — PDF, plain text, URLs?
  • How would you test that the agent is actually retrieving from the documents rather than answering from its training data? Think about questions with answers that only exist in your specific documents.
What the agent gives back

The agent should produce an updated configuration where the Assistant receives a files parameter — a list of file paths or URLs pointing to reference documents. Qwen-Agent automatically enables its built-in retrieval tool when documents are provided. The key addition is simply passing documents at construction time. When asked a question whose answer resides in the documents, the agent should invoke the retrieval tool, extract relevant passages, and cite them in its response. A good test is to provide a document with a specific, unusual fact and then ask the agent about it — if it answers correctly, retrieval is working.

Tip
Qwen-Agent's RAG pipeline uses its own chunking and retrieval logic. For production workloads with large corpora, you may want to replace this with a dedicated vector database (such as Qdrant or Milvus) exposed as an MCP server — combining the patterns from Steps 3 and 4.
How does the agent decide when to retrieve?

The retrieval tool has a description — just like any other tool — that tells the model it can search a document collection. The model evaluates each user query against all available tool descriptions and decides whether retrieval is likely to help. This means the agent will not retrieve for simple conversational queries ('hello') but will retrieve for domain-specific questions ('what was the Q3 revenue?'). The quality of this decision depends on the model's instruction-following capability and the clarity of the retrieval tool's description.

Orchestrate Multi-Step Tasks with Agent Planning

The preceding steps established individual capabilities — tools the agent can invoke one at a time. The final architectural layer is planning: the agent's ability to decompose a complex request into a sequence of tool calls, execute them in order, and synthesize the results into a coherent response.

Qwen3.5 handles planning through its extended reasoning capability. When the model encounters a complex query, it first generates a chain-of-thought plan (visible in the reasoning_content field of the response), then executes the plan step by step. Each step may involve a tool call, a retrieval, or pure reasoning. The Assistant class manages this loop automatically — it continues invoking tools and feeding results back to the model until the model produces a final response with no further tool calls.

The practical implication is that you do not need to write orchestration logic. A well-described set of tools, combined with a sufficiently capable model, produces emergent multi-step behavior. Your role is to ensure that the tool descriptions form a coherent vocabulary — each tool's description should clarify not only what it does in isolation, but how it relates to other available tools.

For this final step, we will craft a system prompt that guides the agent toward planning behavior and test it with a multi-step task.

Ask your agent
Get your AI agent to add a system prompt and test configuration that enables and demonstrates multi-step planning across all available tools.
Think about it
  • What should the system prompt convey to encourage the model to plan before acting? Think about decomposition instructions and tool-use guidelines.
  • How do you make the reasoning process visible — does Qwen-Agent expose the model's chain-of-thought separately from the final response?
  • What would a good multi-step test task look like? It should require at least two different tools in sequence — for example, retrieving information from a document and then writing a summary to a file.
  • How does error handling work if a tool call fails mid-plan? Does the agent retry, skip, or abort?
What the agent gives back

The agent should produce a system prompt that instructs the model to think step by step, identify which tools are needed, and execute them in sequence. A good multi-step test might be: 'Find the key points from the reference documents about Qwen-Agent's MCP support, then write a summary to /tmp/workspace/mcp-summary.txt.' The agent should first invoke the retrieval tool to gather relevant passages, then invoke the filesystem MCP tool to write the summary file. The reasoning_content field in the response should reveal the agent's plan before execution begins.

Tip
The reasoning_content field is Qwen3.5's equivalent of extended thinking. It contains the model's internal deliberation — including its plan for tool use — before it begins generating the visible response. Logging this field is invaluable for debugging agent behavior.
Planning limitations and failure modes

Emergent planning has known failure modes. The model may call tools in a suboptimal order, repeat unnecessary calls, or fail to use a result from an earlier step. These issues are more common with smaller models and less precise tool descriptions. Mitigation strategies include: (1) providing few-shot examples in the system prompt showing correct multi-step behavior, (2) limiting the number of available tools to reduce decision complexity, and (3) using the max_ref_token parameter to control how much retrieved content the model processes at once. For mission-critical workflows, consider using Qwen-Agent's Router agent, which delegates subtasks to specialized sub-agents rather than relying on a single model to plan everything.

Recap

We have constructed a complete tool-calling agent pipeline using Qwen-Agent, progressing through five architectural layers: a model connection, custom tool definition, MCP server integration, retrieval-augmented generation, and multi-step planning. At each stage, the work was performed by prompting an AI coding assistant — our role was to define objectives, evaluate outputs, and iterate.

The central insight of this session is that building an agent is primarily a design activity, not a coding activity. The framework handles execution mechanics. Your leverage lies in three areas: the precision of your tool descriptions (which determine whether the model selects tools correctly), the clarity of your system prompt (which shapes planning behavior), and the composition of your tool set (which defines the agent's capability boundary).

The Qwen-Agent framework, combined with MCP, provides a modular architecture where tools can be developed, tested, and deployed independently of the agent code itself. This separation of concerns — agent logic, tool implementation, and model serving as three distinct layers — is the foundation for building agents that are maintainable and extensible in production.

  • Tool descriptions are prompts to the model — their precision directly determines the reliability of tool selection and invocation.
  • MCP decouples tool implementation from agent code, enabling plug-and-play composition of capabilities from independent servers.
  • Multi-step planning is an emergent property of capable models with well-described tools, not something you need to hard-code.
  • RAG in Qwen-Agent is implemented as a tool, meaning the model decides autonomously when retrieval is necessary — the quality of this decision depends on the retrieval tool's description.
  • The reasoning_content field in Qwen3.5 responses exposes the model's internal planning, providing a critical debugging surface for agent behavior.
