Build an AI Code Verification Agent to Catch Verification Debt

AI generates code in seconds. Verifying it takes hours. We will build an agent that closes that gap.
35 min agentic-coding code-review verification developer-tools

What's happening

A recent article on agentic coding — "Agentic Coding: AI's Adolescence" — crystallized a problem that many engineering teams have begun to feel but few have named: verification debt.

The concept is straightforward. When a human writes code, the act of writing is itself a form of reasoning — the developer thinks through edge cases, considers naming conventions, and builds a mental model of the system as they type. When an AI agent generates code, that reasoning step is skipped. The code appears instantly, and the burden of understanding shifts entirely to the reviewer. Each unreviewed or superficially reviewed AI-generated change accumulates as verification debt: latent risk hiding in a codebase that looks correct but has never been rigorously examined.

Verification debt compounds in three characteristic ways:

  1. Untested edge cases. The agent produces a function that handles the happy path flawlessly but silently fails on empty inputs, Unicode strings, or concurrent access.
  2. Style drift. Over dozens of generated files, naming conventions diverge, error-handling patterns become inconsistent, and the codebase loses the coherence that makes it maintainable.
  3. Hallucinated APIs. The agent calls a method that does not exist in the version of the library you are using — or invents a plausible but fictional configuration option.

In this session, we build a lightweight verification agent: a pipeline that accepts AI-generated diffs, audits them for these failure modes, and produces property-based tests as a safety net. The goal is not to replace human review but to ensure that the most mechanical and error-prone aspects of verification are handled systematically.

1

Define the Verification Schema

Before we ask an agent to build anything, we need a precise specification of what "verification" means in our context. A verification agent without a clear schema is merely a second opinion — useful, but not systematic.

Our schema must answer three questions: What are we examining? (a code diff), What are we looking for? (a taxonomy of failure modes), and How do we report findings? (a structured output format).

The taxonomy of failure modes is the intellectual core of this step. We define five categories:

  • Edge-case gaps: inputs or states the code does not handle.
  • Style violations: deviations from the project's established conventions.
  • API hallucinations: references to functions, methods, or parameters that do not exist.
  • Assumption leaks: implicit assumptions about environment, configuration, or data shape that are never validated.
  • Test absence: logic branches with no corresponding test coverage.

Each finding should carry a severity level (critical, warning, informational) and a location reference (file path and line range). This structured output is what makes the agent's results actionable rather than merely conversational.

Ask your agent
Get the agent to produce a verification schema — a structured data format that represents the input (a code diff) and the output (a list of categorized findings).
Think about it
  • What fields does each finding need to be useful in an automated pipeline? Think about what a downstream tool (a CI check, a dashboard) would consume.
  • How should severity levels be defined — as free-form strings, or as a constrained enumeration? What are the trade-offs?
  • Should the schema include metadata about the diff itself (author, timestamp, originating agent)? When would that context matter?
  • What serialization format makes the schema both human-readable and machine-parseable?
What the agent gives back

The agent should produce a JSON Schema (or equivalent structured type definition) with two main components: a DiffInput object containing the raw diff text and optional metadata, and a VerificationReport object containing a list of Finding entries. Each Finding should include a category field (constrained to the five failure modes above), a severity enum, a location object with file path and line range, a human-readable description, and a suggested_fix field. The schema should be approximately 30–40 lines of JSON Schema or a concise TypeScript interface — ask the agent to keep it minimal.

Tip
A well-defined schema is worth more than a thousand lines of implementation. If the schema is precise, the agent can generate correct parsers, validators, and display logic from it in subsequent steps.
Why a formal schema instead of free-form text?

Free-form review comments are useful for humans reading a pull request, but they cannot be aggregated, filtered, or acted upon programmatically. A formal schema enables three things that free-form text does not: (1) automated severity triage — your CI pipeline can block merges on critical findings without human intervention; (2) trend analysis — you can track whether hallucinated-API findings are increasing over time, which signals a model version problem; and (3) deduplication — identical findings across multiple diffs can be collapsed rather than repeated. The small upfront cost of defining a schema pays for itself the first time you need to answer the question, "How many unresolved critical findings do we have across the last 50 AI-generated PRs?"

2

Build the Diff Analyzer

With our schema defined, we now construct the component that does the actual analytical work: a diff analyzer that reads a unified diff, applies our five-category taxonomy, and emits structured findings.

The key architectural insight is that this analyzer should operate as a two-pass system. The first pass is syntactic: it parses the diff to identify added and modified lines, extracts function signatures, and maps changes to their surrounding context. The second pass is semantic: it evaluates each change against the failure-mode taxonomy, using the surrounding context to assess whether an edge case is missing or an API call is suspicious.

Why two passes? Because conflating parsing and analysis produces brittle, hard-to-debug agents. If the syntactic pass is correct but the semantic analysis misses a finding, you know exactly where to refine your prompt. If both are interleaved, a single misparse can cascade into missed findings with no clear diagnosis.

The semantic pass is where the agent's language-model capabilities become essential. Detecting a hallucinated API, for instance, requires knowledge of library interfaces — something a static linter cannot do without a type database, but which a language model can approximate with high accuracy for popular libraries.

Ask your agent
Get the agent to build a diff analyzer that accepts a unified diff string, parses it into structured change hunks, and then evaluates each hunk against the five failure-mode categories, outputting a VerificationReport.
Think about it
  • What should the agent use for diff parsing — a library, or manual line-by-line parsing? What are the reliability trade-offs?
  • How should the analyzer handle context lines (unchanged lines included in the diff for context)? Are they merely decorative, or do they carry analytical value?
  • For the semantic pass, the agent needs to reason about code it hasn't executed. What instructions would help it distinguish between a genuinely hallucinated API and an uncommon but valid one?
  • Should the analyzer process the entire diff at once, or chunk it by file? Consider token limits and analytical precision.
What the agent gives back

The agent should produce a Python module (or language of your choice) with two clear functions: one that parses a unified diff into a list of structured hunks (each with file path, line range, added lines, removed lines, and surrounding context), and one that takes those hunks and returns a VerificationReport conforming to the schema from Step 1. The semantic analysis function should use an LLM call internally — passing each hunk along with a system prompt that encodes the five failure-mode categories and asks for structured JSON output. The total implementation should be concise; the parsing function is straightforward string processing, and the semantic function is primarily a well-crafted prompt wrapped in an API call.

Warning
When prompting the agent to build the semantic analyzer, be explicit that it should NOT flag findings it is uncertain about as 'critical.' Uncertain findings should be 'informational.' False critical findings erode trust in the pipeline faster than missed true findings.
Tip
Ask the agent to include the raw LLM response in a debug field of each finding. This makes it possible to audit the analyzer's reasoning when you disagree with a classification.
How accurate is LLM-based API hallucination detection?

For widely-used libraries (React, Flask, pandas, Express), current-generation models detect hallucinated methods and parameters with roughly 85–92% precision, based on informal benchmarks. Accuracy drops significantly for niche libraries, internal APIs, or recently released versions. A practical mitigation is to provide the agent with a list of known valid API surfaces for your project's dependencies — either extracted from type stubs or generated from documentation. This narrows the search space and converts a fuzzy reasoning task into a more constrained lookup.

✓
At this point you should have two artifacts: a verification schema that defines the structure of inputs and outputs, and a diff analyzer that parses unified diffs and classifies changes into the five failure-mode categories. Test the analyzer by feeding it a small, hand-crafted diff containing one obvious hallucinated API call (e.g., `requests.get_async()`) and confirming it produces a correctly categorized finding.

Quick Check

The diff analyzer needs to evaluate whether an API call is hallucinated. Which approach produces more reliable results?
✓ Correct! Grounding the analysis in concrete dependency information constrains the model's reasoning. Rather than relying on general training knowledge (which may reflect a different library version), the model can make targeted judgments: 'This project uses requests 2.31; does requests 2.31 expose a get_async method?' This reduces both false positives and false negatives.
✗ Not quite. General knowledge is sufficient for obvious cases but fails on version-specific changes. A method that was deprecated in v3.0 and removed in v4.0 will appear valid to a model trained on v2.x documentation. Without version grounding, the analyzer cannot distinguish between 'this API existed in an older version' and 'this API was hallucinated.' The result is a higher false-negative rate for precisely the most dangerous class of errors.
3

Generate Property-Based Tests as a Safety Net

Detection is valuable; remediation is better. The most durable form of remediation for verification debt is not a comment on a pull request — it is a test that will fail if the identified risk materializes.

Property-based testing is particularly well-suited to AI-generated code verification. Unlike example-based tests (which assert that f(3) returns 9), property-based tests assert invariants — relationships that must hold across all inputs. For instance: "for any non-negative integer n, square(n) returns a value greater than or equal to n." This approach is powerful here because the failure modes we are detecting are precisely the kind that property-based tests catch: edge cases the original author did not consider.

The agent's task is to examine each finding in the verification report and, where applicable, generate a property-based test that would catch the identified issue. Not every finding maps to a test — a style violation, for example, is better addressed by a linter rule. But edge-case gaps, assumption leaks, and untested branches all have natural property-based test expressions.

The conceptual framework is: for each finding, identify the invariant that the code should maintain, and express that invariant as a testable property. The agent should use a property-based testing library appropriate to the project's language — Hypothesis for Python, fast-check for TypeScript, or QuickCheck for Haskell.

Ask your agent
Get the agent to build a test generator that reads a VerificationReport, filters for testable findings, and produces property-based tests targeting each one.
Think about it
  • Which categories of findings are amenable to property-based testing, and which are not? How should the agent handle non-testable findings?
  • A property-based test requires two things: a strategy for generating inputs and a property (invariant) to assert. How should you instruct the agent to derive each from a finding?
  • Should the generated tests be written to a single file or organized by the source file they correspond to? What makes the output more maintainable?
  • How should the agent handle findings where it cannot confidently identify the invariant — skip them, or generate a weaker assertion with a TODO comment?
What the agent gives back

The agent should produce a module with a main function that accepts a VerificationReport and returns a collection of test files. For each testable finding (primarily edge-case gaps and assumption leaks), it should generate a property-based test using Hypothesis (or equivalent). Each generated test should include a docstring referencing the original finding ID, a strategy that generates inputs in the domain identified by the finding (e.g., empty strings, negative numbers, Unicode), and a property assertion. For non-testable findings, the generator should emit a comment-only placeholder explaining why no test was generated. The output should be organized by source file — one test file per source file with findings.

Tip
Instruct the agent to generate tests that are deliberately minimal — one property per test, no setup complexity. Property-based tests that are themselves complex become a new source of verification debt.
Why property-based tests instead of example-based tests?

Example-based tests verify the cases the test author thought of. Property-based tests verify invariants across the space of inputs the test author did not think of — which is exactly the space where verification debt hides. When an AI agent generates a function, it typically handles the examples it was implicitly trained on. The gaps are in the boundaries: what happens at zero, at maximum integer, at empty input, at concurrent access. A property-based test framework like Hypothesis will systematically explore these boundaries, and when it finds a failure, it will shrink the failing input to the minimal reproducing case. This shrinking behavior is especially valuable in an automated pipeline because it produces findings that are immediately actionable rather than buried in noise.

4

Assemble the Pipeline and Define the Feedback Loop

We now have three components: a schema, a diff analyzer, and a test generator. The final step is to compose them into a pipeline that can be invoked on any AI-generated diff — whether triggered manually, by a Git hook, or by a CI system.

The pipeline's architecture is linear: diff input → syntactic parse → semantic analysis → report generation → test generation → output. However, the feedback loop is what transforms this from a one-shot tool into a durable practice.

The feedback loop operates as follows. When the verification agent produces findings, those findings are fed back to the original code-generating agent as context for revision. The code-generating agent addresses the findings and produces a new diff. That new diff is fed through the verification pipeline again. This cycle continues until the report contains no critical findings — or until a human reviewer decides the remaining informational findings are acceptable.

This iterative pattern is the core of responsible agentic coding. The code-generating agent is fast but imprecise; the verification agent is slower but systematic. Together, they approximate the rapid-drafting-then-careful-review cycle that experienced human developers perform internally.

Ask your agent
Get the agent to compose the three components into a single callable pipeline with a clear entry point, and add a feedback-loop mechanism that passes findings back to a code-generating agent for remediation.
Think about it
  • What should the pipeline's entry point accept — a file path to a diff, a raw string, or a Git commit reference? Which is most flexible for both local and CI use?
  • How should the feedback loop terminate? What prevents an infinite cycle of generation and verification?
  • Should the pipeline persist intermediate results (the raw report, the generated tests) or only the final output? Consider debugging and auditability.
  • What happens when the code-generating agent disagrees with a finding — how should that dispute be represented?
What the agent gives back

The agent should produce a pipeline orchestrator — a single entry-point function or CLI command that accepts a diff (as a string, file path, or Git ref), runs it through the analyzer, generates the report, produces property-based tests, and optionally enters a feedback loop. The feedback loop should have a configurable maximum iteration count (defaulting to 3) and a termination condition (no critical findings remaining). Each iteration's report should be persisted to a timestamped file for auditability. The orchestrator should also accept a flag to skip the feedback loop for use in read-only audit mode. The overall design should be a straightforward sequential pipeline — no need for async orchestration or complex state management.

Warning
Set a maximum iteration count for the feedback loop. Without a bound, a pathological interaction between the generator and verifier can produce an infinite cycle — the generator 'fixes' a finding in a way that introduces a new one, which the verifier flags, and so on. Three iterations is a reasonable default; if critical findings persist after three rounds, escalate to human review.
API Key Note
The pipeline makes LLM calls in the diff analyzer and potentially in the test generator. Ensure your API key is available as an environment variable — do not hardcode it in the pipeline. Ask the agent to read it from the environment and fail with a clear error message if it is absent.
How does this fit into an existing CI/CD workflow?

The most natural integration point is as a GitHub Action or GitLab CI job that triggers on pull requests where the author is an AI agent (or where the PR is labeled as AI-generated). The pipeline runs as a check, and its findings are posted as inline review comments using the code review API. Critical findings block the merge; warnings and informational findings appear as suggestions. The generated property-based tests can be committed to a tests/generated/ directory automatically, or offered as a suggested commit. This integration pattern means the verification pipeline operates transparently — developers see its output in the same interface they use for human reviews, and they can dismiss or accept findings using the same workflow.

✓
You should now have a complete verification pipeline: schema → diff analyzer → report → test generator → feedback loop. Test the full pipeline by generating a small Python function with an AI agent (e.g., 'write a function that divides two numbers'), capturing the diff, and running it through your pipeline. The analyzer should identify at least one edge-case gap (division by zero) and the test generator should produce a Hypothesis test for it.

Instrument and Measure Verification Debt Over Time

A verification pipeline that runs once is a tool. A verification pipeline that runs continuously and tracks its findings over time is a practice. The final component of our system is instrumentation: recording findings across runs so that teams can measure their verification debt and observe whether it is growing or shrinking.

The key metrics are:

  • Finding density: the number of findings per 100 lines of AI-generated diff. A rising density suggests the generating agent is degrading in quality or the codebase is becoming more complex.
  • Category distribution: the proportion of findings in each of the five categories. A spike in API hallucinations might indicate a model update or a dependency version change. A steady increase in style drift suggests the project's conventions are not being communicated to the generating agent.
  • Resolution rate: the fraction of findings resolved by the feedback loop without human intervention. A declining resolution rate means the generator and verifier are increasingly disagreeing — a signal that human oversight should increase.
  • Test coverage delta: the number of property-based tests generated per run. This measures the pipeline's remediation output, not just its detection output.

These metrics transform verification debt from an invisible, anxiety-producing concept into a measurable quantity that teams can manage rationally.

Ask your agent
Get the agent to add a metrics collection and reporting layer to the pipeline that persists findings across runs and produces a summary dashboard.
Think about it
  • What is the simplest persistence mechanism that works for a single developer and scales to a team? Consider the trade-off between a database and flat files.
  • How should findings be deduplicated across runs — by exact match, by location, or by semantic similarity?
  • What time granularity is useful for trend analysis — per-commit, daily, weekly? How does this depend on the team's AI-generation volume?
  • Should the dashboard be a CLI summary, an HTML report, or data exported to an existing observability tool? What is most likely to actually be read?
What the agent gives back

The agent should add a lightweight metrics layer that appends each run's VerificationReport to a JSON Lines file (one JSON object per line, trivially parseable). It should include a summary command that reads the log file and prints: total findings by category and severity over a configurable time window, finding density trend (per-run average), resolution rate (findings resolved in feedback loop vs. escalated), and test generation count. The output format should be a clean CLI table for daily use, with an optional JSON export flag for integration with external dashboards. No database, no server — just append-only file storage and a read-time aggregation function.

Tip
The most important metric is not the absolute number of findings — it is the trend. A project with 50 findings that is trending downward is healthier than a project with 10 findings that is trending upward. Frame your dashboard around directional movement, not snapshot counts.
What does healthy verification debt look like?

Zero verification debt is not the goal — it would mean either that AI-generated code is perfect (unlikely) or that the verification criteria are too lenient. Healthy verification debt looks like this: finding density is stable or slowly declining; the category distribution is dominated by informational findings rather than critical ones; the resolution rate is above 70%, meaning the feedback loop handles most issues automatically; and the test coverage delta is positive, meaning each AI-generated change leaves the test suite stronger than it found it. The most important signal is whether critical findings are being caught before merge. If the pipeline's critical-finding escape rate (critical findings discovered in production that the pipeline missed) is near zero, the system is working — even if the total finding count is high.

Recap

We have constructed a five-component verification pipeline: a schema that defines the structure of findings, a diff analyzer that classifies AI-generated changes against a taxonomy of failure modes, a test generator that produces property-based tests for each testable finding, an orchestrator that composes these into a feedback loop with the original code-generating agent, and a metrics layer that tracks verification debt over time.

The central insight of this session is that AI code generation and AI code verification are complementary capabilities that should be deployed together. A code-generating agent without a verification agent is like a writer without an editor — productive, but unreliable. The verification agent does not need to be perfect; it needs to be systematic, catching the classes of errors that human reviewers are most likely to overlook under time pressure: edge cases, hallucinated APIs, and silent style drift.

Verification debt is real, it compounds, and — as of today — it is measurable. The pipeline we built is intentionally lightweight. It uses no specialized infrastructure, no databases, no deployment orchestration. It is a set of functions that call an LLM with well-crafted prompts and structured output schemas. That simplicity is a feature: it means you can integrate this into your workflow tomorrow, not after a quarter of infrastructure work.

  • Verification debt is the gap between the speed of AI code generation and the thoroughness of human review. It compounds silently and manifests as edge-case failures, style drift, and hallucinated API calls in production.
  • A two-pass analysis architecture — syntactic parsing followed by semantic evaluation — produces more reliable and debuggable results than a single-pass approach. Separate what you can verify mechanically from what requires judgment.
  • Property-based tests are the natural remediation for verification debt because they test invariants across input spaces, catching exactly the edge cases that AI-generated code is most likely to miss.
  • The feedback loop between a generating agent and a verifying agent approximates the draft-then-review cycle of experienced developers. Bound it with a maximum iteration count and escalate persistent disagreements to human review.
  • Measuring verification debt over time — finding density, category distribution, resolution rate — transforms an invisible anxiety into a manageable metric.

Where to go next

  • Integrate the pipeline as a Git pre-push hook or CI check that runs automatically on AI-generated pull requests.
  • Expand the failure-mode taxonomy with project-specific categories (e.g., security patterns, performance anti-patterns) by adding new categories to the schema and updating the analyzer's system prompt.
  • Experiment with providing the analyzer access to your project's type stubs and API documentation to improve hallucination detection accuracy for internal and niche libraries.
  • Build a team-level dashboard that aggregates verification metrics across repositories to identify which projects carry the most verification debt.
