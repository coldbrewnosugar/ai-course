Build Reusable Agent Skills with the OpenAI Skills Open Standard

Dissect the skill anatomy, build a custom skill from scratch, and understand how a folder convention could become the package manager for agent capabilities
35 min agent-skills codex open-standard tool-building reusable-modules

What's happening

OpenAI has open-sourced a Skills Catalog — a standardized convention for packaging instructions, scripts, and resources into portable folders that AI coding agents can discover and execute autonomously. The repository, trending on GitHub, introduces what OpenAI calls the Agent Skills open standard: a write-once, use-everywhere model for agent capabilities.

The significance is structural, not merely technical. Today, when a team develops a reliable prompting workflow — say, a code-review checklist or a test-scaffolding procedure — that knowledge lives in someone's head, a wiki page, or a one-off system prompt. The Skills standard proposes a file-system convention that makes such workflows discoverable, portable, and composable. A skill is a folder. That folder contains natural-language instructions (what to do), optional scripts (how to do it), and optional resources (reference material). Any compatible agent can read the folder and execute the skill.

The catalog distinguishes three tiers: system skills (bundled with Codex by default), curated skills (vetted and installable by name), and experimental skills (community-contributed, installed by path or URL). A built-in $skill-installer handles installation. The analogy to package management is deliberate — OpenAI appears to be laying groundwork for an ecosystem where agent capabilities are shared, versioned, and composed much like software libraries.

In this session, we examine the skill anatomy in detail, use an AI agent to construct a custom skill from scratch, install it, and evaluate the portability model that underpins the standard.

1

Understand the Skill Anatomy: What a Skill Actually Is

Before building anything, we need a precise mental model of what a skill is. A skill is not a plugin, not an API integration, and not a code library. It is a folder whose contents follow a convention that agents can parse.

The convention is intentionally simple. At minimum, a skill folder contains an instructions.md file — a natural-language document that tells the agent what the skill does, when to use it, and how to execute it. This is the skill's core: it is prose, not code. The agent reads it the same way a human developer would read a README.

Optionally, a skill folder may also contain:

  • Scripts — executable files (shell scripts, Python scripts) that the agent can invoke as part of the skill's workflow. These handle tasks that benefit from deterministic execution: running linters, executing test suites, making API calls.
  • Resources — reference files (templates, schemas, example outputs) that provide context the agent can consult while executing the skill.

The folder structure typically looks like this:

```

my-skill/

instructions.md

scripts/

run-check.sh

resources/

template.md

```

This simplicity is the point. The skill is human-readable, version-controllable, and agent-executable. There is no compilation step, no manifest schema to memorize, no runtime dependency. The convention is closer to a well-organized documentation folder than to a software package.

What the agent gives back

It should define each of the three tiers: system skills are auto-installed with Codex, curated skills are vetted and installable by name via $skill-installer, and experimental skills require a full path or URL. Tip
The instructions.md file is the single most important component. A skill with excellent instructions and no scripts will outperform a skill with elaborate scripts and vague instructions. The agent's ability to execute the skill depends entirely on the clarity of this document.
Why a folder convention instead of a manifest file?

Many plugin and extension systems require a structured manifest — a package.json, a plugin.yaml, or similar. The Skills standard deliberately avoids this. The reasoning is that agents parse natural language more reliably than they parse configuration schemas. An instructions.md file can contain nuanced context, conditional logic described in prose, and examples — all of which an LLM-based agent handles well. A rigid manifest, by contrast, would constrain what a skill can express and would require the agent to implement a parser for every schema variation. The folder convention optimizes for the agent's strengths: reading and following natural-language instructions.

2

Design a Custom Skill: The Instructions-First Approach

We will now build a custom skill — a code-review skill that instructs an agent to perform a structured review of a pull request or code change. The skill will check for common issues (error handling, naming conventions, test coverage gaps) and produce a formatted review summary.

The critical design decision occurs before any file is created: what should instructions.md contain? This document must accomplish three things simultaneously:

  1. Declare the skill's purpose — a clear, one-sentence statement of what the skill does and when it should be activated.
  2. Specify the procedure — an ordered sequence of steps the agent should follow, written with enough precision that the agent's behavior is predictable and repeatable.
  3. Define the output format — what the agent should produce when the skill completes, so that results are consistent across invocations.

Think of instructions.md as a standing prompt — a prompt that persists across sessions and is automatically injected whenever the skill is triggered. Every principle of effective prompt engineering applies: be specific, provide examples, state constraints explicitly, and anticipate edge cases.

Ask your agent
Ask your agent to draft the `instructions.md` file for a code-review skill. The skill should instruct an agent to review code changes for five specific categories: error handling, naming clarity, test coverage, security concerns, and performance considerations. The output should be a structured review document with severity ratings.
Think about it
  • What activation trigger should the instructions specify — should the skill run automatically on every change, or only when explicitly invoked?
  • How should severity be expressed — numeric scores, categorical labels (critical/warning/note), or something else?
  • What should the agent do if it finds no issues in a category — skip it silently, or explicitly state that the category passed?
  • How much context about the codebase should the instructions assume the agent already has?
What the agent gives back

The agent should produce a well-structured Markdown document of roughly 40–60 lines. It should open with a purpose statement ('This skill performs a structured code review...'), followed by a numbered procedure: (1) identify the files under review, (2) examine each file against the five categories, (3) assign a severity to each finding, (4) compile findings into a summary table. The document should specify that categories with no findings should be reported as 'No issues found' rather than omitted. It should define the output format — likely a Markdown document with a summary section and a per-file findings section, each finding tagged with a severity label.

Warning
Avoid the temptation to make instructions overly prescriptive about implementation. Statements like 'use the ast module to parse Python files' constrain the agent unnecessarily. Instead, state the goal: 'Identify unused imports in Python files.' The agent will select appropriate methods based on context.
How detailed should instructions.md be?

There is a productive tension between brevity and precision. Instructions that are too short leave the agent to improvise, producing inconsistent results. Instructions that are too long may cause the agent to lose focus on key requirements (a well-documented limitation of long-context processing). A practical guideline: each step in the procedure should be one to three sentences. Include one concrete example of expected output — this anchors the agent's behavior more effectively than additional paragraphs of description. If the instructions exceed 100 lines, consider whether the skill should be decomposed into two or more smaller skills.

✓
At this point you should have a clear mental model of the skill folder structure and a drafted `instructions.md` for a code-review skill. The instructions should specify purpose, procedure, and output format. If any of these three elements is missing, revisit Step 2 before continuing.

Add Scripts and Resources to Complete the Skill

With instructions.md drafted, we now add the optional components that make the skill more robust: a helper script and a resource template.

The script serves a specific purpose: it performs a deterministic operation that should not vary between invocations. For our code-review skill, a useful script might collect metadata about changed files — file names, line counts, languages detected — and present this as structured input for the agent's review. The agent can then focus its reasoning on the review itself rather than on file discovery.

The resource provides reference material. For a code-review skill, this might be an output template — a Markdown document with pre-defined sections (Summary, Findings by File, Recommendations) that the agent fills in. Templates improve consistency: every review produced by the skill follows the same structure.

The completed skill folder would contain three items:

```

code-review/

instructions.md

scripts/

collect-changes.sh

resources/

review-template.md

```

The instructions.md references both: it tells the agent to run scripts/collect-changes.sh to gather file information, and to use resources/review-template.md as the output scaffold.

Ask your agent
Ask your agent to produce two files: (1) a short shell script that lists changed files with their line counts and detected languages, and (2) a Markdown template for the review output. Then ask the agent to update `instructions.md` to reference both files.
Think about it
  • What mechanism should the shell script use to detect changed files — should it assume Git, or should it be VCS-agnostic?
  • How should the review template balance structure with flexibility — should every section be mandatory, or should some be conditional?
  • When you ask the agent to update `instructions.md`, how should you indicate *where* in the document the references should appear?
What the agent gives back

The agent should produce a shell script of roughly 10–15 lines that uses git diff --name-only (or a similar Git command) to list changed files and annotates each with basic metadata. The review template should be a Markdown skeleton with clearly labeled sections: an executive summary block, a per-file findings table (columns: file, category, severity, description), and a recommendations section. The updated instructions.md should reference the script in its procedure ('Run scripts/collect-changes.sh to identify files under review') and the template in its output section ('Format the review according to resources/review-template.md').

Tip
Scripts in a skill should be self-contained and idempotent. They should not install dependencies, modify global state, or assume a specific working directory. A script that fails gracefully (with a clear error message) is more valuable than one that handles every edge case but is fragile.
Can a skill contain multiple scripts?

Yes. A skill's scripts/ directory can contain any number of executable files. Common patterns include: a setup script that validates prerequisites, a main execution script, and a cleanup script. The instructions.md file specifies the order and conditions under which each script runs. However, resist the urge to create a complex multi-script pipeline. If a skill requires more than three or four scripts, it likely encompasses too broad a responsibility and should be decomposed into multiple focused skills that can be composed together.

4

Install and Test the Skill

A skill that exists only as files in a folder is a draft. It becomes operational when it is installed into an agent's skill discovery path and tested against real input.

The OpenAI Skills standard provides a $skill-installer command that handles installation within Codex. For curated skills — those vetted and published in the official catalog — installation is by name:

```

$skill-installer gh-address-comments

```

For custom or experimental skills, installation requires a path or URL. If you have published your skill to a GitHub repository, anyone can install it by providing the directory URL:

```

$skill-installer install https://github.com/yourname/skills/tree/main/code-review

```

After installation, the agent must be restarted to discover the new skill. This is an important architectural detail: skill discovery happens at agent startup, not dynamically during a session. The agent scans its skill directories, reads each instructions.md, and indexes the available capabilities.

Testing is straightforward but requires discipline. Run the skill against a known code change — one where you already know what a good review should find. Compare the agent's output against your expectations. If the output misses issues you consider important, the problem is almost certainly in instructions.md, not in the agent itself. Revise the instructions and test again.

Ask your agent
Ask your agent to create a test plan for validating your code-review skill. The plan should define three test cases: a clean file with no issues, a file with obvious problems across multiple categories, and an edge case (e.g., a very large file or a file in an unfamiliar language).
Think about it
  • What makes a good test case for a skill that produces natural-language output rather than deterministic results?
  • How should you evaluate whether the skill's severity ratings are calibrated correctly — what counts as 'critical' versus 'note'?
  • What should happen when the skill encounters a file type it has no specific knowledge about?
What the agent gives back

The agent should produce a structured test plan with three scenarios. Each scenario should specify: the input (a description or example of the code to review), the expected behavior (which categories should flag findings, at what severity), and acceptance criteria (what would constitute a pass or fail). The clean-file case should verify that the skill reports 'No issues found' rather than hallucinating problems. The problematic-file case should verify coverage across all five review categories. The edge case should verify graceful degradation — the skill should acknowledge its limitations rather than produce confident but unreliable analysis.

Warning
After installing a skill, you must restart the agent for it to be discovered. Skills are indexed at startup. If your newly installed skill does not appear, verify that it is in the correct directory and that instructions.md is present and well-formed.
The distribution model: skills as shared infrastructure

The $skill-installer accepting GitHub URLs is a deliberate design choice that mirrors early package management patterns. Any developer can publish a skill by pushing a folder to a public repository. Any other developer can install it by URL. There is no registry to publish to, no approval process for experimental skills, and no versioning mechanism beyond Git itself. This low-friction model encourages experimentation but places the burden of quality assessment on the consumer. Over time, the curated tier — which does involve vetting — may evolve into something resembling a curated package registry, but the current model prioritizes accessibility over governance.

✓
You should now have a complete skill folder with three components: `instructions.md`, a helper script in `scripts/`, and an output template in `resources/`. You should also have a test plan with at least three scenarios. The skill is ready for installation and validation.

Composability and the Ecosystem Ahead

A single skill is useful. A composable set of skills is transformative. The Skills open standard implicitly supports composition: since each skill is an independent folder with its own instructions, an agent with multiple installed skills can draw on any combination of them during a session.

Consider a workflow where three skills are installed: the code-review skill we built, a test-scaffolding skill, and a documentation-generation skill. An agent tasked with 'prepare this PR for review' could invoke all three in sequence — reviewing the code, generating missing tests, and updating documentation — without any explicit orchestration logic. The agent decides which skills to apply based on the instructions it has indexed.

This composability model has implications worth examining:

  • Skill conflicts: What happens when two skills give contradictory instructions for the same task? The standard does not currently define a priority mechanism. In practice, the agent resolves conflicts using its own judgment, which may produce unpredictable results.
  • Skill discovery at scale: As the number of installed skills grows, the agent must search a larger instruction set to find relevant capabilities. This is analogous to the problem of dependency bloat in traditional package managers.
  • Organizational skill libraries: Teams can maintain private skill repositories — a shared folder of vetted skills that encode the team's engineering standards, review checklists, and workflow patterns. This transforms institutional knowledge from implicit (in people's heads) to explicit (in version-controlled folders).

The standard is young. Its conventions will evolve. But the core insight — that agent capabilities can be packaged, shared, and composed using nothing more than folders of natural-language instructions — represents a meaningful step toward a more structured agent ecosystem.

Ask your agent
Ask your agent to design a three-skill composition for a 'PR readiness' workflow. The agent should describe how the code-review, test-scaffolding, and documentation skills would interact, identify potential conflicts, and propose a lightweight coordination strategy.
Think about it
  • Should one skill's output feed into another skill's input, or should they operate independently on the same source material?
  • What happens if the code-review skill flags a function as problematic and the test-scaffolding skill generates tests for that same function — is this a conflict or complementary?
  • How might `instructions.md` in each skill acknowledge the existence of other skills without creating tight coupling?
What the agent gives back

The agent should produce a written analysis (not code) describing the composition. It should identify the execution order (review first to identify issues, then scaffolding for untested code, then documentation for public APIs), explain that each skill operates on the same file set but produces distinct outputs, and note that conflicts are unlikely because the skills address orthogonal concerns. The coordination strategy might suggest that each skill's instructions.md include a section on 'integration with other skills' — a soft convention where a skill acknowledges that its output may be consumed alongside other skills' outputs.

Tip
When designing skills for composition, favor narrow responsibility. A skill that does one thing well composes better than a skill that tries to handle multiple concerns. This mirrors the Unix philosophy: small, focused tools connected through simple interfaces.
Could this become the 'npm for agent capabilities'?

The comparison is instructive but imperfect. npm succeeds because of deterministic execution: npm install express produces identical results everywhere. Agent skills, by contrast, are interpreted by an LLM, so execution is inherently probabilistic. Two agents running the same skill on the same input may produce slightly different results. This means the ecosystem needs different quality signals than traditional package management. Star counts and download numbers are less informative than consistency metrics — how reliably does this skill produce correct output across diverse inputs? The Skills standard does not yet address this, but any mature skill ecosystem will need some form of reliability scoring to complement the distribution mechanism.

Recap

In this session, we examined the OpenAI Agent Skills open standard — a folder-based convention for packaging agent capabilities as portable, discoverable modules. We constructed a complete code-review skill with three components (instructions.md, a helper script, and an output template), designed a test plan to validate it, and explored how skills compose into larger workflows.

The central insight is architectural: by standardizing how agent instructions are organized and discovered, the Skills convention transforms ad-hoc prompting knowledge into shareable, version-controlled infrastructure. The instructions.md file is the critical element — it is a standing prompt that encodes expertise in a form any compatible agent can execute.

The standard is early-stage and deliberately minimal. It does not address versioning, conflict resolution, or reliability metrics. But its simplicity is also its strength: the barrier to creating and sharing a skill is negligible, which is precisely the condition needed for ecosystem growth.

  • A skill is a folder with an instructions.md file (required), optional scripts for deterministic operations, and optional resources for reference material. There is no build step, no manifest, and no runtime dependency.
  • The quality of a skill depends almost entirely on the quality of its instructions.md. Treat it as a standing prompt: be specific about purpose, procedure, and output format. Include examples.
  • Skills compose naturally because they address independent concerns. Design skills with narrow responsibility to maximize composability and minimize conflicts.
  • Distribution is Git-native: push a folder to a repository, install by URL. The low friction mirrors early package ecosystems and prioritizes accessibility over governance.
  • The standard optimizes for LLM strengths — reading and following natural-language instructions — rather than imposing structured configuration that would constrain expressiveness.

  • Experiment with skill composition: install three or more complementary skills and observe how the agent selects and sequences them during complex tasks
