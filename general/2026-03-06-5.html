Build a Tool-Using Agent with Qwen Open-Source Models

Deploy Qwen locally via vLLM and construct an agentic loop with custom tool calls — fully open-source, fully on your hardware
35 min open-source-llm tool-use agents qwen local-deployment

What's happening

Alibaba Cloud's Qwen model family represents a significant milestone in open-source large language models. The series spans from 1.8B to 72B parameters, with each variant available in base, chat-aligned, and quantized (Int4/Int8) forms. Crucially, Qwen-Chat models ship with native tool-use capability — the chat template includes a structured format for declaring functions, receiving tool-call requests from the model, and feeding results back into the conversation.

This matters because tool use is the mechanism that transforms a language model from a text generator into an agent. An agent observes its environment, selects an action (a tool call), interprets the result, and decides what to do next. Qwen's built-in support for this pattern means we can build agentic workflows without relying on proprietary APIs, running everything locally on consumer or workstation GPUs.

The Qwen repository documents support for vLLM serving, GPTQ quantization, Q-LoRA fine-tuning, and a ReAct-style prompting format for tool use. In this session, we assemble these components into a working agent that can plan and execute multi-step tasks using custom Python functions — all orchestrated through careful prompting of an AI coding assistant.

1

Serve a Quantized Qwen-Chat Model Locally with vLLM

Before we can build an agent, we need an inference endpoint. vLLM is a high-throughput serving engine that supports paged attention, continuous batching, and GPTQ-quantized models out of the box. Our first task is to get a quantized Qwen-Chat model running locally behind an OpenAI-compatible API.

The key architectural decision here is quantization level. Qwen offers Int4 and Int8 variants:

  • Int4 (GPTQ-Int4): Reduces model weights to 4-bit precision. A 14B-parameter model fits in roughly 13 GB of VRAM. There is a measurable quality degradation, particularly on tasks requiring precise numerical reasoning, but for tool-use routing — where the model's primary job is selecting the right function and extracting arguments — Int4 is generally sufficient.
  • Int8: Preserves more precision at roughly double the memory cost. Preferable when GPU memory permits, especially for the 7B model where Int8 still fits comfortably on a 16 GB GPU.

The goal in this step is not to memorize command-line flags. It is to understand what we need from a serving layer and how to instruct an AI agent to produce a working deployment script.

Ask your agent

Think about it

  • vLLM can expose an OpenAI-compatible server. What does that mean for how downstream code will interact with the model?
  • What parameters matter for serving — maximum sequence length, GPU memory utilization fraction, tensor parallelism for multi-GPU setups?
  • How should the agent handle the difference between Hugging Face model IDs and local paths?
What the agent gives back

The agent should produce a short shell command or script that installs vLLM and launches the server. The essential elements are: the correct Hugging Face model identifier (e.g., Qwen/Qwen-14B-Chat-Int4), the --quantization gptq flag, a reasonable --max-model-len (e.g., 8192), and the --served-model-name parameter so downstream code can reference a stable name. The server should listen on a configurable port and expose /v1/chat/completions. A minimal example:

```bash

python -m vllm.entrypoints.openai.api_server \

--model Qwen/Qwen-14B-Chat-Int4 \

--quantization gptq \

--max-model-len 8192 \

--gpu-memory-utilization 0.9

```

The agent should note that this exposes an OpenAI-compatible endpoint at http://localhost:8000/v1.

Warning
GPTQ quantization requires the auto-gptq library installed alongside vLLM. If the agent's script omits this dependency, ask it to include installation steps for auto-gptq and optimum.
Tip
If you have fewer than 10 GB of VRAM, consider the Qwen-7B-Chat-Int4 variant (approximately 8.2 GB for 2048-token generation). For CPU-only setups, vLLM is not suitable — you would need llama.cpp with a GGUF conversion, which is outside the scope of this session.
Why vLLM instead of a simple Transformers pipeline?

A raw Hugging Face pipeline loads the model and processes one request at a time. vLLM implements PagedAttention, which manages the KV cache in non-contiguous memory blocks (analogous to virtual memory paging in operating systems). This enables continuous batching — multiple requests share GPU resources concurrently — and dramatically improves throughput. For an agentic loop that may issue many sequential calls, low latency per call is critical, and vLLM's architecture delivers this even on a single GPU.

Int4 vs. Int8: when does quality matter?

Quantization compresses model weights from 16-bit floating point to lower precision. The trade-off is between memory savings and representational fidelity. For tool-use tasks, the model must reliably (1) select the correct function name, (2) extract arguments from user input, and (3) format a valid JSON call. Empirically, Int4 handles these structured-output tasks well because the model's function-selection behavior is robust to moderate precision loss. Where Int4 struggles is in tasks requiring subtle reasoning or precise arithmetic — domains where the lost precision affects intermediate computation. For this workshop, Int4 is the recommended default unless you have ample VRAM.

All subsequent steps will interact with this endpoint as if it were any OpenAI-style API — the only difference is that it runs entirely on your local machine.

Understand Qwen's Tool-Use Prompt Format

Qwen-Chat models support tool use through a specific system prompt structure. This is not a general-purpose hack — it is a format the model was explicitly trained on during supervised fine-tuning. Understanding this format is essential because it determines how the model decides to invoke tools, how it structures its requests, and how it interprets results.

The Qwen tool-use protocol follows a ReAct-style pattern (Reason + Act). At each turn, the model may:

  1. Reason — produce a natural-language thought about what to do next.
  2. Act — emit a structured tool call specifying a function name and arguments.
  3. Observe — receive the tool's return value injected into the conversation.
  4. Repeat or conclude — decide whether further tool calls are needed, or produce a final answer.

The system prompt declares available tools as a JSON schema. Each tool has a name, description, and a parameters object following JSON Schema conventions. The model was trained to recognize this format and respond with structured ✿FUNCTION✿ markers when it decides a tool call is appropriate.

This step focuses on getting the agent to produce a well-structured tool declaration — the system prompt that tells Qwen what tools exist.

Ask your agent
Ask your agent to generate a Qwen-compatible system prompt that declares two tools: a web search function and a Python code execution function.
Think about it
  • What information does each tool declaration need? Think about what the model requires to decide *when* to call a tool and *what arguments* to pass.
  • Qwen expects tools declared in a specific JSON Schema format within the system prompt. How would you ask the agent to match Qwen's expected format rather than, say, OpenAI's function-calling schema?
  • For the web search tool, what parameters are essential? For code execution, what safety considerations should the description convey?
  • How should the tool descriptions guide the model's reasoning about when to use each tool?
What the agent gives back

The agent should produce a system prompt string containing two tool declarations in Qwen's expected format. Each tool should have a name_for_model, name_for_human, description_for_model, and a parameters object with JSON Schema types. The web search tool should accept a query string parameter. The code execution tool should accept a code string parameter. The descriptions should be precise enough that the model can distinguish when to use each tool — e.g., the search tool for factual lookups, the code tool for computation or data manipulation. The overall structure nests these declarations within the Qwen system prompt template using the # Tools section header that the model was trained to recognize.

Tip
Tool descriptions function as instructions to the model. A vague description like 'runs code' gives the model little basis for deciding when to invoke the tool. A precise description like 'Executes Python 3 code in a sandboxed environment and returns stdout/stderr. Use for arithmetic, data transformation, or any task requiring computation' significantly improves tool selection accuracy.
Warning
Qwen's tool-use format differs from OpenAI's function-calling API. If your agent produces OpenAI-style functions declarations, explicitly ask it to use Qwen's native ReAct format with the ✿FUNCTION✿ markers. The model will not reliably respond to formats it was not trained on.
The ReAct pattern explained

ReAct (Reasoning + Acting) is a prompting paradigm introduced by Yao et al. (2023). The core insight is that interleaving natural-language reasoning traces with concrete actions improves both the reliability of action selection and the interpretability of the agent's behavior. Without reasoning traces, a model might select tools based on shallow pattern matching. With them, the model explicitly articulates why it is choosing a particular tool, which creates a self-correcting mechanism: if the reasoning is flawed, it often becomes apparent in the trace before the action is taken. Qwen's training incorporated this pattern, so the model naturally produces Thought: prefixes before ✿FUNCTION✿ calls.

Why JSON Schema for parameters?

JSON Schema is a vocabulary for annotating and validating JSON documents. By expressing tool parameters as JSON Schema, Qwen can leverage the structural constraints during generation — it knows that a parameter of type string should produce a quoted value, that required fields must be present, and that enum constraints limit valid values. This is not merely a convention; it provides the model with machine-readable guardrails that improve the syntactic validity of generated tool calls. When defining your own tools, using precise JSON Schema types (including description fields on individual parameters) gives the model maximum information for correct argument generation.

Build the Agentic Loop: Orchestrating Tool Calls

With a model served and tool declarations defined, we now construct the agentic loop — the control flow that sends messages to the model, detects tool-call requests in the response, executes the corresponding functions, and feeds results back.

Conceptually, an agentic loop is a state machine with three states:

  1. Query — send the current conversation history to the model and receive a response.
  2. Parse — inspect the response for tool-call markers. If none are found, the model has produced a final answer; exit the loop.
  3. Execute — call the indicated function with the provided arguments, append the result to the conversation as an Observation, and return to the Query state.

The loop terminates when the model produces a response without tool-call markers, or when a maximum iteration count is reached (a safety measure against infinite loops).

This is the architectural core of any tool-using agent. The model itself is stateless — it has no memory between calls. The loop maintains state by accumulating the full conversation history, including all prior thoughts, tool calls, and observations. Each iteration, the model sees the entire trajectory and decides the next action.

Ask your agent
Ask your agent to build a Python function that implements the agentic loop: it should send messages to the local vLLM endpoint, parse Qwen's tool-call markers from responses, execute the matching tool function, and iterate until a final answer is produced.
Think about it
  • The loop needs to interact with an OpenAI-compatible API. What client library can talk to a local vLLM server, and what base URL configuration is needed?
  • How should the function detect that the model wants to call a tool? What specific markers does Qwen-Chat use to signal a function call?
  • What data structure should maintain the conversation state across iterations? How do tool results get injected back into the conversation?
  • What termination conditions prevent the loop from running indefinitely? Consider both normal completion and error cases.
What the agent gives back

The agent should produce a Python function (roughly 20–30 lines, but we focus on the logic, not the code) that operates as follows: it initializes an OpenAI client pointed at http://localhost:8000/v1, maintains a messages list, and enters a while loop. On each iteration, it calls client.chat.completions.create(), inspects the response content for Qwen's ✿FUNCTION✿ markers, and if found, extracts the function name and arguments. It then looks up the function in a registry dictionary, calls it, and appends the result as an observation message. The loop exits when no tool-call marker is detected or a max-iteration limit (e.g., 10) is reached. The key structural insight is:

```python

while iterations < max_iterations:

response = client.chat.completions.create(...)

if '✿FUNCTION✿' in response_text:

name, args = parse_tool_call(response_text)

result = tool_registryname

messages.append(observation(result))

else:

return response_text

```

The agent should also define the parse_tool_call helper and the tool registry pattern.

Tip
The tool registry is simply a Python dictionary mapping function names (strings) to callable objects. This pattern decouples tool declaration (what the model knows about) from tool implementation (what actually runs). You can add, remove, or mock tools without modifying the loop logic.
Warning
Always set a maximum iteration limit. A model caught in a reasoning loop — calling the same tool repeatedly or alternating between two tools — will consume GPU cycles and potentially exhaust context length. A limit of 5–10 iterations is reasonable for most tasks.
Why maintain the full conversation history?

Language models are stateless functions: given an input sequence, they produce an output sequence. They have no persistent memory between calls. The agentic loop simulates continuity by appending every prior exchange — user message, model thought, tool call, observation — to the message list before each new call. This means the model 'sees' its own prior reasoning and can build upon it. The cost is that context length grows with each iteration, eventually hitting the model's maximum (8K for Qwen-14B, 32K for Qwen-72B). For long-running agents, strategies like summarizing prior observations or truncating early turns become necessary.

Parsing reliability and failure modes

Qwen's tool-call markers (✿FUNCTION✿, ✿ARGS✿, ✿RESULT✿) are special tokens the model was trained to emit. However, quantized models occasionally produce malformed output — a missing closing marker, invalid JSON in the arguments, or a hallucinated function name not in the registry. Robust agentic loops must handle these cases: validate JSON before parsing, check function names against the registry, and provide informative error messages back to the model as observations so it can self-correct. A well-prompted agent will include these error-handling paths in the generated code.

You now have the three foundational components: a locally served quantized model, a tool-declaring system prompt, and an agentic loop that orchestrates multi-step tool calls. Together, these form a complete — if minimal — tool-using agent running entirely on local hardware.

Implement and Register Custom Tools

The tools declared in the system prompt must correspond to actual Python functions that the agentic loop can invoke. This step focuses on tool implementation — writing the functions that perform real work when the model decides to call them.

A well-designed tool has several properties:

  • Deterministic interface: The function signature must match the JSON Schema declared in the system prompt. If the schema says the search tool accepts a query string, the Python function must accept exactly that parameter.
  • Bounded execution: Tools should have timeouts and resource limits. A code execution tool without a timeout is a denial-of-service vulnerability against your own machine.
  • Informative return values: The string returned to the model becomes an Observation in the ReAct chain. It should contain enough information for the model to reason about the result, but not so much that it overwhelms the context window.
  • Error signaling: When a tool fails, the return value should describe the failure clearly. The model can often recover from a failed tool call by adjusting its approach — but only if it understands what went wrong.
Ask your agent
Ask your agent to implement two tool functions — a web search function using a free API and a sandboxed Python code execution function — along with the registration code that connects them to the agentic loop.
Think about it
  • For web search, what free or open APIs exist that require minimal setup? Think about DuckDuckGo, SearXNG, or similar services. What should the function return — raw HTML, snippets, or structured results?
  • For code execution, what does 'sandboxed' mean in practice? How can the agent restrict the execution environment without requiring Docker or complex infrastructure?
  • How should each function format its return value so the model can effectively reason about the results?
  • What timeout values are appropriate, and how should timeout errors be communicated back to the model?
What the agent gives back

The agent should produce two Python functions. The web search function should call a free search API (such as the duckduckgo_search Python package), accept a query parameter, and return a formatted string containing the top 3–5 result titles and snippets. The code execution function should use Python's subprocess module with a timeout (e.g., 10 seconds) to execute code in a child process, capturing stdout and stderr. It should return the combined output as a string, or a clear error message if execution fails or times out. Both functions should be added to a tool_registry dictionary. The key design point is that return values are concise, informative strings — not raw data dumps.

Warning
Never execute model-generated code without sandboxing. At minimum, use subprocess with a timeout and restricted permissions. For production systems, use Docker containers or nsjail. The model may generate code that performs file operations, network calls, or other actions you did not intend.
Tip
When testing your agent, start with a simple query that requires only one tool call (e.g., 'What is the current population of Tokyo?'). Verify the full cycle — model emits a search call, the function executes, the result is returned, the model produces a final answer — before attempting multi-step tasks.
Designing tool return values for model consumption

The observation returned by a tool becomes part of the model's context. This creates a tension: more information helps the model reason accurately, but excessive information wastes context tokens and may confuse the model. A good heuristic is to return structured summaries rather than raw data. For web search, return titles and two-sentence snippets rather than full page content. For code execution, return stdout (truncated to ~500 characters) and any error messages. If the model needs more detail, it can make a follow-up tool call — that is precisely the advantage of the agentic loop pattern.

Refine Tool-Call Accuracy with Q-LoRA Fine-Tuning

Out-of-the-box Qwen-Chat models handle tool use competently, but for domain-specific tools or complex multi-step workflows, accuracy can degrade. Q-LoRA (Quantized Low-Rank Adaptation) offers a path to improvement without the prohibitive cost of full fine-tuning.

Q-LoRA works by freezing the quantized base model weights and training small adapter matrices (typically rank 8–64) on top. The training data consists of example conversations where the model correctly selects and invokes tools. Because only the adapter weights are updated, the GPU memory requirement is dramatically lower than full fine-tuning — Qwen's documentation reports that Q-LoRA for the 14B model requires approximately 18.7 GB of VRAM.

The critical question is not how to run Q-LoRA (the agent can generate the training script) but what data to train on. Effective fine-tuning data for tool use consists of:

  1. Diverse queries that map to each tool, including edge cases and ambiguous inputs.
  2. Correct tool selections with properly formatted arguments.
  3. Multi-turn sequences where the model must chain multiple tool calls.
  4. Negative examples where the correct action is to answer directly without any tool call.

This step focuses on getting the agent to produce a training data generation strategy and the Q-LoRA configuration.

Ask your agent
Ask your agent to design a training data schema for tool-use fine-tuning and generate a Q-LoRA configuration that could be used with Qwen's training scripts.
Think about it
  • What fields should each training example contain? Think about the full ReAct trajectory: user query, model thought, tool call, observation, and final answer.
  • How many training examples are typically needed for Q-LoRA to show measurable improvement on a narrow task like tool selection? What does the research literature suggest?
  • What hyperparameters matter most for Q-LoRA — learning rate, rank, alpha, number of epochs? How should these be communicated to the agent?
  • How would you evaluate whether fine-tuning actually improved tool-call accuracy? What metrics or test sets would you need?
What the agent gives back

The agent should produce two outputs. First, a JSON schema for training examples that includes fields for the system prompt (with tool declarations), a multi-turn conversation array containing user messages, assistant responses with ReAct formatting, and tool observations. Second, a Q-LoRA configuration specifying key hyperparameters: LoRA rank (e.g., 16), alpha (e.g., 32), target modules (attention projections), learning rate (e.g., 1e-4), and number of epochs (e.g., 3). The agent should recommend 200–500 high-quality examples as a starting point and suggest an evaluation protocol: hold out 20% of examples, measure exact-match accuracy on function name selection, and argument extraction F1 score.

Tip
Quality of training data matters far more than quantity for Q-LoRA. Fifty carefully constructed examples with diverse tool-call patterns will outperform five hundred repetitive examples. Focus on coverage of edge cases: queries that could plausibly map to multiple tools, queries that require no tool at all, and queries that require chaining two or more tools in sequence.
API Key Note
Q-LoRA fine-tuning requires significant GPU time. On a single A100 (80 GB), expect approximately 2–4 hours for 500 examples over 3 epochs on the 14B model. On consumer GPUs (RTX 4090, 24 GB), training is feasible for the 7B model but may require gradient checkpointing for the 14B variant.
LoRA rank selection: the bias-variance trade-off

The rank parameter in LoRA controls the dimensionality of the adapter matrices. Higher rank means more trainable parameters and greater capacity to learn new behavior — but also greater risk of overfitting to the training data. For tool-use fine-tuning, the task is relatively structured (select a function, extract arguments), so low ranks (8–16) are typically sufficient. Ranks above 64 rarely provide additional benefit for this type of task and may cause the model to memorize specific query phrasings rather than learning generalizable tool-selection patterns. A practical approach is to train at ranks 8, 16, and 32, then evaluate on a held-out test set to identify the sweet spot.

Recap

This session constructed a fully open-source tool-using agent from four components: a quantized Qwen-Chat model served via vLLM, a structured system prompt declaring tools in Qwen's native format, an agentic loop implementing the ReAct pattern, and concrete tool implementations with proper sandboxing. We also examined how Q-LoRA fine-tuning can sharpen tool-call accuracy for domain-specific applications.

The central insight is architectural: the model is a stateless reasoning engine, and the agentic loop provides the memory, control flow, and environmental interface that transform it into an agent. Every component — the serving layer, the prompt format, the tool registry, the parsing logic — exists to support this transformation. Understanding this architecture enables you to extend the system with new tools, swap in different models, or adapt the pattern to entirely different domains.

  • An agentic loop is a state machine with three states (query, parse, execute) that gives a stateless LLM the appearance of persistent, goal-directed behavior.
  • Qwen's native tool-use format (ReAct-style with ✿FUNCTION✿ markers) is a trained capability, not a prompt hack — using the correct format is essential for reliable tool selection.
  • Int4 quantization is sufficient for tool-routing tasks on memory-constrained hardware, because function selection and argument extraction are robust to moderate precision loss.
  • Tool descriptions in the system prompt function as instructions to the model — their precision directly determines tool-selection accuracy.
  • Q-LoRA fine-tuning on 200–500 high-quality tool-use examples can substantially improve accuracy for domain-specific tool configurations, at modest computational cost.
