From Seed Information to Emergent Futures

Deploy MiroFish — an open-source swarm-intelligence engine — to simulate thousands of autonomous agents that forecast how real-world events unfold
40 min multi-agent swarm-intelligence GraphRAG simulation prediction

What's happening

Prediction has historically been the province of statistical models operating on structured data — time-series forecasts, regression models, Monte Carlo simulations. These approaches excel when the underlying system is well-characterized and the variables are quantifiable. They falter when the system involves human agency: social dynamics, narrative shifts, cascading public reactions.

MiroFish represents a fundamentally different approach. Rather than modeling aggregate behavior through equations, it spawns thousands of LLM-powered agents — each with a distinct personality, long-term memory, and behavioral logic — and lets them interact freely within a simulated environment seeded from real-world information. The prediction emerges not from a formula but from the collective behavior of the swarm: agents argue, cooperate, react, and adapt, producing outcomes that no single model could anticipate.

The engine accepts remarkably diverse seed materials: breaking news articles, policy drafts, financial signals, even fiction. It constructs a knowledge graph via GraphRAG, generates agent personas from extracted entities, and runs parallel simulations powered by the OASIS engine from CAMEL-AI. The result is both a structured prediction report and an interactive digital world you can query after the simulation completes.

This session walks through deploying MiroFish, configuring its LLM and memory layers, running a simulation from seed material, and interpreting the output — all by directing an AI agent to handle the technical work while we focus on understanding the architecture and making sound decisions.

1

Understand the Architecture Before Deploying

Before touching any configuration, we need a mental model of MiroFish's five-stage pipeline. Each stage transforms the input in a specific way, and understanding this pipeline is essential for diagnosing problems and interpreting results.

Stage 1 — Graph Construction. Seed materials (articles, reports, narratives) are ingested and processed through a GraphRAG pipeline. This extracts entities (people, organizations, locations, concepts) and their relationships, constructing a knowledge graph that serves as the shared world-model for all agents.

Stage 2 — Environment Setup. From the knowledge graph, the system performs entity-relationship extraction and generates agent personas — each with a distinct personality profile, goals, and behavioral tendencies. Simulation parameters are injected into the environment.

Stage 3 — Parallel Simulation. The OASIS engine (developed by CAMEL-AI) runs the actual simulation. Agents interact across dual platforms in parallel, with the system automatically parsing the prediction requirements and dynamically updating temporal memory via the Zep layer.

Stage 4 — Report Generation. A specialized ReportAgent — equipped with a rich tool set — conducts post-simulation analysis by interacting deeply with the simulation environment, producing a structured prediction report.

Stage 5 — Deep Interaction. The simulation world persists. You can converse with any individual agent within it, or query the ReportAgent for further analysis.

The critical insight is that MiroFish is not a monolithic model. It is an orchestration system that coordinates an LLM backend (for agent reasoning), a memory layer (Zep, for temporal and episodic recall), and a simulation engine (OASIS, for agent interaction dynamics). Our deployment task is primarily one of wiring these components together correctly.

Tip
Think of MiroFish as a theater director, not an actor. It does not make predictions itself — it stages a world, casts thousands of characters, gives them memories and motivations, and observes what happens when they interact. The prediction is the emergent plot.
What is GraphRAG, and why does MiroFish use it?

GraphRAG (Graph-based Retrieval-Augmented Generation) extends standard RAG by organizing retrieved information into a structured knowledge graph rather than a flat collection of text chunks. For MiroFish, this is essential because agent personas and their relationships must be relational, not merely textual. A standard RAG system might retrieve that 'Company X announced layoffs' and 'The CEO of Company X is Jane Doe' as separate chunks. GraphRAG connects these into a graph where Jane Doe is linked to Company X with a 'CEO of' relationship, and the layoff event is linked to both. This relational structure is what allows MiroFish to generate agents whose behaviors are contextually grounded — Jane Doe's simulated agent will behave differently from a line worker's agent because the graph encodes their different positions within the system.

What role does Zep play in agent memory?

Zep is a long-term memory service for AI agents. In MiroFish, each simulated agent accumulates experiences during the simulation — conversations, observations, decisions — that must persist across simulation rounds. Without Zep, each agent interaction would be stateless: agents would forget what happened three rounds ago. With Zep, agents develop temporal memory — they remember what they said, what others said to them, and how events unfolded. This is what enables emergent behavior: an agent who 'remembers' being contradicted in round 5 may change strategy in round 12. Zep Cloud offers a free tier sufficient for initial experimentation.

Configure the Environment and Deploy MiroFish

MiroFish requires three external services to be configured: an LLM API endpoint (any provider supporting the OpenAI SDK format), a Zep Cloud account for agent memory, and the runtime environment itself (Node.js 18+ for the frontend, Python 3.11–3.12 for the backend). Both source deployment and Docker deployment are supported.

Rather than manually assembling configuration files, we will direct an AI agent to generate the deployment artifacts and guide us through the setup. The key decision at this stage is choosing your LLM backend — this determines both cost and simulation quality.

Ask your agent
Have the agent produce a complete, annotated `.env` configuration file for MiroFish and a step-by-step deployment checklist for your chosen method (source or Docker).
Think about it
  • What three external credentials does MiroFish require, and where does one obtain each?
  • The LLM API must be OpenAI SDK-compatible. What does that constraint imply about which providers you can use — and what must the `LLM_BASE_URL` look like for each?
  • How should you communicate your deployment preference (source vs. Docker) and your operating system so the agent tailors the instructions?
  • What would you ask the agent to verify or sanity-check about the configuration before you attempt to start services?
What the agent gives back

The agent should produce a complete .env file with three clearly annotated sections: LLM configuration (API key, base URL, and model name — with examples for at least two providers such as OpenAI and Alibaba DashScope), Zep Cloud configuration (API key with a note about the free tier), and any optional parameters. Alongside this, the agent should deliver a sequential deployment checklist — either source-based (clone, npm run setup:all, npm run dev) or Docker-based (docker compose up -d) — with verification commands to confirm each service is running (e.g., checking that ports 3000 and 5001 respond).

API Key Note
MiroFish simulations consume significant LLM tokens — thousands of agents each making multiple API calls per round. The project recommends starting with simulations of fewer than 40 rounds to manage costs. Budget awareness is essential before running your first simulation.
Warning
Python version compatibility is strict: 3.11 or 3.12 only. Python 3.13+ is not supported. If you use Docker, this is handled automatically; for source deployment, verify your Python version before proceeding.
Source deployment vs. Docker: which should I choose?

Docker deployment is simpler for initial experimentation — it encapsulates the correct Python version, Node.js version, and all dependencies in a single docker compose up -d command. Source deployment is preferable if you intend to modify MiroFish's behavior, inspect its internals, or develop custom seed-processing pipelines. For this workshop, either approach works. The functional result is identical: a frontend on port 3000 and a backend API on port 5001.

Why does MiroFish use the OpenAI SDK format rather than a specific provider?

The OpenAI SDK has become a de facto standard interface for LLM APIs. By accepting any provider that implements this format (OpenAI, Azure OpenAI, Alibaba DashScope, local models via Ollama or vLLM), MiroFish avoids vendor lock-in. The LLM_BASE_URL parameter is what makes this possible — it redirects API calls to whichever endpoint you specify. This is a common and sound architectural pattern in multi-agent systems: decouple the orchestration logic from the model provider.

✓
At this point you should have MiroFish deployed and running — the frontend accessible at `localhost:3000` and the backend API responding at `localhost:5001`. Your `.env` file should contain valid credentials for your LLM provider and Zep Cloud. If either service fails to start, check the console output for missing dependencies or invalid API keys before proceeding.
3

Prepare Seed Material and Launch a Simulation

MiroFish's prediction quality depends critically on the seed material you provide. The system accepts diverse inputs — data analysis reports, news articles, policy documents, even fictional narratives — but the richness and structure of that input directly determines the fidelity of the knowledge graph and, consequently, the realism of the simulated agents.

A well-chosen seed document has several properties: it names specific entities (people, organizations, institutions), describes relationships between them, and presents a situation with unresolved tension or uncertainty. A vague summary ('the economy is uncertain') produces vague agents. A detailed report ('Company X's board is divided on the merger, with the CFO opposing and the CEO supporting, while regulators have signaled concern about market concentration') produces agents with grounded, conflicting motivations — which is precisely what generates useful emergent predictions.

The simulation also requires a natural-language description of what you want to predict. This is not a prompt in the traditional sense — it is a prediction requirement that MiroFish uses to focus the simulation and structure the final report.

Ask your agent
Have the agent craft a high-quality seed document and prediction requirement for a scenario of your choosing — something with multiple stakeholders, conflicting interests, and genuine uncertainty about the outcome.
Think about it
  • What makes a seed document useful for entity extraction? Think about the density of named entities, explicit relationships, and described tensions.
  • How specific should your prediction requirement be? Consider whether 'What will happen?' is more or less useful than 'How will stakeholder X respond to event Y within timeframe Z?'
  • Should you ask the agent to structure the seed material in a particular way — or let MiroFish's GraphRAG pipeline handle unstructured text?
  • What simulation parameters matter — specifically, how many rounds should you request for a first run, given the token cost constraints?
What the agent gives back

The agent should produce two artifacts. First, a seed document of 500–1000 words describing a multi-stakeholder scenario — for example, a regulatory decision affecting several companies, a geopolitical negotiation, or a corporate succession crisis — with clearly named entities, explicit relationships, and unresolved tensions. Second, a focused prediction requirement phrased as a specific question: e.g., 'Within 90 days, will the merger proceed, be blocked, or be restructured? Which stakeholder's position will shift, and what will trigger the shift?' The agent should also recommend starting with 20–30 simulation rounds to balance insight against token cost.

Tip
The MiroFish project demonstrates two striking seed-material examples: predicting the lost ending of the classical Chinese novel Dream of the Red Chamber from its first 80 chapters, and simulating public-opinion dynamics around a real university controversy. Both illustrate that the engine handles narrative and social-dynamics inputs equally well.
How does GraphRAG transform seed text into a simulation-ready knowledge graph?

The process has three phases. First, the LLM extracts named entities from the seed text and classifies them (person, organization, event, concept, location). Second, it identifies relationships between entities — 'opposes', 'funds', 'reports to', 'competes with' — and assigns relationship strengths. Third, it constructs a graph where entities are nodes and relationships are edges, with the original text chunks attached as context. This graph becomes the foundation for agent persona generation: each entity that maps to a person or organization role becomes a candidate agent, with the graph encoding that agent's knowledge, relationships, and position within the scenario. Community detection algorithms may further cluster agents into factions or interest groups, which shapes how they interact during simulation.

4

Run the Simulation and Monitor Agent Behavior

With seed material uploaded and the prediction requirement defined, MiroFish initiates its simulation pipeline. This is where the architecture we discussed in Step 1 comes to life: the GraphRAG pipeline constructs the knowledge graph, agent personas are generated and injected with memory via Zep, and the OASIS engine begins running simulation rounds.

During simulation, agents operate on dual platforms in parallel. Each round, agents make decisions, communicate with other agents, observe environmental changes, and update their temporal memory. The system dynamically parses the prediction requirement to focus agent interactions on the relevant domain.

The crucial concept here is emergence — the simulation's value comes not from any single agent's behavior but from the collective patterns that arise from thousands of independent interactions. An agent representing a CFO might initially support a merger, but after simulated interactions with regulatory agents who express concern, that CFO agent may shift position — not because it was programmed to, but because its accumulated memory and behavioral logic lead it to recalculate.

Monitoring the simulation is primarily observational. MiroFish's frontend provides a 'God's-eye view' where you can watch agent interactions unfold and, critically, inject variables dynamically — introducing new information mid-simulation to test how the system responds to shocks.

Ask your agent
Have the agent explain how to interpret the MiroFish simulation dashboard and formulate a dynamic variable injection that would meaningfully stress-test your scenario.
Think about it
  • What kinds of mid-simulation variables would genuinely change the outcome — and what kinds would be noise? Think about what constitutes a 'shock' versus a 'nudge' in your scenario.
  • How would you ask the agent to help you identify which agents are most influential in the simulation — the ones whose behavior shifts cascade to others?
  • What should you watch for as evidence that emergent behavior is occurring, as opposed to agents simply repeating their initial dispositions?
What the agent gives back

The agent should describe the simulation dashboard's key indicators: agent interaction logs, faction-level sentiment trends, and the temporal progression of the prediction-relevant metrics. It should propose 2–3 specific variable injections for your scenario — for instance, 'At round 15, inject a news event that a key regulator has resigned' — and explain the expected downstream effects. The agent should also articulate markers of genuine emergence: agents changing positions that were not predetermined, unexpected coalitions forming, or feedback loops where one agent's decision triggers a chain of reactions.

Tip
Dynamic variable injection is what distinguishes MiroFish from static simulation frameworks. It enables counterfactual reasoning: 'What happens if this unexpected event occurs at this point in the timeline?' This is the engine's core value proposition for decision-makers — a zero-risk environment for testing policy responses to shocks.
How does Zep's temporal memory create realistic agent behavior over time?

Without persistent memory, each simulation round would be independent — agents would have no context from prior rounds and could not exhibit learning, grudge-holding, trust-building, or strategic adaptation. Zep provides each agent with episodic memory (specific past interactions), semantic memory (general knowledge accumulated over time), and temporal ordering (the ability to recall that event A happened before event B). This is what makes long-running simulations valuable: by round 30, agents have developed complex 'histories' with each other, and their behavior reflects those histories. A negotiation agent who was deceived in round 10 will behave differently in round 25 than one who has had only cooperative interactions.

✓
Your simulation should now be complete (or in progress if you chose a higher round count). You should see agent interaction logs in the MiroFish frontend and, upon completion, a generated prediction report. If the simulation stalled or produced errors, check the backend logs for LLM API rate-limiting or Zep connection issues — these are the two most common failure modes.
5

Interpret the Prediction Report and Query the Digital World

The simulation's terminal output is twofold: a structured prediction report generated by the ReportAgent, and a persistent interactive digital world that you can query post-simulation.

The ReportAgent is not a simple summarizer. It is equipped with a rich tool set that allows it to interrogate the simulation environment — querying specific agents about their reasoning, analyzing interaction patterns, identifying turning points, and synthesizing these into a coherent narrative. The resulting report is therefore grounded in the simulation's internal dynamics, not merely in surface-level statistics.

The interactive digital world persists after simulation completion. You can enter conversations with any individual agent — asking the simulated CFO why they changed their merger position, or asking a simulated regulator what information would have changed their decision. This is not merely a novelty; it is a form of interpretable prediction that traditional models cannot provide. When a statistical model predicts a 65% probability of a merger failing, you cannot ask it why. When MiroFish's simulation reaches a similar conclusion, you can trace the causal chain through individual agent decisions.

The analytical task at this stage is critical evaluation: assessing where the simulation's predictions are well-grounded versus where they reflect artifacts of the model's training data or the seed material's biases.

Ask your agent
Have the agent develop a structured evaluation framework for assessing the quality and reliability of a MiroFish prediction report — distinguishing well-supported predictions from speculative extrapolations.
Think about it
  • What would make you trust a simulation prediction? Consider whether the prediction follows logically from the agent interactions you observed, or whether it introduces conclusions not grounded in the simulation data.
  • How should you evaluate whether the agent personas were sufficiently diverse and well-characterized — or whether they collapsed into generic behavior?
  • What are the inherent limitations of LLM-based agent simulation? Think about training-data biases, the tendency toward consensus, and the difference between simulated reasoning and actual human decision-making.
  • How would you use the interactive world to spot-check the report's claims?
What the agent gives back

The agent should produce a structured evaluation rubric with 4–5 criteria: (1) Causal traceability — can each major prediction be traced to specific agent interactions in the logs? (2) Persona fidelity — did agents behave consistently with their assigned personas throughout the simulation, or did they drift toward generic LLM behavior? (3) Emergence evidence — did the simulation produce outcomes not directly implied by the seed material? (4) Bias indicators — are there signs that LLM training-data biases are driving predictions rather than simulation dynamics? (5) Counterfactual sensitivity — did dynamic variable injections produce meaningfully different outcomes, suggesting the simulation is responsive to new information rather than locked into a predetermined trajectory? The agent should also suggest 3–4 specific questions to pose to individual agents in the interactive world as spot-checks.

Warning
LLM-based simulations inherit the biases and limitations of their underlying models. Agents may exhibit a consensus bias (converging on socially expected outcomes), recency bias (over-weighting patterns common in training data), or cultural bias (defaulting to perspectives dominant in the training corpus). Treat MiroFish predictions as structured hypotheses to investigate, not as forecasts to act on uncritically.
How does the ReportAgent differ from simply asking an LLM to summarize the logs?

The ReportAgent has tool-use capabilities — it can programmatically query the simulation database, retrieve specific agent interaction sequences, compute aggregate statistics, and cross-reference events across the simulation timeline. A simple summarization prompt would be limited to whatever text fits in the context window and would lack the ability to perform targeted queries. The ReportAgent operates more like an analyst with database access than a reader with a document. This distinction is architecturally significant: it means the report can surface patterns that are not obvious from reading any single interaction log, because the ReportAgent can aggregate across thousands of interactions.

Recap

This session traced the full arc of deploying and operating MiroFish — from architectural comprehension through deployment, simulation design, execution, and critical evaluation of results.

The core principle throughout has been that multi-agent prediction is fundamentally different from statistical forecasting. Rather than fitting curves to historical data, MiroFish constructs a social simulation — a world populated by agents with distinct identities, memories, and behavioral logics, whose collective interactions produce emergent predictions. The three pillars supporting this architecture are GraphRAG (relational knowledge from seed materials), Zep (temporal agent memory enabling realistic long-term behavior), and OASIS (the simulation engine coordinating parallel agent interactions).

The practical skill developed here is not MiroFish-specific. It is the ability to reason about multi-agent systems: understanding what makes good seed material for agent generation, how to formulate prediction requirements that focus simulation without constraining emergence, how to use dynamic variable injection for counterfactual analysis, and — critically — how to evaluate simulation outputs with appropriate skepticism about the limitations of LLM-based agent reasoning.

  • Multi-agent prediction derives its value from emergence — collective patterns arising from individual agent interactions that no single model could produce in isolation
  • Seed-material quality determines simulation fidelity: entity density, relationship specificity, and described tensions are the three properties that matter most for GraphRAG extraction
  • Temporal memory via Zep is what transforms a stateless round-by-round simulation into a dynamic system where agents develop histories, adapt strategies, and exhibit realistic behavioral evolution
  • Dynamic variable injection enables counterfactual reasoning — the ability to ask 'what if?' mid-simulation — which is the engine's primary value for decision support
  • LLM-based agent simulations inherit model biases; predictions should be treated as structured hypotheses requiring validation, not as authoritative forecasts
