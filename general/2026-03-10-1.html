Build a Self-Improving AI Agent with Hermes Agent's Learning Loop

Deploy a persistent agent that creates skills from experience and remembers across sessions
40 min agents self-improving memory skills open-source

What's happening

Nous Research — the open-source AI lab behind the Hermes family of fine-tuned models — has released Hermes Agent, an open-source CLI agent with a distinctive architectural claim: a closed learning loop. Most agent frameworks treat each session as stateless. The agent executes, forgets, and starts fresh. Hermes Agent inverts this by introducing three persistent feedback mechanisms:

  1. Procedural memory through autonomous skill creation. After completing a complex task, the agent can extract the procedure into a reusable skill — a structured, versioned artifact it can invoke in future sessions without re-deriving the solution.
  2. Cross-session recall via FTS5 search. Conversation history is indexed using SQLite's FTS5 full-text search extension, enabling the agent to search its own past interactions with LLM-powered summarization.
  3. Dialectic user modeling. Through a system called Honcho, the agent maintains and refines an evolving model of the user's preferences, context, and working patterns.

The result is an agent that, in principle, improves with use — accumulating procedural knowledge rather than discarding it. This workshop explores that architecture: how the learning loop works, how to configure it, and how to deploy the agent across platforms.

Hermes Agent is also notable for its model-agnostic design. It supports Nous Portal, OpenRouter (200+ models), OpenAI, and arbitrary endpoints, switchable at runtime with a single command. This decouples the agent's capabilities from any single provider — a meaningful architectural choice in a landscape where provider lock-in remains a persistent concern.

1

Install Hermes Agent and Configure a Model Provider

The foundation of any agent deployment is installation and provider configuration. Hermes Agent ships as a single installer script that handles its own dependency chain — Python, Node.js, and all required packages. It produces a hermes command available in your shell.

The more consequential step is provider configuration. Hermes Agent's model-agnostic architecture means you must select a provider and model before the agent can operate. This is handled through hermes setup, an interactive wizard, or hermes model for subsequent changes. Supported providers include Nous Portal, OpenRouter (which aggregates 200+ models), OpenAI, z.ai/GLM, Kimi/Moonshot, MiniMax, and arbitrary OpenAI-compatible endpoints.

The choice of provider affects latency, cost, capability, and context window — all of which influence the agent's effectiveness at skill creation and memory tasks. For this workshop, any provider with a reasonably capable model (70B+ parameters or equivalent) will suffice.

Installation

The installer is a single curl command:

```bash

curl -fsSL https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.sh | bash

```

After installation, reload your shell and run hermes setup to configure your provider.

Think of provider configuration as selecting the engine for a vehicle. The chassis, steering, and instruments (Hermes Agent's tools, memory, and skill system) remain identical regardless of which engine you install. You can swap engines at any time with hermes model — no rewiring required.

Why model-agnostic design matters

Provider lock-in is a recurring problem in the AI tooling ecosystem. An agent framework coupled to a single provider inherits that provider's pricing changes, outages, and capability limitations. Hermes Agent's provider abstraction means you can start with OpenRouter for cost efficiency, switch to a self-hosted model for privacy-sensitive work, and move to a frontier model for complex reasoning — all without modifying your skills, memory, or configuration.

The hermes model command performs a hot swap: it updates the active provider and model without restarting the agent or losing session state. This is architecturally significant because the skill system and memory layer are provider-independent — they store structured data, not provider-specific artifacts.

2

Understand the Skill Creation Lifecycle

The core differentiator of Hermes Agent is its procedural memory system — the ability to extract reusable skills from completed tasks. This is not prompt caching or conversation replay. It is the creation of structured, versioned skill artifacts that the agent can invoke in future sessions.

The lifecycle has four stages:

  1. Task execution. The agent completes a complex, multi-step task — deploying a service, analyzing a dataset, configuring infrastructure.
  2. Skill extraction. The agent recognizes that the procedure it just executed is generalizable. It creates a skill definition: a structured artifact containing the procedure's steps, required inputs, expected outputs, and invocation pattern.
  3. Skill invocation. In a future session, when the agent encounters a similar task, it retrieves and invokes the stored skill rather than re-deriving the solution from scratch.
  4. Self-improvement. During invocation, the agent may encounter edge cases or improvements. It updates the skill definition — refining parameters, adding error handling, or generalizing the procedure.

This is analogous to how a human develops expertise: the first time you configure a web server, you consult documentation step by step. By the tenth time, you have internalized the procedure and can execute it fluently, adapting to variations without re-reading the manual.

Skills are compatible with the agentskills.io open standard, meaning they can be shared across agents and teams — a form of collective procedural memory.

What a Skill Contains

A skill is not a code snippet. It is a structured definition that includes:

  • A natural language description of the procedure
  • Input parameters and their types
  • The sequence of tool invocations required
  • Validation criteria for the output
  • Version history tracking refinements over time

Tip

Warning
Skill creation requires a sufficiently capable model. Models below approximately 30B parameters may struggle to produce well-structured skill definitions. If skill extraction produces incomplete or malformed results, consider switching to a larger model via hermes model.
How skills differ from prompt templates

A common confusion is equating skills with saved prompts or templates. The distinction is structural. A prompt template is a static string with placeholder variables — it encodes what to ask but not how to execute. A Hermes skill encodes the full execution graph: which tools to invoke, in what order, with what parameters, and how to validate the result.

This means a skill can adapt to context in ways a template cannot. A template for 'deploy to AWS' always produces the same prompt text. A skill for 'deploy to AWS' can inspect the current environment, select the appropriate deployment tool, handle authentication, and verify the deployment — adjusting its execution path based on runtime conditions.

Furthermore, skills carry version history. Each invocation that triggers a refinement creates a new version, allowing the agent (or the user) to inspect how the procedure evolved over time. This is closer to version-controlled runbooks than to prompt engineering.

The agentskills.io standard

Hermes Agent's skills are compatible with the agentskills.io open standard — a specification for portable, agent-executable procedural knowledge. This means skills created by one Hermes Agent instance can, in principle, be exported and imported by another instance or by a different agent framework that supports the standard.

This has implications for team workflows: a senior engineer's agent can create a skill for a complex deployment procedure, export it, and distribute it to the team. Each team member's agent can then invoke that skill, and their individual refinements can be merged back — creating a form of collaborative procedural memory.

Configure Cross-Session Memory and FTS5 Search

A skill system without memory is like a library without a catalog — the books exist, but finding the right one requires luck. Hermes Agent's cross-session memory addresses this through two complementary mechanisms:

  1. FTS5 full-text search. All conversation history is indexed using SQLite's FTS5 extension, a production-grade full-text search engine embedded directly in the agent's local database. This allows the agent to search its own past conversations with sub-millisecond latency — no external search service required.
  1. LLM-powered summarization. Raw search results are passed through the active model for summarization, producing contextually relevant summaries rather than raw text fragments. This transforms search from keyword matching into semantic retrieval.
  1. Periodic memory nudges. The agent periodically prompts itself to persist important information — decisions made, preferences expressed, context established. This is not passive logging; it is active curation, driven by the agent's assessment of what knowledge is worth retaining.

The practical effect is that the agent can answer questions like 'What deployment configuration did we use for the staging environment last month?' by searching its own history, summarizing the relevant exchanges, and presenting a coherent answer — even though the original conversation may have spanned multiple sessions.

The Honcho User Model

Beyond conversation search, Hermes Agent maintains a dialectic user model through a system called Honcho. This is an evolving representation of the user's preferences, technical context, and working patterns. The model is 'dialectic' in that it refines itself through interaction — each session potentially updating the agent's understanding of who you are and how you work.

This means the agent's behavior naturally adapts over time: it learns your preferred coding style, your infrastructure conventions, your communication preferences. This is distinct from system prompts or persona configuration — it emerges from accumulated interaction rather than explicit specification.

Tip
FTS5 search is local and fast — it runs against a SQLite database on the same machine as the agent, with no network latency. This makes cross-session recall practical even on a $5 VPS, which is part of Hermes Agent's design philosophy of running anywhere.
Warning
Memory nudges are model-dependent. The agent's ability to identify what is worth persisting depends on the active model's judgment. If you notice the agent failing to retain important context, check whether your model is sufficiently capable for the meta-cognitive task of evaluating information salience.
FTS5 versus vector search

Many agent frameworks use vector embeddings for memory retrieval — converting text into high-dimensional vectors and finding nearest neighbors. Hermes Agent's choice of FTS5 is a deliberate architectural decision with specific trade-offs.

FTS5 advantages: Zero external dependencies (it is built into SQLite), deterministic ranking, no embedding model required, sub-millisecond query latency, and full transparency — you can inspect exactly what was indexed and why a result was returned.

FTS5 limitations: It operates on lexical matching (with stemming), not semantic similarity. The query 'deployment configuration' will not match a conversation about 'infrastructure setup' unless the words overlap. Hermes Agent compensates for this by layering LLM summarization on top of FTS5 results — using the language model's semantic understanding to bridge the lexical gap.

This hybrid approach (lexical search + LLM summarization) avoids the operational complexity of embedding pipelines while preserving semantic retrieval capability. The trade-off is that semantic matching depends on the active model's quality rather than a dedicated embedding model's quality.

Deploy Across Platforms and Terminal Backends

Hermes Agent's deployment architecture separates the agent process from the interface and the execution environment, creating a flexible topology.

Interfaces: Where You Talk to the Agent

The hermes gateway command starts a messaging gateway that connects the agent to external platforms: Telegram, Discord, Slack, WhatsApp, Signal, and Home Assistant. This means you can start a conversation on the CLI, continue it from Telegram on your phone, and the agent maintains conversational continuity across platforms. Voice memo transcription is supported for audio messages.

This is not multiple agents — it is a single agent process with multiple input/output channels, sharing the same memory, skills, and user model.

Terminal Backends: Where the Agent Executes

When the agent needs to run commands, it does so through a terminal backend. Six backends are available:

  • Local — commands run on the same machine as the agent
  • Docker — commands run in an isolated container
  • SSH — commands run on a remote machine
  • Daytona — serverless container that hibernates when idle
  • Singularity — HPC container runtime for GPU clusters
  • Modal — serverless cloud functions with persistent state

Daytona and Modal are particularly notable for cost efficiency: the execution environment hibernates when the agent is idle and wakes on demand, costing nearly nothing between sessions. This enables persistent agent deployment on infrastructure that scales to zero.

Scheduled Automations

Hermes Agent includes a built-in cron scheduler for unattended task execution. You define automations in natural language — 'every weekday at 9am, review yesterday's commits and post a summary to Slack' — and the agent translates this into a scheduled job that runs without human intervention. Results are delivered to any connected platform.

Tip

Warning
The messaging gateway should be secured with DM pairing — a verification step that ensures only authorized users can send commands to the agent. Without this, anyone who discovers your bot's username could issue commands. Consult the Security section of the documentation for configuration details.
Subagent delegation for parallel workstreams

Hermes Agent can spawn isolated subagents for parallel execution. When a task decomposes into independent workstreams — running tests while building documentation while deploying to staging — the primary agent can delegate each workstream to a subagent that executes concurrently.

Subagents are isolated: they have their own context and tool access, and their results are collected by the primary agent upon completion. This is architecturally similar to thread pools in concurrent programming, but at the agent level — each 'thread' is a full agent instance with tool access.

An advanced pattern is writing Python scripts that call tools via RPC (Remote Procedure Call), collapsing multi-step pipelines into what Hermes Agent calls 'zero-context-cost turns.' Instead of the agent maintaining context for a 20-step pipeline, it writes a script that executes the entire pipeline programmatically, using only a single turn of context window.

Research-ready features

Hermes Agent includes features oriented toward AI research rather than end-user productivity:

  • Batch trajectory generation — recording sequences of agent actions for use as training data
  • Atropos RL environments — reinforcement learning environments for training tool-calling models
  • Trajectory compression — reducing recorded trajectories to their essential components for efficient training

These features position Hermes Agent not just as a productivity tool but as a platform for developing the next generation of tool-calling models. If you are training or fine-tuning models on agentic behavior, these features provide a structured data pipeline from agent execution to training examples.

Observe the Learning Loop in Practice

The learning loop is not a single feature — it is an emergent behavior arising from the interaction of skills, memory, and user modeling over time. Understanding it requires observing the system across multiple sessions, not within a single interaction.

The First Session

In the first session, Hermes Agent operates much like any capable AI agent. It has no skills, no memory, and no user model. It relies entirely on the base model's capabilities and its built-in toolset (40+ tools for file operations, web access, code execution, and system administration).

The Accumulation Phase (Sessions 2–10)

Over subsequent sessions, three things begin to accumulate:

  • Skills form around repeated procedures. If you ask the agent to deploy a service three times with minor variations, it will likely extract a deployment skill after the second or third execution.
  • Memory builds through nudges and indexing. Decisions, preferences, and context accumulate in the FTS5 index and are retrievable in future sessions.
  • The user model refines through interaction. The agent's understanding of your preferences, technical level, and working patterns deepens.

The Compounding Phase (Sessions 10+)

As the skill library and memory corpus grow, the agent's behavior qualitatively changes. It begins to anticipate needs, suggest relevant skills, recall prior decisions without prompting, and adapt its communication style to match your preferences. Tasks that required detailed instruction in session one may require only a brief directive in session twenty.

This is the architectural thesis of Hermes Agent: that persistent procedural memory, combined with evolving user modeling, produces an agent whose value compounds over time rather than remaining static.

Think about it
  • What metrics would meaningfully capture 'improvement'? Consider task completion time, number of clarifying questions needed, accuracy of recalled context, and skill reuse frequency.
  • How would you control for confounding variables — for example, the user getting better at prompting versus the agent genuinely improving?
  • At what intervals should measurements be taken? Daily? Weekly? After specific milestones?

What the agent gives back

The agent should propose a structured observation protocol with four measurement dimensions: (1) Task completion efficiency — tracking the number of turns and elapsed time for comparable tasks across sessions. (2) Memory retrieval accuracy — testing recall of previously established information with queries of varying specificity. (3) Skill utilization rate — measuring how often the agent invokes stored skills versus deriving procedures from scratch. (4) User model fidelity — evaluating whether the agent correctly anticipates preferences without being reminded. The protocol should include a baseline measurement session, weekly checkpoints, and a final assessment at day 30. The agent should note that the primary confound is user adaptation — the user learning to prompt more effectively — and suggest controlling for this by periodically testing with deliberately ambiguous prompts.

Tip
The learning loop's effectiveness depends on consistent use. Sporadic sessions with long gaps provide less signal for skill extraction and user modeling. For meaningful accumulation, aim for at least three sessions per week during the first month.
Limitations of the learning loop

The learning loop is not without limitations, and intellectual honesty requires acknowledging them:

  1. Skill quality depends on model quality. The extraction, structuring, and refinement of skills are all performed by the active language model. A less capable model will produce less useful skills — or may fail to recognize skill-worthy patterns entirely.
  1. Memory is not understanding. FTS5 search with LLM summarization can retrieve and present past information, but it does not constitute genuine understanding or reasoning about that information. The agent can recall that you chose PostgreSQL over MySQL for a project, but its 'understanding' of why is limited to whatever was captured in the original conversation.
  1. User model drift. The dialectic user model can accumulate stale or incorrect information. If your preferences change significantly, the model may lag behind, producing recommendations based on outdated patterns. There is currently no documented mechanism for resetting or auditing the user model.
  1. Skill rot. Skills that reference external tools, APIs, or configurations can become outdated as those dependencies change. A deployment skill created in March may not work in June if the underlying infrastructure has been modified. The self-improvement mechanism can catch some of these issues during invocation, but it is not guaranteed to detect all forms of drift.

Recap

This workshop has explored Hermes Agent's closed learning loop — the architectural pattern that distinguishes it from stateless agent frameworks. We examined five layers of the system:

  1. Installation and provider configuration — establishing the model-agnostic foundation that decouples agent capabilities from provider dependencies.
  2. The skill creation lifecycle — the four-stage process (execution → extraction → invocation → self-improvement) that transforms one-time procedures into persistent, versioned, reusable knowledge.
  3. Cross-session memory — the combination of FTS5 full-text search, LLM summarization, and periodic memory nudges that gives the agent access to its own history.
  4. Multi-platform deployment — the separation of interface, execution environment, and scheduling that enables flexible topologies from a $5 VPS to a GPU cluster.
  5. The learning loop in practice — how skills, memory, and user modeling compound over time to produce an agent whose value increases with use.

The central insight is architectural: by persisting procedural knowledge (skills), declarative knowledge (memory), and relational knowledge (user model), Hermes Agent creates a feedback loop where each session contributes to the agent's future effectiveness. This is a meaningful departure from the prevailing pattern of stateless, session-scoped agent interactions.

  • Procedural memory through skill extraction is fundamentally different from prompt caching or conversation replay — skills encode execution graphs, not text templates, and they self-improve through use.
  • FTS5 full-text search with LLM summarization provides a pragmatic hybrid between lexical and semantic retrieval, avoiding the operational complexity of vector embedding pipelines while preserving semantic capability.
  • Model-agnostic provider switching via hermes model is not merely a convenience feature — it is an architectural decision that prevents vendor lock-in and allows capability-appropriate model selection per task.
  • The learning loop's effectiveness compounds with consistent use but is bounded by the active model's capability for meta-cognitive tasks like skill extraction and information salience assessment.
  • Deployment topology (interface × backend × scheduler) is independently configurable, enabling architectures from single-user CLI to multi-platform team deployments with serverless cost efficiency.
