Automate Google NotebookLM with Python

Build research pipelines and content generation workflows using the unofficial NotebookLM API
35 min tools notebooklm automation python research

What's happening

Google NotebookLM has emerged as one of the most distinctive AI research tools available — a system that ingests sources, synthesizes information, and generates remarkably natural podcast-style audio overviews. Researchers, journalists, and students use it daily. Yet NotebookLM has no public API. Every interaction requires manual work through the web interface: uploading sources one at a time, clicking through generation options, waiting for results.

The notebooklm-py library, created by teng-lin, changes this equation. It reverse-engineers NotebookLM's undocumented internal APIs and wraps them in a clean async Python interface. The result is full programmatic access — including capabilities the web UI does not expose, such as batch downloads, structured quiz export, and mind map data extraction.

This matters for two reasons. First, it transforms NotebookLM from an interactive tool into an automatable pipeline component: ingest fifty URLs, generate audio overviews for each, download the results — all from a script. Second, it can be wired into AI agent workflows, allowing a language model to invoke NotebookLM operations through natural language. In this session, we will set up the library, build an automated research pipeline, and explore agent-driven integration patterns.

1

Set Up Authentication and Environment

Before we can make any API calls, we need to understand how notebooklm-py authenticates with Google's servers. Because this library uses undocumented internal APIs — the same endpoints the NotebookLM web application calls — it cannot use standard OAuth or API key authentication. Instead, it relies on browser session cookies extracted from an authenticated Google session.

This is a common pattern for unofficial API wrappers. The library impersonates your browser session by replaying the authentication cookies that your browser obtained when you logged into NotebookLM. The critical implication: these cookies expire, and Google may invalidate them if it detects unusual usage patterns.

The setup process has two parts: installing the library and providing your authentication credentials. We will ask our agent to guide us through both.

The agent should also note that cookies expire periodically and will need refreshing.

Warning
This library uses undocumented Google APIs. Your authentication cookies are sensitive credentials — they grant full access to your Google account's NotebookLM data. Never commit them to version control, share them, or store them in plaintext outside a .env file. Treat them with the same care as a password.
Tip
If you plan to run pipelines unattended (e.g., on a schedule), be aware that cookie-based authentication will require periodic manual renewal. There is no refresh-token mechanism. Plan for this in any production-adjacent workflow.
Why cookies instead of OAuth?

Google does not offer a public NotebookLM API, so standard OAuth flows are unavailable. The notebooklm-py library operates by calling the same internal HTTP endpoints that NotebookLM's web frontend uses. Those endpoints authenticate requests via session cookies set by Google's login flow. This is the only viable approach for an unofficial client, but it carries inherent fragility: Google can change endpoint signatures, cookie formats, or rate-limiting behavior at any time without notice. This is why the library is best suited for prototyping, research, and personal automation rather than customer-facing production systems.

Build a Source Import Pipeline

The core value proposition of programmatic NotebookLM access is bulk operations. In the web interface, adding sources is a manual, one-at-a-time affair. With the API, we can import dozens or hundreds of sources in a single script execution.

NotebookLM accepts a wide range of source types: URLs (web pages, articles), YouTube videos, uploaded files (PDF, text, Markdown, Word documents, audio, video, images), Google Drive documents, and raw pasted text. Each source type follows a slightly different import path, but the library abstracts these behind a unified interface.

The workflow we want to build: given a list of URLs and files, create a new notebook, import all sources into it, and confirm that each import succeeded. This is the foundation for any research automation pipeline — once sources are loaded, everything else (chat, audio generation, quizzes) builds on top of them.

Ask your agent
Get the agent to build an async Python script that creates a new notebook and bulk-imports a list of sources (URLs and local files) with error handling.
Think about it
  • The library uses async Python patterns. What does this imply about how you structure the script? What keyword do you use to call async functions?
  • When importing many sources, some may fail (broken URLs, unsupported formats). How should the script handle partial failures — stop entirely, or continue and report errors at the end?
  • Sources come in different types (URLs, PDFs, YouTube links). Should the script auto-detect the type, or require the user to specify it? What would be more robust?
  • After importing, how would you verify that all sources were successfully added? What API call would confirm the notebook's contents?
What the agent gives back

The agent should produce an async Python script that: (1) initializes the NotebookLM client with stored credentials, (2) creates a new notebook with a descriptive name, (3) iterates over a list of source items — detecting whether each is a URL, YouTube link, or local file path — and imports each one, (4) wraps each import in error handling so that a single failure does not abort the entire batch, (5) collects results and prints a summary showing which imports succeeded and which failed. The key structural insight is the use of async/await throughout, since the library's API is fully asynchronous. A brief code excerpt might show the import loop pattern:

```python

for source in sources:

try:

await notebook.add_source(source)

results.append((source, 'ok'))

except Exception as e:

results.append((source, str(e)))

```

Tip
When building bulk import scripts, consider adding a short delay between imports (e.g., 1–2 seconds). Undocumented APIs often have undocumented rate limits, and rapid-fire requests increase the risk of throttling or temporary blocks.
Why async? Understanding the concurrency model

notebooklm-py uses Python's asyncio library throughout its API. This is a deliberate design choice: NotebookLM operations are I/O-bound (they involve HTTP requests to Google's servers), and async patterns allow the program to initiate multiple requests without blocking on each one. In practical terms, this means you could import several sources concurrently using asyncio.gather() rather than waiting for each import to complete sequentially. However, be cautious with concurrency against undocumented APIs — parallel requests increase the risk of triggering rate limits. A sequential loop with error handling, as described above, is the safer starting pattern.

Source types and their nuances

Not all sources are equal in NotebookLM. URLs are fetched and parsed server-side by Google — the quality of extraction depends on the page structure. YouTube videos are processed via their transcripts, so videos without captions yield poor results. PDFs are parsed server-side, but heavily formatted or image-heavy PDFs may lose content. Google Drive files require that your Google account has access to the document. Understanding these constraints helps you curate source lists that produce high-quality notebooks.

Generate Audio Overviews and Download Artifacts

NotebookLM's most distinctive feature is its Audio Overview — a podcast-style conversation between two AI hosts who discuss and synthesize the notebook's sources. Through the web interface, you get a single default format. Through the API, you gain access to the full parameter space: four conversation formats (deep-dive, brief summary, critique, and debate), three length options, and support for over fifty languages.

Beyond audio, the API exposes the complete content generation suite: video overviews (with visual styles ranging from classic to anime), slide decks, quizzes, flashcards, infographics, reports, data tables, and mind maps. Each generation type produces downloadable artifacts.

The workflow pattern is consistent across all content types: (1) select a notebook with sources already loaded, (2) call the generation method with desired parameters, (3) poll or await completion, (4) download the resulting artifact. We will focus on audio overviews as the exemplar, but the pattern generalizes to every content type.

  • The API supports four audio formats: deep-dive, brief, critique, and debate. How should these be exposed to the user — as command-line arguments, function parameters, or configuration?
  • Where should the downloaded MP3 be saved? How should the filename be constructed to be meaningful (e.g., including the notebook name and format)?
  • What happens if generation fails partway through? How would you detect and handle this?
What the agent gives back

The agent should produce a script that: (1) connects to an existing notebook by name or ID, (2) initiates an Audio Overview generation with configurable parameters (format, length, language), (3) waits for generation to complete (the library likely handles polling internally via its async interface), and (4) downloads the resulting MP3 file to a local directory with a descriptive filename. The agent should explain that generation can take several minutes depending on the notebook's source volume and the selected format. The script should include a progress indication mechanism (even if just print statements) so the user knows it has not stalled. The key concept is that content generation is a two-phase operation: initiate and retrieve — and the async API bridges this gap.

Tip
The debate format is particularly useful for research: it generates a structured argument between two perspectives on the source material, which can surface tensions and contradictions that a summary might smooth over.
API Key Note
Audio and video generation consume Google's compute resources and may be subject to daily limits on your account. If generation requests start failing, you may have hit an undocumented quota. Wait several hours before retrying.
The full content generation matrix

While Audio Overviews are the flagship feature, the API exposes nine distinct content generation types, each with its own parameter space:

  • Audio Overview: 4 formats × 3 lengths × 50+ languages → MP3/MP4
  • Video Overview: 2 formats × 9 visual styles → MP4
  • Slide Deck: 2 formats, adjustable length, individual slide revision → PDF/PPTX
  • Infographic: 3 orientations × 3 detail levels → PNG
  • Quiz: Configurable quantity and difficulty → JSON/Markdown/HTML
  • Flashcards: Configurable quantity and difficulty → JSON/Markdown/HTML
  • Report: Briefing doc, study guide, blog post, or custom prompt → Markdown
  • Data Table: Custom structure via natural language → CSV
  • Mind Map: Hierarchical visualization → JSON

Several of these — particularly the structured exports (quiz JSON, mind map JSON, PPTX slide decks) — are capabilities that the web UI does not expose at all. The API provides strictly more functionality than the browser interface.

Wire NotebookLM into an AI Agent Workflow

The techniques we have built so far — creating notebooks, importing sources, generating content — are powerful individually but become transformative when composed into an agent tool. Rather than writing a specific script for each research task, we can expose NotebookLM operations as tools that a language model can invoke through natural language.

notebooklm-py ships with built-in support for this pattern: it includes Claude Code skills that can be installed directly, allowing Claude to create notebooks, import sources, generate audio, and download artifacts through conversational commands. But understanding how this integration works is more valuable than simply installing it — the same pattern applies to any agent framework.

The architecture is straightforward. An AI agent tool is a function with a clear interface (name, description, input parameters, output format) that the language model can decide to call. When the model determines that a user's request requires NotebookLM functionality, it invokes the tool with structured parameters, receives the result, and incorporates it into its response. The key design challenge is defining the tool interface: what parameters should be exposed, what should be defaulted, and how should results be formatted for the model to interpret?

Ask your agent
Get the agent to design a set of NotebookLM agent tools — define their interfaces, parameters, and return formats — suitable for integration into a Claude Code or similar agent framework.
Think about it
  • What are the core operations an agent would need? Think about the research workflow: create → populate → generate → retrieve. Does each step need its own tool, or can some be combined?
  • How should you describe each tool so the language model knows when to use it? What makes a good tool description versus a poor one?
  • What should the tools return to the agent? Raw data? A summary? A file path? Consider what the language model needs to formulate a useful response to the user.
  • Should the tools handle authentication internally, or should the agent manage credentials? What is the simpler, more secure pattern?
What the agent gives back

The agent should produce a tool design specification — not necessarily runnable code, but a clear architectural description of 3–5 tools with their names, descriptions, parameter schemas, and return types. A reasonable decomposition would be: (1) create_research_notebook — takes a topic name and list of source URLs, creates a notebook and imports all sources, returns the notebook ID and import summary; (2) generate_content — takes a notebook ID, content type (audio/video/quiz/report), and optional parameters (format, language, difficulty), triggers generation and returns the artifact file path; (3) query_notebook — takes a notebook ID and a question, uses NotebookLM's chat interface to ask a question against the sources, returns the answer with citations. The key insight is that tool descriptions should be written for the language model as the audience — they should clearly state what the tool does, when to use it, and what it returns.

Tip
The library includes a notebooklm skill install command that registers pre-built Claude Code skills. If you want to use NotebookLM from Claude Code immediately without designing custom tools, this is the fastest path. The custom tool design exercise above is for understanding the pattern and building integrations into other agent frameworks.
The tool description is your API contract with the model

When designing agent tools, the natural-language description is arguably more important than the code implementation. The language model uses the description to decide whether to invoke the tool for a given user request. A vague description ('does NotebookLM stuff') leads to the model invoking the tool at the wrong times or not at all. A precise description ('Creates a new NotebookLM notebook with the given name, imports the specified URLs as sources, and returns the notebook ID. Use this when the user wants to start a new research project with web sources.') gives the model clear decision criteria. Write tool descriptions as if you are writing documentation for a junior developer who has never used the tool before.

Combining tools into compound workflows

Individual tools become more powerful when the agent can chain them. For example, a user might say: 'Research recent developments in quantum error correction and create a podcast about it.' A well-designed agent would: (1) use a web search tool to find relevant URLs, (2) call create_research_notebook with those URLs, (3) call generate_content with type 'audio' and format 'deep-dive', (4) return the audio file path to the user. No single tool does all of this — the agent orchestrates the pipeline by composing tools. This is why clean, composable tool interfaces matter: each tool should do one thing well and produce output that other tools can consume.

✓
You should now have a clear mental model of the three-layer architecture: (1) the `notebooklm-py` library provides async Python access to NotebookLM's internal APIs, (2) scripts compose library calls into automated pipelines (import → generate → download), and (3) agent tools wrap pipeline operations behind natural-language-addressable interfaces. Each layer builds on the one below it.

Use the CLI for Quick Operations and Scripting

Not every task requires a custom Python script. notebooklm-py includes a command-line interface that exposes the library's full functionality through shell commands. The CLI is particularly well-suited for three scenarios: quick one-off tasks (list notebooks, check a source's status), shell script integration (chaining NotebookLM operations with other command-line tools), and CI/CD pipelines (automated content generation triggered by repository events).

The CLI follows the subcommand pattern common in modern CLI tools: a top-level notebooklm command followed by a resource noun and an action verb. Understanding the CLI vocabulary lets you perform rapid exploratory operations without writing any Python — useful for testing ideas before committing to a full pipeline implementation.

Tip
The CLI is the fastest way to experiment with notebooklm-py before building a full pipeline. Start by running notebooklm notebooks list to verify your authentication, then try adding a source and generating content interactively before automating anything.
CLI vs. Python API: when to use which

Use the CLI when you need quick interactive exploration, simple shell scripts, or integration with non-Python toolchains (e.g., a bash script that curls articles and feeds them to NotebookLM). Use the Python API when you need complex control flow (conditional logic, error handling with retries), when you are building a larger application that includes NotebookLM as one component, or when you need to process results programmatically before taking action. Use Agent Skills when you want a language model to orchestrate NotebookLM operations through natural language, abstracting away both the CLI syntax and the Python API details.

Recap

We have traced a complete arc from setup to automation to agent integration. The notebooklm-py library transforms NotebookLM from a manual web tool into a programmable research infrastructure component. The key architectural insight is the three-tier access model: the async Python API provides full programmatic control for complex pipelines, the CLI enables rapid experimentation and shell-script integration, and agent skills expose NotebookLM to language models for natural-language-driven research workflows.

The patterns explored here — bulk source import, parameterized content generation, artifact download, and agent tool design — generalize beyond NotebookLM. Any time you encounter a web application without a public API, the same approach applies: identify the internal endpoints, wrap them in a clean interface, and layer automation on top. The agent tool design principles — clear descriptions, composable interfaces, structured return types — apply to any tool-using AI system.

  • notebooklm-py provides async Python access to NotebookLM's full feature set via undocumented internal APIs — including capabilities the web UI does not expose (batch downloads, structured quiz export, PPTX slide decks, mind map JSON).
  • Cookie-based authentication against undocumented APIs is inherently fragile. Design your pipelines with error handling, rate-limit awareness, and the expectation that credentials will need periodic manual renewal.
  • Agent tool design is primarily a description-writing exercise. The natural-language tool description determines when and how a language model invokes the tool — invest more effort in the description than in the implementation.
  • Sequential processing with inter-request delays is the appropriate default for batch operations against unofficial APIs. Concurrency is an optimization to apply only after establishing reliable baseline behavior.
  • The three access tiers (Python API, CLI, Agent Skills) serve different use cases: complex pipelines, quick tasks, and conversational workflows respectively. Choose based on the human-in-the-loop requirements of your use case.
