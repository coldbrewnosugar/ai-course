Supercharge Claude Code with Custom Skills and Context Engineering

Turn Claude Code into a domain-expert pair programmer with 66 specialized skills, context-aware routing, and structured workflows
35 min claude-code developer-tools plugins context-engineering pair-programming

What's happening

Claude Code's plugin ecosystem has matured rapidly. The claude-skills package by Jeffallan — currently trending on GitHub — represents a significant development: a curated collection of 66 specialized skills spanning 12 categories that transform a general-purpose coding assistant into a set of domain experts.

The underlying principle is straightforward but powerful. Rather than relying on a single, undifferentiated prompt surface, the skill system routes your requests to specialized instruction sets. Ask about JWT authentication in NestJS, and the system activates a NestJS expert backed by dedicated reference materials on authentication patterns. Ask about React Server Components, and a different expert activates with its own reference corpus.

This session examines three capabilities that, taken together, constitute a meaningful shift in how developers interact with AI coding assistants:

  1. Context-aware activation — the system selects the appropriate specialist based on the content of your prompt, without explicit routing by the user.
  2. Multi-skill workflow chains — complex development tasks are decomposed across multiple specialists in sequence (e.g., Feature Forge → Architecture Designer → Test Master).
  3. Context engineering via /common-ground — a structured protocol for surfacing and validating the hidden assumptions Claude holds about your codebase.

The broader significance extends beyond this particular plugin. The patterns demonstrated here — specialist routing, workflow chaining, and assumption surfacing — represent design principles applicable to any AI-augmented development workflow.

Install the Skill Pack and Understand the Architecture

Before working with individual skills, we must install the plugin and develop a mental model of how it operates.

The claude-skills package is installed through Claude Code's plugin system in two stages: first, adding the marketplace source, then installing the skill pack itself. Once installed, the system provides 66 skills organized into 12 categories — covering languages (Python, TypeScript, Go, Rust), backend frameworks (NestJS, Django, FastAPI), frontend frameworks (React, Vue, Svelte), infrastructure (Docker, Kubernetes, Terraform), APIs (REST, GraphQL), testing, DevOps, security, data/ML, and platform-specific tooling.

Each skill is defined by a SKILL.md file — a structured Markdown document that provides Claude with domain-specific instructions, conventions, and decision heuristics. Accompanying each skill is a references/ directory containing deep-dive materials (e.g., authentication.md, server-components.md). When a skill activates, both the skill definition and relevant reference files are loaded into Claude's context.

This architecture follows a retrieval-augmented generation (RAG) pattern at the skill level. Rather than encoding all knowledge in a single monolithic prompt, the system dynamically assembles a specialized context window from modular components. The practical consequence: Claude operates with deeper, more current domain knowledge than its base training alone provides.

Tip
The plugin system is distinct from Claude Code's built-in slash commands. Plugins extend Claude's capabilities through structured Markdown instructions and reference materials, while built-in commands control the Claude Code interface itself.
Warning
The skill pack includes 365 reference files. These are not all loaded simultaneously — they are retrieved on-demand when a relevant skill activates. This demand-loading architecture is essential; loading all references at once would exceed context window limits.
How does SKILL.md differ from a system prompt?

A SKILL.md file functions similarly to a system prompt but with important structural differences. System prompts are static and applied uniformly. Skill definitions are modular and conditionally loaded. Each SKILL.md typically includes: a role definition (e.g., 'You are a NestJS expert'), domain-specific conventions and best practices, decision heuristics for common architectural choices, references to companion materials, and trigger conditions that determine when the skill should activate.

This modular approach allows the system to maintain deep expertise across dozens of domains without the context dilution that would occur from combining all instructions into a single prompt. When you ask about NestJS authentication, you receive a specialist — not a generalist who happens to know a little about NestJS.

Context-Aware Activation: Routing Prompts to Specialists

The most immediately useful feature of the skill system is context-aware activation — the automatic selection of an appropriate specialist skill based on the content of your prompt. Understanding how this routing works is essential for writing prompts that reliably engage the correct expert.

The activation mechanism operates on pattern matching against your prompt's content. When you write "Implement JWT authentication in my NestJS API," the system identifies two key signals: the technology (NestJS) and the domain (authentication). These signals route the prompt to the NestJS Expert skill, which in turn loads references/authentication.md — a dedicated reference document on authentication patterns within NestJS.

This routing is not merely cosmetic. Different specialists carry different architectural assumptions. The React Expert assumes a component-based architecture with hooks and composition patterns. The Django Expert assumes an MTV (Model-Template-View) architecture with an ORM layer. The Kubernetes Expert assumes declarative infrastructure management with manifest files. When the correct specialist activates, the response is grounded in framework-appropriate conventions rather than generic patterns.

The practical implication for prompt construction: specificity in your prompts determines routing accuracy. A vague request like "help me build an API" provides insufficient signal for the router. A specific request like "build a REST API with pagination in FastAPI using SQLAlchemy" provides three clear signals (FastAPI, REST, SQLAlchemy) that ensure precise skill activation.

Multi-Skill Workflows: Chaining Specialists for End-to-End Development

Individual skill activation is useful, but the full power of the system emerges when skills are chained into multi-skill workflows. The plugin defines several canonical workflow chains, each decomposing a complex development task into a sequence of specialist handoffs.

Consider the Feature Development workflow:

Feature Forge → Architecture Designer → Fullstack Guardian → Test Master → DevOps Engineer

This chain mirrors how a well-organized engineering team operates. Feature Forge defines the feature scope — user stories, acceptance criteria, and edge cases. Architecture Designer determines the system design — component boundaries, data flow, and integration points. Fullstack Guardian implements with cross-cutting concerns in mind — error handling, logging, and observability. Test Master produces the test suite — unit tests, integration tests, and edge case coverage. DevOps Engineer handles deployment — CI/CD pipeline updates, environment configuration, and monitoring.

The critical insight is that each specialist in the chain receives the output of its predecessor as input context. The Architecture Designer does not operate in isolation; it works with the feature specification produced by Feature Forge. Test Master writes tests against the implementation produced by Fullstack Guardian. This sequential context accumulation is what distinguishes a workflow chain from a set of independent prompts.

Two additional workflow chains merit attention:

  • Bug Investigation: Debugging Wizard → Framework Expert → Test Master → Code Reviewer. This chain progresses from symptom analysis through root cause identification to fix verification.
  • Security Hardening: Secure Code Guardian → Security Reviewer → Test Master. This chain audits existing code for vulnerabilities, recommends remediations, and verifies fixes through security-focused test cases.

  • What information does Feature Forge need to produce a useful specification — user roles, notification channels, priority levels, delivery guarantees?
  • When transitioning from Feature Forge to Architecture Designer, what output from the first stage serves as input to the second?
  • How should you frame the Architecture Designer prompt to ensure it builds upon the Feature Forge output rather than starting from scratch?
  • At the Test Master stage, what testing dimensions matter — unit tests for individual handlers, integration tests for the notification pipeline, edge cases for delivery failures?
What the agent gives back

The workflow proceeds in three stages. Feature Forge produces a structured specification: user stories (e.g., 'As a user, I receive real-time notifications for mentions'), acceptance criteria, notification channel definitions (in-app, email, push), priority levels, and edge cases (offline users, notification batching, rate limiting). Architecture Designer takes that specification and produces a system design: an event-driven architecture with a notification service, channel adapters (one per delivery method), a preference store, and a queue for reliable delivery. It identifies component boundaries and data flow. Test Master then produces a test strategy covering unit tests for each channel adapter, integration tests for the end-to-end notification pipeline, and edge case tests for failure scenarios (channel unavailable, user preferences changed mid-delivery). Each stage explicitly references the output of its predecessor.

Tip
You do not need to execute the entire workflow chain in a single session. It is often preferable to complete one stage, review and refine its output, and then proceed to the next stage with validated input. The workflow is a sequence, not a monolithic operation.
Warning
Context window limits apply across the chain. If the accumulated output from earlier stages is very large, summarize the key decisions and interface contracts before passing context to subsequent stages. Preserve architectural decisions and constraints; trim implementation details.
Designing your own workflow chains

The three canonical workflows (Feature Development, Bug Investigation, Security Hardening) are starting points, not prescriptions. You can compose custom chains from the available 66 skills to match your team's process.

A useful heuristic for chain design: identify the phases of the task, then map each phase to the specialist whose domain most closely matches the required expertise. For a database migration task, you might compose: Architecture Designer (schema design) → Python/TypeScript Expert (migration scripts) → Test Master (data integrity tests) → DevOps Engineer (deployment and rollback procedures).

The ordering principle is: upstream stages produce constraints and specifications; downstream stages implement within those constraints. Reversing this order — implementing before designing, or deploying before testing — produces the same quality problems in AI-assisted workflows as it does in traditional development.

Context Engineering with /common-ground

The /common-ground command addresses a subtle but pervasive problem in AI-assisted development: hidden assumptions. When Claude analyzes your codebase, it forms implicit beliefs about your architecture, conventions, and constraints. These assumptions influence every recommendation it makes — but they remain invisible unless explicitly surfaced.

Consider an analogy. When a new developer joins a team, they bring assumptions from their previous projects. They might assume the codebase uses REST when it actually uses GraphQL, or assume PostgreSQL when the project uses MongoDB. These mismatches produce subtly wrong recommendations until corrected. The same phenomenon occurs with AI assistants, but with an additional complication: the AI cannot volunteer its assumptions unprompted.

/common-ground solves this by initiating a structured dialogue in which Claude explicitly states its beliefs about your project — the technology stack, architectural patterns, coding conventions, testing strategies, deployment pipeline, and other foundational assumptions. You then validate, correct, or refine each assumption. The result is a shared understanding — a common ground — that improves the accuracy of all subsequent interactions.

This is a form of context engineering: the deliberate construction and maintenance of the contextual information that shapes an AI's behavior. Rather than hoping the AI infers your conventions correctly from code samples, you establish them explicitly through a validation protocol.

The practical value compounds over time. A corrected assumption about your authentication pattern, for example, influences every subsequent recommendation involving user sessions, API security, and data access control. One /common-ground session can prevent dozens of downstream misalignments.

Ask your agent

Think about it
  • Before running /common-ground, what aspects of your project are most likely to be incorrectly assumed — the ORM, the authentication strategy, the deployment target, the testing framework?
  • When Claude presents its assumptions, what criteria distinguish a 'close enough' assumption from one that requires correction?
  • How should you phrase corrections — is it sufficient to say 'wrong, we use X' or should you explain *why* X was chosen to give Claude better calibration for future recommendations?
  • After establishing common ground, how would you test whether subsequent prompts reflect the corrected assumptions?
What the agent gives back

Running /common-ground initiates a structured exchange. Claude presents its inferred assumptions about the project, organized by category: technology stack ('I believe this project uses Express with TypeScript and PostgreSQL'), architecture ('The application follows a layered architecture with controllers, services, and repositories'), conventions ('Error handling uses custom exception classes with HTTP status codes'), and testing ('Tests use Jest with a repository-mock pattern'). You review each assumption and provide corrections where needed — for instance, confirming the stack but correcting the architecture from layered to hexagonal, or noting that the project uses Vitest rather than Jest. After the exchange, Claude retains these corrections as established context for the remainder of the session.

Tip
Run /common-ground at the start of a new project session or when switching between significantly different codebases. The assumptions Claude forms about a React SPA will be entirely wrong when applied to a Go microservice.
API Key Note
The corrections established through /common-ground persist for the current session but may not carry over to new sessions. For persistent context, consider documenting the validated assumptions in a project-level configuration file that Claude can reference.
Context engineering as a development practice

Context engineering — the deliberate shaping of an AI's operational context — is emerging as a distinct skill for developers who work extensively with AI assistants. It encompasses several practices:

Assumption surfacing (as with /common-ground): explicitly extracting and validating the AI's beliefs about your project.

Context priming: providing relevant architectural documents, ADRs (Architecture Decision Records), or style guides at the start of a session to establish conventions before any code is generated.

Context hygiene: periodically reviewing and refreshing the AI's understanding as the project evolves, particularly after major refactors or architectural changes.

Constraint specification: explicitly stating what the AI should not do — frameworks to avoid, deprecated patterns to exclude, performance budgets to respect.

These practices share a common principle: the quality of AI-generated output is bounded by the quality of the context it operates within. Investing time in context engineering yields compounding returns across all subsequent interactions.

Project Workflow Commands: Managing the Development Lifecycle

The final capability layer is the set of 9 workflow commands that manage development epics from discovery through retrospective. These commands integrate with Atlassian tools (Jira and Confluence) via an MCP (Model Context Protocol) server, bridging the gap between AI-assisted coding and project management.

The workflow commands map to the standard phases of an agile development lifecycle. Rather than switching between your IDE, Jira, and Confluence to manage a feature's journey, the workflow commands allow you to drive the entire lifecycle from within Claude Code.

The conceptual model is important: these commands do not replace your project management tools. They provide a unified interface that reads from and writes to those tools, with Claude serving as an intelligent intermediary. When you initiate an epic, Claude can create the corresponding Jira epic, break it into stories based on the feature specification, and generate Confluence documentation — all informed by the specialist knowledge from the relevant skills.

This integration creates a feedback loop between the code-level skills and the project-level workflow. The Feature Forge skill produces a specification; the workflow command translates that specification into Jira stories. The Test Master skill produces a test plan; the workflow command records that plan in Confluence. Each layer reinforces the other.

The workflow commands span the full lifecycle: discovery (scoping), planning (story creation), implementation (linking to skill-generated code), review (coordinating with Code Reviewer skill output), and retrospective (analyzing cycle time, scope changes, and outcomes). The key integration point is that Claude can reference the active Jira context when making implementation decisions, ensuring code changes align with story requirements.

Warning

The skill-based features (context-aware activation, multi-skill workflows, /common-ground) function independently of the Atlassian integration.
Tip
Even without Atlassian integration, the workflow command patterns are instructive. The underlying principle — structuring AI interactions around development lifecycle phases — applies regardless of which project management tools you use.
MCP servers and tool integration

The Model Context Protocol (MCP) is a standard for connecting AI assistants to external tools and data sources. An MCP server acts as a bridge: it exposes capabilities (reading Jira issues, creating Confluence pages) through a standardized interface that Claude Code can invoke.

The Atlassian MCP server used by the workflow commands is one example, but the pattern is general. MCP servers exist for databases, cloud providers, monitoring tools, and many other services. The key architectural insight is that MCP allows Claude to act on external systems, not merely generate text about them.

For the workflow commands specifically, the MCP server handles authentication, API versioning, and data transformation. Claude expresses intent ('create a story with these acceptance criteria'), and the MCP server translates that intent into the appropriate Jira API calls. This separation of concerns allows the skill definitions to focus on development logic while the MCP server handles integration mechanics.

Recap

This session examined the claude-skills plugin ecosystem as a case study in structured AI-assisted development. We progressed through three layers of capability — from individual skill activation, through multi-skill workflow chains, to context engineering — each building on the previous to create a coherent methodology for working with AI coding assistants.

The fundamental shift this system represents is from ad-hoc prompting to structured specialist routing. Rather than crafting prompts that attempt to convey all necessary context in a single message, the skill system decomposes expertise into modular, conditionally-activated specialists, each backed by curated reference materials. This mirrors how effective engineering organizations operate: through specialization, handoff protocols, and shared context.

The /common-ground pattern deserves particular emphasis. Context engineering — the deliberate construction of the operational context in which an AI functions — is arguably the highest-leverage skill for developers working with AI assistants. A single session of assumption validation can prevent systematic errors across an entire development cycle.

These patterns are not unique to this particular plugin. The principles of specialist routing, workflow chaining, and assumption surfacing apply to any AI-assisted development workflow, regardless of the specific tools or models involved.

  • Context-aware activation routes your prompts to domain specialists based on technology and task signals in your prompt — specificity determines routing accuracy.
  • Multi-skill workflow chains decompose complex tasks into specialist handoffs where each stage's output serves as the next stage's input context, mirroring how effective engineering teams operate.
  • The /common-ground command implements context engineering by surfacing and validating Claude's implicit assumptions about your codebase — correcting one upstream assumption prevents many downstream errors.
  • Prompt specificity is the primary lever for skill activation quality: include the technology, the framework, and the task domain to ensure the correct specialist engages.
  • Workflow commands bridge AI-assisted coding and project management by providing a unified interface to Jira and Confluence through MCP server integration.
