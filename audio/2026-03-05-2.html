Your CLI Has a New User — and It Doesn't Have Fingers

AI agents are the fastest-growing consumer of command-line tools. Let's build a CLI designed for them from Day One.
40 min cli-design agent-dx mcp schema-introspection tool-building

What's happening

Google engineer Justin Poehnelt dropped a blog post yesterday that's been lighting up Hacker News: "You Need to Rewrite Your CLI for AI Agents." His core argument? Human DX and Agent DX are fundamentally different design goals. Humans need discoverability and forgiveness — tab completion, helpful error messages, --help flags. Agents need predictability and defense-in-depth — structured I/O, self-describing schemas, and guardrails against their own hallucinations.

Poehnelt didn't just theorize about this. He built gws, a CLI for Google Workspace where agents were the primary consumer from Day One. And his design choices are counterintuitive enough to be worth studying: raw JSON payloads instead of bespoke flags, runtime-queryable schemas instead of docs stuffed into prompts, and safety rails like dry-run modes baked into every command.

Today we're going to build a small CLI tool that follows these same principles. Not a toy — a reusable template you can adapt for wrapping any API. By the end, you'll have a CLI that an AI agent can pick up and use without ever reading documentation.

1

The Core Insight: Why Flags Are the Wrong Abstraction

Before we build anything, let's make sure the why clicks.

Imagine you're ordering food at a restaurant. A human menu has pictures, descriptions, categories — it's optimized for browsing and discovery. Now imagine a kitchen order ticket: it's terse, structured, and maps directly to what the kitchen needs to produce. No pictures. No flowery descriptions. Just the data.

Traditional CLI flags are the human menu. They're great for discovery (--help tells you what's available) but terrible for machines. Ten flat flags can't express nested data without inventing conventions like --sheet-title vs --title. An agent generating those flags has to know your custom naming scheme.

A raw JSON payload is the kitchen ticket. It maps 1:1 to the API schema. An agent that knows the API schema can generate the JSON trivially — zero translation loss.

Here's the mental model: flags are a lossy compression of the API surface. Every flag you invent is a mapping the agent has to learn. With --json, the agent just speaks the API's native language.

Let's get our agent to build a CLI skeleton that accepts both modes.

Ask your agent
Get the agent to scaffold a CLI tool in Python (or Node — your pick) that accepts a subcommand and a `--json` flag for raw payloads, alongside traditional flags for the same operation.
Think about it
  • What's a simple domain to model? Think: a task manager, a bookmark tool, a note-taker — something with nested data.
  • How should the CLI handle the case where someone passes BOTH --json AND individual flags? Which wins?
  • What library handles CLI argument parsing well? (Click, argparse, Commander, yargs...)
  • Should the JSON be validated before being sent anywhere, or just passed through?
What the agent gives back

The agent should give you a CLI with a create subcommand that works two ways. In "human mode" you'd pass flags like --title and --tags. In "agent mode" you pass --json '{...}' with the full payload. The key design choice: --json overrides individual flags when both are present. The agent will probably use Click or argparse in Python, or Commander/yargs in Node. Look for the part where it merges flag-based input into the same internal structure as the JSON input — that's the important bit. It should be maybe 30-40 lines total, and you really only care about the routing logic:

```python

The key pattern: both paths produce the same dict

if json_input:

payload = json.loads(json_input)

else:

payload = {"title": title, "tags": tags.split(",")}

```

Everything downstream just works with payload — it doesn't know or care how it was constructed.

Tip
Poehnelt's gws CLI uses --params for query parameters and --json for request bodies. This maps directly to HTTP semantics — query string vs. body. Keeping that separation clean is surprisingly powerful.
But won't this confuse human users?

This is the most common pushback, and it's fair. The answer is: you serve both audiences in the same binary. The human types mytool create --title "My Note" and gets a nice experience. The agent sends mytool create --json '{...}' and gets exactly what it needs. The trick is that both paths converge to the same internal representation before any business logic runs. You're not maintaining two tools — you're maintaining two entry points into the same tool.

Poehnelt also suggests auto-detecting the consumer: if stdout isn't a TTY (meaning output is being piped to another program or captured by an agent), default to JSON output. If it IS a TTY (a human is watching), use pretty formatting. Most modern CLIs already do this for color codes — extend the same idea to output format.

Schema Introspection: Let the CLI Be Its Own Documentation

Here's a scenario that happens constantly in agentic workflows: an agent needs to call your CLI, but it doesn't know the exact shape of the JSON payload. In the old world, you'd stuff API docs into the system prompt — burning tokens and going stale immediately.

Poehnelt's insight is elegant: make the CLI itself queryable. Add a schema subcommand that dumps the expected JSON shape for any operation.

Think of it like this: instead of giving someone a paper map of a city (static docs), you give them a GPS that they can query in real time (runtime schema). The GPS is always up-to-date because it's connected to the same system it's describing.

This is also what makes the MCP (Model Context Protocol) connection work. MCP tools are fundamentally self-describing — they tell the model what parameters they accept via JSON Schema. We're applying the same principle to a CLI.

Ask your agent
Extend your CLI with a `schema` subcommand that dumps the JSON Schema for any operation's expected payload.
Think about it
  • Where should the schema definitions live? Inline in the code, or in separate files?
  • Should the schema command output human-readable descriptions too, or pure machine-readable JSON Schema?
  • What if an agent wants to know ALL available operations — not just the schema for one? How would it discover them?
  • Think about required vs. optional fields — how does JSON Schema express that?
What the agent gives back

The agent should add a schema subcommand so you can run something like mytool schema create and get back a JSON Schema describing the expected payload. It'll include field types, which ones are required, descriptions, and maybe even examples. The important design choice is having a registry or dictionary that maps operation names to their schemas — this becomes the single source of truth for both validation AND documentation.

You should also get a schema --list or schema operations mode that dumps all available operations, so an agent can self-discover what's possible.

```python

The schema registry pattern — this IS your docs

SCHEMAS = {

"create": {

"type": "object",

"required": ["title"],

"properties": {

"title": {"type": "string"},

"tags": {"type": "array", "items": {"type": "string"}}

}

}

}

```

That registry is doing triple duty: it validates input, generates help text, and serves as runtime documentation for agents.

Tip
Pro move: use the same schema definitions to VALIDATE incoming --json payloads. Now your schemas aren't just documentation — they're guardrails. One definition, three jobs: docs, validation, and agent introspection.
Warning
Don't over-engineer the schema definitions. Start with the basics — type, required, description. You can always add enum constraints, pattern regexes, and $ref composition later. Ship the simple version first.
How does Google's gws CLI handle this?

Poehnelt's gws CLI does something really clever: it pulls schemas from Google's Discovery Document — a machine-readable description of every Google API endpoint. The CLI resolves $ref pointers dynamically, so when Google updates an API, the schema output updates automatically without a CLI release. You probably don't have a Discovery Document for your API, but the principle transfers: derive your CLI's schema output from the same source of truth that your API uses. If your API has OpenAPI specs, generate your CLI schemas from those. If you're using a typed language with Zod, Pydantic, or similar, generate schemas from your type definitions.

The MCP connection

If you squint at this pattern, it's basically what MCP (Model Context Protocol) does. An MCP server exposes tools with JSON Schema descriptions. The model reads those schemas to understand what it can call and what shape the arguments should be. Our schema subcommand is the non-MCP version of the same idea — it lets any agent, not just MCP-connected ones, understand the CLI's interface at runtime.

In fact, if you wanted to wrap this CLI as an MCP tool later, the schema registry is exactly what you'd feed into the MCP tool definition. You've already done the hard work.

✓
At this point, you should have a CLI with two superpowers: it accepts raw JSON payloads via `--json` (alongside traditional flags), and it can describe its own expected input shapes via a `schema` subcommand. An AI agent could call `mytool schema create`, read the output, generate a valid JSON payload, and pass it to `mytool create --json '{...}'` — all without ever seeing documentation.

✗ Not quite. Flat flags can't naturally express nested structures. What happens with multiple sheets? --sheet-title-1, --sheet-title-2? You're inventing a custom encoding that the agent has to learn, and it'll hallucinate flag names that don't exist. Every flag is a mapping the agent can get wrong.

Safety Rails: Protecting Against the Agent's Own Hallucinations

Here's the uncomfortable truth about AI agents using CLIs: they will hallucinate flags, invent parameters, and confidently execute destructive operations. This isn't a bug to fix in the model — it's a reality to design around.

Poehnelt calls this "defense-in-depth," and it's borrowed from security engineering. You don't trust any single layer to be perfect. You stack protections:

  1. Dry-run mode: Every mutating command should support --dry-run, which shows what would happen without doing it. Think of it as a flight simulator — the agent gets to practice before flying the real plane.
  1. Confirmation gates: For destructive operations (delete, overwrite, transfer ownership), require explicit --confirm or --yes flags. An agent that forgets to pass --confirm gets a safe failure instead of deleting your production data.
  1. Structured error output: When something goes wrong, return machine-readable errors (JSON with error codes), not just human-readable messages. An agent can parse {"error": "missing_field", "field": "title"} and fix its request. It can't reliably parse "Oops! Looks like you forgot the title 😅".
  1. Schema validation as a guardrail: Remember those schemas from Step 2? Use them to validate --json input before hitting the API. Catch hallucinated fields early.

Let's add these rails to our CLI.

Ask your agent
Add dry-run mode, confirmation gates for destructive ops, and structured JSON error output to your CLI.
Think about it
  • Which operations in your CLI are destructive (delete, update, overwrite)? Those need confirmation gates.
  • What should dry-run output look like? The agent needs to understand what WOULD have happened.
  • How should errors be structured? Think about what an agent needs to automatically retry or fix its request.
  • Should there be a global `--dry-run` flag, or per-command? What's easier for an agent to remember?
What the agent gives back

The agent should add three things:

Dry-run mode: A global --dry-run flag that, when set, validates the input and prints what would happen (as JSON!) without executing. The output might look like {"action": "create", "payload": {...}, "status": "dry_run", "would_affect": "notes collection"}.

Confirmation gates: Destructive commands (like delete) require --confirm to actually execute. Without it, they print a warning and exit with a non-zero code. This is the seatbelt.

Structured errors: Instead of printing "Error: title is required", output {"error": true, "code": "VALIDATION_ERROR", "details": [{"field": "title", "message": "required"}]}. An agent can parse this and fix its next attempt.

The key insight: all three features are nearly free to implement once you have the schema registry from Step 2. Validation? Schema. Dry-run output? The validated payload. Error structure? Schema violation details.

Warning
Never make --confirm the default. Agents hallucinate flags they've seen before, but they're less likely to hallucinate a flag they've never encountered. Making confirmation opt-IN means the safe path is the default path.
Tip
Exit codes matter more than you think. An agent can check $? instantly. Use 0 for success, 1 for validation errors (agent should fix and retry), 2 for auth errors (agent should re-authenticate), and non-zero for everything else. Semantic exit codes are free documentation.
The 'blast radius' mental model

Think about every CLI command in terms of its blast radius — how much damage can it do if called incorrectly?

  • Read-only (list, get, schema): Zero blast radius. No guardrails needed. Let agents call these freely.
  • Create: Small blast radius. Worst case, you create junk data. Dry-run is nice but not critical.
  • Update: Medium blast radius. You could overwrite good data with bad data. Dry-run recommended.
  • Delete: Maximum blast radius. Data loss is irreversible. Confirmation gate required.

Map your safety rails to blast radius. Don't put a confirmation gate on list — that's just annoying. Don't skip it on delete — that's just reckless.

Structured Output: Speaking the Agent's Language

We've been focused on input — how agents send data to your CLI. But output matters just as much. An agent that gets back a pretty table with emoji headers and ANSI colors is going to have a bad time parsing it.

The principle is simple: JSON in, JSON out. But there's a slick trick from Poehnelt's design that makes this work for both humans and agents without maintaining two output formatters.

Check if stdout is a TTY (a terminal with a human watching) or a pipe (output being captured by a program or agent). If it's a TTY, format nicely. If it's a pipe, output raw JSON. Most shells expose this — in Python it's sys.stdout.isatty(), in Node it's process.stdout.isTTY.

This means: mytool list shows a pretty table for a human. But result=$(mytool list) or an agent capturing the output gets clean JSON. Same command, two experiences, zero configuration.

Ask your agent
Add auto-detecting output formatting to your CLI: pretty output for humans (TTY), JSON for agents (piped/captured).
Think about it
  • How do you detect whether stdout is a terminal or a pipe?
  • Should there be a way to force JSON output even in a terminal? (Think: `--output json`)
  • What about the reverse — forcing pretty output even when piped? When would you want that?
  • Should error output (stderr) always be structured, regardless of TTY detection?
What the agent gives back

The agent should add output formatting that auto-detects the environment. The core logic is tiny — it's basically an if/else on isatty(). When it detects a terminal, it formats output as a readable table or summary. When piped, it dumps JSON.

It should also respect a --output flag (or OUTPUT_FORMAT environment variable) to override auto-detection. Errors on stderr should always be JSON — agents need parseable errors regardless of how stdout is configured.

```python

import sys, json

def output(data):

if sys.stdout.isatty() and not force_json:

print_pretty_table(data) # human-friendly

else:

print(json.dumps(data)) # agent-friendly

```

That's really it. Five lines that make your CLI bilingual.

Tip
NDJSON (Newline-Delimited JSON) is even better than regular JSON for streaming operations. Each line is a valid JSON object, so an agent can process results as they arrive instead of waiting for the entire output. Think of it as the difference between downloading an entire movie vs. streaming it.
Why environment variables matter for agents

Agents operating in agentic loops (like Claude Code or AutoGPT-style systems) often set environment variables before running commands. Supporting OUTPUT_FORMAT=json means the agent can configure your CLI once and then run multiple commands without repeating --output json every time. It's the difference between telling a coworker "please write in bullet points" before every email vs. setting that as a preference.

Poehnelt's gws CLI also uses environment variables for auth tokens and default project settings — same principle. Reduce per-invocation boilerplate.

✓
Nice — you now have a CLI with four agent-first superpowers: (1) raw JSON payloads via `--json`, (2) runtime schema introspection via `schema`, (3) safety rails with dry-run and confirmation gates, and (4) auto-detecting output formatting. This is a reusable template. Any API you want to wrap for agents can follow this exact pattern.

Quick Check

✓ Correct! This is defense-in-depth in action. The agent gets a machine-readable error it can understand, the operation safely fails, and the agent can retry with --confirm if it truly intends to delete. The safe path is the default path.
✗ Not quite. Interactive prompts are a human UX pattern. An agent can't type 'y' into a prompt — it'll hang forever or crash. Agent-first CLIs must never block on interactive input. Use explicit flags instead.
✗ Not quite. Agents hallucinate. They confidently call delete on the wrong resource, with wrong IDs, on wrong accounts. 'The agent knows what it's doing' is the most dangerous assumption in agentic tool design. Always require explicit confirmation for destructive ops.

From CLI to MCP Tool: The Natural Evolution

Here's the payoff for all this careful design: an agent-first CLI is 90% of the way to an MCP tool already.

MCP (Model Context Protocol) is Anthropic's standard for connecting AI models to external tools. An MCP tool is, at its core, three things:

  1. A name
  2. A JSON Schema describing its inputs
  3. A function that accepts those inputs and returns structured output

Look familiar? You already have all three:

  • Your subcommands are tool names
  • Your schema registry defines the inputs
  • Your command handlers are the functions

The mental model: your CLI is an MCP server that speaks argv instead of JSON-RPC. Wrapping it as an actual MCP server is mostly plumbing — and exactly the kind of thing an AI agent is great at generating for you.

This is why Poehnelt's approach is so powerful. He didn't just build a nice CLI. He built an architecture that's portable across every way an agent might want to consume it: direct CLI invocation, MCP tool, or even an HTTP API wrapper.

Ask your agent
Ask your agent to explain (not build — just explain) how you'd wrap your CLI as an MCP tool, and what would change vs. stay the same.
Think about it
  • What parts of your CLI can be reused directly as MCP tool definitions?
  • What's the difference between how a CLI receives input (argv, flags) and how an MCP tool receives input (JSON-RPC)?
  • Where does the schema registry do the heavy lifting in the MCP conversion?
What the agent gives back

The agent should explain that the conversion is mostly about swapping the transport layer. Your schema registry becomes the MCP tool's inputSchema directly — no rewriting. Your command handlers become the tool's execution functions. The main changes are: (1) instead of parsing argv, you receive structured JSON from the MCP runtime, (2) instead of printing to stdout, you return a result object, and (3) you add MCP server boilerplate to handle the JSON-RPC protocol.

The punchline: if you designed your CLI well (schemas separate from parsing, handlers that take a dict and return a dict), the MCP conversion touches almost none of your business logic. That's the whole point of the architecture.

Tip
You don't have to choose between CLI and MCP. Many tools ship both: the CLI for humans and scripts, the MCP server for AI agents. With the schema registry pattern, maintaining both is nearly free.
The bigger picture: agent-first tool design

Poehnelt's article is really about something larger than CLIs. It's about a design philosophy for any tool that agents will consume:

  1. Self-describing interfaces — tools should explain themselves at runtime, not rely on external docs.
  2. Native data formats — accept and return the same structures the agent already thinks in (JSON, not bespoke encodings).
  3. Defense-in-depth — assume the consumer will make mistakes. Build guardrails, not just error messages.
  4. Dual-mode operation — serve humans and agents from the same codebase by detecting the consumer.

These principles apply to APIs, SDKs, MCP tools, and yes, CLIs. The CLI is just the most concrete place to practice them.

Recap

We started with a simple question: what does it look like to build a CLI where AI agents are the primary consumer? And we landed on four design principles that are broadly applicable to any tool agents will touch.

We built a CLI template with: raw JSON payloads via --json (eliminating the lossy compression of flat flags), runtime schema introspection via a schema subcommand (so agents self-serve instead of burning tokens on stale docs), defense-in-depth safety rails with dry-run mode and confirmation gates (because agents will hallucinate), and auto-detecting structured output that speaks JSON to agents and pretty-prints for humans.

The sneaky lesson: none of these features are expensive to build. They're design choices — decisions about how data flows in and out of your tool. And they make your CLI better for humans too: the schema command is great documentation, dry-run mode helps during development, and structured errors are easier to debug.

  • Agent DX and Human DX are different design goals — optimize for both by supporting raw JSON payloads alongside convenience flags, not by picking one
  • Runtime schema introspection (a schema subcommand) replaces static docs in prompts — it's cheaper, always current, and the agent self-serves
  • Defense-in-depth means safe defaults: destructive ops require --confirm, every mutation supports --dry-run, errors are structured JSON with semantic exit codes
  • Auto-detect the consumer via TTY detection — pretty output for humans, JSON for agents, same command, zero config
  • An agent-first CLI is 90% of the way to an MCP tool — the schema registry and handler architecture port directly

Where to go next

  • Wrap a real API you use daily as an agent-first CLI — pick something with nested data to really feel the --json advantage
  • Try pointing Claude Code at your CLI with zero documentation and see if the schema command alone is enough for it to figure out the interface
  • Read Poehnelt's full article for the Google Workspace-specific details and the Rust implementation insights
  • Explore MCP tool authoring — take your CLI's schema registry and convert it into an MCP server to see how little changes
