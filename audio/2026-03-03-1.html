Audio Files → Markdown → LLM Magic

Microsoft's MarkItDown can now transcribe audio files and spit out clean Markdown. Let's build a pipeline that turns any audio file into structured text your AI tools can actually work with.
30 min speech-to-text transcription MCP pipeline markdown

What's happening

Here's what's cool: Microsoft's MarkItDown library — which already handled PDFs, Word docs, and images — now supports audio files. It pulls out EXIF metadata and runs speech transcription to produce Markdown output.

Why does this matter? Because every LLM on the planet speaks Markdown fluently. If you can turn audio into Markdown, you can feed meeting recordings, podcast episodes, or voice memos straight into your AI workflow. Summarize a meeting. Extract action items. Build a searchable archive of podcast transcripts.

Even better, MarkItDown now ships an MCP server — that's the Model Context Protocol that lets tools like Claude Desktop call it directly. So you can say "hey Claude, transcribe this audio file" and it just... works.

Today we'll go from zero to a working audio-to-Markdown pipeline, and then wire it into Claude Desktop so it's always available.

1

Get Your Environment Ready

Before we can transcribe anything, we need MarkItDown installed with its audio extras. Think of it like buying a Swiss Army knife (MarkItDown) and making sure the corkscrew attachment (audio support) is included.

MarkItDown uses OpenAI's Whisper under the hood for speech-to-text — either running locally on your machine or hitting the OpenAI API. It also needs ffmpeg installed on your system, because audio files come in a zoo of formats (MP3, WAV, M4A, OGG...) and ffmpeg is the universal translator that normalizes them all.

Let's ask our agent to help us set everything up.

Ask your agent
Get the agent to write you installation commands for MarkItDown with audio support, including system dependencies.
Think about it
  • What system-level tool does audio processing usually depend on?
  • MarkItDown uses optional dependency groups — how would you install just the audio extras vs. everything?
  • What if the user is on macOS vs. Linux — does the system dependency install differ?
What the agent gives back

The agent should give you two things: first, the system dependency (ffmpeg) with install commands for your OS — brew install ffmpeg on Mac, sudo apt install ffmpeg on Linux. Second, the pip install with audio extras: pip install 'markitdown[audio]'. It might also mention creating a virtual environment first, which is good practice. If you want everything MarkItDown can do, use 'markitdown[all]' instead.

Tip
If you plan to use the OpenAI Whisper API (cloud-based, faster, no GPU needed) instead of local Whisper, you'll also need pip install openai and an API key. Local Whisper is free but compute-heavy — on a laptop without a GPU, a 5-minute audio file might take a while.
Warning
ffmpeg is non-negotiable. Without it, MarkItDown can't decode MP3s or most audio formats. If your transcription silently fails, this is probably why.
Local Whisper vs. OpenAI API — which should I pick?

Local Whisper (openai-whisper package) runs entirely on your machine. Free, private, no API key needed — but it downloads a ~1.5GB model and needs decent compute. Great for privacy-sensitive audio or if you're processing a ton of files.

OpenAI Whisper API costs a fraction of a cent per minute of audio, is much faster, and works on any machine. You just need an API key. For most people getting started, this is the easier path.

MarkItDown supports both — you just pass an OpenAI client when initializing if you want the cloud version.

2

Transcribe Your First Audio File

Alright, let's actually transcribe something. MarkItDown gives you two ways to do this: the command line (great for quick one-offs) and the Python API (great for building into a larger pipeline).

The mental model is simple: you hand MarkItDown an audio file, it runs Whisper on the speech, extracts any metadata baked into the file (like recording date, duration, artist tags), and gives you back a single Markdown document with everything.

Think of it like a court stenographer who also notes down what brand of microphone was used and what time the recording started.

Ask your agent
Ask your agent to show you how to transcribe an audio file using both the CLI and Python API — including how to configure it for the OpenAI Whisper API.
Think about it
  • What's the simplest CLI command to convert a file and see the output?
  • In the Python API, how do you tell MarkItDown to use the cloud Whisper API instead of local?
  • What does the output object look like — how do you get the actual text?
What the agent gives back

For the CLI, it's dead simple: markitdown meeting.mp3 -o meeting.md. That's it — one command, Markdown out.

For the Python API, the key insight is that you create a MarkItDown() instance and call .convert(). If you want to use the OpenAI API instead of local Whisper, you pass an OpenAI client when creating the instance. The result comes back as an object with a .text_content property that holds your Markdown. The whole thing is about 5 lines of Python.

API Key Note
If using the OpenAI Whisper API, set your key as an environment variable (OPENAI_API_KEY) or pass it directly to the OpenAI client. Never hardcode API keys in your scripts.
What does the Markdown output actually look like?

The output typically has two sections:

  1. Metadata — pulled from the audio file's EXIF/ID3 tags. Things like duration, sample rate, artist, album, recording date. This shows up as a Markdown list or table at the top.
  1. Transcript — the actual speech-to-text output. It's plain text organized into paragraphs. Don't expect speaker labels or timestamps by default — Whisper gives you a continuous transcript.

Honestly, the metadata part is underrated. If you're processing a batch of podcast episodes, having the episode title and duration right there in the Markdown is super handy for downstream processing.

✓
At this point, you should be able to take any audio file — a podcast clip, a voice memo, a meeting recording — and turn it into a Markdown file using either the command line or a few lines of Python. If `markitdown yourfile.mp3` produces text, you're golden.
3

Chain the Transcript into an LLM for Analysis

Here's where it gets interesting. A transcript by itself is nice, but what if we could immediately hand it to an LLM and say "summarize this meeting" or "extract all action items" or "identify the key arguments in this podcast"?

This is the pipeline pattern: Audio → Markdown → LLM prompt → Structured output.

Think of it like an assembly line. MarkItDown is the first station (raw audio → text). Your LLM is the second station (text → insights). You're the factory manager deciding what each station does.

The beautiful thing is that since MarkItDown outputs Markdown — and LLMs are basically Markdown-native — there's no awkward format conversion in between. It just flows.

Ask your agent
Get your agent to build a small Python script that transcribes an audio file with MarkItDown, then sends the transcript to an LLM (via the OpenAI API) with a customizable prompt — like 'summarize this' or 'extract action items'.
Think about it
  • How should you structure this so the analysis prompt is easy to swap out?
  • Should the transcript go in the system message or the user message?
  • What if the transcript is really long — how might you handle that?
What the agent gives back

The agent should give you a script with two clear phases. Phase one: use MarkItDown to convert the audio file and grab the .text_content. Phase two: send that text to an LLM (like GPT-4o or Claude) as the user message, with a system prompt that describes the task (e.g., 'You are a meeting summarizer. Extract key decisions and action items.'). The key design choice is making the analysis prompt a variable or argument so you can reuse the same script for different tasks. The whole thing should be maybe 15-20 lines — nothing fancy.

Tip
Pro move: ask your agent to make the analysis prompt a command-line argument. That way you can run python pipeline.py meeting.mp3 "extract all action items with owners" without editing the script every time.
What about really long recordings?

LLMs have context windows — typically 128K tokens for modern models, which is roughly 90,000 words. A one-hour meeting transcript is about 8,000-10,000 words, so you're usually fine.

But for, say, an 8-hour conference recording? You'll need to chunk it. The simplest approach: split the transcript into segments (by paragraph or by time if you have timestamps), summarize each chunk, then summarize the summaries. Your agent can build this for you — just describe the chunking strategy and ask for a recursive summarization pipeline.

Alternatively, some people split long audio into 10-minute segments before transcription, transcribe each one, and then process them individually. Both approaches work.

4

Wire Up the MCP Server for Claude Desktop

MCP — Model Context Protocol — is basically a universal adapter that lets AI assistants call external tools. Think of it like USB-C for AI: a standard plug that works everywhere.

The markitdown-mcp package wraps MarkItDown as an MCP server. It exposes a convert_to_markdown tool that any MCP client can call. Claude Desktop is one such client. Once configured, you can literally tell Claude "transcribe this audio file at /path/to/meeting.mp3" and it'll call MarkItDown behind the scenes and show you the result.

The setup is just editing one JSON config file. Let's do it.

Ask your agent
Ask your agent how to configure the markitdown-mcp server for Claude Desktop, including where the config file lives and what to put in it.
Think about it
  • Where does Claude Desktop store its configuration on your OS?
  • There are multiple ways to run the MCP server — uvx, python -m, Docker. Which is simplest?
  • What happens after you edit the config — do you need to restart anything?
What the agent gives back

The agent should tell you to edit Claude Desktop's config file — on Mac it's at ~/Library/Application Support/Claude/claude_desktop_config.json, on Windows it's in %APPDATA%\Claude\. The simplest config uses uvx (so you don't even need to pre-install anything):

```json

{

"mcpServers": {

"markitdown": {

"command": "uvx",

"args": ["markitdown-mcp"]

}

}

}

```

Then restart Claude Desktop. You should see a convert_to_markdown tool appear in Claude's available tools. That's it — you're wired up.

Tip
If you already have other MCP servers configured, just add the "markitdown" entry inside your existing mcpServers object — don't replace the whole file!
Warning
The MCP server inherits whatever MarkItDown extras are installed. If markitdown[audio] isn't installed in the environment the MCP server runs in, audio transcription won't work even though the server starts fine. If using uvx, make sure the resolved environment has audio dependencies.
What is MCP, actually?

Model Context Protocol is an open standard (created by Anthropic) that defines how AI assistants communicate with external tools. It's like a two-way API:

  1. The server (MarkItDown in our case) advertises what tools it offers — "I can convert files to Markdown"
  2. The client (Claude Desktop) discovers these tools and can call them when needed
  3. Communication happens over stdio (standard input/output) — the simplest possible transport

The beauty is that it's a standard. Any MCP server works with any MCP client. So if tomorrow you switch from Claude Desktop to some other MCP-compatible app, your MarkItDown integration comes along for free.

It's the same idea behind USB: agree on the plug shape, and everything just connects.

5

Build a Batch Processing Pipeline

One file at a time is fine for testing, but the real power move is batch processing. Imagine you have a folder of meeting recordings from the past month, or a season of podcast episodes. You want to transcribe all of them and generate a summary for each.

This is where we combine everything: MarkItDown for transcription, an LLM for analysis, and a bit of glue to iterate over files and save results.

The pattern is embarrassingly simple: loop over files, transcribe each one, send the transcript to an LLM with your analysis prompt, save the result. Your agent can build this whole thing.

Ask your agent
Ask your agent to build a batch processing script that takes a folder of audio files, transcribes each one with MarkItDown, runs an LLM analysis on each transcript, and saves both the raw transcript and the analysis as separate Markdown files.
Think about it
  • What audio file extensions should it look for?
  • How should output files be named — based on the input filename?
  • Should it skip files that have already been processed (so you can re-run safely)?
  • What if one file fails — should it crash or log the error and continue?
What the agent gives back

The agent should give you a script that scans a directory for audio files (mp3, wav, m4a, etc.), loops through them, and for each one: (1) transcribes it with MarkItDown, (2) saves the transcript as filename_transcript.md, (3) sends the transcript to an LLM with a configurable prompt, and (4) saves the analysis as filename_analysis.md. Good extras: skip files that already have output files (idempotent re-runs), try/except around each file so one failure doesn't kill the batch, and a progress indicator. The whole script should be straightforward — maybe 30-40 lines.

Tip
Ask your agent to add a --dry-run flag that lists which files would be processed without actually doing it. Super helpful for sanity-checking before kicking off a big batch.
How much does batch processing cost?

If you're using the OpenAI Whisper API for transcription, it's about $0.006 per minute of audio. So a 60-minute meeting costs ~$0.36 to transcribe.

The LLM analysis step depends on the model. GPT-4o runs about $2.50 per million input tokens. A typical meeting transcript is maybe 10,000 tokens, so the analysis costs ~$0.025.

Total for a one-hour meeting: roughly $0.39. A folder of 20 meetings? About $7.80. That's... basically nothing for what you're getting.

Local Whisper is free but slower. If you have a GPU, it's a great option for large batches where cost would add up.

✓
You should now have: (1) MarkItDown installed with audio support, (2) the ability to transcribe individual files from CLI or Python, (3) an MCP server hooked into Claude Desktop, and (4) a batch processing pipeline for folders of audio. That's a complete audio-to-LLM workflow!

Recap

We just built a complete audio-to-intelligence pipeline. Starting from a raw audio file — a meeting recording, a podcast, a voice memo — we can now automatically produce clean Markdown transcripts, pipe them into any LLM for analysis, and even do it at scale across entire folders of files.

The secret sauce here isn't any one tool. It's the composability. MarkItDown handles the messy audio-to-text conversion. Markdown serves as the universal interchange format that every LLM understands natively. And MCP glues it all together so your AI assistant can call the whole pipeline on demand.

This is what modern AI engineering looks like: not building everything from scratch, but wiring together good tools with clear interfaces. And your AI agent did most of the coding for you — you just had to know what to ask for.

  • MarkItDown converts audio to Markdown using Whisper under the hood — either locally (free, slow) or via the OpenAI API (cheap, fast). The output includes both metadata and transcript.
  • The MCP server (markitdown-mcp) turns MarkItDown into a tool that Claude Desktop can call directly — one JSON config change and you're done.
  • The real power is the pipeline pattern: Audio → Markdown → LLM → Structured Output. Because Markdown is the lingua franca of LLMs, there's zero friction between steps.
  • Batch processing lets you scale this to entire folders of audio files. Combined with a good LLM prompt, you can generate meeting minutes, podcast summaries, or searchable archives automatically.

Where to go next

  • Try MarkItDown's other converters — feed it PDFs, PowerPoint decks, and images alongside audio to build a universal document processing pipeline
  • Experiment with different LLM analysis prompts: sentiment analysis, topic extraction, speaker identification, or compliance checking on call recordings
  • Explore building a custom MCP server that chains MarkItDown with your own post-processing logic — so Claude gets transcription AND analysis in one tool call
  • Look into adding speaker diarization (who said what) by combining Whisper with tools like pyannote-audio before feeding into MarkItDown
