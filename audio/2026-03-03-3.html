Your Audio AI Lab — In a Box You Can Throw Away

Use OpenSandbox to run Whisper, TTS, and music generation models in isolated Docker containers — no GPU drivers polluting your laptop, no dependency nightmares, just clean ephemeral environments you spin up and tear down with a single API call.
40 min OpenSandbox code-interpreter TTS speech-to-text Docker sandboxing Whisper MusicGen

What's happening

Why This Matters Right Now

Alibaba just open-sourced OpenSandbox — a general-purpose sandbox platform designed specifically for AI workloads. Think of it like having a stack of fresh, disposable laptops you can boot up in seconds, run messy AI experiments on, and then crush into nothing when you're done.

Here's the thing that makes this interesting for audio AI specifically: models like Whisper, Bark, and MusicGen are notorious dependency hogs. They want specific PyTorch versions, CUDA libraries, ffmpeg builds — the kind of stuff that makes your system Python cry. OpenSandbox gives you Docker-based isolation with a clean Code Interpreter API on top, so you can run all this heavy audio processing without touching your actual machine.

The platform ships with multi-language SDKs (Python, TypeScript, Java, Go on the roadmap), a unified sandbox lifecycle API (create → use → kill), and support for both local Docker and Kubernetes runtimes. Today we're going to use it to build a little audio AI playground where we can transcribe speech, generate speech from text, and even create music — all running safely inside throwaway containers.

And yes — we're going to ask our AI agent to build basically all of it for us. Our job is to understand the architecture and guide the prompts.

1

Get the Sandbox Server Running

Before we can do anything fun with audio models, we need the sandbox platform itself. OpenSandbox has two pieces: a server that manages container lifecycles, and client SDKs that talk to it.

Think of it like a hotel front desk (the server) and your phone app for booking rooms (the SDK). The front desk handles all the messy stuff — provisioning rooms, cleaning up after guests, security — while you just tap "book" and "checkout" from your app.

The server runs locally and talks to Docker under the hood. You'll need Docker running on your machine — that's the one real prerequisite here.

What the agent gives back

The agent should give you a short shell script that does three things: (1) installs opensandbox-server and opensandbox-code-interpreter via uv pip, (2) generates the Docker config file at ~/.sandbox.toml using the init-config command with the docker example flag, and (3) starts the server. It should also mention that Docker Desktop (or the Docker daemon) needs to be running first. The key commands boil down to:

```bash

```

The agent might also add a health check curl command or a note about which port the server listens on.

Warning
Docker must be running before you start the OpenSandbox server. If you're on macOS, make sure Docker Desktop is up — the whale icon in your menu bar should be steady, not animating.
Tip
OpenSandbox uses uv for Python package management. If you don't have uv yet, install it with curl -LsSf https://astral.sh/uv/install.sh | sh. It's blazing fast compared to pip.
What's actually in that config file?

The ~/.sandbox.toml file tells the server how to manage containers. The Docker example config sets up local Docker as the runtime, configures networking, and sets resource limits for sandboxes. You can tweak things like default memory limits, timeout durations, and network policies. For now the defaults are fine — we'll revisit this if we need more resources for the bigger audio models.

Docker vs Kubernetes runtime — when would you pick which?

Docker runtime is perfect for local development — it's what we're using today. The Kubernetes runtime is for when you need to scale: think running 50 sandboxes in parallel for batch transcription, or providing sandbox-per-user in a multi-tenant app. Kubernetes also gives you better resource scheduling and isolation guarantees. But for learning and prototyping? Docker all the way.

Your First Sandbox — Hello World with Audio Dependencies

Now for the mental model that makes everything click.

A sandbox in OpenSandbox has a lifecycle: create → do stuff → kill. It's like a whiteboard in a meeting room. You check out the room (create), draw all over the whiteboard (run code, install packages, process files), and when you leave, housekeeping wipes the board clean (kill). Nothing persists unless you explicitly save it.

The SDK is async-first and uses Python's async with pattern, which means cleanup happens automatically even if your code crashes. That's a big deal when you're running heavy ML models that might OOM or timeout.

Here's what we want: a sandbox with the right Docker image, pre-loaded with audio processing tools like ffmpeg, and the ability to run Python code inside it via the Code Interpreter. Let's get our agent to build this foundation.

Ask your agent

Think about it
  • The sandbox needs a Docker image — OpenSandbox provides `opensandbox/code-interpreter:v1.0.1`. What entrypoint and env vars does it need?
  • Audio processing needs ffmpeg and PyTorch inside the container. How do you run shell commands in a sandbox?
  • How do you verify the installs worked? Think about what the Code Interpreter lets you do...
  • What timeout makes sense for installing large ML packages like torch?
What the agent gives back

The agent should give you an async Python module that: (1) creates a sandbox using Sandbox.create() with the code interpreter image, a 10-minute timeout, and Python 3.11 configured via env vars, (2) runs shell commands inside the sandbox to install ffmpeg via apt-get and torch/torchaudio via pip, (3) creates a CodeInterpreter instance and runs a quick Python snippet that imports torch and prints the version. The key pattern looks like:

```python

sandbox = await Sandbox.create(

"opensandbox/code-interpreter:v1.0.1",

entrypoint=["/opt/opensandbox/code-interpreter.sh"],

timeout=timedelta(minutes=10),

)

interpreter = await CodeInterpreter.create(sandbox)

```

The agent should wrap everything in async with sandbox: for automatic cleanup, and use sandbox.commands.run() for the shell installs.

Tip
The async with sandbox: pattern is your safety net. Even if your code crashes mid-way, the sandbox gets cleaned up. No orphaned Docker containers eating your RAM at 3 AM.
Why install packages at runtime instead of baking a custom image?

Great question — and honestly, for production you would bake a custom Docker image with all your audio deps pre-installed. But for prototyping and learning, runtime installation is way more flexible. You can experiment with different package versions, add new models on the fly, and iterate without rebuilding images every time. Once you've nailed down your stack, then freeze it into a custom image. OpenSandbox supports custom images — just point Sandbox.create() at your own image name.

What's the Code Interpreter actually doing under the hood?

The Code Interpreter is a process running inside the sandbox container that accepts code snippets via the OpenSandbox API, executes them in a managed Python (or other language) kernel, captures stdout/stderr and return values, and sends them back. It's similar to how Jupyter notebooks work — there's a persistent kernel that maintains state between executions. So if you define a variable in one run() call, it's still there in the next one. This is crucial for our audio pipeline where we'll load a model once and use it across multiple calls.

✓
At this point you should have the OpenSandbox server running locally and a working Python module that can create a sandbox, run shell commands inside it, and execute Python code via the Code Interpreter. If `interpreter.codes.run('print("hello")')` returns 'hello', you're golden.

The model stays in memory for the sandbox's lifetime. This avoids re-downloading/loading a 3GB model for every single audio file.
✗ Not quite. This would mean downloading and loading the 3GB Whisper model from scratch every time someone wants to transcribe audio. Speech-to-Text: Whisper in a Box

Now the fun part. We're going to run OpenAI's Whisper model — the open-source speech recognition powerhouse — entirely inside our sandbox. No API keys needed, no data leaving your machine. Just a model running in an isolated container.

The flow is: upload an audio file into the sandbox → load Whisper → transcribe → pull the text back out. OpenSandbox's file API handles the upload/download, and the Code Interpreter runs the actual model inference.

Here's a mental model: imagine you're sending a package to a secure facility. You drop the package at the gate (file upload), workers inside process it (Whisper transcription), and they send back a report through the gate (read the result). You never go inside, and nothing from inside leaks out.

  • Whisper comes in different sizes (tiny, base, small, medium, large). Which makes sense for a quick demo vs. production accuracy?
  • The Code Interpreter keeps state between runs. Should you load the model in the same call as the transcription, or separately? Why?
  • How do you get the transcription result back? Think about what `interpreter.codes.run()` returns...
What the agent gives back

The agent should give you an async function that: (1) reads the local audio file as bytes, (2) uploads it into the sandbox at a path like /tmp/audio.wav using sandbox.files.write_files() with a WriteEntry, (3) runs a Code Interpreter snippet that loads Whisper (using whisper.load_model('base') for demo speed) and calls model.transcribe('/tmp/audio.wav'), (4) returns the transcription text from the result. The agent should probably split this into two interpreter calls — one to load the model (slow, do once), one to transcribe (fast, do per-file). The result comes back as result.result[0].text from the interpreter. The function signature is something like async def transcribe(sandbox, interpreter, audio_path: str) -> str.

Tip
Start with Whisper 'base' (140MB) for fast iteration. The 'large' model is 2.9GB and way more accurate, but you'll be staring at a progress bar for a while. Scale up once you've confirmed the pipeline works.
Warning
If you're running without a GPU (likely on most laptops), Whisper will run on CPU. It works fine — just slower. A 30-second audio clip takes about 15-30 seconds to transcribe with the base model on CPU.
How does file transfer actually work in OpenSandbox?

OpenSandbox's file API uses the WriteEntry model which takes a path (where to put the file inside the container), data (the file contents as bytes or string), and mode (Unix permissions like 644). Under the hood, it's sending the file contents over the sandbox API to the container. For reading files back out, sandbox.files.read_file(path) pulls bytes back from the container to your host. This means your local filesystem and the sandbox filesystem are completely separate — files only cross the boundary when you explicitly transfer them.

Text-to-Speech: Making the Sandbox Talk Back

Now let's go the other direction — text in, audio out. We're going to run a TTS model inside the same sandbox.

For TTS, we've got a few open-source options. Bark by Suno is super expressive (it can do laughter, music, and different speakers) but it's heavy and slow on CPU. Kokoro is lighter and faster. For our sandbox demo, we'll go with whichever our agent recommends based on our constraints — which is a perfect exercise in prompt engineering.

The pattern is almost the mirror image of Step 3: send text into the sandbox → run TTS → pull the generated audio file back out to our local machine. Same secure facility analogy — just in reverse.

Tip
Here's a pro move: since the Code Interpreter keeps state, you can load both Whisper AND the TTS model in the same sandbox session. They coexist in memory. This means you could build a pipeline that transcribes audio, processes the text, and generates new speech — all in one sandbox.
The TTS model landscape in 2026 — a quick lay of the land

The open-source TTS space has exploded. Here's the quick rundown:

  • edge-tts: Microsoft's edge voices via API. Great quality, super fast, but needs internet access (which your sandbox may or may not have depending on egress config).
  • Bark (Suno): Amazingly expressive — handles laughter, singing, accents. But it's huge and slow on CPU.
  • Kokoro: Good balance of quality and speed. Runs okay on CPU.
  • Coqui TTS / XTTS: Great for voice cloning and multilingual. Medium weight.
  • Piper: Optimized for on-device/edge. Lightning fast, decent quality.

For a sandbox demo, Piper or Kokoro are probably your best bets if you're CPU-only. If the sandbox has internet egress enabled, edge-tts is the easiest win.

Tie It Together: A Flask Service That Orchestrates Sandbox Audio AI

Right now we've got individual functions floating around. Let's wrap them in a small web service — a single Flask app with endpoints for transcription, speech generation, and (bonus) music generation.

The architecture is clean: Flask handles HTTP requests from the outside world, and for each request, it talks to the OpenSandbox API to do the heavy lifting inside a container. Your Flask app is essentially a concierge — it doesn't do any audio processing itself, it just knows which sandbox to call and how to shuttle files back and forth.

This is also where sandbox lifecycle management gets real. Do you create a new sandbox per request? Keep a pool of warm sandboxes? Reuse one sandbox for everything? These are real architectural decisions — let's see what our agent suggests.

Ask your agent

Think about it

  • Each endpoint needs to handle file upload/download differently. What content types and response formats make sense?
  • For the music generation endpoint, MusicGen by Meta takes a text prompt and outputs audio. How would you describe this to the agent?
  • Error handling in sandboxes is important — what happens if the sandbox times out or the model crashes?
What the agent gives back

The agent should give you a Flask app with three POST endpoints. The key architectural decision is sandbox management — the agent will likely suggest creating the sandbox once at app startup and reusing it (with the models pre-loaded) rather than per-request creation. Each endpoint follows the same pattern: receive input → transfer to sandbox → run model via Code Interpreter → transfer output back → respond. For /transcribe, it accepts an audio file upload and returns JSON with the transcript. For /speak, it accepts JSON with text and returns an audio file. For /generate-music, it accepts a text prompt and returns a generated audio clip. The agent should include error handling for sandbox timeouts and model failures. The whole app is maybe 80-100 lines, but the important part is the sandbox initialization at startup:

```python

@app.before_serving # or at startup

async def init_sandbox():

app.sandbox = await Sandbox.create(...)

app.interpreter = await CodeInterpreter.create(app.sandbox)

Pre-load all three models

```

The agent might suggest using Quart instead of Flask since OpenSandbox is async-first — that's a smart suggestion if it comes up.

Warning
MusicGen is a big model (~3.3GB for the small version). If you're on a laptop with limited RAM, you might want to skip the music generation endpoint or use the 'musicgen-small' variant. Tell your agent about your hardware constraints — it should adapt the suggestions accordingly.
Tip
OpenSandbox supports network egress controls per sandbox. If you're running this in production, you'd want to lock down the sandbox's internet access so models can't phone home. Check the egress controls section of the sandbox config.
Sandbox pooling — the production pattern

For a real service, you'd maintain a pool of pre-warmed sandboxes with models already loaded. When a request comes in, grab a sandbox from the pool, use it, and return it. If the pool is empty, create a new one (and maybe grow the pool). If a sandbox gets into a bad state (OOM, crashed kernel), kill it and create a fresh one. This is basically connection pooling but for compute environments. OpenSandbox's Kubernetes runtime makes this much easier at scale — each sandbox is a pod, and K8s handles the scheduling.

Why not just run the models directly in the Flask process?

You totally could — and for a personal project, that's fine. But sandboxing gives you three things:

  1. Isolation: If Whisper has a memory leak or MusicGen segfaults, it kills the sandbox, not your web server.
  2. Security: If you're accepting user-uploaded audio files, you really don't want them processed in the same environment as your server code. Malicious audio files exploiting model bugs would be contained to the sandbox.
  3. Reproducibility: Every sandbox starts from the same clean state. No 'works on my machine' issues from accumulated state.

It's the same reason we use containers in general — but with a nicer API for the AI-specific workflow of 'run this code, give me the output.'

Recap

What We Built

We went from zero to a working audio AI service that runs three different AI models — Whisper for transcription, a TTS model for speech generation, and MusicGen for music creation — all inside isolated, ephemeral Docker containers managed by OpenSandbox.

The core pattern we learned is beautifully simple: create a sandbox → load models → shuttle files in and out → tear it down when done. OpenSandbox handles all the container lifecycle headaches, and the Code Interpreter gives us a persistent Python session inside the sandbox where models stay loaded across requests.

More importantly, we practiced the agent-first workflow: instead of writing all this code ourselves, we described what we wanted, thought carefully about the right questions to guide our prompts, and let the AI agent handle the implementation details. Our job was architecture and judgment — deciding on sandbox lifecycle strategy, picking the right model sizes, understanding the tradeoffs between isolation and convenience.

This is genuinely useful infrastructure. Next time you need to run untrusted code, process user-uploaded files through ML models, or just keep your dev environment clean while experimenting with heavy AI models — you've got the pattern.

  • OpenSandbox gives you Docker-based isolation with a clean async Python API — create, use, and kill sandboxes programmatically without touching Docker commands directly
  • The Code Interpreter inside a sandbox maintains state between calls, just like Jupyter cells — load a model once, use it many times, and it persists for the sandbox's lifetime
  • The sandbox file API (write_files / read_file) is your airlock for getting data in and out of the isolated environment — nothing crosses the boundary unless you explicitly transfer it
  • For audio AI specifically, sandboxing solves the dependency nightmare — PyTorch, CUDA, ffmpeg, and model weights all live inside the container and vanish when you kill the sandbox
  • The agent-first approach works especially well here: describe the architecture and constraints, let the agent handle the OpenSandbox API boilerplate, and focus your energy on the interesting decisions (which models, what lifecycle strategy, how to handle errors)

Where to go next

  • Build a custom Docker image with your audio models pre-baked — this eliminates the model download step and makes sandbox creation nearly instant
  • Explore OpenSandbox's Kubernetes runtime for running sandboxes at scale — perfect for batch transcription or a multi-user audio processing service
  • Add streaming support: OpenSandbox's Code Interpreter can stream output, which is great for long-running transcriptions where you want to show partial results
  • Try the sandbox egress controls to lock down network access — critical if you're processing user-uploaded audio files in production
