Your Own Private AI That Listens and Talks Back

We're going to self-host Khoj — an open-source AI assistant — and wire up voice input and output so you can have a spoken conversation with an AI that actually knows your stuff.
40 min speech-to-text text-to-speech self-hosted voice-assistant RAG

What's happening

Here's what's going on: the AI assistant space has been dominated by cloud services — ChatGPT, Gemini, Claude. They're great, but your data lives on someone else's servers, and you're locked into their voice interface decisions.

Khoj flips that script. It's a fully open-source, self-hostable AI assistant that supports voice interaction out of the box — speech-to-text for input, text-to-speech for playback. You can feed it your own documents (PDFs, markdown, Notion exports, Word files) and it'll do RAG-powered Q&A over them. Talk to it. It talks back. And everything stays on your machine.

What makes this particularly interesting right now is that Khoj isn't just a chatbot wrapper. It supports custom agents with tunable personalities, dedicated knowledge bases, and configurable tools. Imagine a voice-first research assistant that knows your entire notes archive, or a study buddy agent that quizzes you on your own documents — all running locally.

Today we're going to stand this whole thing up, configure the voice pipeline, feed it personal documents, and build a custom voice agent. Let's go.

1

Get Khoj Running Locally with Docker

First things first — we need Khoj running on your machine. Khoj is a web application with a Django backend, and the easiest way to get it going is Docker Compose. Think of it like installing an app, except the app is an entire AI assistant platform.

The architecture is straightforward: Khoj runs as a web server on your machine, you access it through your browser (usually localhost:42110), and it connects to whichever LLM backend you configure — could be a local model via Ollama, or a cloud API like OpenAI or Claude.

Here's where your AI agent comes in. Instead of hunting through docs and writing Docker configs by hand, let's have the agent set this up for us.

  • Where should persistent data live so it survives container restarts?
  • What port does Khoj serve on by default, and how do you expose that?
  • What environment variables might you need for connecting to an LLM provider?
What the agent gives back

The agent should give you a docker-compose.yml with two services: the Khoj app (using ghcr.io/khoj-ai/khoj image) and a PostgreSQL database. It'll map port 42110 to your host, set up a Docker volume for the database so your data persists, and include environment variables for the database connection. The key thing to notice: Khoj itself doesn't bundle an LLM — it connects to one. So you'll configure that through the web UI after first launch, not in the Docker file.

The compose file is only about 20 lines. The important bits are the image name, the port mapping (42110:42110), and the Postgres connection string passed as an environment variable.

Tip
If you want to go fully local (no cloud APIs at all), you can run Ollama alongside Khoj and point it at models like Llama 3 or Qwen. Ask your agent: "Add an Ollama service to this Docker Compose so I can use local models with Khoj."
Warning
Make sure Docker Desktop (or Docker Engine on Linux) is actually running before you docker compose up. Sounds obvious, but this trips people up more than any config issue.
Why PostgreSQL and not SQLite?

Khoj uses PostgreSQL for its vector search capabilities via the pgvector extension. When you upload documents, Khoj generates embeddings (numerical representations of meaning) and stores them in Postgres. This is what powers the semantic search and RAG — when you ask a question, Khoj searches for the most relevant document chunks by comparing embedding vectors. SQLite can't do that efficiently. Think of it like this: SQLite is a filing cabinet (great for looking things up by label), while Postgres with pgvector is more like a librarian who understands what you mean and can find related material even if the exact words don't match.

2

Configure the Voice Pipeline — Speech In, Speech Out

Okay, this is the fun part. Khoj supports voice interaction through two separate pipelines:

  1. Speech-to-Text (STT): You talk into your microphone, and Khoj transcribes it into text that gets sent to the LLM. Under the hood, this typically uses OpenAI's Whisper model.
  2. Text-to-Speech (TTS): The LLM's text response gets converted to audio and played back to you.

Think of it like a translator sitting between you and the AI. You speak English → the STT translator writes it down → the AI reads it and writes a response → the TTS translator reads it out loud to you.

The cool thing is that these pipelines are modular. You can use cloud APIs (OpenAI's Whisper API for STT, their TTS API for output) or run models locally. Let's get the agent to help us configure both.

Ask your agent
Get the agent to explain how to configure Khoj's speech-to-text and text-to-speech settings, including what API keys or local models you need.
Think about it
  • What's the difference between running Whisper locally vs. using the Whisper API?
  • What tradeoffs matter for TTS — speed, voice quality, cost, privacy?
  • Where in Khoj's admin interface would these settings live?
  • What happens if you want completely offline voice — no API calls at all?
What the agent gives back

The agent should walk you through Khoj's server admin panel (accessible at localhost:42110/server/admin). For STT, Khoj can use OpenAI's Whisper API (needs an API key) or you can configure a local Whisper model. For TTS, it supports OpenAI's TTS voices or ElevenLabs.

The key insight: voice config happens through the admin UI under the 'Speech to Text' and 'Text to Speech' model settings — you're basically telling Khoj which service to call when it needs to convert audio. If you set an OpenAI API key in Khoj's LLM config, the voice features can piggyback on that same key.

For fully local STT, the agent might suggest running whisper.cpp or faster-whisper as a sidecar service.

API Key Note
If you're using OpenAI's Whisper API for speech-to-text, you'll need an OpenAI API key. The good news: Whisper transcription is cheap — roughly $0.006 per minute of audio. A typical voice query costs a fraction of a cent.
Tip
Want the best local STT experience? Ask your agent about faster-whisper — it's a CTranslate2-optimized version of Whisper that runs 4x faster than the original with the same accuracy. Great for real-time voice input.
How does speech-to-text actually work?

Whisper (the model Khoj uses for STT) is a transformer model trained on 680,000 hours of multilingual audio. It works differently from old-school speech recognition that tried to match sounds to words.

Old approach: sound → phonemes → words → sentence (brittle, needed careful tuning)

Whisper approach: sound → spectrogram image → transformer → text (end-to-end, handles accents and noise gracefully)

It literally converts your audio into a visual representation (a spectrogram — think of it as a heatmap of sound frequencies over time) and then "reads" that image to produce text. It's basically the same trick as image captioning, but for audio. This is why it handles background noise, accents, and even code-switching between languages so well — it learned from the messy reality of how people actually talk.

Latency considerations for voice UX

Voice interaction feels terrible if there's lag. Here's what contributes to the delay:

  1. STT latency (~0.5-2s): Converting your speech to text. Local Whisper is faster for short queries; the API can be faster for longer ones.
  2. LLM thinking time (~1-5s): The model generating a response. This is usually the biggest bottleneck.
  3. TTS latency (~0.5-1s): Converting the response to speech.

Total round-trip: 2-8 seconds typically. That's... okay. Not instant, but usable. The trick is streaming: Khoj can start the TTS as soon as the first sentence of the LLM response arrives, rather than waiting for the whole thing. This makes it feel much more conversational.

3

Feed It Your Documents — Building the Knowledge Base

An AI assistant that can talk is cool. An AI assistant that can talk about your stuff is actually useful.

This is where Khoj's RAG (Retrieval-Augmented Generation) pipeline comes in. The idea is simple but powerful:

  1. You upload your documents (PDFs, markdown files, Notion exports, Word docs)
  2. Khoj chops them into chunks and generates embeddings for each chunk
  3. When you ask a question, Khoj finds the most relevant chunks and feeds them to the LLM along with your question

Think of it like giving the AI a cheat sheet before an exam. Instead of relying purely on what the model was trained on, it gets to peek at your actual documents first. This is why you can ask "What were the key decisions from last month's meeting notes?" and get a real answer, not a hallucinated one.

Content source config: For larger collections, you configure a folder path in Khoj's admin panel. Khoj watches this folder and automatically indexes new or updated files. This is how you'd connect your Obsidian vault, a folder of PDFs, or a Notion export.

The key insight: you want your chunks small enough to be specific but large enough to contain complete thoughts. Khoj's defaults are usually sensible, but if you're getting vague answers, your chunks might be too big.

Tip
Pro move: if you use Obsidian for notes, Khoj has a dedicated Obsidian plugin that syncs your vault automatically. Ask your agent about setting up the Khoj Obsidian plugin for real-time note indexing.
Embeddings explained — GPS coordinates for meaning

Here's a mental model that really clicked for me: embeddings are like GPS coordinates, but for meaning instead of location.

When you embed the sentence "The quarterly revenue exceeded expectations," you get a list of numbers (a vector) like [0.23, -0.87, 0.14, ...] — maybe 384 or 768 numbers long. These numbers represent where that sentence lives in meaning-space.

Sentences with similar meanings end up at nearby coordinates. So "Q3 earnings beat forecasts" would have a very similar vector, even though it uses completely different words. Meanwhile, "The cat sat on the mat" would be far away in this space.

When you ask Khoj a question, it embeds your question into the same space and then looks for document chunks that are nearby. It's literally finding the closest neighbors in meaning-space. This is why semantic search is so much better than keyword search — it understands what you mean, not just what you said.

Build a Custom Voice Agent with Personality

Here's where Khoj gets really interesting. Instead of just having one generic assistant, you can create custom agents — each with its own personality, knowledge base, and set of tools.

Imagine creating:

  • A Research Advisor agent that only references your uploaded papers and speaks in a formal academic tone
  • A Study Buddy that quizzes you on your notes using a casual, encouraging voice
  • A Meeting Summarizer that processes your meeting notes and gives you crisp spoken summaries

Each agent gets its own system prompt (personality), can be restricted to specific document collections (knowledge base), and can use different tools (web search, code execution, etc.). And because Khoj supports voice, you can have a spoken conversation with any of these agents.

Let's build one.

Think about it

  • How do you restrict an agent to only use certain document collections?
  • What tools should this agent have access to? What should it NOT have?
  • How would you tell the agent to keep responses short enough for comfortable listening?
What the agent gives back

The agent should help you craft a configuration for a custom Khoj agent. The key pieces:

System prompt / personality: Something like "You are a concise research assistant. When answering, lead with the key finding, then provide 1-2 supporting details. Keep responses under 3 sentences for voice delivery. Always cite which document your answer comes from."

Knowledge base: Restricted to your uploaded document collection — the agent won't hallucinate from general training data.

Tools: Document search enabled, web search disabled (you want answers from YOUR docs, not the internet).

The agent should explain that you create custom agents through Khoj's web UI under the Agents section, or through the admin panel at /server/admin. The critical voice-first insight: your system prompt should explicitly tell the agent to give spoken-friendly responses — short sentences, no bullet points or markdown formatting, clear signposting ("Based on your paper from March...").

Tip
Voice-first prompt design is its own skill. Written AI responses use headers, bullet points, and code blocks. Spoken responses need short sentences, verbal signposting ('first... second... finally...'), and natural pauses. Ask your agent: 'Help me rewrite this system prompt to optimize for spoken delivery.'
The system prompt is everything for voice agents

Here's something that honestly surprised me: the difference between a good and bad voice agent is almost entirely in the system prompt, not the model or the pipeline.

A default ChatGPT-style response sounds terrible when read aloud — it's full of markdown headers, numbered lists, asterisks for emphasis, and paragraphs that are way too long for listening. Your ears don't have a scrollbar.

The fix is dead simple: tell the agent in its system prompt how to format for voice. Key rules:

  • Max 3 sentences per response unless asked for detail
  • Lead with the answer, then support it (inverted pyramid, like journalism)
  • No visual formatting — no bullets, headers, or markdown
  • Use verbal transitions — "That said," "On the other hand," "Here's the key thing"
  • Cite sources conversationally — "According to your March meeting notes" not "[Source: meeting_notes_march.md]"

This one change — just the system prompt — transforms the experience from "robot reading a Wikipedia article" to "knowledgeable friend answering your question."

Agent architecture — how Khoj agents actually work

Under the hood, a Khoj agent is essentially a bundle of configuration:

  1. System prompt: Sets personality, response style, and behavioral rules
  2. Knowledge scope: Which document collections the agent can search (or all, or none)
  3. Tool access: Which capabilities are enabled — document search, web search, code execution, image generation
  4. Chat model: Which LLM powers this particular agent (you can mix — GPT-4 for complex analysis, a faster model for quick lookups)

When you talk to an agent, here's the flow:

  • Your voice → STT → text query
  • Text query → embedding → semantic search over the agent's knowledge scope
  • Retrieved chunks + system prompt + your query → LLM
  • LLM response → TTS → spoken audio back to you

Every agent shares the same STT/TTS pipeline but has its own brain (prompt + knowledge + model). That's why you can have a formal research agent and a casual study buddy running side by side.

Automate It — Scheduled Research and Voice Briefings

The last piece of the puzzle is automation. Khoj has a built-in automation feature that lets you schedule recurring tasks — think of it as cron jobs for your AI assistant.

Here's a scenario that makes this click: imagine every morning at 8 AM, your Khoj research agent scans your recently added documents, identifies anything new or important, and prepares a spoken briefing for when you sit down with your coffee. Or every Friday, it summarizes the week's meeting notes and sends you an audio digest.

This moves Khoj from a "tool you use" to an "assistant that works for you." And because it's self-hosted, these automations run on your schedule, with your data, on your machine.

Ask your agent
Get the agent to help you design a morning research briefing automation in Khoj — something that summarizes recent document additions and surfaces key insights.
Think about it
  • What should the automation's prompt look like to generate a good spoken briefing?
  • How do you tell it to focus only on recently added or modified documents?
  • What delivery method should it use — email, push notification, or just have it ready when you open Khoj?
  • How would you schedule this to run at a specific time each day?
What the agent gives back

The agent should explain that Khoj's automations are configured through the web UI. You create an automation by specifying:

  • A natural language prompt: Something like "Review any documents I added in the last 24 hours. Summarize the key points in 3-4 short sentences suitable for a voice briefing. Highlight anything that connects to or contradicts my existing notes."
  • A schedule: Daily at 8 AM, or whatever cadence you want.
  • Delivery: Khoj can send results via email or have them waiting in your chat when you next open the app.

The agent should note that this works best when combined with the custom agent you built in the previous step — you assign the automation to your Research Advisor agent so it uses the right personality and knowledge scope.

The clever part: because Khoj does RAG over your entire document collection, the morning briefing doesn't just summarize new additions — it can spot connections between new content and your existing knowledge base.

Tip
Khoj automations can be triggered by starting a message with /automate in the chat. Try: '/automate Every Monday at 9am, summarize any new research papers I've added and suggest three questions I should investigate this week.'
Why self-hosted automation matters for privacy

Here's the thing about cloud AI assistants with automation — you're giving a company's servers continuous, scheduled access to your most personal or sensitive documents. For a lot of use cases (medical records, legal documents, financial data, proprietary research), that's a non-starter.

Self-hosted Khoj with local automation means:

  • Your documents never leave your machine (or your private network)
  • The automation runs on your hardware, on your schedule
  • No API calls to third parties if you're using a local LLM via Ollama
  • Full audit trail of what was accessed and when

This is genuinely one of the strongest arguments for self-hosting over cloud AI. The automation features of cloud assistants are great — but only if you're comfortable with your data living on their infrastructure forever.

Recap

Let's zoom out and look at what we've built today. You now have a fully self-hosted, voice-enabled AI assistant running on your own machine. It can:

  • Listen to your spoken questions via speech-to-text
  • Search through your personal documents using semantic retrieval
  • Generate contextual answers grounded in YOUR data, not just training data
  • Speak those answers back to you with text-to-speech
  • Run automated briefings on a schedule
  • Support multiple custom agents, each with their own personality and knowledge scope

The bigger picture here isn't just about Khoj specifically — it's about the architecture pattern. Voice + RAG + self-hosting is a combo that solves a real problem: making your personal knowledge base accessible through the most natural interface we have (speech), without giving up privacy or control.

The AI agent workflow we used today is transferable too. Every step was about describing what you want to an agent and iterating on the output — not hand-coding solutions. That's the skill that compounds. The tools will change (maybe next month it's a different platform), but the ability to clearly prompt for infrastructure, pipelines, and agent configurations? That carries over to everything.

  • Voice interaction with AI requires two separate pipelines — STT (input) and TTS (output) — and optimizing each independently is key to a good experience
  • RAG quality depends more on how you chunk and retrieve documents than on which LLM you use. Fix retrieval first, upgrade the model second.
  • System prompts for voice agents are fundamentally different from text agents — short sentences, verbal signposting, no visual formatting. This one change transforms the experience.
  • Self-hosting isn't just about privacy — it gives you full control over the voice pipeline, document processing, agent configuration, and automation scheduling that cloud services lock down
  • Custom agents are defined by four knobs: personality (system prompt), knowledge (document scope), tools (capabilities), and model (which LLM). Mix and match for different use cases.
