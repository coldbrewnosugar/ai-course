Voice-to-Code with Aider: Speak Your Features Into Existence

What if you could lean back, describe a feature out loud, and watch the code write itself? Today we set that up.
35 min speech-to-text voice-coding aider developer-tools

What's happening

Aider has been quietly becoming one of the most popular open-source AI coding tools — it's been trending on GitHub consistently, and for good reason. It's a terminal-based AI pair programmer that can work with Claude, GPT-4o, DeepSeek, and basically any LLM you throw at it.

But here's the feature that caught our eye: voice-to-code. You literally talk to Aider, it transcribes your speech, feeds that transcription to an LLM, and the LLM edits your actual codebase. It's like dictating a letter, except the letter is a working React component.

The pipeline is deceptively simple: microphone → speech-to-text → LLM prompt → code edit → git commit. Each link in that chain is a piece of technology you could use independently, but Aider chains them together into something that genuinely feels like the future.

Today we'll set the whole thing up, understand how each piece works under the hood, and build a small project entirely by voice. Even if you never use voice coding daily, understanding this pipeline will change how you think about developer tooling.

1

Get Aider Running — Your AI Pair Programmer

Before we can talk to our code, we need Aider installed. The good news: it's a Python package with a one-liner installer. The tricky part is getting the audio dependencies right — your computer needs to be able to actually listen to your microphone, which requires a system-level library called PortAudio.

Think of it this way: Aider is the brain, but PortAudio is the ear. Without the ear, the brain can't hear you.

The installation has two layers:

  1. Aider itself — the Python tool that talks to LLMs and edits your code
  2. Audio dependencies — PortAudio (system-level) and SpeechRecognition (Python-level) so Aider can use your microphone

Here's where your AI agent comes in handy — let it figure out the right install commands for YOUR operating system.

Think about it
  • What operating system are you on? macOS, Linux, Windows? The audio library installs differently on each.
  • Aider needs both itself AND audio libraries — how would you tell your agent you need both?
  • What about API keys? Aider needs to talk to an LLM. Which model do you want to use?
What the agent gives back

The agent should give you a short sequence of commands: first install PortAudio via your system package manager (brew on macOS, apt on Linux), then install Aider via pip or its installer script, then install the SpeechRecognition and PyAudio Python packages. It should also remind you to set an API key environment variable for whatever LLM you want (Anthropic, OpenAI, etc.). The whole thing should be 4-5 commands total.

API Key Note
Aider's voice feature uses OpenAI's Whisper API for transcription by default, so you'll need an OpenAI API key even if you're using Claude as your coding LLM. You can set it with export OPENAI_API_KEY=your-key. The Whisper usage is very cheap — pennies per session.
Tip
If PyAudio gives you install grief (it's famously finicky), ask your agent: "PyAudio won't install on my machine, what are the common fixes?" — it's one of the most-asked questions on Stack Overflow and your agent will know the workarounds.
Why does Aider need PortAudio specifically?

PortAudio is a cross-platform audio I/O library — think of it as a universal translator between your microphone hardware and any software that wants to record audio. Python's PyAudio package is just a wrapper around PortAudio. Without PortAudio installed at the system level, PyAudio can't compile, and without PyAudio, Aider can't access your mic.

This is a common pattern in audio/video tooling: a system-level C library does the heavy lifting, and a Python package provides a friendly interface on top. Same idea as how OpenCV works for video.

2

Understanding the Voice Pipeline — From Sound Waves to Git Commits

Before we start talking to our code, let's understand what actually happens when you say "add a login form with email and password fields" into your microphone. This is the cool part.

The pipeline has four stages, and honestly, once you see it laid out, you'll realize you could build something like this yourself:

  1. Capture — Your microphone records raw audio. Aider listens until you stop talking (it detects silence).
  2. Transcribe — The audio gets sent to a speech-to-text model (Whisper by default). You get back plain text.
  3. Prompt — That text becomes the exact same prompt you'd type manually into Aider. No magic — it literally types your words for you.
  4. Edit & Commit — The LLM reads your codebase, generates the code changes, and Aider applies them and auto-commits to git.

The beautiful insight here is that stage 3 is the key. The voice part is just an input method. Once your speech becomes text, it's indistinguishable from typing. This means everything you've learned about writing good prompts for coding agents applies directly to voice coding — you just say the prompt instead of typing it.

Ask your agent
Ask your agent to explain how OpenAI's Whisper API works and what alternatives exist for speech-to-text
Think about it
  • What's the difference between the Whisper API (cloud) and running Whisper locally?
  • What about latency — how fast does transcription need to be for voice coding to feel natural?
  • Are there privacy implications of sending your voice to a cloud API?
What the agent gives back

Your agent should explain that Whisper is OpenAI's speech-to-text model, available both as a cloud API and as an open-source model you can run locally. The cloud API is fast (~1-2 seconds for a typical utterance) and cheap. Running locally via whisper.cpp or faster-whisper avoids sending audio to the cloud but requires a decent GPU for real-time speed. For voice coding, the cloud API latency is fine because you're waiting for LLM code generation anyway, which takes longer than transcription.

Tip
Here's a mental model: voice-to-code is like speech-to-text-to-code. The 'speech-to-text' part is a solved problem. The 'text-to-code' part is what LLMs are great at. Aider just glues them together. Once you see it that way, voice coding feels less magical and more like a natural evolution.
Could you swap in a different STT engine?

Absolutely, and this is where it gets interesting for tinkerers. Aider uses Python's SpeechRecognition library, which supports multiple backends: Google Speech Recognition (free tier), Whisper API, local Whisper, even IBM Watson or Microsoft Azure speech services.

If you wanted to go fully local and private, you could run faster-whisper on your machine and point Aider at it. The transcription quality with Whisper large-v3 is remarkably good — it handles technical jargon, variable names, and even some programming keywords surprisingly well.

The broader lesson: any pipeline that takes text input can theoretically become voice-driven. Chat apps, CLI tools, search — if it accepts a string, you can put a microphone in front of it.

Your First Voice Session — Talking a Project Into Existence

Time to actually use this thing. We're going to create a small project entirely by voice. The project: a command-line weather tool that fetches weather data and displays it nicely. Simple enough to build in a few voice commands, interesting enough to feel real.

Here's how a voice session works in practice:

  1. Navigate to an empty project directory
  2. Launch Aider with your preferred model: aider --model sonnet
  3. Type /voice to activate voice mode
  4. Talk. Describe what you want. Aider records until you pause.
  5. Watch your transcribed words appear, then watch the code materialize.

The first time you do this, it's genuinely a bit surreal. You say "create a Python script that takes a city name as an argument and fetches the current temperature from the wttr.in API" and... it just does it.

But here's the thing — how you speak matters, just like how you type prompts matters. Let's figure out what makes a good voice prompt.

Ask your agent
Ask your agent for tips on how to structure voice prompts differently from typed prompts — what works better when spoken vs. typed?
Think about it
  • When you talk, you tend to ramble and add filler words. How does that affect the prompt quality?
  • Can you say things like 'variable name user underscore input'? How does Whisper handle technical terms?
  • Should you describe the WHAT (behavior) or the HOW (implementation details) when speaking?
What the agent gives back

The agent should explain that voice prompts work best when you focus on describing behavior and outcomes rather than specific syntax. Say 'create a function that fetches weather data for a given city and returns the temperature as a number' rather than trying to dictate code. Whisper handles natural English well but can stumble on camelCase variable names or unusual syntax. The sweet spot is speaking like you'd explain something to a coworker: 'I need a CLI tool that takes a city name, hits the wttr.in API, parses the JSON response, and prints the temperature in both Celsius and Fahrenheit.'

Warning
The /voice command records until it detects a pause in your speech. If you're a slow, thoughtful speaker with long pauses, it might cut you off mid-thought. Speak in complete thoughts, then pause deliberately when you're done. Think of it like a walkie-talkie — say your piece, then release.
Tip
Pro move: use /voice for the big-picture descriptions ('build me a weather CLI with these features') and then switch back to typing for precise tweaks ('rename the variable on line 12 from temp to temperature_celsius'). Voice for vision, keyboard for precision.
How does Aider know which files to edit?

This is one of Aider's cleverest features: it builds a repository map — a condensed representation of your entire codebase's structure. Think of it like a table of contents for your code: function names, class definitions, imports, file relationships.

When you say 'add error handling to the weather fetching function,' Aider uses this map to figure out which file contains that function, what it looks like, and what context the LLM needs to make the edit. You don't have to say 'in weather.py on line 34' — Aider figures that out.

The map uses tree-sitter for parsing, which means it understands code structure across 100+ languages. It's not just grepping for text — it actually understands that def fetch_weather is a function definition.

The Voice-Driven Workflow — Iterating by Conversation

Here's where voice coding goes from a cool trick to a genuinely different way of working. Instead of the usual cycle of think → type → read output → type corrections, you enter a conversational flow: think → speak → watch → speak corrections.

It's the difference between texting someone and calling them. Both communicate the same information, but the conversational flow is fundamentally different. When you're speaking, you tend to be more direct, more natural, more focused on what you actually want rather than how to express it in code-speak.

Here's a realistic voice session building our weather tool, showing how each voice command builds on the last:

  • Voice 1: "Create a Python CLI tool that takes a city name and shows the current weather from wttr.in"
  • Voice 2: "Now add a flag for Fahrenheit vs Celsius, default to Celsius"
  • Voice 3: "Add some color to the output — make the temperature blue if it's cold and red if it's hot"
  • Voice 4: "Write three tests for the temperature conversion logic"

Each of those is a single /voice command. Each one produces a working, committed change. After four sentences, you have a tested, colorful CLI tool. That's the pitch.

But let's be honest about the rough edges too.

Ask your agent
Ask your agent about the limitations and failure modes of voice-to-code — when does it break down?
Think about it
  • What happens when Whisper mistranscribes a technical term?
  • Are there types of coding tasks that are just better done by typing?
  • How do you handle the 'undo' case — when the voice command produces bad code?
What the agent gives back

The agent should be honest: voice coding struggles with exact variable names, regex patterns, complex file paths, and anything that requires precise syntax. Whisper can mishear 'camelCase' names or confuse similar-sounding words. The undo story is actually great though — since Aider auto-commits every change, you can just /undo to revert the last voice command. Best workflow: use voice for describing features and behavior, switch to typing for debugging, exact refactors, or anything involving specific identifiers. Also, noisy environments kill the experience.

Tip
Aider's /undo command is your safety net. Every change gets its own git commit, so you can undo any voice command that went sideways. Think of it as a conversational 'actually, never mind' — something we do naturally in speech but rarely have in code editors.
What about accessibility?

Voice-to-code isn't just a productivity hack — it's a genuine accessibility win. Developers with RSI, motor disabilities, or conditions that make extended typing painful have found voice coding transformative. One Aider user in the GitHub discussions mentioned continuing to be productive while recovering from hand surgery.

This is worth remembering when people dismiss voice coding as a gimmick. For some developers, it's the difference between being able to code and not. And the technology is only getting better — Whisper's accuracy has improved dramatically with each version, and local models are approaching real-time speeds on consumer hardware.

✓
You should now have a working voice-to-code setup, have spoken at least a couple of commands to Aider using /voice, and have a small project (or the start of one) that was built by speaking. You should also have a feel for what kinds of prompts work well spoken vs. typed. If you haven't tried /voice yet, now's the moment — the rest of this workshop assumes you've felt the experience firsthand.

Beyond Aider — Building Your Own Voice-Driven Dev Tools

Now that you've seen the pipeline in action, let's zoom out. The real takeaway isn't "Aider has a cool voice feature" — it's that any text-based tool can become voice-driven with a surprisingly simple pipeline.

The recipe is always the same:

  1. Capture audio from a microphone
  2. Send it to a speech-to-text model
  3. Feed the resulting text into whatever tool accepts text input

That's it. Three steps. You could voice-enable your git workflow, your task manager, your database queries, your CI/CD pipeline — anything that takes a string.

Let's think about what a custom voice-driven tool might look like for YOUR workflow.

Ask your agent
Ask your agent to sketch out how you'd build a minimal voice-to-anything pipeline in Python — not the full code, just the architecture and key libraries
Think about it
  • What's the simplest way to record audio from a microphone in Python?
  • If you wanted to use Whisper locally instead of the API, what would that look like?
  • How would you make this work as a reusable function you could plug into any CLI tool?
What the agent gives back

The agent should describe a simple three-part architecture: (1) use sounddevice or PyAudio to record audio until silence is detected, (2) send the audio buffer to either the Whisper API or a local faster-whisper model for transcription, (3) return the text string so you can pipe it wherever. The key insight is that this entire pipeline can be a single function — record_and_transcribe() — that returns a string. Maybe a 5-line code sketch showing the function signature and the Whisper API call, but the emphasis should be on the architecture, not the implementation.

Tip
If you want to go fully local and private, look into faster-whisper — it's a CTranslate2-optimized version of Whisper that runs 4x faster than the original. On a decent GPU, you get near-real-time transcription without sending anything to the cloud.
Real-world voice-driven dev tools people have built

The voice-to-text-to-action pattern is showing up everywhere:

  • Voice git: Speak your commit messages and branch names. "Create a branch called feature slash user-auth" → git checkout -b feature/user-auth
  • Voice SQL: Describe the data you want in plain English, get the SQL query generated and executed. "Show me all users who signed up last month and haven't made a purchase"
  • Voice documentation: Narrate what a function does while looking at it, and an LLM formats your speech into proper docstrings
  • Voice testing: Describe test scenarios out loud — "test that the login fails with an empty password and returns a 401" — and get test code generated

The common thread: voice is great for intent and description, LLMs are great for translating intent into precise syntax. They complement each other perfectly.

Recap

We started with a simple question — what if you could talk to your code? — and built up a complete understanding of how voice-to-code actually works.

The pipeline is simpler than it seems: microphone → speech-to-text → text prompt → LLM → code edit → git commit. Each piece is mature technology. Aider's contribution is gluing them together into a smooth developer experience.

More importantly, we saw that voice coding isn't about replacing your keyboard. It's about adding a new input modality that's better for certain tasks — describing features, explaining bugs, thinking out loud about architecture. Voice for vision, keyboard for precision.

And the pattern generalizes. Any tool that accepts text can become voice-driven. The speech-to-text problem is solved. The text-to-action problem is what LLMs excel at. You're just connecting a microphone to an AI agent and letting it translate your intent into whatever structured output you need.

  • Voice-to-code is a pipeline, not magic: capture audio → transcribe with Whisper → prompt an LLM → apply code changes. Understanding the pipeline lets you customize or rebuild each piece.
  • Speak intent, not syntax. Voice prompts work best when you describe what you want the code to DO ('add caching with a 10-minute TTL') rather than how to write it ('create a dictionary called cache_store'). Let the LLM handle implementation details.
  • The voice-to-text-to-action pattern extends far beyond coding. Any text-based interface — git, SQL, documentation, CI/CD — can be made voice-driven with the same three-step recipe.
  • Aider's /undo + auto-commit gives you a safety net that makes voice coding less scary. Every voice command is its own commit, so bad generations are one undo away from gone.

Where to go next

  • Try building an entire feature by voice in an existing project — start with something small like adding a new API endpoint, and see how it feels compared to typing
  • Explore running Whisper locally with faster-whisper for a fully private, offline voice coding setup
  • Build your own voice-to-X tool using the pipeline pattern — voice commits, voice SQL queries, or voice documentation generation
  • Check out Aider's other power features: /architect mode for high-level design, repo mapping for large codebases, and multi-file editing workflows
