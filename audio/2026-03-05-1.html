Teaching Your AI Agent New Tricks

Skills are like apps for your AI — modular expertise packages that turn a generalist agent into a domain specialist. Today we'll install some, understand how they work, and build one from scratch.
40 min agent-skills claude-code plugins modular-ai

What's happening

Two things just landed that are worth paying attention to. First, Anthropic open-sourced their official Agent Skills repo — the same skill system that powers Claude's built-in document capabilities (PDF, DOCX, PPTX, XLSX). You can now see exactly how those production skills are structured and build your own.

Second, the community responded fast. A developer named Alireza Rezvani published a library of 86 production-ready skill packages covering everything from security auditing to RAG architecture to financial modeling. It's already at 2,300+ stars.

The big idea here is modular AI expertise. Instead of writing one massive system prompt that tries to cover everything, you install purpose-built skill packages that the agent loads on demand. Think of it like the difference between a Swiss Army knife and a full workshop — sometimes you need the right specialized tool.

This matters because it changes the economics of AI agent work. Instead of spending 30 minutes crafting the perfect prompt every time you need your agent to do something specialized, you install a skill once and it just knows that domain. And because the format is standardized (a folder with a SKILL.md file), skills are portable across Claude Code, Claude.ai, and the Claude API.

Today we'll go from zero to having a custom audio-workflow skill that we built ourselves — and along the way, we'll understand the architecture well enough to turn any domain expertise into a reusable skill package.

1

Understand the Skill Architecture

Before we build anything, let's grok what a skill actually is. Honestly, when I first heard "agent skills" I expected something complicated — SDKs, APIs, registration flows. Nope.

A skill is just a folder with a markdown file called SKILL.md. That's it. That's the whole thing.

Think of it like a recipe card. The front of the card has the name and a description ("when should the chef use this recipe?"). The back has the actual instructions — step-by-step guidance the agent follows when it activates that skill.

The SKILL.md file has two parts:

  1. YAML frontmatter — metadata at the top between --- markers. Just two required fields: name and description.
  2. Markdown body — the actual instructions, examples, and guidelines Claude follows.

The description field is secretly the most important part. It's how Claude decides when to activate the skill. It's like the label on a toolbox — if you label it poorly, you'll never grab the right tool.

More complex skills can also include:

  • Python CLI tools alongside the SKILL.md (scripts the agent can invoke)
  • Reference documentation (specs, style guides, API docs)
  • Example files the agent can reference

But the core is always that one markdown file. Let's have our agent show us what one looks like.

Ask your agent
Get the agent to explain the SKILL.md structure by generating a simple example skill
Think about it
  • What kind of simple, everyday task would make a good example skill? Think of something your agent does repeatedly.
  • What two pieces of metadata does the frontmatter need?
  • What should go in the description to help the agent know WHEN to activate this skill?
  • What kind of instructions, examples, and guardrails would make the skill reliable?
What the agent gives back

The agent should give you a simple SKILL.md file — maybe a code-review skill or a git-commit-message skill. The key thing to notice is the structure:

```yaml

---

name: my-skill-name

description: When to use this skill

---

```

Followed by markdown with sections like ## Instructions, ## Examples, and ## Guidelines. The description field should read almost like a trigger condition — "Use this when the user asks for X or when Y happens." The instructions section is where the real expertise lives — the specific steps and rules the agent follows.

Tip
The description field isn't just documentation — it's the activation trigger. Claude reads it to decide whether this skill is relevant to the current task. Write it like you're telling a coworker: "Hey, use this when someone asks you to..."
How does Claude actually load skills?

When you install a skill, it goes into ~/.claude/skills/ (for Claude Code). When you start a conversation, Claude scans available skills and reads their descriptions. If your request matches a skill's description, Claude loads that skill's full instructions into its context window — like pulling a reference manual off the shelf mid-conversation.

This is why the description matters so much. A vague description means the skill might get loaded when it shouldn't (wasting context) or not loaded when it should (missing the opportunity).

It's also why skills should be focused rather than broad. Each skill you load takes up context window space, so a focused skill that does one thing well beats a bloated skill that tries to cover everything.

What about those document skills Anthropic uses in production?

The open-sourced repo includes the actual skills that power Claude's PDF, DOCX, PPTX, and XLSX handling. These are source-available (not Apache 2.0 like the examples), but studying them is incredibly useful. They show how a production skill handles edge cases, error recovery, and complex multi-step workflows. They're the "reference implementation" of what a serious skill looks like. Check skills/pdf and skills/docx in the anthropics/skills repo.

Install Skills from the Plugin Marketplace

Now that we know what skills are, let's install some. Anthropic built a plugin marketplace right into Claude Code — it's like an app store for agent capabilities.

The flow is dead simple. You register a GitHub repo as a "marketplace" (basically telling Claude Code where to find skills), then browse and install whatever you need. The two big marketplaces right now are:

  • anthropics/skills — Anthropic's official skills, including the document skills that power Claude.ai
  • alirezarezvani/claude-skills — 86 community skills across 9 domains (engineering, product, marketing, compliance, finance, and more)

The community library is particularly wild. There are skills for things like:

  • agent-designer — helps Claude design multi-agent orchestration systems
  • rag-architect — builds RAG pipelines with chunking optimization
  • security-auditor — scans code for vulnerabilities
  • iso-13485 — knows medical device quality management standards (!)

Let's install some skills and see how the marketplace works. You don't need to write any code here — just Claude Code commands.

Design Our Custom Audio Workflow Skill

Here's where it gets fun. We're going to build a custom skill for audio workflows — think transcription, speaker diarization, audio summarization, podcast processing, that kind of thing.

Why audio? Because it's a domain where Claude is generally helpful but not specialized. Without a skill, if you ask Claude to help process a podcast episode, it'll give you reasonable but generic advice. With a well-crafted skill, it'll know the specific tools (Whisper, pyannote, ffmpeg), the optimal workflow order, common pitfalls, and output formats.

Before we write anything, let's think about what makes a great skill. This is the most important step — the design — and it's where your agent can really help.

A good skill has three layers:

  1. When — a precise description of when this skill should activate
  2. What — the specific expertise and workflow steps
  3. How — examples, guardrails, and edge case handling

Think of it like training a new team member. You wouldn't just say "handle audio stuff." You'd explain the specific scenarios they'll encounter, the tools they should reach for, and the mistakes to avoid.

Ask your agent
Get the agent to help you design the SKILL.md for an audio-workflow skill — focus on the description field and the main instruction sections
Think about it
  • What specific audio tasks should this skill cover? Transcription? Diarization? Noise reduction? Summarization?
  • What tools and libraries should the skill know about? (Think ffmpeg, Whisper, pyannote...)
  • What's the typical workflow ORDER for processing audio? Does it matter?
  • What are common mistakes people make with audio processing that the skill should warn about?
What the agent gives back

The agent should give you a well-structured SKILL.md with a description like: "Activate when the user needs to transcribe, process, analyze, or summarize audio files. Covers speech-to-text, speaker diarization, audio cleaning, and content extraction workflows."

The instructions section should cover a logical workflow: (1) Audio preparation (format conversion, noise reduction with ffmpeg), (2) Transcription (Whisper model selection based on quality/speed needs), (3) Post-processing (speaker diarization, timestamp alignment), (4) Content extraction (summarization, key moments, action items).

It should include guidelines like: "Always check audio format before processing — convert to WAV 16kHz mono for best transcription results" and "For files over 25MB, chunk the audio rather than processing all at once."

The whole thing should be one markdown file — no code to write, just domain expertise encoded as instructions.

Tip
The best skills encode workflow knowledge, not just tool knowledge. Anyone can Google "how to use Whisper." The value of a skill is knowing the ORDER of operations, the decision points ("use the large model for this, the tiny model for that"), and the gotchas ("always convert to mono 16kHz first").
What makes a skill description great vs. mediocre?

Mediocre: "A skill for audio processing."

Good: "Activate when the user needs to transcribe, process, analyze, or summarize audio files including podcasts, interviews, meetings, and voice memos. Covers speech-to-text with Whisper, speaker diarization with pyannote, audio format conversion with ffmpeg, and content extraction workflows."

The good version lists specific trigger scenarios (podcasts, interviews, meetings) and specific capabilities (Whisper, pyannote, ffmpeg). This helps Claude match the skill to the right requests. The mediocre version would match too broadly (any mention of audio) or miss specific use cases ("summarize this meeting recording" might not trigger "audio processing").

Can skills include CLI tools?

Yes! More advanced skills can include Python scripts alongside the SKILL.md. For our audio skill, you could add a tools/ directory with scripts for common operations — like a chunk_audio.py that splits large files, or a format_transcript.py that converts raw Whisper output into a clean document. The SKILL.md would reference these tools in its instructions: "For files over 25MB, use the chunk_audio tool to split into segments." Claude Code can then execute these tools as part of the workflow. This is how the production document skills work — they have significant Python tooling alongside their SKILL.md.

Build and Install the Skill Locally

Now let's take our design and make it real. The beauty of the skill system is that "installing" a custom skill is just putting a folder in the right place. No build step, no compilation, no registration API.

For Claude Code, custom skills go in ~/.claude/skills/. For Claude.ai, you upload them through the UI. For the API, you pass them in the request.

We're going to have our agent create the final polished SKILL.md and put it in the right location. But here's the thing — this is also where we add the good stuff: specific examples that show Claude exactly what good output looks like, edge case handling, and maybe even a helper script or two.

Think of examples in a skill like unit tests for behavior. They don't just document what to do — they anchor the agent's understanding of what "good" looks like for this domain.

Ask your agent
Get the agent to create the final audio-workflow skill folder with a polished SKILL.md and install it locally
Think about it
  • Where exactly should the skill folder go on your filesystem?
  • What examples would help Claude understand what good audio workflow output looks like?
  • Should you include a helper script for any common operations like audio chunking?
  • How will you test that the skill activates correctly — what test prompt would you try?
What the agent gives back

The agent should create a folder at ~/.claude/skills/audio-workflow/ containing a polished SKILL.md. The file should have:

  • Frontmatter with name: audio-workflow and a detailed description
  • A ## Workflow section with the step-by-step processing order
  • A ## Tool Reference section listing ffmpeg, Whisper, and pyannote with when to use each
  • A ## Examples section with 2-3 scenarios ("User has a 2-hour podcast episode" → here's the workflow)
  • A ## Common Pitfalls section with things like sample rate mismatches and memory limits

The agent might also create a simple tools/chunk_audio.py helper script. After creation, test it by asking: "I have a 90-minute meeting recording in MP3 format. Help me transcribe it and identify who said what."

Tip
Test your skill with prompts that DON'T mention audio processing directly. Try 'I recorded a client call and need to pull out the action items.' A well-described skill should still activate because the description covers meeting recordings and content extraction.
API Key Note
If your skill references tools like Whisper's API, the user will need their own OpenAI API key for the transcription step. Make sure your skill instructions mention this requirement clearly — "Requires: OpenAI API key for Whisper, or local Whisper installation."
How to distribute your skill to others

Once your skill works locally, sharing it is straightforward. Push the skill folder to a GitHub repo (can be part of a larger skills collection). Others can either:

  1. Clone and copy — just drop the folder into their ~/.claude/skills/
  2. Use the marketplace — if you structure your repo correctly with a plugin manifest, others can install via /plugin marketplace add your-username/your-repo and then /plugin install your-skill@your-marketplace
  3. Upload to Claude.ai — the skill folder works directly with Claude.ai's skill upload feature
  4. Use via API — pass the SKILL.md contents in API requests

The format is intentionally simple so that skills are maximally portable. One SKILL.md works everywhere.

✓
You should now have a working custom audio-workflow skill installed at `~/.claude/skills/audio-workflow/`. Test it by asking Claude a question about processing audio — it should respond with noticeably more specific, structured guidance than before. If the responses feel generic, your description might need to be more precise about when to activate.
5

Skill Chaining — Making Skills Work Together

Here's where skills go from "neat trick" to "actually powerful." Skills can reference other skills. Your audio-workflow skill can say, in its instructions, "After transcription is complete, use the content-summarization skill to generate a brief."

This is skill chaining, and it's the pattern that makes the whole system scale. Instead of one monolithic mega-skill that tries to handle everything, you build focused skills that hand off to each other — like a relay team.

The community library's agent-workflow-designer skill actually knows five different chaining patterns:

  • Sequential — A → B → C (our audio pipeline)
  • Parallel — A and B run simultaneously, C combines results
  • Router — a decision skill picks which specialist skill to invoke
  • Orchestrator — a coordinator skill manages multiple workers
  • Evaluator — a quality-check skill that reviews another skill's output

For our audio skill, the natural chain is sequential: audio-preparation → transcription → diarization → summarization. But you could also imagine a parallel pattern where transcription and noise analysis happen simultaneously.

Let's design a chain that connects our audio skill to a summarization or action-item extraction step.

Ask your agent
Get the agent to add skill chaining instructions to your audio-workflow skill — specifically, connecting it to a downstream summarization or action-item extraction step
Think about it
  • What should the handoff look like? What data passes from one skill to the next?
  • Should the chain be hardcoded, or should the skill suggest the next step and let the user decide?
  • What if someone wants just transcription without summarization — how do you keep the chain flexible?
  • Could you use a router pattern where the skill decides between 'meeting notes,' 'podcast show notes,' and 'interview highlights' based on the content?
What the agent gives back

The agent should update your SKILL.md with a ## Workflow Chaining section that describes how this skill connects to others. The approach should be flexible — not hardcoded chains, but suggested next steps.

Something like: "After transcription, suggest the appropriate follow-up based on content type: for meetings → action-item extraction, for podcasts → show notes generation, for interviews → highlight compilation."

The key design insight is that the skill should suggest the chain, not enforce it. The instructions might say: "Once transcription is complete, ask the user if they'd like to proceed with content analysis. If the transcript appears to be a meeting, recommend action-item extraction. If it's a podcast or interview, recommend highlights and summary."

This is the router pattern in action — the audio skill acts as both a processor and a smart dispatcher.

Tip
The best skill chains feel invisible. The user says 'process this meeting recording and give me the action items,' and the agent seamlessly flows through audio prep → transcription → diarization → action-item extraction — each step using the right specialized knowledge.
Subagents — the next level of skill architecture

The community library includes an agent-designer skill that knows how to build multi-agent systems. In Claude Code, skills can spawn subagents — separate Claude instances with their own skill loadouts, working on subtasks. Imagine your audio skill spawning a transcription subagent (optimized for Whisper workflows) and a summarization subagent (optimized for content analysis) running in parallel.

This is still emerging, but the pattern is powerful: instead of one agent switching between skills sequentially, you have specialized agents working simultaneously. The agent-workflow-designer skill in the community library encodes the patterns for designing these multi-agent systems — sequential, parallel, router, orchestrator, and evaluator topologies.

Cross-platform portability — one skill, three platforms

Here's something genuinely cool: the SKILL.md format works identically across Claude Code, Claude.ai, and the Claude API. Your audio-workflow skill folder works in all three without modification. In Claude Code, it's a local folder. In Claude.ai, you upload it. In the API, you pass the content in your request. The community library even includes install scripts for OpenAI Codex and OpenClaw — the format is deliberately simple enough that any AI agent platform can parse it. It's essentially an emerging standard for packaging agent expertise.

Recap

We just went from "what even is an agent skill?" to building and installing a custom audio-workflow skill with chaining. Let's take stock of what we actually learned.

The skill system is deceptively simple in architecture — a folder with a markdown file — but surprisingly powerful in practice. The key insight is that skills encode domain expertise as structured instructions, not code. You're not writing a plugin or an API integration. You're writing a really good briefing document that turns a generalist AI into a specialist.

We installed skills from both Anthropic's official marketplace and the 86-skill community library. We designed and built a custom audio-workflow skill from scratch, focusing on the three layers that matter: when it should activate (the description), what it knows (workflow steps and tool knowledge), and how it handles edge cases (examples and guardrails).

Then we leveled up by adding skill chaining — making our audio skill smart enough to suggest the right follow-up step based on what it just processed. This router pattern is the key to building AI workflows that feel seamless rather than stitched together.

The meta-lesson here is about how to think about AI agent architecture. Skills are the building block. Chains are the wiring. And the marketplace is where expertise compounds across the community. Your audio skill could help the next person who needs it, just like the 86 community skills are already helping you.

  • Skills are just folders with SKILL.md files — YAML frontmatter (name + description) plus markdown instructions. No code required, though you can include CLI tools for complex skills.
  • The description field is the activation trigger — write it like you're telling a coworker when to reach for this particular playbook. Specific scenarios and capabilities beat vague labels.
  • Skill chaining turns individual skills into intelligent workflows — use the router pattern (suggest next steps based on content) rather than hardcoded pipelines.
  • Skills are portable across Claude Code, Claude.ai, and the Claude API — one SKILL.md works everywhere, making your expertise investment highly reusable.
  • The community library (86+ skills) proves this architecture scales — from security auditing to financial modeling to medical device compliance, domain expertise is becoming modular and shareable.
